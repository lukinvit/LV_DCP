# LV_DCP ŌĆö Developer Context Platform

**Local-first engineering memory.** Turns projects on macOS into a queryable context layer for Claude, IDE agents, and humans. Supports Python, TypeScript/JS, Go, Rust, Java, Kotlin, and Swift. Reduces token cost of repeated code reading, builds a relation graph, and makes agent edits safer.

## What it actually does

You ask a question in plain language ŌĆö _"where is refresh token logic?"_ or _"change login validation"_ ŌĆö and LV_DCP returns a **2ŌĆō20 KB markdown pack** with the 2ŌĆō5 most relevant files and symbols. Your LLM agent reads that pack instead of grep-walking the whole repository.

For edit tasks, the pack groups files by role ŌĆö **target files**, **impacted tests**, **impacted configs** ŌĆö and surfaces dependencies found through the relation graph, not just keyword matching. A test file gets included even if it doesn't contain the keywords, because the graph knows it imports the file you're changing.

```bash
$ ctx pack . "refresh token rotation" --mode edit

# Context pack ŌĆö edit
**Project:** my-api
**Intent:** refresh token rotation
**Coverage:** high

## Target files
- app/services/auth.py (score 12.34)
- app/handlers/auth.py (score 8.21)

## Impacted tests
- tests/test_auth.py

## Impacted configs
- config/settings.yaml

## Candidate symbols
- app.services.auth.refresh_access_token
- app.services.auth.ACCESS_TTL
- app.handlers.auth.refresh

## Reminder: edit discipline
1. Build minimal plan before patching multiple files
2. Never touch write_protected_paths
3. Run lint + typecheck + tests after every change
4. Summarize the diff when done
```

## Core value proposition

- **10├Ś fewer tokens** when Claude/Cursor/Cline works on a large Python codebase. The agent reads a 5 KB pack instead of 50ŌĆō80 KB of raw files.
- **Automatic invocation via MCP.** Once `ctx mcp install` is run, Claude Code (CLI and VS Code extension) calls LV_DCP automatically before answering architectural or edit questions ŌĆö zero ceremony for the user.
- **No context slips through.** Edit-mode packs include impacted tests and configs discovered via graph walk ŌĆö the kind of files `grep` misses.
- **Local-only.** No network calls, no telemetry, no SaaS. Indexes live in `.context/` next to each project. Secrets in source files are detected by regex and excluded from the search index.
- **Deterministic and measurable.** A retrieval eval harness with 32 golden queries is a first-class citizen; every retrieval change must not regress the metrics.

## Status

**v0.8.51 (2026-04-25)** ŌĆö `ctx pack --json` for scriptable structured pack payload (CLI/MCP parity for the primary retrieval surface). The `ctx pack` command is the **primary retrieval interface** of LV_DCP ŌĆö every script, dashboard, retrieval-quality benchmark, hash-comparison drift detector, and LLM-on-LLM evaluation harness ultimately calls `ctx pack` to get the ranked context. Until this release, the CLI emitted only the human-readable assembled markdown ŌĆö automation could see the rendered pack body, but had no access to `coverage` (the high/medium/ambiguous confidence signal that drives `lvdcp_explain` and edit-mode gating), `trace_id` (the retrieval trace ID for `ctx history` lookup and F1.B sparkline correlation), or the ranked `retrieved_files` / `retrieved_symbols` lists (the structured signals that downstream re-ranking heuristics, hash-comparison workflows, and evaluation judges actually consume). The MCP `lvdcp_pack` tool already returned the full structured payload via its `PackResult` schema since the MCP feature shipped ŌĆö we had two transports for the same retrieval pipeline emitting two contracts (markdown-only vs. structured-object), inviting drift every time the pipeline added a field. This release adds a `--json` flag on `ctx pack` that emits a single JSON object **mirroring the MCP `PackResult` schema 1:1**: `{markdown (the assembled pack body ŌĆö the same string the text view prints; included so a consumer that wants the rendered pack still gets it via `jq -r .markdown` and a consumer that wants only the structured signals can ignore the field; stripping it would create a third contract divergent from MCP ŌĆö wrong tradeoff), trace_id (the retrieval trace ID ŌĆö round-trips into `ctx history` / `lvdcp_explain` lookup and F1.B sparkline correlation; persisted via `idx.save_trace(trace_with_project)` so CLI-originated queries land in the same trace store as MCP-originated ones), coverage (one of `"high"` / `"medium"` / `"ambiguous"` ŌĆö the confidence signal that gates edit-mode workflows; `jq -e '.coverage == "high"'` is the natural CI gate for "the pack is confident enough to act on"), retrieved_files (ranked list of file paths ŌĆö converted to a list from the `ContextPack` tuple because JSON has no tuple type; `jq -r '.retrieved_files[]'` feeds directly into hash-comparison drift detection and per-query coverage analysis), retrieved_symbols (ranked list of fully-qualified symbol names ŌĆö same tupleŌåÆlist conversion; `jq '.retrieved_symbols | length'` surfaces symbol-density signal for the eval harness)}`. **Single object, not a bare array** ŌĆö `ctx pack` is a single-result retrieval call with a body (`markdown`) plus structured metadata (`trace_id`, `coverage`) plus two ranked sub-arrays. Same single-object discipline as v0.8.48 `wiki cross-project` (single-result generate command with run-level summary fields) and v0.8.50 `obsidian status` (tri-state vault contract). Compare with the bare-array siblings (`wiki status` v0.8.46 per-module, `wiki lint` v0.8.47 per-finding, `watch list` v0.8.49 per-project): those are pure per-row reads with no top-level metadata to encode. Same scriptability discipline as v0.8.38 prune / v0.8.40 restore / v0.8.42 scan / v0.8.43 inspect / v0.8.44 memory list / v0.8.45 reconcile / v0.8.46 wiki status / v0.8.47 wiki lint / v0.8.48 wiki cross-project / v0.8.49 watch list / v0.8.50 obsidian status: pure data on stdout (no human chrome; structlog routed to `sys.stderr` since v0.8.42 carries forward unchanged), error path identical in both modes (un-indexed project ŌåÆ stderr message + exit 1, **never** `{"error": "..."}` swallowed into stdout JSON ŌĆö same v0.8.42-v0.8.50 discipline of "exit code is the gate, structured payload is for the success path"), trace persistence preserved across both modes so F1.B sparklines and `lvdcp_explain` see CLI-originated queries identically to MCP-originated ones. **CLI/MCP parity is the design goal** ŌĆö two contracts for the same retrieval pipeline would invite subtle drift; one shared schema (`apps.mcp.tools.PackResult` Ōåö `apps.cli.commands.pack._pack_to_json`) forces every shape change to be reviewed against both consumers. CLI-only change ŌĆö no library code touched (no mutation to `libs/context_pack/`, `libs/project_index/`, or any retrieval pipeline code; `_pack_to_json` lives next to `pack` in `apps/cli/commands/pack.py`, sister-helper to v0.8.46-v0.8.50 CLI-shell payload shapers). Four new test invariants lock the contract ŌĆö text path emits the legacy markdown body byte-identically with **no JSON leak** asserted via `pytest.raises(json.JSONDecodeError)` (locks the v0.8.42-v0.8.50 "text path untouched" promise; a future refactor that promotes JSON to the default render breaks this test instead of silently degrading every shell-pipeline consumer that pipes `ctx pack` to `pandoc` or another markdown processor), navigate-mode JSON emits a single object with the schema-locked key set (`_PACK_JSON_KEYS = {markdown, trace_id, coverage, retrieved_files, retrieved_symbols}` frozenset; `coverage` asserted in `_VALID_COVERAGE_VALUES = {high, medium, ambiguous}`; markdown body still carries the navigate-template signature `Context pack`; expected file `auth.py` surfaces from the seeded `login endpoint` query in the fixture), edit-mode JSON emits the **same** schema as navigate (locks render-mode-invariance of the JSON contract ŌĆö only the markdown body shape differs, schema is invariant; markdown body asserted to carry the `Target files` edit-template signature so a refactor that accidentally routes edit mode through the navigate template breaks this test), error path under `--json` mode does **not** swallow into a `{"error": "..."}` stdout payload (un-indexed project exits 1 in both text and JSON modes; if any output parses as JSON, asserted **not** to be the success-shape dict ŌĆö same discipline shared with v0.8.42-v0.8.50). All 7 tests in the file pass (3 pre-existing + 4 new). This unblocks workflows that the rolling `--json` rollout's earlier ticks couldn't reach: retrieval-quality dashboards (`for q in queries.txt; do ctx pack . "$q" --json | jq -c '{q: $q, cov: .coverage, files: (.retrieved_files | length)}'; done > eval.ndjson`), hash-comparison drift detection (store `retrieved_files` per query, alert when set diff exceeds threshold across reindexes), cross-project benchmark sweeps (same pack call across N projects, fold `coverage` distribution into hygiene reports), LLM-on-LLM evaluation (feed the structured pack to a judge model without re-parsing markdown), agentic re-ranking heuristics (downstream callers can re-rank `retrieved_files` against task-specific signals without a markdown-parsing step). Scope locks: only one new CLI flag (`--json` on `pack`), no library code touched (`libs/context_pack/builder.py::build_navigate_pack` / `build_edit_pack` and `ContextPack` dataclass signatures unchanged; `libs/project_index/index.py::ProjectIndex.retrieve` / `save_trace` signatures unchanged), no new on-disk artifacts (no extra trace records, no extra cache writes ŌĆö `--json` is a pure render switch), MCP `lvdcp_pack` tool **untouched** (`apps.mcp.tools.PackResult` is the canonical schema this release mirrors ŌĆö schema flows MCPŌåÆCLI, never CLIŌåÆMCP, to preserve the LLM-facing contract as the source of truth), text output **byte-identical** byte-by-byte (every existing pack test passes; `make typecheck` strict-mode 393 source files clean), trace persistence semantic preserved across both modes so F1.B sparklines and `lvdcp_explain` see CLI-originated queries identically. **v0.8.25 (2026-04-24)** ŌĆö Swift `INHERITS` edges (single-colon heritage). Third and final deepening-increment on the inheritance-edge axis after v0.8.23 (Java) and v0.8.24 (Kotlin). With this ship, every tree-sitter parser on the platform that targets a language with a nominal class hierarchy (Java, Kotlin, Swift) emits `INHERITS` edges; Go and Rust have structural conformance rather than nominal inheritance, and the Python parser treats call-graph as its primary edge type with inheritance as a separate future pass. So this release closes the **inheritance-edge axis across language coverage**, not a single-parser gap. `SwiftParser.parse()` now overrides the base, calls it, then walks the CST a second time with a scope-aware stack to emit `Relation(relation_type=INHERITS)` edges for every heritage clause ŌĆö same structural shape as the Java and Kotlin ports (`_TYPE_DECLARATION_NODES` frozenset, `parse()` override, `_extract_inherits()`, `_collect_heritage()`, `_user_type_name()`). The one structural difference from the JVM pair is that Swift has **no heritage wrapper node**: unlike Java's `super_interfaces` and Kotlin's `delegation_specifiers`, Swift places each `inheritance_specifier` as a direct sibling of the type name inside `class_declaration` / `protocol_declaration`, separated by `,` tokens. `_collect_heritage` therefore iterates `type_node.children` directly and picks out `inheritance_specifier` kinds, naturally filtering out non-heritage siblings (`type_parameters` on the source type, `type_constraints` for `where` clauses, `class_body` / `enum_class_body` / `protocol_body`, `:` / `,` tokens). Swift overloads the `:` heritage syntax across an unusually broad set of declarations ŌĆö `class` / `struct` / `enum` / `actor` / `extension` (all under one unified `class_declaration` grammar node) plus `protocol_declaration` ŌĆö and all six forms can carry heritage; every case lands on the same `RelationType.INHERITS` edge (class inheritance, struct/enum/actor protocol conformance, protocol composition, extension conformance). **Extensions are the interesting case**: `class_declaration` for an `extension User: P` has its `name` field set to the *extended* type (so `child_by_field_name("name").text == b"User"`), and the base walker from v0.8.22 already emits a duplicate CLASS symbol named `User` for every extension. This release's heritage walker therefore produces `<module>.User INHERITS P` for the conformance, which correctly matches the duplicate symbol's `fq_name` ŌĆö a graph query for "does this codebase conform `User` to `Equatable`?" now answers yes whether the conformance is declared in `User`'s primary definition or in a retroactive `extension User: Equatable {}`, regardless of which file the extension lives in. That is the shape Swift users actually reason about. Type-name extraction: Swift's `user_type` grammar uses `type_identifier` (not Kotlin's `identifier`) for each dotted-path segment, with an optional trailing `type_arguments` child for generics. `_user_type_name` walks only direct `type_identifier` children and joins them with `.`, so `Container<String>` emits `"Container"`, `Comparable<Box<T>>` emits `"Comparable"`, and scoped paths like `pkg.lib.Deep<T>` emit `"pkg.lib.Deep"`. The node-type filter in `_collect_heritage` excludes `type_constraints` naturally, so no `Constrained INHERITS Hashable` edge leaks ŌĆö `test_where_clause_does_not_leak` locks this. Design lock: `INHERITS` still collapses class inheritance + protocol conformance + extension conformance + protocol composition + enum raw-value bases ŌĆö same "one edge type" decision as v0.8.23 Java and v0.8.24 Kotlin, same downstream-consumer audit-cost rationale (a future `CONFORMS_TO` sub-type would force Obsidian sync, pack enrichment, dashboard graph rendering to all decide the split with no driving use case). 17 new tests in `TestInheritsEdges` (single class inheritance, multi-protocol class conformance, struct conformance, protocol inherits-many, enum raw-value+protocol, actor conformance, extension conformance uses extended type, generic stripping with `src != dst` guard, scoped path preservation, nested class uses outer fq, where-clause does-not-leak, plain class has none, `src_ref` resolves-to-symbol invariant, relation-shape invariant, SPM module-fq respect, empty file, pre-v0.8.25 fixture regression guard on the v0.8.22 `SWIFT_CODE` showing `ViewController INHERITS UIViewController` + `User INHERITS CustomStringConvertible`). Full suite: **1387 passing** (+17 vs v0.8.24 baseline 1370). Ruff clean on 380 files, ruff format clean, mypy strict clean on 380 files. ## Phase 3c.1 ŌĆö LLM Summaries (new in 0.3.2)

Pluggable LLM provider with file-level summaries:

```bash
# Configure provider (default: OpenAI)
export OPENAI_API_KEY=sk-...                  # or ANTHROPIC_API_KEY, etc.
uv run ctx ui                                  # open http://127.0.0.1:8787/settings
# toggle "Enable LLM summaries", save

# Generate summaries for a project
uv run ctx summarize /path/to/project
```

Summaries are cached in `~/.lvdcp/summaries.db` keyed on file content hash + prompt version + model name, so re-running on unchanged files has zero cost.

Supported providers out of the box:
- **OpenAI** (default): `gpt-4o-mini`, `gpt-5-mini`, `gpt-5`
- **Anthropic**: `claude-haiku-4-5`, `claude-sonnet-4-6`
- **Ollama** (local, zero-cost): `qwen2.5-coder:7b`, `qwen2.5-coder:32b`, `llama3.3:70b`

### Rate limits on OpenAI tier-1

OpenAI tier-1 accounts have a 200K TPM (tokens per minute) limit. Cold-scanning a
large project with the default concurrency (4) stays under this limit for most
projects. If you hit HTTP 429 errors, re-run `ctx summarize` ŌĆö already-completed
files will be cached and only the failed ones will retry. Tier-2+ users can use
`--concurrency 10` for faster cold scans.

The `openai_client` adapter honors `Retry-After` headers automatically and retries
up to 3 times per file before giving up.

### Cost tracking

Dashboard topbar shows `$X.XX / $25 monthly` budget usage. `ctx mcp doctor` now runs 9 checks (was 7) ŌĆö the new check 8 verifies provider connectivity, check 9 warns at 80% and fails at 100% of monthly budget.

### Settings UI

`/settings` page lets you switch provider, model, and budget without editing config files. API keys are stored as environment variables only ŌĆö the UI displays the env var *name* and its set/unset status, never the key value itself.

### What gets indexed

Supported languages:
- **Python** ŌĆö via stdlib `ast`, extracts classes, functions, methods, constants, imports, same-file calls
- **TypeScript / JavaScript** ŌĆö via tree-sitter, extracts classes, functions, interfaces, enums, imports, constants
- **Go** ŌĆö via tree-sitter, extracts functions, methods, types, imports
- **Rust** ŌĆö via tree-sitter, extracts functions, structs, traits, enums, use declarations
- **Markdown** ŌĆö heading extraction as navigation anchors
- **YAML / JSON / TOML** ŌĆö syntax validation, config role detection

Automatic ignore list: `.git/`, `.venv/`, `node_modules/`, `__pycache__/`, `.mypy_cache/`, `.ruff_cache/`, `.pytest_cache/`, `dist/`, `build/`, `.context/`, `secrets/`, `credentials/`, plus `.env`, `.env.local/production/staging/development`, `credentials.json`, `secrets.json`.

### Privacy model

- **All processing is local.** LV_DCP never makes network calls on its own. The only outbound data is what your LLM client (e.g. Claude Code) sends to its own API ŌĆö and LV_DCP reduces that payload rather than expanding it.
- **Secret pattern detection** ŌĆö 12 regex patterns for AWS keys, OpenAI/Stripe/GitHub/Slack tokens, JWTs, PEM private key headers. Files matching these are indexed by path only; their content is excluded from the full-text index and context packs.
- **Deny list** ŌĆö env files and credentials files are ignored at the path level before any content is read.
- **Nothing leaves the machine.** Indexes live in `.context/` directories inside each project root. Delete the directory to remove the index.
