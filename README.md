# LV_DCP — Developer Context Platform

**Local-first engineering memory.** Turns projects on macOS into a queryable context layer for Claude, IDE agents, and humans. Supports Python, TypeScript/JS, Go, Rust, Java, Kotlin, and Swift. Reduces token cost of repeated code reading, builds a relation graph, and makes agent edits safer.

## What it actually does

You ask a question in plain language — _"where is refresh token logic?"_ or _"change login validation"_ — and LV_DCP returns a **2–20 KB markdown pack** with the 2–5 most relevant files and symbols. Your LLM agent reads that pack instead of grep-walking the whole repository.

For edit tasks, the pack groups files by role — **target files**, **impacted tests**, **impacted configs** — and surfaces dependencies found through the relation graph, not just keyword matching. A test file gets included even if it doesn't contain the keywords, because the graph knows it imports the file you're changing.

```bash
$ ctx pack . "refresh token rotation" --mode edit

# Context pack — edit
**Project:** my-api
**Intent:** refresh token rotation
**Coverage:** high

## Target files
- app/services/auth.py (score 12.34)
- app/handlers/auth.py (score 8.21)

## Impacted tests
- tests/test_auth.py

## Impacted configs
- config/settings.yaml

## Candidate symbols
- app.services.auth.refresh_access_token
- app.services.auth.ACCESS_TTL
- app.handlers.auth.refresh

## Reminder: edit discipline
1. Build minimal plan before patching multiple files
2. Never touch write_protected_paths
3. Run lint + typecheck + tests after every change
4. Summarize the diff when done
```

## Core value proposition

- **10× fewer tokens** when Claude/Cursor/Cline works on a large Python codebase. The agent reads a 5 KB pack instead of 50–80 KB of raw files.
- **Automatic invocation via MCP.** Once `ctx mcp install` is run, Claude Code (CLI and VS Code extension) calls LV_DCP automatically before answering architectural or edit questions — zero ceremony for the user.
- **No context slips through.** Edit-mode packs include impacted tests and configs discovered via graph walk — the kind of files `grep` misses.
- **Local-only.** No network calls, no telemetry, no SaaS. Indexes live in `.context/` next to each project. Secrets in source files are detected by regex and excluded from the search index.
- **Deterministic and measurable.** A retrieval eval harness with 32 golden queries is a first-class citizen; every retrieval change must not regress the metrics.

## Status

**v0.8.36 (2026-04-25)** — `ctx registry prune --missing` for absent-root cleanup. v0.8.33 gated pruning on staleness (`packs_total=0 AND last_scan>30d`) and kind (real/transient/all) — which works for entries whose *scanning* is dormant but whose *directory* still exists on disk. It misses a common failure mode: a worktree deleted the same day (`git worktree remove`, `rm -rf /tmp/scratch`, a project folder moved across a drive) turns into a **tombstone** — path gone, staleness gate doesn't fire for 30 days, `ctx status` keeps pinging a dead filesystem root. v0.8.35 prevents *new* transient accrual, but existing tombstones still need a catch-up command. This release adds a `--missing` flag on `ctx registry prune` that treats root-directory absence (`Path(entry.root).exists() is False`) as an **independent** eligibility predicate — ignores `--older-than` (a gone root is gone regardless of age), ignores `--kind` (a gone transient and a gone real project are equally dead), preserves the v0.8.33 safety contract (dry-run default, `--yes` gate, pre-mutation `*.bak` sidecar, atomic `save_config`). Layering kept: library-layer logic in `libs/status/registry_prune.py` (`plan_prune` and `prune_stale` gain a `missing_only: bool = False` kwarg — when `True`, the candidate filter checks `Path(row.root).exists()` and short-circuits past the `older_than_days` / `kind` branches), thin Typer shell in `apps/cli/commands/registry_cmd.py::prune_cmd` adds `--missing` alongside `--older-than` / `--kind` / `--yes`, output copy reports `missing root` vs `>Nd` on the nothing-to-do / candidates path. Eight new test invariants lock the contract (5 library-layer, 3 CLI-layer): `test_plan_prune_missing_lists_absent_roots` (the predicate catches deleted paths while sparing live directories), `test_prune_missing_ignores_older_than_gate` (a worktree registered this morning, scanned at 9 a.m., deleted at noon: staleness-mode returns `[]`, missing-mode returns it), `test_prune_missing_ignores_kind_filter` (both a gone-real and a gone-transient entry are caught even with `kind="transient"`), `test_prune_missing_applies_with_backup` (`apply=True` mutates exactly the gone entries and writes `*.bak` with the pre-mutation bytes), `test_prune_missing_no_candidates_is_graceful` (when every root exists, missing-mode is a pure read), `test_registry_prune_missing_dry_run_lists_deleted_root` (CLI dry-run prints `missing root` + `dry-run` and leaves config untouched), `test_registry_prune_missing_yes_removes_and_backs_up` (CLI `--yes` applies + reports `REMOVED` + `backup saved`), `test_registry_prune_missing_nothing_to_do` (CLI no-op path prints the `missing root` variant of the nothing-to-do message). Rejected alternatives: auto-prune on every `ctx status` / daemon wake (hidden destructive behavior — the v0.8.33 dry-run-by-default contract is the right UX, the user chooses when to apply), `.context/cache.db` existence instead of root directory (a user who runs `rm -rf /path/to/proj/.context` but keeps the project would lose their entry — root-directory absence is the correct signal), combined `--stale-or-missing` mode (overspecified — two independent flags, compose-mode added only if driven by a real use case), `Path(root).is_dir()` instead of `.exists()` (a regular-file tombstone from a stray `touch` is still a dead entry — `.exists()` catches both), gating `--missing` behind `--kind all` (friction for the common case with no safety win — `--yes` is already the mutation gate). Scope locks: no registry schema change, no public signature change on read-side helpers (`audit_registry`, `is_stale`, `list_projects`, `save_config`), default behavior of `ctx registry prune` without `--missing` is unchanged, no new on-disk artifacts (only the v0.8.33 `*.bak` sidecar), no `ctx registry ls --missing` yet (symmetric read-side surface is a plausible follow-up, but `ls --json | jq '.[] | select(...)'` already handles the use case). Full docs in [release note](docs/release/2026-04-25-v0.8.36-prune-missing-roots.md).

No registry schema change, no public signature change on `list_projects` / `save_config` / `remove_project` / `update_last_scan`, no migration (existing transient entries can be cleaned with `ctx registry prune --yes` which already shipped in v0.8.33). **v0.8.34 (2026-04-24)** — single atomic write path for `~/.lvdcp/config.yaml`. While shipping v0.8.33's `ctx registry prune` I noticed two `save_config()` implementations — the v0.8.33 one in `libs/core/projects_config.py` (atomic: tmp + `fsync` + `rename`) and an older one in `apps/agent/config.py` (non-atomic `path.write_text`). The older one was the actual writer for every code path except prune: daemon scan completion via `update_last_scan`, `add_project`/`remove_project`, CLI setup wizard, UI settings form. So the registry was being written non-atomically dozens of times per day while the atomic path was used once a month during prune. A crash or OOM kill mid-write could leave truncated YAML and wipe out the user's entire project registry on next boot. This release unifies to a single atomic writer: `apps/agent/config.py` now re-exports `save_config` from `libs/core`, the chmod 0o600 moves into the shared impl wrapped in `contextlib.suppress(OSError)` (skips on filesystems without POSIX perms instead of crashing), and `yaml.safe_dump(..., allow_unicode=True)` from the v0.8.33 path propagates — so Cyrillic / emoji project names like `/Users/me/Проект_X5` now render as readable YAML instead of `\uXXXX` escapes. Five new test invariants lock the contract: chmod 0o600 on POSIX, original file preserved on simulated `Path.replace` failure, unicode round-trip, save survives chmod failure, and `apps.agent.config.save_config is libs.core.projects_config.save_config` — a one-liner that catches any future regression reintroducing a second writer. Full unit suite: **1289 passing** (+5 vs v0.8.33 baseline 1284). Rejected: keeping both implementations (divergence invites drift), promoting the non-atomic one (loses the property v0.8.33 needed), `fcntl.flock()` (overkill — registry is single-writer, atomic-rename is enough), moving the domain helpers down to `libs/core` (would pull `datetime.now()` and "register a project" semantics into the core layer, violating CLAUDE.md "libs/core зависит только от stdlib + pydantic"). No schema change, no public signature change, no migration needed — first write after upgrade is atomic. Full docs in [release note](docs/release/2026-04-24-v0.8.34-atomic-config-write.md).

**v0.8.33 (2026-04-24)** — `ctx registry prune` with dry-run default. v0.8.32 surfaced the pruning candidates via `ctx registry ls --stale`, but the user still had no safe remediation command — abandoned worktree entries had to be cleaned up via ad-hoc `yq`/`sed` scripts and the daemon kept waking up for dead projects. This release closes the loop: new `ctx registry prune` subcommand, gated by three independent safety layers — dry-run is the default (must pass `--yes` to mutate), the default `--kind` is `transient` (worktree artifacts and test fixtures — near-zero false-positive cost — real user projects require explicit `--kind real`), and a pre-mutation `*.bak` sidecar is written before the atomic rewrite starts so recovery is always a one-liner (`mv ~/.lvdcp/config.yaml{.bak,}`). Layered per CLAUDE.md: mutation logic lives in a new `libs/status/registry_prune.py` (`plan_prune`, `prune_stale`, `PruneResult`) so `registry_audit.py` stays read-only per its docstring invariant; `libs/core/projects_config.py` gains a new `save_config()` helper with atomic tmp + `fsync` + `rename` + temp-file cleanup on error; `apps/cli/commands/registry_cmd.py` appends the thin Typer shell. 16 new tests across three test files — 8 library-layer cases (dry-run is a byte-level no-op, `apply=True` removes matching entries and writes `*.bak` with the pre-mutation bytes verbatim, `--kind real` spares transient, `--kind all` prunes both buckets, `--older-than` respects recent scans, invalid kind raises `ValueError`, empty registry is graceful), 5 CLI-layer cases (default is dry-run, `--yes` applies, invalid `--kind` exits 2, `--older-than 0` exits 2, `--kind real` against transient-only stale set reports "nothing to do"), 3 core-layer cases for `save_config` (round-trips, no `*.tmp` leak, overwrites existing atomically). Full unit suite: **1284 passing**. Scope locks: only registry entries are removed (no filesystem mutation on the project roots themselves), no `~/.lvdcp/config.yaml` schema change, no MCP surface (destructive operations should never be reachable via a Claude-invokable tool), survivor entries keep their original ordering. Rejected alternatives: auto-prune on scan (too invasive — the user should decide what leaves the registry), `--force` instead of `--yes` (`--yes` matches `apt` / `brew` muscle memory), in-place rewrite without backup (one 2 KB sidecar is free insurance), `typer.confirm` prompt (breaks non-interactive CI / hook usage), cascade-delete of `.context/` dirs of pruned projects (separate concern — future `ctx registry clean`). Full docs in [release note](docs/release/2026-04-24-v0.8.33-registry-prune.md).

**v0.8.30 (2026-04-24)** — VS Code extension activation-event narrowing. Phase 10 polish item from the v0.8.29 roadmap. The extension always declared `"activationEvents": ["onStartupFinished"]`, which fires on every VS Code cold start — so the "LV_DCP" status bar item appeared in every window, even for projects that had nothing to do with LV_DCP (a web frontend scratch repo, a friend's Elm project, a dotfiles folder). The noisier status-bar item is the surface we actually care about narrowing. **v0.8.28 (2026-04-24)** — Obsidian nightly sync (`ctx obsidian sync-all`). Closes the last Phase 10 non-parser item on the v0.8.27 roadmap. The Obsidian publisher has been live since Phase 2 wired two ways: manual (`ctx obsidian sync <project> --vault <vault>`) and after-scan (daemon's `apps.agent.obsidian_worker.run_obsidian_sync()`, debounced by `~/.lvdcp/config.yaml:obsidian.debounce_seconds`, default 1h, via `.context/obsidian_last_sync`). Missing was **nightly** — the case where a project was scanned yesterday, the debounce elapsed, but no scan fired today, so the vault drifts. This release ships `ctx obsidian sync-all` which walks every project in `~/.lvdcp/config.yaml:projects`, decides which ones need a sync via a single `--stale-hours N` gate (default 24h; `0` force-syncs everything; missing/corrupt marker counts as stale because "registered but never synced" is the common first-run case), and shells out to `ctx obsidian sync` per project via `subprocess.run` with a `--timeout SECS` isolation (default 300s). `--dry-run` prints the plan (`sync` / `skip: no cache.db` / `skip: fresh (Xh old)` per project) without invoking the inner CLI — essential for arming launchd/cron without silent misconfigurations. One flaky project doesn't block the others: the loop catches `CalledProcessError` / `TimeoutExpired` / `OSError` per project and continues; exit code is `1` iff any project failed, with "`Synced: N, skipped: N, failed: N`" on stdout. On success the same `.context/obsidian_last_sync` marker the after-scan worker uses gets advanced, so the daemon won't re-fire inside its own debounce window after the nightly run — one marker, two writers, same gate. Design choice lock: the repo ships the command, **not** the schedule. LV_DCP's constitution (§I.3 and CLAUDE.md Принцип 5) is "local-first, minimal moving parts, modular monolith not microservices, простота эксплуатации превыше модной архитектуры" — an in-process scheduler inside the daemon is an extra moving part that fires only when the daemon runs (laptop closed = no sync), while launchd/cron is the macOS-native way and already what every power user trusts for scheduled jobs. The release note documents the two canonical setups: a launchd `StartCalendarInterval` at 03:00 and a `0 3 * * *` crontab, both pointing at `/path/to/LV_DCP/.venv/bin/ctx obsidian sync-all`. Neither is installed by the LV_DCP installer — `ctx mcp install` touches `~/.claude/` because that's Claude's territory, a scheduled daemon touching `~/Library/LaunchAgents/` is a different trust level that the user should own explicitly. Helper shape: `_read_last_sync(project_root)` reads the marker as `float | None` (corrupt file → None, silently treated as stale), `_is_stale(project_root, stale_seconds, now)` combines the missing-marker-is-stale and `stale_seconds <= 0 = force` branches into one gate, `@app.command("sync-all")` validates config (missing file / disabled / empty `vault_path` / no projects all exit cleanly or with a typed error), builds a plan of `(root, reason)` tuples, branches on `--dry-run`, and for the live path writes the marker only after a zero-return-code `subprocess.run`. 18 new tests in `tests/unit/cli/test_obsidian_sync_all.py`: config validation (missing config file exits 1, disabled exits 0, empty vault_path exits 1, no projects exits 0), dry-run plan (fresh project shows `sync`, cacheless shows `skip: no cache.db`, recently-synced shows `skip: fresh (Xh old)`, `--stale-hours 0` forces sync even on fresh projects), live sync (subprocess invoked with correct args + marker advanced, fresh project skipped with `mock_run.assert_not_called()`, mixed stale/fresh respects the gate and calls subprocess exactly once with the stale project's path, subprocess failure on one of two projects exits 1 but both are attempted and the succeeded one gets its marker while the failed one doesn't, `TimeoutExpired` treated as failure, nonexistent-project-root skipped not crashed), plus a parametrized `_is_stale` invariant locking never-synced/fresh/stale/force-hours-zero semantics. Zero changes to the daemon, zero changes to the publisher, zero changes to the after-scan worker — all new code lives in `apps/cli/commands/obsidian_cmd.py`. Full Python suite: **1405 passing** (+18 vs v0.8.27 baseline 1388). Ruff clean, ruff format clean, mypy strict clean. Known limits: sequential not parallel (one subprocess at a time — 20-project sweeps run in well under a minute on warm SQLite, parallelization would need a shared vault-lock strategy, deferred until >60 registered projects); no per-project interval override (the `--stale-hours` gate is global — "small projects nightly, big project weekly" would need two launchd entries with two `--config` files, deemed low-priority); no built-in scheduler (deliberate — the day a Windows/Linux user hits this and the `launchd` path doesn't apply, a `ctx watch install-service`-style subcommand becomes the natural follow-up); exit-code granularity is `0` or `1`, no "`2` means partial" — scripts that care about failure counts parse the `"failed: N"` tail line; ~20 ms of Python startup per project via `subprocess.run` (irrelevant for nightly, a future in-process path would lose the timeout-isolation + error-isolation properties that are the whole point of the shelling-out design).

**v0.8.25 (2026-04-24)** — Swift `INHERITS` edges (single-colon heritage). Third and final deepening-increment on the inheritance-edge axis after v0.8.23 (Java) and v0.8.24 (Kotlin). With this ship, every tree-sitter parser on the platform that targets a language with a nominal class hierarchy (Java, Kotlin, Swift) emits `INHERITS` edges; Go and Rust have structural conformance rather than nominal inheritance, and the Python parser treats call-graph as its primary edge type with inheritance as a separate future pass. So this release closes the **inheritance-edge axis across language coverage**, not a single-parser gap. `SwiftParser.parse()` now overrides the base, calls it, then walks the CST a second time with a scope-aware stack to emit `Relation(relation_type=INHERITS)` edges for every heritage clause — same structural shape as the Java and Kotlin ports (`_TYPE_DECLARATION_NODES` frozenset, `parse()` override, `_extract_inherits()`, `_collect_heritage()`, `_user_type_name()`). The one structural difference from the JVM pair is that Swift has **no heritage wrapper node**: unlike Java's `super_interfaces` and Kotlin's `delegation_specifiers`, Swift places each `inheritance_specifier` as a direct sibling of the type name inside `class_declaration` / `protocol_declaration`, separated by `,` tokens. `_collect_heritage` therefore iterates `type_node.children` directly and picks out `inheritance_specifier` kinds, naturally filtering out non-heritage siblings (`type_parameters` on the source type, `type_constraints` for `where` clauses, `class_body` / `enum_class_body` / `protocol_body`, `:` / `,` tokens). Swift overloads the `:` heritage syntax across an unusually broad set of declarations — `class` / `struct` / `enum` / `actor` / `extension` (all under one unified `class_declaration` grammar node) plus `protocol_declaration` — and all six forms can carry heritage; every case lands on the same `RelationType.INHERITS` edge (class inheritance, struct/enum/actor protocol conformance, protocol composition, extension conformance). **Extensions are the interesting case**: `class_declaration` for an `extension User: P` has its `name` field set to the *extended* type (so `child_by_field_name("name").text == b"User"`), and the base walker from v0.8.22 already emits a duplicate CLASS symbol named `User` for every extension. This release's heritage walker therefore produces `<module>.User INHERITS P` for the conformance, which correctly matches the duplicate symbol's `fq_name` — a graph query for "does this codebase conform `User` to `Equatable`?" now answers yes whether the conformance is declared in `User`'s primary definition or in a retroactive `extension User: Equatable {}`, regardless of which file the extension lives in. That is the shape Swift users actually reason about. Type-name extraction: Swift's `user_type` grammar uses `type_identifier` (not Kotlin's `identifier`) for each dotted-path segment, with an optional trailing `type_arguments` child for generics. `_user_type_name` walks only direct `type_identifier` children and joins them with `.`, so `Container<String>` emits `"Container"`, `Comparable<Box<T>>` emits `"Comparable"`, and scoped paths like `pkg.lib.Deep<T>` emit `"pkg.lib.Deep"`. The `test_generic_type_arguments_stripped` invariant asserts `String` and type-parameter substitutions never surface as bases, and adds a `src != dst` guard that catches the `Box<T>: Comparable<Box<T>>` failure mode where a type parameter could leak its own name back through generics recursion. `where` clauses must not leak: `class Constrained<T>: Base where T: Hashable {}` places `type_constraints` as a sibling after the last `inheritance_specifier`, with an inner `inheritance_constraint → user_type` (`Hashable`) that's *syntactically* identical to a heritage entry but *semantically* a generic-parameter constraint, not an inheritance edge. The node-type filter in `_collect_heritage` excludes `type_constraints` naturally, so no `Constrained INHERITS Hashable` edge leaks — `test_where_clause_does_not_leak` locks this. Scope-aware walker is a structural clone of v0.8.23 Java / v0.8.24 Kotlin: `stack: list[tuple[Node, str]]` tracks `(node, enclosing_fq)`, type-declaration nodes compute `type_fq = f"{scope}.{name}"`, emit edges with `src_ref=type_fq`, and propagate `type_fq` as child scope — so `class Outer { class Inner: pkg.lib.Deep {} }` emits `<module>.Outer.Inner INHERITS pkg.lib.Deep` matching the symbol walker's own fq_name output, locked by `test_inherits_src_ref_matches_symbol_fq`. Design lock: `INHERITS` still collapses class inheritance + protocol conformance + extension conformance + protocol composition + enum raw-value bases — same "one edge type" decision as v0.8.23 Java and v0.8.24 Kotlin, same downstream-consumer audit-cost rationale (a future `CONFORMS_TO` sub-type would force Obsidian sync, pack enrichment, dashboard graph rendering to all decide the split with no driving use case). 17 new tests in `TestInheritsEdges` (single class inheritance, multi-protocol class conformance, struct conformance, protocol inherits-many, enum raw-value+protocol, actor conformance, extension conformance uses extended type, generic stripping with `src != dst` guard, scoped path preservation, nested class uses outer fq, where-clause does-not-leak, plain class has none, `src_ref` resolves-to-symbol invariant, relation-shape invariant, SPM module-fq respect, empty file, pre-v0.8.25 fixture regression guard on the v0.8.22 `SWIFT_CODE` showing `ViewController INHERITS UIViewController` + `User INHERITS CustomStringConvertible`). Full suite: **1387 passing** (+17 vs v0.8.24 baseline 1370). Ruff clean on 380 files, ruff format clean, mypy strict clean on 380 files. Known limits: `dst_ref` is a bare type name, not a fully-qualified symbol, so `: Data` after `import Foundation` emits `dst_ref="Data"` not `"Foundation.Data"` (import-based resolution is a separate graph-enrichment pass — same gap as Java / Kotlin); `typealias` and `associatedtype` declarations are not in `_symbol_type_map` and therefore cannot be INHERITS sources (future sub-increment); `deinit_declaration` has no name and is excluded from symbol emission; no `SAME_FILE_CALLS` edges (same gap as Java / Kotlin / Go / Rust); `where`-clause conformances deliberately do not emit edges (constraint on `T`, not inheritance of `C`).

**v0.8.8 (2026-04-24)** — HTMX live-polling on the `wiki_refresh` panel. New public helper `libs.status.aggregator.build_wiki_refresh` lets the polling endpoint skip the full `build_project_status` pipeline, keeping each tick cheap. A new fragment route `GET /api/project/<slug>/wiki-refresh` re-renders just the partial; the outer wrapper's presence-or-absence of `hx-get` is the cancellation signal, so polling self-cancels the moment the refresh transitions to idle.

## Phase 3c.1 — LLM Summaries (new in 0.3.2)

Pluggable LLM provider with file-level summaries:

```bash
# Configure provider (default: OpenAI)
export OPENAI_API_KEY=sk-...                  # or ANTHROPIC_API_KEY, etc.
uv run ctx ui                                  # open http://127.0.0.1:8787/settings
# toggle "Enable LLM summaries", save

# Generate summaries for a project
uv run ctx summarize /path/to/project
```

Summaries are cached in `~/.lvdcp/summaries.db` keyed on file content hash + prompt version + model name, so re-running on unchanged files has zero cost.

Supported providers out of the box:
- **OpenAI** (default): `gpt-4o-mini`, `gpt-5-mini`, `gpt-5`
- **Anthropic**: `claude-haiku-4-5`, `claude-sonnet-4-6`
- **Ollama** (local, zero-cost): `qwen2.5-coder:7b`, `qwen2.5-coder:32b`, `llama3.3:70b`

### Rate limits on OpenAI tier-1

OpenAI tier-1 accounts have a 200K TPM (tokens per minute) limit. Cold-scanning a
large project with the default concurrency (4) stays under this limit for most
projects. If you hit HTTP 429 errors, re-run `ctx summarize` — already-completed
files will be cached and only the failed ones will retry. Tier-2+ users can use
`--concurrency 10` for faster cold scans.

The `openai_client` adapter honors `Retry-After` headers automatically and retries
up to 3 times per file before giving up.

### Cost tracking

Dashboard topbar shows `$X.XX / $25 monthly` budget usage. `ctx mcp doctor` now runs 9 checks (was 7) — the new check 8 verifies provider connectivity, check 9 warns at 80% and fails at 100% of monthly budget.

### Settings UI

`/settings` page lets you switch provider, model, and budget without editing config files. API keys are stored as environment variables only — the UI displays the env var *name* and its set/unset status, never the key value itself.

See [docs/adr/006-llm-provider-abstraction.md](docs/adr/006-llm-provider-abstraction.md) for the pluggable design rationale.

Upgrading: `git pull && uv sync --extra dev && uv run ctx mcp install`.

## Installation

### One-time setup

```bash
git clone https://github.com/lukinvit/LV_DCP.git
cd LV_DCP
uv sync --all-extras
make lint typecheck test  # verify toolchain
```

### Global `ctx` CLI (optional)

To run `ctx` from any directory without `uv run`:

```bash
mkdir -p ~/bin
ln -sf "$(pwd)/.venv/bin/ctx" ~/bin/ctx
# ensure ~/bin is on PATH (add to ~/.zshrc or ~/.bashrc if missing):
#   export PATH="$HOME/bin:$PATH"
```

Then from any project directory:

```bash
cd /path/to/your-project
ctx scan .                 # incremental index
ctx scan --full .          # full re-parse
ctx pack . "your question"
ctx inspect .              # index stats
ctx wiki update .          # refresh dirty wiki articles
```

Do **not** put `uv run` in front of a wrapper-installed `ctx` — the venv already
knows its interpreter, and `uv run` inside another project's directory will try
to resolve a non-existent local environment and recurse.

### Project onboarding

```bash
# One guided command for a new project
uv run ctx setup /absolute/path/to/your/project --open-ui
```

`ctx setup` reuses the existing install/scan/wiki/watch/UI primitives and ends
with a readiness summary:

- `base mode` = local scan/index/packs/UI baseline
- `full mode` = hybrid/vector retrieval quality

For `full mode`, LV_DCP requires:

- `Qdrant`
- a real embedding provider/API key (for example `OPENAI_API_KEY`)

For wiki generation, LV_DCP also requires:

- `claude` CLI on `PATH`

### Register with Claude Code (recommended)

```bash
# Register the MCP server so Claude Code calls it automatically
claude mcp add --scope user lvdcp -- \
  uv run --directory /absolute/path/to/LV_DCP python -m apps.mcp.server

# Verify
claude mcp list
# Should show: lvdcp: ... - ✓ Connected
```

Replace `/absolute/path/to/LV_DCP` with your actual clone path.

After this, restart your Claude Code session (or VS Code if using the extension). Claude will now see `lvdcp_pack`, `lvdcp_scan`, `lvdcp_inspect`, `lvdcp_explain`, `lvdcp_neighbors`, `lvdcp_history`, `lvdcp_cross_project_patterns`, `lvdcp_memory_propose`, `lvdcp_memory_list` as available tools and will call them automatically per the behavioral rules in `~/.claude/CLAUDE.md`.

### What gets indexed

Supported languages:
- **Python** — via stdlib `ast`, extracts classes, functions, methods, constants, imports, same-file calls
- **TypeScript / JavaScript** — via tree-sitter, extracts classes, functions, interfaces, enums, imports, constants
- **Go** — via tree-sitter, extracts functions, methods, types, imports
- **Rust** — via tree-sitter, extracts functions, structs, traits, enums, use declarations
- **Markdown** — heading extraction as navigation anchors
- **YAML / JSON / TOML** — syntax validation, config role detection

Automatic ignore list: `.git/`, `.venv/`, `node_modules/`, `__pycache__/`, `.mypy_cache/`, `.ruff_cache/`, `.pytest_cache/`, `dist/`, `build/`, `.context/`, `secrets/`, `credentials/`, plus `.env`, `.env.local/production/staging/development`, `credentials.json`, `secrets.json`.

### Privacy model

- **All processing is local.** LV_DCP never makes network calls on its own. The only outbound data is what your LLM client (e.g. Claude Code) sends to its own API — and LV_DCP reduces that payload rather than expanding it.
- **Secret pattern detection** — 12 regex patterns for AWS keys, OpenAI/Stripe/GitHub/Slack tokens, JWTs, PEM private key headers. Files matching these are indexed by path only; their content is excluded from the full-text index and context packs.
- **Deny list** — env files and credentials files are ignored at the path level before any content is read.
- **Nothing leaves the machine.** Indexes live in `.context/` directories inside each project root. Delete the directory to remove the index.

- **v0.8.19** (done) — outage-duration signal on the `wiki_refresh` recovery toast. Since v0.8.11 the green recovery toast has read "Wiki refresh status recovered — `<slug>` is reachable again. Resuming normal polling." — the copy told the user *that* recovery happened but never *how long* they were looking at the yellow degraded card, a gap explicitly documented when v0.8.15 added `trigger=manual|poll` telemetry. This release closes it: a user returning to their dashboard after lunch now sees "reachable again after 1h 14m" instead of a generic "reachable again", short blips read "after 47s" / "after 2m 17s". Design: the route holds **no server-side state** tracking outage starts — no session, no cookie, no Redis entry, the dashboard stays stateless HTMX end-to-end. The outage-start timestamp instead rides on the browser via an HTMX `hx-headers` round-trip, identical to the v0.8.11 `X-LV-DCP-Was-Degraded` marker and the v0.8.15 `X-LV-DCP-Retry-Source` classifier. On the first degraded render the route's `except` branch stamps `degraded_since = int(time.time())` into the template context and the degraded wrapper's `hx-headers` JSON includes `"X-LV-DCP-Degraded-Since": "<ts>"` alongside the was-degraded marker; on every subsequent degraded tick HTMX echoes the header back, the route parses it via `_parse_degraded_since(request)`, and the new degraded render re-stamps the **same** value — so the original start time is preserved indefinitely across a long outage rather than resetting every tick. On the recovery tick the same echo is translated into `outage_seconds = now - since`, formatted via `_format_outage_duration(seconds)` as a short human label (`"<1s"` / `"47s"` / `"2m 17s"` / `"1h 23m"` — only the two largest non-zero units, so `1h 23m` not `1h 23m 47s` for scannability), and the template splices it into the toast copy via `{% if outage_duration_label %}is reachable again after {{ outage_duration_label }}. {% else %}is reachable again. {% endif %}`. A missing / malformed / future header collapses cleanly to the pre-v0.8.19 copy so browser tabs that opened on older builds still get the recovery toast on rolling upgrades — they just don't get the duration suffix. The Retry button's `hx-headers` JSON now carries all three markers (was-degraded + retry-source + degraded-since) so manual-retry recoveries surface the duration identically to poll-initiated recoveries — no split UX where one path flashes "recovered after 2m" and the other flashes a generic "recovered". Design choice lock: the repo ships the command, not the schedule — LV_DCP's "local-first, minimal moving parts" constitution disfavours an in-process scheduler inside the daemon (extra moving part, fires only when the daemon runs, laptop-closed = no sync), while launchd/cron is the macOS-native way and already what every power user trusts for scheduled jobs. The release note documents the two canonical setups (launchd `StartCalendarInterval` and a `0 3 * * *` crontab) but the LV_DCP installer touches neither — `ctx mcp install` writes to `~/.claude/` because that's Claude's territory, a scheduled daemon writing to `~/Library/LaunchAgents/` is a different trust level the user should own explicitly. 18 new tests in `tests/unit/cli/test_obsidian_sync_all.py` covering config validation (missing / disabled / empty vault_path / no projects), dry-run plan (fresh / cacheless / recently-synced / `--stale-hours 0` force), live sync (subprocess invoked with correct args + marker advanced, fresh skipped, mixed stale/fresh respects gate, one failure doesn't block others, `TimeoutExpired` treated as failure, nonexistent root skipped not crashed), plus parametrized `_is_stale` invariants. All new code lives in `apps/cli/commands/obsidian_cmd.py`: `_read_last_sync` returns `float | None`, `_is_stale` combines missing-marker-is-stale and force-hours-zero into one gate, `@app.command("sync-all")` validates config → builds plan → branches on dry-run → shells out per project → writes marker only on zero-return-code. Zero changes to the daemon, publisher, or after-scan worker. Full suite: **1405 passing** (+18 new sync-all tests). Ruff clean, ruff format clean, mypy strict clean. Known limits: sequential not parallel (20-project sweeps run in <1 min on warm SQLite, parallelization would need vault-lock strategy); no per-project interval override (global `--stale-hours`, "small nightly, big weekly" needs two launchd entries with two `--config` files); no built-in scheduler (deliberate — Windows/Linux users who can't `launchd` would need a `ctx watch install-service`-style follow-up); exit code `0` or `1` not `2`-means-partial (scripts parse the `"failed: N"` tail).
