# v0.8.44 — `ctx memory list --json` for scriptable memory review queue

**Date:** 2026-04-25
**Previous:** [v0.8.43 — `ctx inspect --json` for scriptable index stats](2026-04-25-v0.8.43-inspect-json.md)
**Type:** UX (CLI scriptability — extends the v0.8.38/v0.8.40/v0.8.42/v0.8.43 `--json` rollout to the memory subcommand group)
**Risk:** Low — additive flag; existing text path byte-identical; only one new CLI flag; no library signature changes; no on-disk artifact changes; `accept_cmd` and `reject_cmd` untouched.

## Problem

`ctx memory list` has been the human-readable review queue for proposed project memories since the feature shipped, but its CLI output stayed text-only:

```
[proposed ] mem_abc123  Auth refresh token rotation  (2026-04-20T14:33:01Z)
[accepted ] mem_def456  Qdrant payload schema        (2026-04-21T09:12:44Z)
[rejected ] mem_ghi789  Old worker queue             (2026-04-22T15:01:09Z)
```

That's fine for humans but useless for any automation that wants to act on the review queue programmatically:

- A weekly digest script that posts "you have 7 unreviewed memory proposals" to Slack needs `jq 'map(select(.status == "proposed")) | length'`, not a brittle `grep "^\[proposed"` over prose.
- A CI guard for "no rejected memories accidentally got accepted between PRs" needs `jq -r '.[] | select(.status == "accepted") | .id'` against a JSON array, not a parsed table.
- A workspace-sweep — `find ~/dev -maxdepth 2 -name '.context' -exec ctx memory list --project {}/.. --json` — wants per-project payloads it can fold into a single review-queue dashboard.
- A pre-commit hook that surfaces "this branch has unreviewed memories from your last `ctx memory propose` session" needs structured access to `id` + `topic` + `created_at_iso` without a regex.

Same gap that v0.8.38 closed for `prune`, v0.8.40 for `restore`, v0.8.42 for `scan`, v0.8.43 for `inspect`. Time to close it for `memory list`.

## Solution

A `--json` flag on `ctx memory list` that emits a bare JSON array — one entry per memory:

```bash
$ ctx memory list --project /path --json
[
  {
    "id": "mem_abc123",
    "status": "proposed",
    "topic": "Auth refresh token rotation",
    "tags": ["auth", "security"],
    "created_at_iso": "2026-04-20T14:33:01Z",
    "created_by": "claude",
    "path": "/path/.context/memory/mem_abc123.md"
  },
  ...
]
```

### Schema notes

- **1:1 mirror of the `Memory` dataclass minus `body`.** The markdown body of a memory can be arbitrarily large (multi-paragraph notes, code snippets, decision rationales). Including it inflates list payloads on projects with hundreds of memories — which is the exact use case `--json` exists to support — without serving the consumer pattern (`list` is the **index** view; reading a single memory's body is `cat $(jq -r '.[0].path')`).
- **`tags` is a JSON array, not a tuple.** The dataclass stores `tags: tuple[str, ...]` for hashability, but JSON has no tuple type and downstream serializers (Python `json`, `jq`, JS `JSON.parse`) all map to lists anyway. The serializer eagerly calls `list(memory.tags)` to lock the contract.
- **`status` is the enum value (`"proposed"` / `"accepted"` / `"rejected"`)**, not the enum object. Lets `jq 'map(select(.status == "proposed"))'` work without nested object access.
- **Bare array, not `{"memories": [...]}`.** Matches the v0.8.32 `ls --json` and v0.8.40 `restore --json` precedent. Empty list (`[]`) is the contract for "no memories matched" — never `null`, never the human-prose `(no memories)` marker. Consumers do `jq length` without a None-guard.
- **Composes with `--status`.** `ctx memory list --status accepted --json` returns only accepted entries, identical filter semantics to the text path. The render switch (`--json`) is orthogonal to the filter axis (`--status`).

### Error path

No new error cases. `MemoryStatus(status)` raises `ValueError` on a bad `--status` value the same way it did before — Typer surfaces the traceback. The `--json` flag never swallows that into a JSON `{"error": ...}` payload — same discipline as v0.8.42 / v0.8.43: exit code is the script gate, stderr carries the human message, stdout is exclusively the success payload.

## Library / CLI layering

CLI-only change. No new library helpers, no schema modifications to `libs/memory/models.py` or `libs/memory/store.py`.

A small `_memory_to_json(memory: Memory) -> dict[str, object]` helper in `apps/cli/commands/memory_cmd.py` builds the per-row payload. Lives next to `list_cmd` (the only consumer) — when a future v0.8.x extends the CLI surface (e.g. a `ctx memory show --json`), it can either grow that helper or extract it into `libs/memory/`. Premature to extract now: `accept_cmd` and `reject_cmd` already echo single-line status confirmations; they don't need a JSON view (`ls --json` after the mutation is the script-side contract for "did the accept land").

## New invariants

Three new tests in `tests/unit/cli/test_memory_cmd.py` (5 existing + 3 new):

- **`test_memory_list_json_emits_well_formed_array`** — bare JSON array, exact key set per entry (`id, status, topic, tags, created_at_iso, created_by, path`), `tags` is a list (not tuple), `status` is the enum value, `path` ends in `.md`. Schema-locked surface — adding a key requires bumping the `_MEMORY_LIST_JSON_KEYS` frozenset + the helper at the same time.
- **`test_memory_list_json_empty_returns_bare_list`** — empty project (no memories) returns `[]`, not `null`, not the `(no memories)` prose marker. Locks the empty-list contract so consumers can do `jq length` without a None-guard.
- **`test_memory_list_json_composes_with_status_filter`** — `--status accepted --json` returns `[]` before accept, returns the row after `accept_memory()` flips it, and `--status proposed --json` returns `[]` after the flip. Locks render-vs-filter orthogonality.

## Rejected alternatives

- **Include `body` in the JSON output.** Inflates the list payload (memory bodies are often hundreds of lines of markdown — multi-paragraph notes, code snippets), which is the exact use case `--json` exists to support. Recoverable with `cat $(jq -r '.[].path')` for the rare consumer that needs it. A future `ctx memory show <id> --json` is the right surface for body-level access.
- **Wrap in `{"memories": [...]}`.** Breaks the v0.8.32 / v0.8.40 / v0.8.43 bare-array precedent; forces every consumer to write `.memories[]` instead of `.[]`. The wrapping object would be useful only if we needed sibling top-level keys (e.g. a `total` count) — which `jq length` already gives.
- **Schema-version the JSON output (`"_schema_version": 1`).** Premature ceremony, same v0.8.38 / v0.8.40 / v0.8.42 / v0.8.43 argument. Add when there's a real consumer that needs to gate on schema versions.
- **Add `--json` to `accept` / `reject` too.** Premature — no current consumer needs structured confirmation of the mutation. `ctx memory list --status accepted --json` after the accept is the right script-side contract for "did the accept land". Symmetry-for-symmetry's-sake adds CLI surface for zero use case.
- **Make the path field a relative path inside the project.** The absolute path is more useful for `find -exec` / multi-project sweeps that need to bind memories back to their on-disk markdown source; the basename / project-relative form is trivially recoverable via `jq '.path | split("/") | last'`.
- **Sort the array by `created_at_iso` descending so newest are first.** Order is currently controlled by `list_memories` in the library — changing it here would split the source of truth and surprise consumers of the text path who already see one ordering. Order-sensitive consumers can `jq 'sort_by(.created_at_iso) | reverse'`.

## Scope locks

- Only one new CLI flag (`--json`) on `list_cmd`.
- `accept_cmd` and `reject_cmd` **untouched**.
- `libs/memory/models.py::Memory` and `libs/memory/store.py::list_memories` library signatures **unchanged**.
- No new on-disk artifacts.
- No MCP surface — `lvdcp_memory_list` already returns structured data; duplicating with a `--json` flag would split the source of truth.
- Text output **byte-identical** — every existing memory test passes.
- structlog routing introduced in v0.8.42 (`apps/cli/main.py`) carries forward unchanged — JSON consumers benefit automatically.

## Follow-ups explicitly NOT done here

- A `ctx memory show <id> --json` for body-level access. Different consumer pattern (single memory, full body) — separate design when there's a forcing function.
- A `--limit N` flag for paginating large memory queues. `jq '.[:50]'` handles slicing for now.
- A `ctx memory list --since <iso>` filter. `jq 'map(select(.created_at_iso > "2026-04-01"))'` handles this; promote to a flag only if the pattern repeats.
- A README badge tracking the JSON-scriptable surface coverage (e.g. "JSON-scriptable: scan ✓, inspect ✓, prune ✓, restore ✓, ls ✓, memory ✓"). Premature — release notes carry this discoverability and a badge would need maintenance every time another command grows `--json`.
- `--json` on `accept` / `reject` for structured confirmation of mutations. `list --status accepted --json` after the call is the right contract; promote only if a real consumer needs it.
