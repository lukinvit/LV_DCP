# v0.8.27 ‚Äî VS Code extension user-configurable settings

**Date:** 2026-04-24
**Scope:** `apps/vscode/` only ‚Äî adds `contributes.configuration`, refactors `ctxClient.ts` signatures, threads config through `extension.ts`. No backend change, no Python change.

## Why

v0.8.26 shipped marketplace-readiness (metadata, README, CHANGELOG, LICENSE, tightened `.vscodeignore`) and explicitly scope-locked **three** known gaps:

1. Actual marketplace publication ‚Äî user-gated (Publisher + PAT)
2. PNG icon ‚Äî designer-gated
3. **Extension settings** ‚Äî "`ctx` must currently be on `PATH`; there's no `lvdcp.cliPath` configuration. Users running `ctx` via `uv run python -m apps.cli` need a wrapper shell script. Future increment will add `configuration` contribution points."

This release closes gap 3. It's the cheapest of the three to ship from an autonomous-mode increment: pure TypeScript + VS Code config contribution, no user-level action, no design-asset dependency.

The gap is real. A user who installed LV_DCP with `uv sync` has `ctx` reachable via `uv run python -m apps.cli` but not on `PATH` ‚Äî until this release, the extension would `ENOENT` on every command with no in-extension configuration to fix it. The only workaround was a shell wrapper + symlink into `~/.local/bin` ‚Äî a reasonable maintainer workflow but a hostile onboarding story.

## Design

### `apps/vscode/package.json` ‚Äî new `contributes.configuration`

Three settings under `lvdcp.*`:

| Setting | Type | Default | Purpose |
| --- | --- | --- | --- |
| `lvdcp.cliPath` | `string` | `"ctx"` | Path to the `ctx` executable. Accepts bare name (PATH lookup), absolute path (`/usr/local/bin/ctx`), or a wrapper script (`/Users/you/bin/ctx-lvdcp`). |
| `lvdcp.defaultMode` | `string` (enum `"navigate" \| "edit"`) | `"navigate"` | Default retrieval mode for `LV_DCP: Get Context Pack`. `LV_DCP: Show Impact` always uses `"edit"` regardless. |
| `lvdcp.cliTimeoutMs` | `number` (range 1000‚Äì300000) | `30000` | Timeout (ms) for `ctx` subprocess calls. Previously hardcoded: 30s in `getContextPack`, 15s in `getInspect`. |

Rationale on each:

- **`cliPath` default `"ctx"`** ‚Äî zero-config default preserves v0.8.26 behaviour exactly.
- **`defaultMode` enum + `enumDescriptions`** ‚Äî the Settings UI renders a dropdown with per-option descriptions; free-text would require users to remember the strings and typos would silently fall through to the default.
- **`cliTimeoutMs` range `1000`‚Äì`300000`** ‚Äî hard floor avoids 0/negative values (would cause immediate `ETIMEDOUT`); hard ceiling is 5 min, which is already generous for a synchronous context pack.
- **No `lvdcp.backendUrl`** ‚Äî the extension doesn't talk to the backend directly; it shells out to `ctx`, which handles backend discovery via `.context/config.yaml`. Adding an extension-side `backendUrl` would be dead config.

### `apps/vscode/src/ctxClient.ts` ‚Äî explicit `CtxConfig` parameter

Before: `getContextPack(projectPath, query, mode = "navigate")` ‚Äî `"ctx"` and `30000` baked in as string literals.

After:

```ts
export interface CtxConfig {
    cliPath: string;
    timeoutMs: number;
}

export async function getContextPack(
    projectPath: string,
    query: string,
    mode: "navigate" | "edit",
    config: CtxConfig,
): Promise<PackResult> { /* ... */ }

export async function getInspect(
    projectPath: string,
    config: CtxConfig,
): Promise<string> { /* ... */ }
```

Signature change, not behaviour change. Default values live at the call site (in `extension.ts`), read from `vscode.workspace.getConfiguration("lvdcp")`, not baked into `ctxClient.ts` ‚Äî the client is now a pure thin wrapper over `execFile`.

The `mode` parameter on `getContextPack` lost its `= "navigate"` default. The old default was a call-site convenience; under the new model the call site always computes the mode explicitly (from `readDefaultMode()` for `Get Pack`, or hardcoded `"edit"` for `Show Impact`), so the default would be dead code.

### `apps/vscode/src/extension.ts` ‚Äî config readers

Two tiny helpers, called per-command-invocation:

```ts
function readCtxConfig(): CtxConfig {
    const cfg = vscode.workspace.getConfiguration("lvdcp");
    return {
        cliPath: cfg.get<string>("cliPath", "ctx"),
        timeoutMs: cfg.get<number>("cliTimeoutMs", 30000),
    };
}

function readDefaultMode(): "navigate" | "edit" {
    const cfg = vscode.workspace.getConfiguration("lvdcp");
    const mode = cfg.get<string>("defaultMode", "navigate");
    return mode === "edit" ? "edit" : "navigate";
}
```

**Why per-invocation, not cached / subscribed.** `vscode.workspace.getConfiguration(...)` returns a live view and is cheap (O(1) hash lookup in the VS Code config cache). Caching into a module-level variable would require an `onDidChangeConfiguration` subscription + disposal wiring, and the only UX win would be eliminating a sub-millisecond read on the already-slow path of spawning a subprocess. The per-invocation read also means edits in the Settings UI take effect immediately for the next command ‚Äî a small but real UX win.

**Why `readDefaultMode` narrows the string.** `cfg.get<string>("defaultMode", ...)` returns `string`, not the literal union. A user who hand-edits `settings.json` with a typo (`"ediit"`) would pass an invalid value to `getContextPack`, which is typed as `"navigate" | "edit"`. Narrowing with `mode === "edit" ? "edit" : "navigate"` keeps the extension type-safe AND gives the user the default mode on typo rather than an `ENOENT`-ish surprise.

### `apps/vscode/README.md` ‚Äî Settings section + `uv run` example

New "Settings" section between "Commands" and "Design philosophy", with the three-row table and a concrete wrapper-script example:

```sh
#!/usr/bin/env bash
exec uv run --project /path/to/LV_DCP python -m apps.cli "$@"
```

Saved to `~/bin/ctx-lvdcp`, referenced via `"lvdcp.cliPath": "/Users/you/bin/ctx-lvdcp"`. This is the exact pattern that was called out as "needs a wrapper shell for now" in v0.8.26's known-limits ‚Äî now documented as the supported solution.

The "Known limitations" section loses the "no extension settings" bullet; the remaining two bullets are unchanged.

## Known limitations (explicit scope locks)

- **Per-invocation config read, not subscribed.** A user editing `cliPath` while a pack is mid-flight gets the old value for the in-flight call. `onDidChangeConfiguration` subscription would close this but the corner case is rare and the wiring complexity isn't justified for v1.
- **No `lvdcp.backendUrl`.** Backend URL is a `ctx`-side concern via `.context/config.yaml`, not an extension concern. Would be dead config on the extension side.
- **No workspace-level override surface.** All three settings live in User / Workspace / Folder scope as VS Code natively provides ‚Äî no extension-side logic to route, e.g., per-multi-root-workspace-folder config. Not a gap, just explicitly noted.
- **No automated TS tests.** `npm run compile` typechecking is the only gate. `@vscode/test-electron` integration tests are a plausible future increment but need a CI runner willing to bring up an xvfb-driven VS Code.
- **Still not published to the Marketplace.** Same gate as v0.8.26 ‚Äî requires Publisher registration + PAT, out of scope for autonomous mode.
