# v0.8.52 — `ctx eval --json` for scriptable retrieval-quality metrics

**Date:** 2026-04-25
**Tag:** `v0.8.52`
**Theme:** CLI scriptability rollout — quality measurement surface

## Summary

Adds `--json` to `ctx eval`, exposing the full `EvalReport` payload as
a single structured object so retrieval-quality dashboards, regression
gates, and trend dashboards can consume per-query and aggregate
metrics without parsing the human-readable markdown report.

This is the natural follow-up to v0.8.51 (`ctx pack --json`): pack is
the **production retrieval surface**, eval is the **measurement
surface** for that retrieval. Both now expose structured payloads —
the closed loop "did our pack quality regress on this branch" finally
has a clean CI gate without re-parsing markdown tables.

## Schema

`ctx eval --json` returns a single JSON object that exactly mirrors
the `EvalReport` dataclass plus invocation-parameter round-trip
fields:

```json
{
  "project": "/abs/path/to/project",
  "queries_path": "/abs/path/to/queries.yaml",
  "impact_queries_path": null,
  "summary": {
    "recall_at_5_files": 0.85,
    "precision_at_3_files": 0.72,
    "recall_at_5_symbols": 0.0,
    "mrr_files": 0.66,
    "impact_recall_at_5": 0.0
  },
  "query_results": [
    {
      "query_id": "q01",
      "mode": "navigate",
      "retrieved_files": ["app/auth.py", "..."],
      "retrieved_symbols": ["app.auth.login"],
      "expected_files": ["app/auth.py"],
      "expected_symbols": []
    }
  ]
}
```

### Why this exact shape?

- **`project` + `queries_path` round-tripped.** Same v0.8.48 / v0.8.50
  precedent for "did this run actually use the input I intended" —
  catches `--queries` typos without echoing arbitrary input flags
  back. `impact_queries_path` is `null` when `--impact-queries` was
  not passed, distinguishing "no impact queries supplied" from
  "impact queries supplied but yielded zero results" (both would
  collapse to `impact_recall_at_5=0.0` without this field).
- **`summary` nested instead of flat top-level.** Aggregate metrics
  live under `summary` rather than at the top-level so the
  `query_results` array sits cleanly alongside without ambiguity
  about which fields are aggregates vs. invocation parameters. The
  schema-locked test asserts both `_EVAL_JSON_KEYS` (top-level) and
  `_EVAL_SUMMARY_KEYS` (nested) — adding a metric forces a
  reviewed update to both frozensets.
- **`query_results` order matches input file order.** Same semantic
  as the markdown report; lets dashboards correlate per-query rows
  across runs by index without re-keying on `query_id`. Order-
  sensitive consumers can `jq 'sort_by(.query_id)'` if they want
  alphabetical instead.

## The `--baseline` deferred decision

`--json` is **not yet supported with `--baseline`** — the combination
exits 2 with a helpful error message. Rationale:

- The default-mode `EvalReport` is a single-payload single-shape
  contract; the comparison-report mode produces two `EvalReport`s
  plus diff metrics, which would force either a single object with
  conditional fields (ugly schema split) or a schema bump in this
  ship without a real consumer driving the design.
- Same scope discipline as v0.8.42-v0.8.51: keep increments small.
  When a real consumer needs comparison `--json`, the right factoring
  is a separate ship that designs the comparison shape against the
  actual use case (drift detection? regression gate? both?).
- The gate is the cleanest possible signal: error message tells the
  user the combination is unsupported, suggests both alternatives
  ("run without `--baseline` to get the structured payload, or omit
  `--json` for the markdown comparison report"), and the test locks
  the rejection so a refactor that silently accepts the combo
  breaks the test instead of shipping a half-defined payload.

## `--output` composition

- Without `--json`: `--output` writes the markdown report to the
  file (existing behavior, unchanged).
- With `--json`: `--output` writes the JSON payload to the file.
  The `wrote: ...` confirmation goes to **stderr** in JSON mode (not
  stdout) — keeps stdout pure for scripts that want the byte-
  identical "redirect-then-read" pattern. Markdown mode keeps the
  confirmation on stdout for byte-identical text-mode preservation.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/eval/` — `EvalReport`,
  `QueryResult`, `run_eval`, and the report generators all unchanged.
  `_eval_report_to_json` lives next to `eval_cmd` in
  `apps/cli/commands/eval_cmd.py` — sister-helper to v0.8.46-v0.8.51
  CLI-shell payload shapers.
- **Schema-locked tests.** Three frozensets in
  `tests/unit/cli/test_eval_cmd.py` assert the exact key set on
  every emission: `_EVAL_JSON_KEYS` (top-level), `_EVAL_SUMMARY_KEYS`
  (nested aggregate metrics), `_EVAL_QUERY_RESULT_KEYS` (per-query
  rows). Adding a new field to `EvalReport`, `QueryResult`, or the
  CLI helper without updating the corresponding frozenset will fail
  the test instead of silently diverging.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  human-readable markdown report. Pre-existing pipes that consume
  `ctx eval` output as plain markdown see no diff.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42. The
  `wrote: ...` confirmation in `--json --output` mode lands on
  stderr to keep stdout clean.

## Tests

Four new tests in `tests/unit/cli/test_eval_cmd.py`:

| Test | Mode | Asserts |
|---|---|---|
| `test_eval_cmd_text_output_unchanged` | text | markdown body preserved + no JSON leak (`pytest.raises(json.JSONDecodeError)`) |
| `test_eval_cmd_json_emits_well_formed_payload` | json | `_EVAL_JSON_KEYS` schema lock, `summary` is nested dict with `_EVAL_SUMMARY_KEYS` and float-in-`[0,1]` values, `query_results` array is per-row schema-locked, order matches input file |
| `test_eval_cmd_json_with_output_writes_json_to_file` | json + output | file receives JSON payload (not markdown), schema-locked when read back |
| `test_eval_cmd_json_with_baseline_rejects_combo` | json + baseline | combination rejects with exit 2 + helpful error mentioning both flags; no half-defined JSON on stdout |

All 9 tests in the file pass (5 pre-existing + 4 new).

## Why this matters

Before this ship: every retrieval-quality dashboard, regression gate,
or trend tracker had to either (a) parse the markdown table back into
structure (brittle, lossy — per-query expected-vs-retrieved diffs are
embedded in formatted lists), or (b) duplicate `run_eval` via a
custom Python script (operational overhead, drift risk against the
shipped CLI). After this ship: `ctx eval ... --json` returns the
`EvalReport` dataclass shape directly. Same pipeline, same metrics,
zero markdown parsing.

This unblocks workflows that v0.8.51 (`pack --json`) couldn't reach
on its own:

- **CI regression gates**: `ctx eval . --queries q.yaml --json | jq -e '.summary.recall_at_5_files >= 0.7'` — fail the PR if retrieval quality drops below threshold
- **Per-query trend tracking**: `ctx eval . --queries q.yaml --json | jq -c '.query_results[] | {q: .query_id, hit: (.retrieved_files | inside(.expected_files))}'` for "which queries regressed this week"
- **Cross-project quality sweeps**: same eval call across N projects, fold per-project `summary.recall_at_5_files` into a fleet retrieval-quality dashboard
- **Closing the v0.8.51 loop**: `pack --json` produces the retrieval, `eval --json` measures it — both now consume / emit structured data, the closed-loop "ship a retrieval change → run eval → gate on metric drop" pipeline finally has a clean Unix surface
