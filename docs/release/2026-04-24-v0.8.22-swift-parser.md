# v0.8.22 — Swift tree-sitter parser

**2026-04-24**

## Why

Third shipped increment of Phase 10. v0.8.20 shipped the Java parser,
v0.8.21 shipped the Kotlin parser. With the JVM pair complete, Swift
is the natural next language: a self-contained deliverable, published
`tree-sitter-swift` wheel is available with stable bindings, and the
LV_DCP side of the work is a ~130-line parser + three one-line
allowlist additions that follow the exact `TreeSitterParser` hook
shape established by the five prior parsers (Go / Rust / TS/JS / Java
/ Kotlin).

Swift is a strategically valuable target: macOS-first means local-first
LV_DCP users are disproportionately Swift developers (iOS/macOS apps,
Swift Packages, server-side Swift via Vapor/Hummingbird). Indexing
Swift lets LV_DCP answer cross-language questions inside mixed
Swift/Objective-C codebases and inside SPM-published libraries that
App developers consume.

## Design

`libs/parsers/swift.py` is a 130-line `TreeSitterParser` subclass
with `language = "swift"`. All non-trivial Swift-specific decisions
are hook overrides.

### Symbol type mapping

| Swift AST node | LV_DCP `SymbolType` |
|----------------|---------------------|
| `class_declaration` | `CLASS` |
| `protocol_declaration` | `CLASS` |
| `function_declaration` | `METHOD` |
| `init_declaration` | `METHOD` |
| `protocol_function_declaration` | `METHOD` |
| `property_declaration` | `VARIABLE` |

`class_declaration` covers `class`, `struct`, `actor`, `enum`, AND
`extension Foo { ... }`. The tree-sitter-swift grammar uses one node
kind for all of these — the distinguishing keyword (`struct` / `class`
/ `actor` / `enum` / `extension`) appears as a child of the
`class_declaration` node. For v1 we don't branch on the keyword — a
type declaration is a type declaration.

`protocol_declaration` is a separate node kind with a `name` field.
Protocols map to `CLASS` (matches Kotlin `interface` → CLASS): they
declare a named type, they participate in inheritance graphs, they
are the natural symbol granularity for "find the Drawable protocol"
queries.

`init_declaration` → METHOD. The node's `name` is the literal string
`"init"`. Swift allows multiple initializers (designated + any number
of convenience), so multiple METHOD symbols named `"init"` under the
same parent class is both expected and correct.

`protocol_function_declaration` → METHOD. Protocol bodies use a
distinct grammar node kind for their method requirements (`func
draw()` inside `protocol Drawable` is `protocol_function_declaration`,
not `function_declaration`). Surfacing them as METHOD matches the
in-class case — protocol methods are part of the protocol's API
contract and should be findable by name.

### Extension handling

Extensions compile as `class_declaration` nodes with the extended
type's name as the `name` field — `extension User: CustomStringConvertible`
emits a CLASS symbol named `"User"`, distinct from (but with the same
name as) the primary `struct User` declaration.

v1 design decision: **accept the apparent duplicate**. It's semantically
correct — the graph now records every file that touches `User` (primary
declaration + every extension site), and methods inside each extension
correctly associate with the extended type via `parent_fq_name`.
`test_extension_emits_duplicate_class_symbol` and
`test_extension_method_parent_is_extended_type` lock this behaviour.

### Init and property extraction

Both `init_declaration` and `property_declaration` nodes expose their
name directly via the `name` field, so the default
`_extract_name = node.child_by_field_name("name")` works unchanged.

Swift simple property declarations like `let MAX_SIZE: Int = 1024` and
`var name: String` parse cleanly with the default name extraction.
Computed properties (`var description: String { return name }`) also
work — the name field anchors to the identifier before the type
annotation.

### Static / const property handling

Swift's `static let SHARED_INSTANCE = Counter()` is the closest thing
the language has to Java's `static final` or Kotlin's `const val`.
v1 maps **all** `property_declaration` nodes (including `static let`)
to `VARIABLE` uniformly — matches the Java v1 decision to map
`static final` to `VARIABLE` and the Kotlin v1 decision to map
`const val` to `VARIABLE`. Promoting static / const properties to
`CONSTANT` requires inspecting the property's `modifiers` child at
parse time; the test `test_static_let_maps_to_variable` locks the
current behaviour so a future promotion becomes an explicit contract
break.

### Import relations

`import_declaration` emits `RelationType.IMPORTS` edges with
`dst_type="module"`:

| Swift import form | `dst_ref` |
|-------------------|-----------|
| `import Foundation` | `Foundation` |
| `import UIKit` | `UIKit` |
| `import class Foundation.NSData` | `Foundation.NSData` |
| `import struct Foundation.URL` | `Foundation.URL` |
| `import func Swift.print` | `Swift.print` |

All variants share the same grammar shape: `import_declaration` with
an `identifier` child holding the full module path. For symbol-kind
imports (`import class ...`, `import struct ...`, `import func ...`,
`import protocol ...`, etc.) the kind keyword is a sibling before the
identifier; the identifier itself still contains the full
`Module.Symbol` path. Walking children for the first `identifier`
therefore naturally yields `Foundation` for a plain import and
`Foundation.NSData` / `Swift.print` for scoped symbol imports.

Scoped imports surface the full path rather than trimming to the
module — matches the Java v1 decision to surface
`java.util.Arrays.asList` for static imports. Graph consumers still
see the dependency; scoped / symbol imports convey more specific
information and that extra precision is preserved.

### Role detection

Swift projects span Swift Package Manager (`Sources/<Target>/` +
`Tests/<Target>Tests/`) and Xcode (flat app folders + `<Name>Tests/`
companion dirs). Role detection walks directory fragments first, then
filename fallback:

**Directory fragments** that flip role to `test`:
- `/Tests/` (SPM convention)
- `/tests/`, `/Test/`, `/test/` (broader Unix conventions, parity
  with other parsers)

**Path prefixes** (start-of-path) that flip to `test`:
- `Tests/`, `tests/`, `Test/`, `test/` — explicit startswith check
  so bare root-level `Tests/MyLibTests/...` works without false
  positives from substring matches on names like `SpecialTests/`

**Filename fallback** (strips `.swift` first):
- `*Tests` suffix (e.g. `FooTests.swift`)
- `*Test` suffix (e.g. `ConfigTest.swift`)
- `Test*` prefix (e.g. `TestUser.swift`)

Everything else → `source`. The filename fallback catches Xcode-style
`MyAppTests/FooTests.swift` paths as well as `app/src/ConfigTest.swift`
files outside a dedicated test directory.

### Module FQ derivation

`_module_fq()` strips SPM-convention source roots along with the
target-name segment, then dots the remainder:

| Input path | `_module_fq` output |
|------------|---------------------|
| `Sources/MyLib/Parser.swift` | `Parser` |
| `Sources/MyLib/Utils/Parser.swift` | `Utils.Parser` |
| `Tests/MyLibTests/FooTests.swift` | `FooTests` |
| `modules/foo/Sources/MyLib/Parser.swift` | `Parser` |
| `Sources/Parser.swift` | `Parser` |
| `MyApp/Foo.swift` | `MyApp.Foo` (Xcode flat) |
| `Foo.swift` | `Foo` |
| `Package.swift` | `Package` |

Nested multi-package SPM layouts (`modules/foo/Sources/MyLib/...`)
work because the search looks for the `Sources/` / `Tests/` marker
anywhere in the path, not just at the start — parity with Java's
handling of `modules/foo/src/main/java/...` and Kotlin's handling of
`modules/foo/src/main/kotlin/...`.

`.swift` is the only extension; stripped before the source-root logic
runs.

## Known limitations / non-goals

1. **No `INHERITS` edges** — `class Foo: Bar`, `struct User:
   CustomStringConvertible`, and `protocol P: Q` are visible in the
   AST (`inheritance_specifier` nodes) but v1 doesn't emit graph
   edges for them. Java and Kotlin have the same v1 gap; all three
   would benefit from a shared future inheritance extractor.
2. **No `SAME_FILE_CALLS` edges** — matches Go / Rust / Java /
   Kotlin shape; only the Python parser emits call-graph edges
   today.
3. **`static let` maps to `VARIABLE`, not `CONSTANT`** — documented
   above, locked by test, parity with Java `static final` and
   Kotlin `const val`.
4. **Extensions emit duplicate CLASS symbols with the extended
   type's name** — documented above, locked by test, semantically
   correct for graph lookup but visible as duplicate entries in
   raw symbol listings.
5. **`typealias_declaration` not in the symbol map** — `typealias
   UserID = Int` is a type-level name but not a type. Including
   would need a dedicated SymbolType decision. Natural future
   sub-increment.
6. **`deinit_declaration` not in the symbol map** — `deinit` has
   no name and is effectively an anonymous destructor; skipping
   it matches Kotlin's treatment of companion_object (no name →
   not a symbol, but children still walk). Not a future item —
   nothing to surface.
7. **`enum_case_declaration` not surfaced as separate symbols** —
   cases live inside enum bodies; graph lookup for the enum still
   succeeds. Natural future sub-increment if a driving use case
   arrives.
8. **`protocol_property_declaration` not in the symbol map** —
   protocol property requirements (`var size: Int { get }`) use
   a distinct grammar node with an awkwardly-shaped name that
   includes the `var` / `let` keyword. Skipping for v1; could be
   added with a custom `_extract_name` override in a future
   sub-increment.

## Roadmap positioning

Third shipped increment of Phase 10. With Java (v0.8.20), Kotlin
(v0.8.21), and Swift (v0.8.22) done, the first Phase 10 axis —
**language coverage for the big three ecosystems we didn't yet
support** — is complete. Next plausible increments:

- **VS Code marketplace publication** — the extension has been in
  `apps/vscode` since Phase 6; marketplace listing requires
  publisher verification + CI wheel bundling.
- **Obsidian nightly sync** — the wiki publisher in `libs/wiki`
  already produces a vault-compatible output; a scheduled runner
  would close the loop to an always-fresh knowledge base.
- **Language-depth sub-increments** — inheritance edges for Java /
  Kotlin / Swift (shared extractor), data class / record / primary
  constructor property extraction for Kotlin and Swift, Swift
  typealias and enum case symbols.

Prioritisation from here is driven by concrete user signals rather
than a fixed order.
