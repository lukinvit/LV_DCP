# v0.8.59 — `ctx timeline enable --json` opens the timeline-flag mutation surface

**Date:** 2026-04-25
**Tag:** `v0.8.59`
**Theme:** CLI scriptability rollout — opens the timeline-flag mutation surface

## Summary

Adds `--json` to `ctx timeline enable`, opening the timeline-flag
mutation surface. Single-object payload mirroring v0.8.57 `memory
accept --json` / v0.8.58 `memory reject --json` shape: round-trips the
project root + flag path and surfaces the post-mutation `enabled` boolean
so a script can confirm the state transition without a follow-up
`status --json` call.

This opens the timeline-flag operator surface — the matching `disable
--json` is queued for v0.8.60 to close it end-to-end:

- **Read side** (already shipped): `ctx timeline status --json` — single
  object with `{enabled, total_events, last_scan_commit_sha, ...}`
- **Enable side** (this ship): `ctx timeline enable --json` — single
  object with `enabled: true` post-mutation
- **Disable side** (v0.8.60): `ctx timeline disable --json` — single
  object with `enabled: false` post-mutation (will reuse the same
  `_flag_to_json` helper and `_TIMELINE_FLAG_JSON_KEYS` frozenset)

## Schema

`ctx timeline enable --json` returns a single JSON object — three keys,
schema-locked via `_TIMELINE_FLAG_JSON_KEYS`:

```json
{
  "project_root": "/abs/path/to/project",
  "enabled": true,
  "flag_path": "/abs/path/to/project/.context/timeline.enabled"
}
```

The `enabled` field round-trips as `true` post-mutation so a script can
`jq -e '.enabled == true'` to confirm the state landed without a
follow-up `status --json` call. `flag_path` surfaces the actual on-disk
artifact the daemon / scanner reads, so a debugging script can
`cat $(jq -r .flag_path)` to inspect the raw `on` / `off` content.

## Idempotent re-enable semantic

`enable_cmd` has no state-machine guard — calling it on an already-enabled
project is a no-op-on-state-but-write-on-disk that returns the same
payload. Locked by `test_timeline_enable_json_idempotent_re_enable` which
asserts both call payloads are bytewise identical.

This mirrors v0.8.57's `test_memory_accept_json_idempotent_re_accept`:
proves a script can safely re-run `enable --json` without branching on
"was it already on" — same render-switch-not-semantic-change discipline.

## Cross-state flip semantic

Calling `disable` then `enable --json` flips the flag cleanly with a
**stable** `flag_path` (same on-disk marker rewritten in place, no fresh
file). Locked by `test_timeline_enable_json_overrides_disabled_state`,
mirroring v0.8.58's `test_memory_reject_json_overrides_accepted_status`
cross-state flip — proves no duplicate file is created on the state
transition, just an in-place rewrite of the existing `.context/timeline.enabled`
marker.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/symbol_timeline/store.py` —
  `_write_flag` signature unchanged.
- **Helper introduced** for shared shape across enable/disable.
  `_flag_to_json` is the single point that emits the three-key payload.
  When v0.8.60 lands the `disable --json` slice, it will call the same
  helper with `enabled=False` — one helper, two surfaces, one frozenset
  to bump if a future field appears on the flag.
- **Schema lock** via `_TIMELINE_FLAG_JSON_KEYS` frozenset (new in this
  ship) prepares the cross-surface lock the v0.8.60 `disable` ship will
  inherit. Same pattern as v0.8.44 `_MEMORY_LIST_JSON_KEYS` →
  v0.8.57/v0.8.58 cross-surface span.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  single-line `timeline: enabled for <root>` footer. Pre-existing pipes
  see no diff. Locked by `test_timeline_enable_text_output_unchanged`.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Why this matters

Before this ship: every timeline-flag automation that wanted to
programmatically enable timeline capture (a per-project setup script
that bulk-enables timeline across all registered projects, an IDE
"enable timeline" toggle that needs to confirm the transition before
re-rendering its UI, a daemon-bootstrap step that ensures every newly
registered project has timeline capture on) had to grep the text footer
(`timeline: enabled for <root>`) or rely on the exit code alone — losing
the round-tripped `flag_path` that confirms which on-disk marker was
just written.

After this ship: `ctx timeline enable --json` returns a structured
payload directly. Same flag-write semantic, same on-disk artifact, zero
text parsing.

This unblocks workflows like:

- **Bulk enable across registered projects**: `ctx registry ls --json | jq -r '.[] | select(.scanned == true) | .root' | xargs -I{} ctx timeline enable --project {} --json | jq -s 'map({project: .project_root, enabled: .enabled})'` — auto-enable timeline capture for every scanned project, collect the post-mutation state for an audit log
- **IDE / dashboard toggle confirmation**: `ctx timeline enable --project "$root" --json | jq -e '.enabled == true'` — assert the transition landed before re-rendering the UI; exit 1 from `jq -e` if the toggle silently failed
- **Daemon-bootstrap ensure-on**: `ctx timeline status --project "$root" --json | jq -e '.enabled == true' || ctx timeline enable --project "$root" --json` — idempotent ensure-enabled with no spurious writes; the `--json` payload from `enable` confirms the resulting state in one call
- **Setup-script audit trail**: `ctx timeline enable --project "$root" --json | jq '{when: now, project: .project_root, flag: .flag_path, decision: "enabled"}' >> setup_audit.jsonl` — append-only audit log of timeline-enable events with flag-path provenance preserved
