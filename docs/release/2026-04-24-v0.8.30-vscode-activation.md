# v0.8.30 — VS Code extension activation-event narrowing

**Date:** 2026-04-24
**Scope:** `apps/vscode/package.json` + `apps/vscode/CHANGELOG.md` + new `tests/unit/vscode/test_extension_manifest.py` contract test. No TypeScript source changes, no `out/` rebuild required, no backend change.

## Why

The VS Code extension has always declared `"activationEvents": ["onStartupFinished"]`, which fires on every VS Code cold start regardless of what's in the workspace. Consequences:

- The "LV_DCP" status bar item appears in **every** VS Code window the user opens, even for projects that have nothing to do with LV_DCP — a web frontend scratch repo, a friend's Elm project, a dotfiles folder.
- `activate()` runs, registering two commands and creating a status bar item, even when there's zero chance the user will use them this session.
- In a multi-window workflow (one VS Code per project) this becomes visible noise — the extension looks pushy rather than passive.

The activation cost itself is cheap (sub-millisecond registrations), so this isn't a performance release. It's a **UX correctness** release: the extension should be invisible until there's a reason to be visible.

The Phase 10 roadmap from v0.8.29 explicitly listed "VS Code extension: activation-event narrowing (avoid `onStartupFinished` cold-start cost)" as a polish item. This ships it.

## Design

### The narrowing pattern: `workspaceContains:**/.context/cache.db`

The standard VS Code narrowing lever is `workspaceContains:<glob>` — the extension host evaluates the glob against the workspace root(s) on startup and only activates when a match is found. `.context/cache.db` is the canonical LV_DCP marker: every indexed project has it, no non-indexed project has it. One glob, zero false positives.

```json
"activationEvents": [
  "workspaceContains:**/.context/cache.db"
]
```

Alternative patterns considered and rejected:

- **`workspaceContains:.context/cache.db`** (no `**/`). Wouldn't match monorepos where the index lives in a subdirectory (e.g. `services/api/.context/cache.db`). Strict worse; the `**/` form is the correct default.
- **`onLanguage:python`** plus one per TS/JS/Go/Rust/Java/Kotlin/Swift. Seven activation triggers, fires on any file of those languages — much broader than needed, and still fires for non-LV_DCP projects that happen to be Python.
- **Per-command-only**, i.e. drop `activationEvents` entirely and rely on VS Code ≥1.74's implicit command activation. Status bar item would never appear until the user ran a command for the first time — the affordance that "you can query your code here" is gone.

`workspaceContains:**/.context/cache.db` is the precise trigger: "this workspace has an LV_DCP index, so the extension is relevant here."

### Implicit command activation covers the palette

Since VS Code 1.74, commands declared in `contributes.commands` get implicit activation (same as listing `onCommand:<id>` in `activationEvents`). The engine requirement is `^1.85.0`, well past that threshold, so dropping `onStartupFinished` doesn't break Command Palette invocation in non-indexed workspaces — the command still fires, just activates on demand rather than pre-emptively.

### What changes, what doesn't

| Scenario | Before | After |
| --- | --- | --- |
| Indexed project (has `.context/cache.db`) | Status bar + sidebar + commands | Same — no behaviour change |
| Non-indexed project | Status bar + commands, sidebar empty | **Silent** — no UI until user runs a command |
| Command Palette in non-indexed project | Works | Works (implicit command activation) |
| Cold-start cost per VS Code window | `activate()` always runs | `activate()` runs only if workspace matches |
| Sidebar "LV_DCP" activity-bar icon | Always visible | Always visible (declared contribution, not activation-gated) |

The activity-bar icon is the one surface that doesn't narrow — icons come from `contributes.viewsContainers` which is a declaration, not a runtime activation. Making the icon conditional requires a `when` clause, which is a larger change with real tradeoffs (the view container becomes undiscoverable in a non-indexed project). The status bar item is the more offensive of the two; narrowing it alone captures ~90% of the perceived clutter for ~10% of the design surface.

### The contract test

VS Code extensions ship to the marketplace as a pure `package.json` + `out/*.js` bundle. There's no Python in the production path — but this repo's CI runs the Python test suite on every PR, and the manifest is the source of truth. A regression here (someone copies the 0.8.26/0.8.27 manifest back, or re-adds `onStartupFinished` "for testing") wouldn't break any test, wouldn't be caught by mypy, and would silently ship to users.

`tests/unit/vscode/test_extension_manifest.py` loads `apps/vscode/package.json` as plain JSON and asserts six properties:

1. All marketplace-required fields are present (`name`, `displayName`, `description`, `version`, `publisher`, `license`, `repository`, `engines`).
2. `activationEvents == ["workspaceContains:**/.context/cache.db"]` exactly — catches both additions and removals.
3. `onStartupFinished` does not appear — explicit negative, calls out in the error message that re-adding it requires updating the v0.8.30 release note first.
4. `lvdcp.getPack` and `lvdcp.showImpact` commands stay contributed — guards against the Command Palette regressing when activation is narrowed.
5. `engines.vscode >= 1.74` — implicit command activation only works on VS Code ≥1.74; if the engine requirement ever drops below that, the "drop onStartupFinished, rely on implicit" design assumption is no longer valid.
6. `version >= 0.8.30` — prevents a stray merge from reverting the version bump.

No `npm`, no `@vscode/test-electron`, no Node toolchain in CI — just `json.load()` and asserts. Cost: 0.02s in the test suite.

## Manual verification path

1. Install the `.vsix` in a VS Code instance.
2. Open an unrelated project (no `.context/` directory). Expect: no "LV_DCP" in the status bar. `Ctrl+Shift+P` → "LV_DCP" still lists both commands.
3. Invoke `LV_DCP: Get Context Pack` from the palette. Expect: input prompt appears (command activates on demand); status bar item now shows.
4. Close VS Code, reopen. Expect: status bar item gone again (activation is scoped per window session).
5. Open an LV_DCP-indexed project. Expect: status bar item appears immediately, sidebar tree is ready to receive results.

## Known limitations (explicit scope locks)

- **Activity-bar icon is still always visible.** The `viewsContainers.activitybar` contribution is not activation-gated — it's a declaration. Hiding it conditionally requires a `when` clause on the container, which makes discovery of the extension harder in a freshly-scanned project (the icon wouldn't appear until activation fires, after a scan). Net worse UX. Keep the icon always-visible; narrow the noisier status bar item.
- **Multi-root workspaces.** VS Code re-evaluates `workspaceContains` globs when workspace folders change, so adding a second folder with `.context/cache.db` to an open workspace will activate the extension. Single-folder projects work with no special handling.
- **No `onCommand:` explicitness.** We rely on VS Code ≥1.74's implicit command activation. If the engine requirement ever drops (unlikely — we need the pickers + tree API from 1.85), the command-only activation must be made explicit. The contract test guards the engine floor.
- **Status bar item only tracks the current window.** Opening a second window with a different project means two status-bar items race for the global command binding. Not a v0.8.30 regression — same behaviour as v0.8.27.
- **No rebuild of `out/`.** The manifest change doesn't touch TypeScript. Publishing the extension will run `vscode:prepublish` which re-runs `tsc`, so the published `.vsix` rebuilds anyway; the checked-in `out/*.js` stays unchanged and is not part of this commit.
