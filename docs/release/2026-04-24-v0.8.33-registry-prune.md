# v0.8.33 ŌĆö `ctx registry prune` with dry-run default

**Date:** 2026-04-24
**Previous:** [v0.8.32 ŌĆö `ctx registry ls` for auditing registered projects](2026-04-24-v0.8.32-ctx-registry-ls.md)
**Type:** Destructive CLI surface, gated by dry-run + explicit `--yes`
**Risk:** Medium ŌĆö mutates `~/.lvdcp/config.yaml`, protected by three layers (dry-run default, `--kind transient` default, pre-mutation `*.bak` sidecar)

## Problem

v0.8.32 surfaced pruning candidates via `ctx registry ls --stale`. On my workspace that listed ~50 transient worktree entries ŌĆö abandoned `.claude/worktrees/vX.Y.Z-*` roots left over from prior ship ceremonies. Manual cleanup via `yq`/`sed` is error-prone, and running the daemon with dead entries wastes cycles (each one gets a watchdog attempt, logs a failure, then retries).

The visibility layer gave the data; the user still had no safe remediation command.

## Solution

New `ctx registry prune` subcommand, safe by design:

```text
ctx registry prune                           # dry-run, kind=transient, >30d
ctx registry prune --older-than 60           # raise the staleness cutoff
ctx registry prune --kind real               # target dormant real projects
ctx registry prune --kind all                # both buckets
ctx registry prune --yes                     # actually mutate the config
```

Without `--yes`, the command is a pure read: it prints the entries that *would* be removed and exits without touching the file. With `--yes`, it:

1. Writes a `*.bak` sidecar (`~/.lvdcp/config.yaml.bak`) holding the pre-mutation bytes verbatim.
2. Atomically rewrites the config via tmp file ŌåÆ `fsync` ŌåÆ `rename`.
3. Prints a summary naming the backup path.

Recovery after a mis-prune is a one-liner: `mv ~/.lvdcp/config.yaml{.bak,}`.

## Three layers of safety

1. **Dry-run default.** The CLI `--yes` flag defaults to `False`; `prune_stale(apply=False)` is also the library default. A caller who forgets both gates gets a preview, never a deletion.
2. **`--kind transient` default.** Worktree artifacts and test fixtures have near-zero false-positive cost (regenerate on next scan). Real user projects require explicit `--kind real` ŌĆö you have to type the opt-in for the destructive population.
3. **Pre-mutation backup.** A `*.bak` sidecar is written *before* `save_config()` starts rewriting the registry. If the atomic rewrite crashes mid-fsync, the user still has an authoritative copy.

## Architecture

Kept `libs/status/registry_audit.py` read-only per its docstring invariant. Mutation logic lives in a sibling:

- **`libs/status/registry_prune.py`** (new) ŌĆö `plan_prune()` preview + `prune_stale()` destructive variant + `PruneResult` dataclass.
- **`libs/core/projects_config.py`** ŌĆö added `save_config(path, config)`: atomic tmp + `fsync` + `rename`, with temp-file cleanup on error. No other caller exists yet; this is the minimal surface for the prune flow.
- **`apps/cli/commands/registry_cmd.py`** ŌĆö appended `prune_cmd` Typer subcommand. Thin shell ŌĆö all business logic in `libs/`.

Layering (`apps/* ŌåÆ libs/*`, never reverse) per CLAUDE.md.

## Rejected alternatives

- **Auto-prune on scan.** Too invasive. The user should decide what leaves the registry.
- **`--force` instead of `--yes`.** `--yes` matches `apt`, `brew`, and dozens of other tools ŌĆö the muscle memory is right. `--force` suggests "override a guard," which isn't what's happening here.
- **In-place rewrite without backup.** Power-user ergonomics beaten by the one time you regret it. A 2 KB sidecar is free insurance.
- **Delete the sidecar on success.** Keeps the user guessing whether they have an undo handle. Better to leave it; they can `rm` it when they're confident.
- **Cascade delete `.context/` dirs of pruned projects.** Out of scope. Registry removal is one concern; on-disk cleanup is another. The CLAUDE.md `write_protected_paths` discipline plus the "single-writer" rule argues for keeping them separate ŌĆö a future `ctx registry clean` could handle `.context/` scrubbing, ideally only for roots no longer present in the registry.
- **Confirmation prompt (typer.confirm) instead of `--yes`.** Breaks non-interactive usage (CI, scripts, background hooks). A boolean flag works everywhere and is scriptable.

## Scope locks

- **Only removes registry entries.** No filesystem mutation on the project roots themselves.
- **No change to the `~/.lvdcp/config.yaml` schema.**
- **No MCP surface.** Destructive operations should never be reachable via a Claude-invokable tool.
- **Preserves entry ordering** for survivors (we rebuild the list from `config.projects` in original order minus the removed set).

## Testing

- **Library layer (`tests/unit/status/test_registry_prune.py`)** ŌĆö 8 cases:
  1. `plan_prune` lists candidates without mutation
  2. `prune_stale(apply=False)` is a byte-level no-op (config unchanged, no backup written)
  3. `apply=True` removes matching entries, writes `*.bak`, preserves survivors
  4. `--kind real` doesn't touch transient entries
  5. `--kind all` prunes both buckets
  6. `--older-than` respects recent `last_scan_at_iso`
  7. Invalid `kind` raises `ValueError`
  8. Empty registry is graceful
- **CLI layer (`tests/unit/cli/test_registry_cmd.py`)** ŌĆö 5 new cases via Typer's `CliRunner`:
  1. Default invocation is dry-run (config untouched, no backup)
  2. `--yes` applies, writes `*.bak` with pre-mutation bytes, removes stale entry
  3. Invalid `--kind` exits 2
  4. `--older-than 0` exits 2 (positive-only contract)
  5. `--kind real` against a transient-only stale set reports "nothing to do"
- **Core layer (`tests/unit/core/test_projects_config.py`)** ŌĆö 3 new cases for `save_config`:
  1. Round-trips a `DaemonConfig` through `save_config` ŌåÆ `load_config`
  2. Doesn't leak `*.tmp` on success
  3. Overwrites existing file atomically (stale content replaced)

All 16 new tests in-memory, run in <1s. Full unit suite (1284 tests) green.

## Invocation safety note

The ship pipeline does NOT exercise `ctx registry prune --yes` against the developer's real `~/.lvdcp/config.yaml`. That's a user-owned, interactive decision ŌĆö the command ships, the destructive invocation is the user's call from their own terminal. The tests cover the mutation path using `tmp_path` fixtures only.

## Follow-ups explicitly NOT done here

- `ctx registry clean` ŌĆö on-disk cleanup of orphaned `.context/` directories for pruned roots
- `ctx registry add <path>` / `rm <path>` ŌĆö single-entry mutation commands
- `ctx registry export` / `import` ŌĆö backup/restore beyond the auto-sidecar
- Exposing any of this as an MCP tool ŌĆö destructive ops stay CLI-only
