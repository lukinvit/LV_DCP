# v0.8.60 — `ctx timeline disable --json` closes the timeline-flag mutation surface

**Date:** 2026-04-25
**Tag:** `v0.8.60`
**Theme:** CLI scriptability rollout — closes the timeline-flag mutation surface

## Summary

Adds `--json` to `ctx timeline disable`, closing the timeline-flag
mutation surface end-to-end. Reuses the v0.8.59 `_flag_to_json` helper
and `_TIMELINE_FLAG_JSON_KEYS` frozenset — one helper, two surfaces, one
schema lock. The `enabled` field round-trips as `false` post-mutation
so a script can `jq -e '.enabled == false'` to confirm the state
transition without a follow-up `status --json` call.

This **closes** the timeline-flag operator surface that v0.8.59 opened:

- **Read side**: `ctx timeline status --json` — single object with
  `{enabled, total_events, last_scan_commit_sha, ...}`
- **Enable side** (v0.8.59): `ctx timeline enable --json` — single
  object with `enabled: true` post-mutation
- **Disable side** (this ship): `ctx timeline disable --json` — single
  object with `enabled: false` post-mutation

The cross-surface span (v0.8.59 + v0.8.60) parallels the v0.8.57
(`memory accept --json`) + v0.8.58 (`memory reject --json`) memory-review
operator-surface closure: same single-object shape, same shared helper
+ frozenset, same idempotent + cross-state-flip discipline.

## Schema

`ctx timeline disable --json` returns a single JSON object — three keys,
schema-locked via `_TIMELINE_FLAG_JSON_KEYS` (shared with `enable`):

```json
{
  "project_root": "/abs/path/to/project",
  "enabled": false,
  "flag_path": "/abs/path/to/project/.context/timeline.enabled"
}
```

The `enabled` field round-trips as `false` post-mutation so a script can
`jq -e '.enabled == false'` to confirm the state landed without a
follow-up `status --json` call. `flag_path` surfaces the actual on-disk
artifact the daemon / scanner reads, so a debugging script can
`cat $(jq -r .flag_path)` to inspect the raw `off` content.

## Idempotent re-disable semantic

`disable_cmd` has no state-machine guard — calling it on an
already-disabled project is a no-op-on-state-but-write-on-disk that
returns the same payload. Locked by
`test_timeline_disable_json_idempotent_re_disable` which asserts both
call payloads are bytewise identical.

This mirrors v0.8.59's `test_timeline_enable_json_idempotent_re_enable`:
proves a script can safely re-run `disable --json` without branching on
"was it already off" — same render-switch-not-semantic-change
discipline.

## Cross-state flip semantic

Calling `enable` then `disable --json` flips the flag cleanly with a
**stable** `flag_path` (same on-disk marker rewritten in place, no
fresh file). Locked by `test_timeline_disable_json_overrides_enabled_state`,
mirroring v0.8.59's `test_timeline_enable_json_overrides_disabled_state`
cross-state flip — proves no duplicate file is created on the state
transition, just an in-place rewrite of the existing
`.context/timeline.enabled` marker.

Together with the v0.8.59 enable counterpart this **locks the
bidirectional flip semantic**: `enable` → `disable` → `enable`
round-trips through the same on-disk artifact, no fresh writes on
either edge. A script that toggles capture state across a fleet of
projects will not leave behind orphaned marker files.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/symbol_timeline/store.py` —
  `_write_flag` signature unchanged.
- **Helper reused** — `_flag_to_json` (v0.8.59) called with
  `enabled=False`. Zero new helpers introduced; the shape is locked at
  the v0.8.59 declaration site.
- **Schema lock reused** — `_TIMELINE_FLAG_JSON_KEYS` frozenset
  (v0.8.59) covers both surfaces. Adding a key requires bumping the
  frozenset + `_flag_to_json` together; both surfaces inherit the
  change atomically.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  single-line `timeline: disabled for <root>` footer. Pre-existing
  pipes see no diff. Locked by
  `test_timeline_disable_text_output_unchanged`.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Why this matters

Before this ship: every timeline-flag automation that wanted to
programmatically disable timeline capture (a per-project teardown
script that bulk-disables timeline before archiving a project, an IDE
"disable timeline" toggle that needs to confirm the transition before
re-rendering its UI, a daemon-shutdown step that ensures a project's
capture is off before unmounting its volume) had to grep the text
footer (`timeline: disabled for <root>`) or rely on the exit code
alone — losing the round-tripped `flag_path` that confirms which
on-disk marker was just written.

After this ship: `ctx timeline disable --json` returns a structured
payload directly. Same flag-write semantic, same on-disk artifact,
zero text parsing. And — critically — the **enable + disable** pair
share the same shape, so a single `jq` filter handles both directions:

```bash
ctx timeline enable  --project "$root" --json | jq -e '.enabled == true'
ctx timeline disable --project "$root" --json | jq -e '.enabled == false'
```

This unblocks workflows like:

- **Bulk disable across registered projects**: `ctx registry ls --json | jq -r '.[] | select(.scanned == true) | .root' | xargs -I{} ctx timeline disable --project {} --json | jq -s 'map({project: .project_root, enabled: .enabled})'` — bulk-disable timeline capture before archival, collect the post-mutation state for an audit log
- **IDE / dashboard toggle confirmation**: `ctx timeline disable --project "$root" --json | jq -e '.enabled == false'` — assert the transition landed before re-rendering the UI; exit 1 from `jq -e` if the toggle silently failed
- **Daemon-shutdown ensure-off**: `ctx timeline status --project "$root" --json | jq -e '.enabled == false' || ctx timeline disable --project "$root" --json` — idempotent ensure-disabled with no spurious writes; the `--json` payload from `disable` confirms the resulting state in one call
- **Teardown audit trail**: `ctx timeline disable --project "$root" --json | jq '{when: now, project: .project_root, flag: .flag_path, decision: "disabled"}' >> teardown_audit.jsonl` — append-only audit log of timeline-disable events with flag-path provenance preserved
- **Bidirectional toggle**: `for op in disable enable; do ctx timeline $op --project "$root" --json | jq -c .; done` — a single shell loop drives both transitions through the same parser, no per-direction branching
