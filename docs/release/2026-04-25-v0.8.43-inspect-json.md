# v0.8.43 — `ctx inspect --json` for scriptable index stats

**Date:** 2026-04-25
**Previous:** [v0.8.42 — `ctx scan --json` for scriptable scan results](2026-04-25-v0.8.42-scan-json.md)
**Type:** UX (CLI scriptability — symmetric with v0.8.42 scan; closes the read-side gap)
**Risk:** Low — additive flag; existing text path byte-identical; only one new CLI flag; no library signature changes; no on-disk artifact changes.

## Problem

v0.8.42 brought scan results into the scriptable surface (`ctx scan --json`), but the read-side inspect command stayed text-only:

```
project: LV_DCP
files: 412
  python: 318
  yaml: 24
  ...
symbols: 4127
  function: 2891
  class: 412
  ...
relations: 1837
  calls: 1102
  imports: 521
  ...
```

That's fine for humans, useless for any monitoring or dashboard that wants to track index health programmatically:

- A dashboard widget plotting "languages by file count" over time needs the count distribution as JSON, not regex-extracted from prose.
- A CI guard that asserts "after the refactor, no Go files were accidentally promoted to the index" needs `jq '.language_counts.go // 0'`, not a brittle `grep "go: " | awk`.
- A multi-project sweep — `find ~/dev -maxdepth 2 -name '.context' -exec ctx inspect {}/.. --json` — wants per-project payloads it can fold into a single workspace report.
- A `ctx watch` cron that checks for "did the index gain >100 new symbols since yesterday's snapshot" needs `symbols` as an int, not a parsed string.

Same gap that v0.8.38 closed for `prune`, v0.8.40 for `restore`, v0.8.42 for `scan`. Time to close it for `inspect`.

## Solution

A `--json` flag on `ctx inspect` that emits a schema-locked payload:

```bash
$ ctx inspect /path --json
{
  "path": "/path",
  "files": 412,
  "language_counts": {
    "python": 318,
    "yaml": 24,
    ...
  },
  "symbols": 4127,
  "symbol_type_counts": {
    "function": 2891,
    "class": 412,
    ...
  },
  "relations": 1837,
  "relation_type_counts": {
    "calls": 1102,
    "imports": 521,
    ...
  }
}
```

### Schema notes

- **`path`**: resolved scan target (absolute). Useful for `find -exec` workflows binding multiple inspects back to their roots — without this, scripts have to track the path themselves.
- **`*_counts`**: dicts insertion-ordered by descending count (mirrors `Counter.most_common`). Scripts can do `jq '.language_counts | to_entries[0].key'` for the most-frequent language without an explicit sort. Same ordering as the human text view.

### Error path

Un-indexed project surfaces error on stderr + exit 1 in **both** text and JSON modes:

```bash
$ ctx inspect /not/a/scanned/project --json
$ echo $?
1
$ # stderr: "Project not indexed at /not/a/scanned/project — run `ctx scan` first"
$ # stdout: empty
```

Discipline: `--json` does **not** swallow the error into stdout JSON (`{"error": "..."}`). Scripts gate on exit code (`set -e`) and read the human message from stderr. Splitting that contract — sometimes JSON, sometimes prose, both on stdout — would force every consumer to parse-then-check-keys instead of just relying on the gate.

## Library / CLI layering

A new `InspectPayload` TypedDict gives mypy strict catch on field-name typos in the renderer and gives JSON consumers a documented contract:

```python
class InspectPayload(TypedDict):
    path: str
    files: int
    language_counts: dict[str, int]
    symbols: int
    symbol_type_counts: dict[str, int]
    relations: int
    relation_type_counts: dict[str, int]
```

Adding a field to the TypedDict is the **single point** that surfaces it in both text and JSON outputs by design. No drift between the two views.

Two helpers split the responsibilities:

- `_collect_inspect_payload(idx, *, path) -> InspectPayload` — pure-data builder. Walks the index, builds the counters via `most_common()`, returns the typed dict.
- `_render_text(payload, *, name) -> str` — text renderer that walks the same payload. Single source of truth for row order and labels keeps text and JSON paths in lockstep.

`inspect()` signature also dropped its `typer.Argument` defaults in favor of plain `Path` + keyword-only `as_json: bool = False` — same direct-call ergonomics the scan command adopted in v0.8.42 (the inner function is called from `apps/cli/main.py::inspect`, not by Typer itself, so `OptionInfo` truthiness leaks would silently misbehave).

## New invariants

Five tests in `tests/integration/test_cli_inspect.py` (1 existing + 4 new):

- **`test_inspect_reports_stats`** *(existing)* — text path passes byte-identically.
- **`test_inspect_json_emits_well_formed_object`** — exact key set on the JSON payload (locked surface). `set(payload.keys()) == _INSPECT_JSON_KEYS`. `path` matches `resolved`. `files`, `symbols`, `relations` are ints. The three `*_counts` are dicts.
- **`test_inspect_json_suppresses_human_text_on_stdout`** — JSON-only output on stdout, no human prose or structlog leaks. `json.loads(result.stdout)` succeeds against the entire stdout — same guarantee v0.8.42 introduced via the structlog-to-stderr routing.
- **`test_inspect_json_counts_are_descending_ordered_dicts`** — `*_counts` dicts preserve insertion order = descending by count. Locks the `most_common()` semantic so a refactor that drops it breaks this test instead of silently reshuffling consumer dashboards.
- **`test_inspect_not_indexed_error_goes_to_stderr_with_exit_1`** — un-indexed project surfaces error on stderr + exit 1 in BOTH text and JSON modes. Exits 1, stdout is empty, stderr carries the message. No `{"error": ...}` leaks into stdout.

## Rejected alternatives

- **Include `relations` graph adjacency in the JSON output.** Different consumer pattern — `ctx pack` (and the `lvdcp_pack` MCP tool) already returns adjacency-aware payloads. `inspect` is the aggregate-counts surface; adding adjacency conflates two different consumer intents.
- **Schema-version the JSON output (`"_schema_version": 1`).** Premature ceremony, same v0.8.38 / v0.8.40 / v0.8.42 argument. Add when there's a real consumer that needs to gate on schema versions.
- **Expose `path` as the project name (basename) only.** The resolved absolute path is more useful for `find -exec` workflows binding multiple inspects back to their roots; the basename is trivially recoverable via `jq '.path | split("/") | last'`.
- **Make the error path emit `{"error": "...", "exit_code": 1}` JSON in `--json` mode.** Would split the contract — exit code is the script gate, stderr carries the human message, stdout is exclusively the success payload. Forcing JSON consumers to also branch on `if .error` defeats the gate.
- **Add a `--no-counts` flag to suppress per-type breakdowns for compactness.** Premature — `jq 'del(.language_counts, .symbol_type_counts, .relation_type_counts)'` does this without a flag, and the typical bandwidth cost is negligible (a few hundred bytes even for large repos).
- **Sort `*_counts` alphabetically by key for "stable" output.** Loses the most-frequent-first ordering that mirrors the human view; consumers that want alphabetic order can do `jq '.language_counts | to_entries | sort_by(.key) | from_entries'`.

## Scope locks

- Only one new CLI flag (`--json`).
- `ProjectIndex.iter_files` / `iter_symbols` / `iter_relations` library signatures **unchanged**.
- No new on-disk artifacts.
- No MCP surface — the MCP `lvdcp_inspect` tool already returns structured data; duplicating with a `--json` flag would split the source of truth.
- Text output **byte-identical** — every existing inspect test passes.
- No embedding / Qdrant / git / wiki behavior change.
- structlog routing introduced in v0.8.42 (`apps/cli/main.py`) carries forward unchanged — JSON consumers benefit automatically.

## Follow-ups explicitly NOT done here

- A `--counts-only` mode that drops `path`, `files`, `symbols`, `relations` and returns just the three `*_counts` dicts. `jq` slicing is sufficient for now; flag clutter not warranted without a forcing function.
- Time-series metadata in the payload (`scan_completed_at`, `index_built_at`). Useful for dashboard "last scanned" widgets but better surfaced via `ctx scan --json` on the actual scan event rather than recomputed at inspect time.
- Per-language symbol-type breakdown (`{"python": {"function": 200, "class": 30}, "go": {...}}`). Useful for cross-language refactor reports but doubles the payload size and isn't a current friction.
- A `--diff <other-path>` mode to compute deltas between two project indexes. Different consumer pattern (compare two snapshots), separate design.
- A README badge tracking the JSON-scriptable surface coverage (e.g. "JSON-scriptable: scan ✓, inspect ✓, prune ✓, restore ✓, ls ✓"). Premature — release notes carry this discoverability and a badge would need maintenance every time another command grows `--json`.
