# v0.8.38 — `ctx registry prune --json` for scriptable cleanup

**Date:** 2026-04-25
**Previous:** [v0.8.37 — `ctx registry ls --missing` for tombstone audit](2026-04-25-v0.8.37-ls-missing.md)
**Type:** Feature / UX (registry hygiene — scripting surface)
**Risk:** Low — new optional flag; default behavior unchanged; pure read of an existing dataclass.

## Problem

`ctx registry ls` has had `--json` since v0.8.32, but `ctx registry prune` has been text-only across v0.8.33 → v0.8.37. That's an asymmetry CI hygiene scripts trip over: a nightly job that wants to detect registry drift and warn (`prune --kind transient | grep "would remove"`) is brittle to copy changes, and there's no clean way to count or process candidate roots without `awk`/`sed` parsing of human-readable output.

The `PruneResult` dataclass already has every field a script needs — `kept`, `removed`, `applied`, `backup_path`, `config_path`. It's been computed all along; we just weren't exposing it.

## Solution

A new `--json` flag on `ctx registry prune` that emits the `PruneResult` as a JSON object instead of human-readable text. Composes with every existing flag:

- `prune --json` (dry-run, default) → preview as JSON
- `prune --json --yes` (apply) → live result with `applied=true` and `backup_path` populated
- `prune --json --missing` → tombstone preview as JSON
- `prune --json --kind real --older-than 60` → custom-policy preview as JSON

```bash
# preview as data
$ ctx registry prune --missing --json
{
  "kept": ["/Users/v/projects/AliveProject"],
  "removed": ["/tmp/scratch/worktrees/deleted-today"],
  "applied": false,
  "backup_path": null,
  "config_path": "/Users/v/.lvdcp/config.yaml"
}

# count what would be cleaned up
$ ctx registry prune --missing --json | jq '.removed | length'
3

# CI hygiene check: fail if there are tombstones
$ test "$(ctx registry prune --missing --json | jq '.removed | length')" -eq 0

# apply and capture the backup path for paranoid undo
$ BACKUP=$(ctx registry prune --missing --yes --json | jq -r '.backup_path')
$ cp "$BACKUP" "/safe/place/$(date +%F).bak"
```

CLI layer: `apps/cli/commands/registry_cmd.py::prune_cmd` adds `as_json: bool` flag. When set, the function builds the dict via a new `_prune_result_to_json` helper (string-converts `Path` fields for JSON serialization) and returns immediately — no hint text, no header lines, no "nothing to do" message. The text output path is unchanged.

Library layer: **untouched**. `prune_stale` already returns the structured result; the JSON layer is purely a CLI rendering concern.

## Schema

The JSON object is a 1:1 mirror of the `PruneResult` dataclass. No flag echo, no derived fields, no metadata. That gives scripts the smallest stable surface to bind against.

```jsonc
{
  "kept": ["abs/path", ...],          // post-mutation registry (or current, for dry-run)
  "removed": ["abs/path", ...],       // what was/would be removed
  "applied": false,                   // true after --yes mutation, false otherwise
  "backup_path": null | "abs/path",   // populated only on actual mutation
  "config_path": "abs/path"           // always populated
}
```

## New invariants

Five new CLI-layer tests lock the contract:

- **`test_registry_prune_json_dry_run_shape`** — exact key set `{kept, removed, applied, backup_path, config_path}`, `applied=false`, `backup_path=null`, candidate root present in `removed`, config and `*.bak` untouched.
- **`test_registry_prune_json_yes_applies_and_reports_backup`** — `applied=true`, `backup_path` populated and equal to `<config>.bak`, `kept` reflects post-mutation YAML state.
- **`test_registry_prune_json_nothing_to_do_returns_empty_removed`** — when no candidates, still emits well-formed JSON with `removed=[]` and `applied=false` (no special-case empty stdout that scripts have to guard against).
- **`test_registry_prune_json_missing_dry_run_lists_tombstones`** — `--missing` predicate composes with `--json`; tombstone path appears in `removed`, config untouched.
- **`test_registry_prune_json_suppresses_human_hint_text`** — JSON mode emits **only** the JSON object: no `dry-run` hint, no `Pass --yes`, no `REMOVED` header, no `nothing to do`. `json.loads(stdout)` succeeds on a single object.

## Rejected alternatives

- **Echo invocation parameters in the JSON (`{"predicate": "missing", "older_than": 30, ...}`).** Brittle — changes the schema every time we add a flag. Scripts that want to know which mode was used can pass `--missing` themselves and remember.
- **Output NDJSON (one row per kept/removed entry).** Would force consumers to handle two record types. The current single-object shape is what `jq` users actually want — `.removed[]` already gives streaming row access.
- **Add `prune --plan` separately from `--json`.** Conflates output format with semantics. `--json` is purely a render switch; the dry-run vs apply axis is `--yes`. Two orthogonal flags are simpler than one combined `--plan` mode.
- **Make `--json` imply `--yes` (because "scripts want to apply").** No — scripts want **both** preview and apply, and the dry-run preview is the more common scripting case (CI checks that warn but don't mutate). Composing with `--yes` per-call is the right ergonomic.
- **Pretty-print as YAML instead of JSON.** YAML is a worse scripting target (whitespace-sensitive parsing, no canonical streaming parser in shell). The existing config file is YAML; the result rendering doesn't need to match.

## Scope locks

- Library layer untouched — `PruneResult` shape is unchanged; the JSON serializer lives entirely in the CLI module so the dataclass stays free of presentation concerns.
- No registry schema change.
- No public signature change on `prune_stale` / `plan_prune` / `audit_registry` / `is_missing` / `is_stale`.
- Default behavior of `ctx registry prune` (without `--json`) is unchanged — every existing text-output assertion still holds.
- No new on-disk artifacts — JSON goes to stdout, nothing else writes anywhere.
- No MCP surface — destructive operations stay CLI-only (same lock as v0.8.33).

## Follow-ups explicitly NOT done here

- Symmetric `--json` flag with stable schema versioning (`{"schema_version": 1, ...}`). The schema is a thin mirror of `PruneResult` and has been stable across 4 releases — premature versioning would add ceremony without solving an actual versioning problem. Add when a backward-incompatible field rename actually arrives.
- A `ctx registry restore` command that swaps `~/.lvdcp/config.yaml.bak` back into place. Would round out the prune undo story (the `*.bak` is currently a manual `mv`); useful but not blocking — the manual one-liner is documented in v0.8.33's release note.
- Coloring `removed` paths red in the human-readable text output. Same deferral as in v0.8.37 — global TTY/`NO_COLOR` policy decision, not a one-flag drop-in.
