# v0.8.24 ‚Äî Kotlin `INHERITS` edges (single-colon heritage)

**Date:** 2026-04-24
**Scope:** single-language deepening; no new dependencies; no new route; no config changes.

## Why

v0.8.23 closed the inheritance-edge gap for Java. Kotlin was shipped in v0.8.21 with the same explicit v1 scope-lock ‚Äî "no inheritance edges yet" ‚Äî and the v0.8.23 release notes explicitly named Kotlin `INHERITS` as the next plausible increment. This release delivers it.

Kotlin is a natural next step after Java (not Swift) because the two JVM parsers share the most behaviour: Maven/Gradle source roots, dotted package paths, same `TreeSitterParser` hook shape. Porting v0.8.23's Java pattern to Kotlin only requires swapping one AST layer ‚Äî the heritage-clause shape ‚Äî and the rest of the scaffolding (scope-aware walker, `src_ref = type_fq`, `INHERITS` relation type, the "`src_ref` must resolve to a Symbol" invariant test) reapplies verbatim.

Swift follows next as a separate increment; its grammar (`inheritance_specifier` under a `class_declaration` / `protocol_declaration`) is different enough that merging it here would bloat the change.

## Design

`KotlinParser` now overrides `parse()` to call the base `TreeSitterParser.parse()` first, then walks the CST a second time with a scope-aware stack to emit `Relation(relation_type=INHERITS)` edges for every heritage clause it finds. This is a direct structural clone of `JavaParser.parse()` ‚Äî the only differences are the set of type-declaration nodes and the heritage-extraction helpers, both encapsulated in helpers that Swift can reuse when its own port lands.

### Heritage clause shape (single `:`, not two keywords)

Unlike Java's `extends` / `implements` split, Kotlin uses one colon and one production (`delegation_specifiers`) for every base type. Each entry inside is a `delegation_specifier` wrapping one of three shapes:

| Shape | Example | Notes |
| --- | --- | --- |
| `user_type` (bare) | `class Cat : Feline` | Interface conformance |
| `constructor_invocation` ‚Üí `user_type` | `class Dog : Animal()` | Class extension (parentheses mandatory) |
| `explicit_delegation` ‚Üí `user_type` | `class Impl(x) : Iface by x` | Kotlin-specific "implements by delegation" |

All three collapse onto a single `RelationType.INHERITS` edge. Kotlin itself has no syntactic class-vs-interface distinction at the heritage position (the declaration keyword carries it), so the graph consumer cannot distinguish them anyway ‚Äî and keeping one edge type mirrors the Java design from v0.8.23.

**Explicit delegation is an intentional is-a edge.** When `class Impl(x) : Iface by x` is encoded as `Impl INHERITS Iface`, impact-analysis can answer "what subclasses conform to `Iface`?" with Impl included. The delegation mechanism is a Kotlin-specific implementation detail ‚Äî for graph purposes it's still "is-a".

### Type-name extraction from `user_type`

Kotlin represents a scoped base type as a flat sequence of `identifier` and `.` tokens under `user_type`, not as a nested `scoped_type_identifier` node (Java's shape). Generics appear as a trailing `type_arguments` child that must be ignored:

| Node layout | Example | `dst_ref` |
| --- | --- | --- |
| `user_type` with one `identifier` | `Feline` | `"Feline"` |
| `user_type` with `identifier . identifier . identifier` | `pkg.Outer.Inner` | `"pkg.Outer.Inner"` |
| `user_type` with `identifier` + `type_arguments` | `Container<String>` | `"Container"` (args stripped) |
| Nested type_arg ‚Üí deeper `user_type` | `Comparable<Box<T>>` | `"Comparable"` (base only) |

Implementation: `_user_type_name` walks only direct `identifier` children and joins them with `.`. `type_arguments` is simply never iterated into, so generics are stripped without special-casing. The `test_generic_type_arguments_stripped` invariant asserts `String` and `Int` never surface as bases ‚Äî the same safety test v0.8.23 shipped for Java.

### Scope-aware source_ref derivation (identical to Java)

The walker tracks an `enclosing_fq` alongside each CST node on its work stack. When entering a `class_declaration` or `object_declaration`:

1. Computes `type_fq = f"{enclosing_fq}.{name}"` ‚Äî matching the base walker's `parent_fq.name` convention
2. Emits every heritage edge with `src_ref=type_fq`
3. Propagates `type_fq` as the enclosing scope for the node's children

`class Outer { class Inner : Base {} }` therefore produces `<module>.Outer.Inner INHERITS Base`. The locked invariant `test_inherits_src_ref_matches_symbol_fq` asserts every `INHERITS.src_ref` resolves to an actual Symbol in the same file ‚Äî any divergence between the symbol walker's scope logic and the heritage walker's scope logic is caught by that test.

### What is intentionally excluded

**`companion_object` heritage.** `class K { companion object : Holder }` has a heritage clause, but the `companion_object` node has no `name` field and the base walker does not emit a Symbol for it. Emitting an edge with a companion as source would violate the `src_ref ‚Üí symbol.fq_name` invariant. The walker therefore excludes `companion_object` from `_TYPE_DECLARATION_NODES`. Graph queries that want "which companions inherit from Holder?" are out of v1 scope; if a consumer surfaces that need, a follow-up increment can make companion objects first-class symbols.

**`annotation class`.** These are still `class_declaration` in the grammar (`annotation` lives under `modifiers`), but Kotlin annotation classes cannot legally carry delegation specifiers. They produce zero edges naturally, with no explicit filter needed. `test_annotation_class_never_source` locks this as a positive regression test.

### INHERITS vs. IMPLEMENTS: still one edge type

Same decision as v0.8.23 Java: interface conformance, class extension, and explicit delegation all collapse onto `RelationType.INHERITS`. Downstream consumers (Obsidian wiki sync, context pack enrichment, dashboard graph rendering) treat "is-a" as one relation. A future `IMPLEMENTS` sub-type would require a `RelationType` enum addition and a coordinated downstream audit ‚Äî no concrete consumer driving it yet.

## Known limitations (explicit scope locks)

- Only Kotlin closes in this release. Swift (`class Foo: Base, P`, `struct S: P`, `protocol P: Q`) uses a different AST (`inheritance_specifier` under `class_declaration` / `protocol_declaration`) and is a separate sub-increment.
- `INHERITS` collapses class extension + interface conformance + explicit delegation. A future `IMPLEMENTS` / `DELEGATES_TO` sub-type would require a `RelationType` enum addition.
- `dst_ref` is a bare type name (`Base`, `pkg.Outer.Inner`), not a fully-qualified symbol. Kotlin lets you refer to `List` after `import kotlin.collections.List`, so `: List<String>` emits `dst_ref="List"`, not `"kotlin.collections.List"`. Import-based resolution is a separate graph-enrichment pass.
- Companion objects are excluded per the fq-invariant (see Design above). Explicit delegation via a companion (`class K { companion object : Holder }`) produces zero edges.
- No `SAME_FILE_CALLS` edges ‚Äî same gap as Java / Swift / Go / Rust; the Python parser's call-graph remains the only one generalised to method invocations.
- Type aliases (`typealias StringList = List<String>`) are not tracked as inheritance ‚Äî they don't surface as symbols today.
