# v0.8.67 — Obsidian wikilinks: cross-references between Wiki/ and Modules/

**Date:** 2026-04-25
**Tag:** `v0.8.67`
**Theme:** Bidirectional graph navigation in Obsidian

## What ships

v0.8.66 mirrored the per-project wiki into the vault but left both halves
of the graph (the LLM-generated `Wiki/` tree and the auto-generated
`Modules/` tree) as **disconnected islands**. The mirrored articles still
used portable markdown links (`[text](modules/foo.md)`) which Obsidian
renders as inert spans with no graph edges, no backlinks, no preview on
hover. The auto-generated `Modules/<name>.md` and `Home.md` pages had no
mention of the wiki at all.

This ship turns those two trees into a navigable graph by:

1. **Rewriting markdown links to wikilinks** when mirroring `INDEX.md`,
   `architecture.md`, and every `modules/*.md` from `.context/wiki/` into
   the vault. The source `.context/wiki/` keeps its portable markdown
   form (still read by `lvdcp_pack` and any non-Obsidian consumer).
2. **Appending a `## See also` footer** to every mirrored article that
   links back to `Projects/<name>/Home`, `Projects/<name>/Wiki/INDEX`,
   and (for `<short>-*.md` articles) the matching auto-generated
   `Projects/<name>/Modules/<short>` stats page.
3. **Prepending a navigation blockquote** to the mirrored `INDEX.md` so
   the wiki landing page links out to Home / Modules / Recent Changes
   without burying the original H1.
4. **Adding cross-links from `Modules/<name>.md` and `Home.md`** to the
   wiki: every Modules page now has a `## Wiki articles` section
   pointing at the wiki INDEX with the slug-prefix hint
   (e.g. `apps-*`), and Home's Navigation list has a new entry for
   `[[Wiki/INDEX|Wiki index]]`.

End result: clicking any module name in the auto-generated tree reaches
the prose articles in one hop; clicking any wiki article reaches the
structural stats and the project home in one hop; Obsidian's graph view
finally shows the wiki and the auto-generated tree as one connected
component instead of two disjoint clouds.

## Converter contract (pinned by `test_wikilinks.py`)

The converter is deliberately conservative: false negatives (leaving a
link as plain markdown) are far cheaper than false positives (rewriting
an external link as an unresolvable wikilink). Each rule below is a
checked test case.

| Input | Output |
|---|---|
| `[apps-cli](modules/apps-cli.md)` | `[[modules/apps-cli\|apps-cli]]` |
| `[foo](modules/foo)` (no extension) | `[[modules/foo\|foo]]` |
| `[Foo](modules/Foo.MD)` | `[[modules/Foo\|Foo]]` (case-insensitive `.md` strip) |
| `[the rules](modules/constitution.md#rules)` | `[[modules/constitution#rules\|the rules]]` |
| `[karpathy](https://karpathy.ai/wiki)` | unchanged (external) |
| `[team](mailto:noreply@example.com)` | unchanged (scheme link) |
| `![diagram](diagrams/arch.png)` | unchanged (image syntax) |
| `[intro](#introduction)` | unchanged (in-page anchor) |
| `[click](modules/foo.md)` *inside* ` ``` `…` ``` ` *or* `~~~`…`~~~` | unchanged (code fence body) |
| `[a](modules/a.md) and [b](modules/b.md)` | both rewritten on the same line |

Idempotent: `convert(convert(x)) == convert(x)` because `[[wikilinks]]`
don't match `[text](url)`. Trailing newline preserved verbatim.

## Slug-format mismatch and the `_short_module_for` fallback

The wiki uses 2-segment dash-joined slugs (`apps-cli`, `libs-wiki`)
because the underlying source path was `apps/cli` / `libs/wiki` (see
`libs.wiki.state.wiki_filename`). The publisher's auto-generated
`Modules/` tree slugs at the **first segment only** (`apps`, `libs`).

There is no 1:1 mapping. `_short_module_for(stem)` picks the first
dash-separated chunk so the footer cross-link can target a real
auto-generated page (`apps-cli.md` → `Modules/apps`). Slugs without a
dash (`README.md`) get `None` back and the footer omits the Modules
link entirely rather than pointing at a nonsense target.

The Modules-side cross-link uses the same first-segment heuristic in
reverse: `Modules/apps.md` links to the wiki INDEX with the hint
"articles starting with `apps-*`" so the user lands somewhere useful
even when there are 5+ wiki articles for that module.

## Why this matters

Before this ship: the user opened the vault, found the auto-generated
Modules tree, found the LLM-generated Wiki tree, and had no clickable
path between them. Two trees, no graph edges. Searching for a module's
prose article meant manually navigating to `Wiki/INDEX.md` and reading
the table of contents.

After this ship: every Obsidian backlink pane on a wiki article shows
the corresponding `Modules/<short>.md` and the project `Home.md`. Every
backlink pane on a Modules page shows the wiki INDEX. The graph view
shows the wiki and the auto-generated tree as one connected component.
The user's request — "в обсидиан и вики должны быть перекрестные
ссылки" — is satisfied at every junction.

The source `.context/wiki/` directory keeps its portable markdown form,
so `lvdcp_pack` enrichment, MCP tools, and any future non-Obsidian
consumer continue to read the same articles unchanged. Wikilinks live
only in the vault copy.

## Recovery / migration

- **No schema migration, no config change.** Existing vault contents
  get rewritten on the next sync.
- **Forcing a re-mirror right now:** `ctx obsidian sync-all --stale-hours 0`
  rewrites every project's vault `Wiki/` folder with the new wikilink
  conversion + cross-link footers + index header.
- **Wiki articles regenerated since v0.8.66 don't need another rebuild**
  — the conversion happens at mirror time on the vault copy, not at
  generation time on `.context/wiki/`. Nothing in the source needs to
  change.
