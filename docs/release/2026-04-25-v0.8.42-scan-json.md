# v0.8.42 — `ctx scan --json` for scriptable scan results

**Date:** 2026-04-25
**Previous:** [v0.8.41 — `ctx registry ls` surfaces backup-availability footer](2026-04-25-v0.8.41-ls-backup-footer.md)
**Type:** UX (CLI scriptability — bring `scan` in line with v0.8.38 / v0.8.40 `--json` discipline)
**Risk:** Low — additive flag; existing text path byte-identical; only one new CLI flag; no library signature changes; no on-disk artifact changes.

## Problem

The full / incremental scan surface has been the foundation of every other LV_DCP feature since day one (parsing, indexing, embedding, timeline, wiki dirty-tracking, history events all hang off `scan_project`). Its CLI output, however, has stayed text-only:

```
scanned 42 files in /path (3 reparsed, 0 stale removed), 187 symbols, 12 reparsed / 64 total relations in 0.31s
```

That's fine for humans, useless for scripts. Concrete drivers:

- A cron-driven daily full re-scan that posts a Slack alert when `files_reparsed` spikes (indicating a bulk refactor or a corrupted cache rebuild).
- A CI guard that asserts a bulk refactor reparses ≥N files (catches "did the refactor actually touch the files we expected?").
- `find -exec ctx scan --json` over a workspace bundling per-project `ScanResult`s into a single dashboard payload.
- A pre-commit hook that runs `ctx scan --json` and exits non-zero if the scan elapsed > a threshold (catches scan regressions).

Up until now the only path was a brittle regex over the human prose. Same gap that v0.8.38 closed for `prune` and v0.8.40 for `restore`. Time to close it for `scan`, the most-invoked command in the surface.

## Solution

A `--json` flag on `ctx scan` that emits a 1:1 mirror of the `ScanResult` dataclass plus three CLI-side fields the library doesn't know about:

```bash
$ ctx scan /path --json
{
  "path": "/path",
  "mode": "incremental",
  "files_scanned": 42,
  "files_reparsed": 3,
  "stale_files_removed": 0,
  "symbols_extracted": 187,
  "relations_reparsed": 12,
  "relations_cached": 64,
  "elapsed_seconds": 0.31,
  "wiki_dirty_count": 0,
  "qdrant_warnings": []
}
```

The three CLI-side fields:

- **`path`**: the resolved scan target (absolute). Scripts often want to bind multiple scan results back to their roots — `find -exec` workflows otherwise lose this association.
- **`mode`**: `"full"` or `"incremental"`. Echoed so a script can defensively assert the run actually used the mode it asked for (e.g., a daily `--full --json | jq -e '.mode == "full"'`).
- **`qdrant_warnings`**: `list[str]`, empty list when embedding config is healthy. Same ergonomic discipline as v0.8.38 — `payload.qdrant_warnings | length` works without a None-guard.

Pure data — no hints, no headers, no advisory text. Side effects (auto-register in `~/.lvdcp/config.yaml`, `CLAUDE.md` discipline section update) still run on `--json` because they're part of "scan" semantics, not output rendering.

## Bundled discovery: structlog → stderr

While shipping the JSON path, integration tests revealed structlog defaulted to `PrintLoggerFactory`, which writes to **stdout**. Timeline / wiki / embedding diagnostics (e.g. `timeline.scan_bracket added=1 commit_sha=...`) interleaved with the `typer.echo` JSON payload and broke `jq` parsing:

```
2026-04-25 03:34:09 [info     ] timeline.scan_bracket          added=1 ...
{
  "path": "/path",
  ...
}
```

Fix is a one-line `structlog.configure(logger_factory=WriteLoggerFactory(file=sys.stderr))` at the CLI entry point in `apps/cli/main.py`, applied **before** any subcommand module imports because structlog caches the factory at `get_logger()` time. Logs now go to stderr uniformly across every CLI subcommand (not just `--json` ones), which is the right Unix discipline anyway — JSON consumers were the forcing function, but the fix benefits every CLI invocation that pipes stdout downstream.

## Library / CLI layering

Same internal refactor as v0.8.38 prune / v0.8.40 restore:

- `_collect_qdrant_warnings()` — extracted helper. The warning-collection logic was already inline in `scan()`; now it's a named function reused by both text and JSON paths.
- `_scan_result_to_json(result, *, path, mode, qdrant_warnings)` — pure-data builder. Schema-locked dict that's `json.dumps`-able with `indent=2`.

`scan()` signature also dropped its `typer.Argument` / `typer.Option` defaults in favor of plain `bool`:

```python
def scan(
    path: Path,
    *,
    full: bool = False,
    as_json: bool = False,
) -> None:
```

Direct callers (other CLI commands invoking it) and tests that omit kwargs no longer trip on `typer.OptionInfo` truthiness ambiguity. The Typer-decorated wrapper lives in `apps/cli/main.py::scan` and re-declares the validation surface — the inner function is called directly from there, not by Typer itself.

## New invariants

Four CLI tests in `tests/integration/test_cli_scan.py` (all 4 existing scan tests continue to pass byte-identically — the text path is untouched):

- **`test_scan_json_emits_well_formed_object`** — exact key set on the JSON payload (locked surface). `set(payload.keys()) == _SCAN_JSON_KEYS`. `path` matches `resolved`. `mode == "incremental"` by default. `files_scanned >= 1` for the sample repo. `qdrant_warnings` is a list. `elapsed_seconds` is numeric and ≥ 0.
- **`test_scan_json_suppresses_human_text_on_stdout`** — JSON-only output on stdout, no human prose leaks. `json.loads(result.stdout)` succeeds against the entire stdout — confirms no text leak. Spot-checks for the `scanned ` prose marker absent.
- **`test_scan_json_full_mode_echoes_full`** — `--full --json` reports `mode=full`, letting scripts assert intent. The motivating case is a daily `--full --json | jq -e '.mode == "full"'`.
- **`test_scan_json_qdrant_warnings_is_empty_list_when_disabled`** — `qdrant_warnings == []` (empty list, not `null`) when Qdrant is disabled. Empty-list contract is the ergonomic choice for `jq` consumers — `payload.qdrant_warnings | length` works without a None-guard.

## Rejected alternatives

- **Emit JSON only when piped to non-tty stdout.** Magic; `--json` is explicit. Scripts that wrap `ctx scan` in unusual environments (CI without a TTY, but with `stdout` going to a log file) deserve a deterministic flag, not heuristic detection.
- **Include the resolved `.context/` artifact paths in the JSON.** Scripts can derive them from `path` (`{path}/.context/cache.db`, etc.) — adds noise without an information delta.
- **Schema-version the JSON output (`"_schema_version": 1`).** Premature ceremony, same v0.8.38 / v0.8.40 argument. Add when there's a real consumer that needs to gate on schema versions.
- **Separate `--json-pretty` vs `--json-compact` flags.** Premature — `jq -c` handles compaction and the default 2-space indent reads fine in cron logs.
- **Exit non-zero when `qdrant_warnings` is non-empty.** Warnings are advisory, not errors — same v0.8.41 discipline of not promoting nudges to gates. Scripts that want to fail on Qdrant misconfiguration can do `jq -e '.qdrant_warnings | length == 0'` themselves.
- **Add `--quiet` to suppress text output without enabling JSON.** Solves a non-problem — `> /dev/null` already does this. `--json` is the only output-shape variant we need.

## Scope locks

- Only one new CLI flag (`--json`).
- `scan_project` library signature **unchanged** — still returns the same `ScanResult` dataclass; CLI layer composes JSON from the dataclass, library doesn't know about JSON.
- No new on-disk artifacts.
- No MCP surface — the MCP `lvdcp_scan` tool already returns structured data; duplicating with a `--json` flag would split the source of truth between CLI and MCP. Scripts that want to call MCP-style structured scan from a shell should use `ctx scan --json`; scripts that want it from inside Claude should use `lvdcp_scan` directly.
- Text output **unchanged byte-identically** — every existing scan test passes.
- No embedding / Qdrant behavior change — best-effort upsert still runs on incremental scans with `changed_for_embed` populated; the SSL / network failure modes are still caught at the `scan_project` level and surface as `qdrant_warnings` or silent debug logs.
- structlog routing change is at the CLI entry point only — library code that uses structlog (timeline, wiki, embedding) still gets a configured logger; the difference is just where it writes. No log message content change. No log level change. No filter change.

## Follow-ups explicitly NOT done here

- A `--json-lines` mode that emits one JSON object per processed file (for streaming consumers tailing a long full re-scan). Useful but premature — current `ScanResult` aggregate is the discoverable shape, per-file streaming is a separate consumer pattern.
- Surfacing `wiki_dirty_count` breakdown by module in the JSON (currently a flat int). Scripts that want the breakdown can call `ctx wiki status --json` after the scan — separation of concerns.
- Cross-CLI structlog log-format flag (e.g. `--log-format=json` for structured stderr logs that a daemon supervisor can ingest). Useful for the `apps/agent` daemon path but not the `ctx` CLI path; deferred until the daemon's launchd plist actually wires up such a consumer.
- `ctx watch --json` event stream over the daemon's pending-events socket. Different consumer pattern (push vs pull), separate design.
- A README badge for the JSON contract surface (e.g. "JSON-scriptable: scan, prune, restore, ls"). Premature — the release notes carry this discoverability and a badge would need maintenance every time another command grows `--json`.
