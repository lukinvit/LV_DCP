# v0.8.48 — `ctx wiki cross-project --json` for scriptable cross-project run summary

**Date:** 2026-04-25
**Previous:** [v0.8.47 — `ctx wiki lint --json` for scriptable wiki hygiene gating](2026-04-25-v0.8.47-wiki-lint-json.md)
**Type:** UX (CLI scriptability — completes the wiki subgroup read-side scriptability triad: `wiki status` ✓ v0.8.46, `wiki lint` ✓ v0.8.47, `wiki cross-project` ✓ this release)
**Risk:** Low — additive flag; existing text path byte-identical; only one new CLI flag; no library signature changes; no on-disk artifact changes; the file-write side effect of cross-project (global INDEX + pattern articles) is unchanged.

## Problem

`ctx wiki cross-project` aggregates wiki INDEXes across all registered projects, builds a global INDEX at `~/.lvdcp/wiki/INDEX.md`, and (when Claude CLI is available) detects cross-project patterns into `~/.lvdcp/wiki/patterns/`. Until now the CLI emitted only a single human-readable line:

```
Cross-project wiki generated: 3 pattern(s) written to /Users/me/.lvdcp/wiki
```

That's fine for humans, useless for automation that wants to act on the structured run result:

- A nightly cron that runs `ctx wiki cross-project` across CI workspaces and warns on regressions ("you have 8 registered projects but only 3 patterns were written — Claude CLI may have failed midway") needs `jq -e '.patterns_written < (.projects_registered / 2)'`, not a brittle `awk '{print $4}'` over the prose.
- A multi-config sweep — `for cfg in workspace_*/config.yaml; do ctx wiki cross-project --config "$cfg" --json; done | jq -s 'map({cfg: .config_path, patterns: .patterns_written})'` — wants per-run aggregation into a single workspace-level dashboard.
- A pre-deploy gate that asserts "the cross-project wiki is non-empty before publishing" needs `jq -e '.patterns_written > 0'` — currently impossible without parsing the prose count out of the single-line text output.
- A workspace-onboarding script that surfaces "you have N registered projects but the cross-project run found 0 patterns — check that each project has run `ctx wiki update` recently" needs the structured `projects_registered` field, which the text view doesn't even mention.

`generate_cross_project_wiki()` in `libs/wiki/cross_project.py` already returned the patterns count since the wiki feature shipped — and `list_projects()` in `libs/core/projects_config.py` is the canonical read for the registered-projects count. We just weren't composing them into a structured CLI payload. Same gap that v0.8.38 closed for `prune`, v0.8.40 for `restore`, v0.8.42 for `scan`, v0.8.43 for `inspect`, v0.8.44 for `memory list`, v0.8.45 for `timeline reconcile`, v0.8.46 for `wiki status`, v0.8.47 for `wiki lint`. Time to close it for `wiki cross-project` — the last read-side surface in the wiki subgroup.

## Solution

A `--json` flag on `ctx wiki cross-project` that emits a single JSON object summarising the run:

```bash
$ ctx wiki cross-project --config ~/.lvdcp/config.yaml --json
{
  "wiki_dir": "/Users/me/.lvdcp/wiki",
  "config_path": "/Users/me/.lvdcp/config.yaml",
  "projects_registered": 8,
  "patterns_written": 3
}
```

### Schema notes

- **Single object, not a bare array.** `wiki status` (v0.8.46) and `wiki lint` (v0.8.47) emit bare arrays because they're per-row reads (one entry per module / one entry per finding). `wiki cross-project` is a generate command with one result payload — a single run produces one output object. Schema-locked frozenset (`_WIKI_CROSS_PROJECT_JSON_KEYS`) still applies; the only difference is array-vs-object.
- **`wiki_dir` and `config_path` are stringified absolute paths** — consumers can `find -exec` / multi-config-sweep without re-resolving, and the round-trip `config_path` lets a script confirm the run actually used the config it intended (catches typos in the `--config` flag without echoing back the input flags).
- **`projects_registered` is the count from `list_projects(config_path)`** — total registered projects, not just those with INDEX.md present. The gap between `projects_registered` and `patterns_written` is the script-side ergonomic: `jq -e '.patterns_written < .projects_registered'` flags the case where Claude CLI didn't produce a pattern per project (which is normal — patterns aggregate across projects, not per-project — but the metric is useful for "is this run healthy?" gating). For the precise `projects_with_index_md` count, scripts can post-process by walking the registered roots (cheap, ~1 stat per entry).
- **`patterns_written` is a pass-through of `generate_cross_project_wiki()`'s return value** — same int the text view interpolates into the prose line. Zero is a valid healthy state (e.g. only one project registered, the prompt explicitly emits "## No patterns detected" and we count zero). Empty registry also returns `patterns_written=0`.

### Why a single object instead of `{"summary": {...}, "patterns": [...]}`

A naive expansion would be to also emit the per-pattern list (file names of patterns written under `~/.lvdcp/wiki/patterns/`). Rejected for this release because:

1. **Library signature change.** `generate_cross_project_wiki` returns `int`; expanding to also return the list of pattern names requires either a new dataclass result type or a breaking signature change. The CLI-only discipline (sister to v0.8.46 / v0.8.47 — no library changes when not needed) keeps the blast radius minimal.
2. **Pattern names are filesystem-recoverable.** A consumer that wants the list can `ls $(jq -r '.wiki_dir')/patterns/*.md` — no need to bake it into the payload.
3. **The summary-level fields are the higher-leverage gate.** `patterns_written` and `projects_registered` answer the most common questions ("did the run produce useful output? did it cover all my projects?"). Per-pattern inspection is a deeper-dive workflow that benefits from `wiki lint` or direct file reading.

If a real consumer surfaces wanting the per-pattern list, the natural follow-up is a separate `ctx wiki cross-project ls --json` read-side subcommand — cleaner than overloading the generate command's payload.

### Error path

`generate_cross_project_wiki` swallows `RuntimeError` from a missing Claude CLI and returns 0 (still writes the global INDEX) — no exit-1 path. So `--json` mode cleanly emits `patterns_written=0` and exits 0 in the Claude-CLI-missing case. Scripts gate on `patterns_written == 0 && projects_registered > 1` to detect "Claude CLI failed silently" — same shape as v0.8.42 `qdrant_warnings == []` for the embedding-disabled case.

The harder error path (`config_path` doesn't exist) is handled by `load_config` raising — the CLI surface lets the exception propagate (Typer renders the traceback; not a contract we promise), so `--json` mode never tries to wrap it. Same v0.8.42-v0.8.47 discipline of "exit code is the gate; structured payload is for the success path."

## Library / CLI layering

CLI-only change. No new library helpers, no schema modifications to `libs/wiki/cross_project.py` or `libs/core/projects_config.py`. `generate_cross_project_wiki` and `list_projects` already returned the values the JSON payload composes.

A small `_cross_project_to_json(*, wiki_dir, config_path, projects_registered, patterns_written) -> dict[str, object]` helper in `apps/cli/commands/wiki_cmd.py` builds the payload. Lives next to `cross_project` (the only consumer). Sister-helper to `_module_to_json` (v0.8.46) and `_issue_to_json` (v0.8.47) — same pattern of CLI-shell payload shaping, but emits a single object instead of a per-row dict (different shape because cross-project is a single-result generate command, not a per-row read).

## New invariants

Three new tests appended to `tests/unit/cli/test_wiki_cmd.py` (now 11 tests total in the file: 4 status + 4 lint + 3 cross-project):

- **`test_wiki_cross_project_text_output_unchanged`** — baseline: text path emits the legacy `Cross-project wiki generated: N pattern(s) written to ...` single-line format byte-identically. Locks the v0.8.42-v0.8.47 "text path untouched" promise. Asserts no JSON braces leak (`not result.output.lstrip().startswith("{")`) so a future refactor that accidentally promotes JSON to the default breaks this test.
- **`test_wiki_cross_project_json_emits_well_formed_object`** — `--json` returns a single object, exact key set per payload (`_WIKI_CROSS_PROJECT_JSON_KEYS` frozenset = `{wiki_dir, config_path, projects_registered, patterns_written}`), counts match the seeded fixture (2 registered projects, monkeypatched `generate_cross_project_wiki → 5`). `wiki_dir` is asserted via suffix match (`endswith("/.lvdcp/wiki")`) since the absolute path is user-home-dependent. Asserts no human-prose markers (`Cross-project wiki generated`) leak into the JSON output.
- **`test_wiki_cross_project_json_empty_registry_returns_zero_counts`** — empty registry → `projects_registered=0`, `patterns_written=0`, schema invariant still holds. Locks the empty-state contract: scripts can detect "no work to do" via `jq -e '.patterns_written == 0'` without a None-guard. Same discipline as v0.8.46 wiki status `[]` and v0.8.47 wiki lint `[]`.

Existing 4 wiki status + 4 wiki lint tests pass byte-identically — text path untouched.

## Rejected alternatives

- **Wrap in `{"run": {...}, "patterns": [{name, file}, ...]}`** — requires a library signature change to expose pattern names, and pattern names are trivially recoverable from the filesystem (`ls $(jq -r '.wiki_dir')/patterns/`). Defer to a separate `cross-project ls --json` if a real consumer needs it. See "Why a single object" above.
- **Emit a bare array of per-pattern entries** (matching v0.8.46 / v0.8.47 shape). Wrong unit of analysis — cross-project is a single-result generate command, not a per-row read. Forcing it into the bare-array shape would either mean wrapping every payload field in a single-element array (ugly) or losing the run-level summary fields (`projects_registered`).
- **Schema-version the JSON output (`"_schema_version": 1`).** Premature ceremony, same v0.8.38-v0.8.47 argument. Add when there's a real consumer that needs to gate on schema versions.
- **Include `claude_cli_available: bool` in the payload.** Would require library changes to surface the swallowed `RuntimeError` from `_detect_patterns_via_claude`. Inferable from `patterns_written == 0 && projects_registered > 1` (the same heuristic v0.8.42 uses for `qdrant_warnings`-based degradation detection); promote to an explicit field only if a consumer needs the precise signal.
- **Echo invocation parameters in the JSON shape (`config_path` already does this; also `--json`, etc).** Brittle — schema would change with every new flag. `config_path` is already round-tripped because it's the most useful field for "did this run use the config I intended"; other flags are ephemeral.
- **Add a `--dry-run` flag that emits the JSON payload without performing the file writes.** Tempting for "show me what cross-project would do" workflows, but `wiki cross-project` is fundamentally a generate command (it can't know `patterns_written` without actually invoking Claude). Dry-run would degenerate to `{wiki_dir, config_path, projects_registered, patterns_written: null}` — adds CLI surface for one nullable field. Reject until a real use case surfaces.
- **Separate `ctx wiki cross-project status --json` read-side subcommand** that reports the existing global INDEX state without re-running. Plausible follow-up but out of scope here (would benefit from a real consumer asking for it; current need is the run-level structured result, not a separate read surface).
- **Add `--json` to `wiki update` in the same release.** Keep increments small. `update` has mutation semantics with per-module progress streaming that benefits from a separate ship — the streaming-vs-terminal question for `--json` deserves its own analysis (NDJSON for per-module progress events vs. single payload at the end vs. both via a flag).

## Scope locks

- Only one new CLI flag (`--json`) on `wiki cross-project`.
- `wiki update`, `wiki status`, `wiki lint` subcommands **untouched** (status / lint already shipped JSON in v0.8.46 / v0.8.47; update deserves its own ship).
- `libs/wiki/cross_project.py::generate_cross_project_wiki` and `libs/core/projects_config.py::list_projects` **unchanged**.
- File-write side effect (global INDEX at `~/.lvdcp/wiki/INDEX.md`, pattern articles under `~/.lvdcp/wiki/patterns/`) **unchanged** in both modes — `--json` is a render switch, not a behavior switch.
- No new on-disk artifacts beyond what `cross-project` already writes.
- No MCP surface — cross-project is an operator-driven aggregation, not retrieval context for LLMs (the pattern articles themselves are accessible via the file system).
- Text output **byte-identical** — the new `test_wiki_cross_project_text_output_unchanged` test locks this.
- structlog routing introduced in v0.8.42 carries forward unchanged.

## Follow-ups explicitly NOT done here

- `ctx wiki update --json` — has mutation semantics with per-module progress streaming; deserves a separate ship to think through whether the JSON contract should be NDJSON (one event per module) or a single terminal summary. The streaming-vs-terminal question is the same one that applies to any long-running command's `--json` mode (e.g. `ctx scan` with --full could benefit from progress events; we shipped the terminal-summary form in v0.8.42 and deferred streaming).
- Promote `_detect_patterns_via_claude`'s `RuntimeError` into a structured `claude_cli_available: false` field on the payload. Requires library refactor; infer from `patterns_written == 0 && projects_registered > 1` for now.
- A separate `ctx wiki cross-project ls --json` read-side subcommand listing pattern names + paths. Cleaner than overloading the generate command's payload; ship when a real consumer needs it.
- A README badge tracking the JSON-scriptable surface coverage (e.g. "JSON-scriptable: scan ✓, inspect ✓, prune ✓, restore ✓, ls ✓, memory ✓, reconcile ✓, wiki status ✓, wiki lint ✓, wiki cross-project ✓"). Premature — release notes carry this discoverability.
