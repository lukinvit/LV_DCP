# v0.8.32 ‚Äî `ctx registry ls` for auditing registered projects

**Date:** 2026-04-24
**Previous:** [v0.8.31 ‚Äî value metrics: separate transient from real projects](2026-04-24-v0.8.31-value-metrics-transient-split.md)
**Type:** Read-only CLI surface
**Risk:** Low (new command group, no mutation, no changes to existing code paths)

## Problem

v0.8.31 classified registry entries as real vs transient on the dashboard, but left the operator without a terminal-native way to audit the registry. To see which entries were dormant candidates for future pruning, I had to either:

- Open the dashboard in a browser, click through the "+N transient" subtext, and read the tooltip, or
- Run an ad-hoc Python one-liner against `collect_value_metrics()`.

Neither works in a scripting context, and neither surfaces the *staleness* dimension (packs_total == 0 AND last_scan > 30d ago) ‚Äî which is the actual signal for "candidate for pruning."

## Solution

New `ctx registry ls` command group with one subcommand (`ls`) and three flags:

```text
ctx registry ls                        # all entries, tabular
ctx registry ls --kind real            # filter to non-worktree/non-fixture
ctx registry ls --kind transient       # filter to worktree/fixture
ctx registry ls --stale                # only dormant candidates (pruning hit-list)
ctx registry ls --json                 # machine-readable for scripts
```

Output columns: `NAME  KIND  SCAN  7D  TOTAL  LASTSCAN  PATH`. Each row combines:

- **Static** ‚Äî registry entry (name, root, last_scan_at_iso)
- **Derived** ‚Äî `kind` via the v0.8.31 `_is_transient(root)` classifier
- **Live** ‚Äî `scanned` via `.context/cache.db` existence check
- **Activity** ‚Äî `packs_7d` and `packs_total` from `retrieval_traces` table

All non-destructive. The `--stale` flag surfaces pruning candidates; it does not prune anything.

## Architecture

**Layered correctly per CLAUDE.md:**

- `libs/status/registry_audit.py` ‚Äî pure business logic: `audit_registry(config_path)` returns `list[ProjectAudit]`, plus `is_stale(row)` heuristic. No I/O beyond reading registry + cache.db. Injectable `now` for deterministic tests.
- `apps/cli/commands/registry_cmd.py` ‚Äî thin Typer shell: renders text/JSON and applies filters. No business logic.

Both layers import the v0.8.31 `_is_transient` classifier directly from `libs.status.value_metrics` ‚Äî single source of truth for the rule set.

## Staleness heuristic ‚Äî `is_stale(row, *, older_than_days=30)`

```python
if row.packs_total > 0:          return False  # actively used ‚Üí keep
if row.last_scan_age_hours is None: return True   # never scanned ‚Üí stale
return row.last_scan_age_hours >= older_than_days * 24
```

Rationale:

- **"packs_total > 0" short-circuits first.** A registry entry that has ever served a pack is valuable ‚Äî even if it's been dormant for months, the user might want it back.
- **Never-scanned entries count as stale.** This catches the accidental `lvdcp_scan <path>` against a path that was never followed up with actual use ‚Äî a common ship-ceremony leftover.
- **30 days is the default cutoff.** Configurable via keyword arg for callers that want different thresholds (e.g., a future `ctx registry prune --older-than 90` would pass 90).

## Rejected alternatives

- **`ctx project ls`** ‚Äî name clash. `ctx project` already exists as the copilot wrapper (check/refresh/wiki/ask). Adding `ls` there would conflate "registry audit" with "project copilot."
- **Flag on `ctx scan --dry`** ‚Äî wrong surface. Scan is about indexing; this is about inventory.
- **Putting the logic in `libs/status/value_metrics.py`** ‚Äî that module is already doing workspace-level aggregation. Mixing per-entry audit there would grow the single-responsibility violation. A sibling module keeps both focused.
- **Auto-pruning `--prune --yes`** ‚Äî deliberately deferred. Pruning is destructive, needs explicit user ACK, and a dry-run preview mode. Shipping classification and visibility first gives the user the data to make the decision themselves; the prune command can land in a subsequent release once the user has reviewed what `--stale` surfaces on their actual workspace.

## Scope locks

- **No mutation.** `ls` reads only. No `prune`, no `rm`, no `update`, no `touch`. Those belong in a separate gated release.
- **No change to `~/.lvdcp/config.yaml` schema.** We enrich the existing rows; we don't add new fields to the file.
- **No MCP surface.** The audit is an operator CLI, not a tool for Claude. Exposing it as an MCP tool would invite accidental automation of pruning logic.
- **Preserves registry ordering.** Output follows the YAML file order so the user's mental map stays stable.

## Testing

- **Lib layer (`tests/unit/status/test_registry_audit.py`)** ‚Äî 6 cases:
  1. Real/transient classification + pack counting on fresh registry
  2. Never-scanned entry has `scanned=False`, packs=0
  3. `last_scan_age_hours` math is correct within tolerance
  4. Stale-by-age detection (40d old, 0 packs)
  5. Active entry is not stale (recent packs)
  6. Never-scanned treated as stale
- **CLI layer (`tests/unit/cli/test_registry_cmd.py`)** ‚Äî 6 cases via `typer.testing.CliRunner`:
  1. Default text output includes both entries
  2. `--json` produces valid array with expected keys
  3. `--kind real` filters to 1 entry
  4. `--kind transient` filters to 1 entry
  5. Invalid `--kind` exits 2 with error
  6. `--stale` surfaces the dormant transient entry, hides the active real one

All in-memory. Full new-test suite runs in <4s.

## Follow-ups explicitly NOT done here

- `ctx registry prune [--dry-run | --yes]` ‚Äî destructive, needs its own release with explicit user ACK ceremony
- `ctx registry add <path>` / `rm <path>` ‚Äî registry mutation, out of scope
- Sorting flags (`--sort packs|age|name`) ‚Äî not needed at current registry sizes (<100 entries)
- Exposing the audit as an MCP tool ‚Äî not a Claude-facing surface
