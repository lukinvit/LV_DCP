# v0.8.58 — `ctx memory reject --json` closes the memory-review operator surface

**Date:** 2026-04-25
**Tag:** `v0.8.58`
**Theme:** CLI scriptability rollout — closes the memory-review operator surface

## Summary

Adds `--json` to `ctx memory reject`, completing the memory-review
operator surface. Same single-object payload as v0.8.57 `memory accept
--json` — just with `status: "rejected"` post-mutation instead of
`status: "accepted"`.

This closes the memory-review operator surface end-to-end:

- **Read side** (v0.8.44): `ctx memory list --json` — bare array of
  all memory entries (composes with `--status` for filtering)
- **Accept side** (v0.8.57): `ctx memory accept --json` — single
  object emitting the accepted entry with `status: "accepted"`
- **Reject side** (this ship): `ctx memory reject --json` — single
  object emitting the rejected entry with `status: "rejected"`

A script can now `list` to enumerate proposals, `accept` the keepers,
`reject` the noise, and verify the final state — all with `jq` and
exit codes, no markdown parsing.

## Schema

`ctx memory reject --json` returns a single JSON object — same per-row
schema as v0.8.44 `memory list --json` rows and v0.8.57 `memory
accept --json`:

```json
{
  "id": "mem_abc123",
  "status": "rejected",
  "topic": "Auth flow",
  "tags": [],
  "created_at_iso": "2026-04-25T10:14:00.123456+00:00",
  "created_by": "claude-opus",
  "path": "/abs/path/to/.context/memory/mem_abc123.md"
}
```

The post-mutation `status` field round-trips as `"rejected"` so a
script can confirm the state transition landed without a follow-up
`list --json` call. `body` intentionally omitted (recoverable via
`cat $(jq -r .path)`) — same v0.8.44 / v0.8.57 lean-payload precedent.

## Cross-state flip semantic

`reject_memory` has no state-machine guard — rejecting an
already-accepted memory is a no-op-on-state-but-write-on-disk that
returns the entry with status flipped to `"rejected"`. Locked by
`test_memory_reject_json_overrides_accepted_status` which:

1. Accepts a proposed memory → emits `status: "accepted"`
2. Rejects the same accepted memory → emits `status: "rejected"`
3. Asserts both emissions share the same `path` (no fresh on-disk
   write) — proves the cross-state flip mutates the existing file
   rather than creating a duplicate

This mirrors v0.8.57's `test_memory_accept_json_idempotent_re_accept`
but cross-state instead of idempotent. A regression that gated reject
behind "must-be-proposed" would break this test.

## Error-vs-success boundary

The `--json` flag is a **render switch**, not a semantic change:

- **Unknown memory id** → exit 2 with `error: <msg>` on stderr, no
  JSON payload on stdout. Locked by
  `test_memory_reject_json_unknown_id_exits_nonzero_no_payload`.
- **Successful reject (any prior state)** → exit 0 with the JSON
  payload showing `status: "rejected"`.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/memory/store.py` —
  `reject_memory` signature unchanged.
- **Helper reused, not duplicated.** `_memory_to_json` (introduced
  v0.8.44) emits the single-object payload directly — no new
  helpers, no new imports.
- **Cross-surface schema lock** via `_MEMORY_LIST_JSON_KEYS`
  frozenset (v0.8.44) now spans **three** surfaces: list rows,
  accept emission, reject emission. A future `Memory` field
  addition has ONE frozenset to bump.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  single-line `rejected: <id>  <topic>` footer. Pre-existing pipes
  see no diff. Locked by `test_memory_reject_text_output_unchanged`.
- **Symmetric mirror of v0.8.57 accept.** Implementation is a
  deliberate near-clone — same try/except shape, same JSON branch
  pattern, same docstring discipline. The two commands now read as
  obvious symmetric siblings in the source file.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Why this matters

Before this ship: every memory-review automation that wanted to
batch-decline noisy proposals (a daily cron rejecting all memories
older than 30 days that never got accepted, a topic-filter pipeline
that auto-rejects memories matching a deny-list regex, an IDE
"reject" button that needs to confirm the state transition before
re-rendering the review queue) had to grep the text footer
(`rejected: <id>`) or rely on the exit code alone — losing the
round-tripped `Memory` shape that confirms which on-disk entry was
just mutated.

After this ship: `ctx memory reject --json` returns a structured
payload directly. Same mutation semantic, same on-disk write, zero
text parsing.

This unblocks workflows like:

- **Batch rejection of stale proposals**: `for id in $(ctx memory list --status proposed --json | jq -r --arg cutoff "$(date -u -v-30d +%Y-%m-%dT%H:%M:%SZ)" '.[] | select(.created_at_iso < $cutoff) | .id'); do ctx memory reject "$id" --json; done | jq -s 'map(.id)'` — auto-decline proposals older than 30 days, collect the rejected ids
- **Topic-filter cleanup**: `ctx memory list --status proposed --json | jq -r '.[] | select(.topic | test("(?i)debug|temp|wip")) | .id' | xargs -I{} ctx memory reject {} --json | jq -s 'group_by(.topic) | map({topic: .[0].topic, count: length})'` — reject memories with throwaway topics, group by topic for an audit summary
- **Closing the memory-review operator surface**: read side via `memory list --json` (v0.8.44), accept side via `memory accept --json` (v0.8.57), reject side via `memory reject --json` (this ship) — together they let a script manage the memory-review queue end-to-end with `jq` and exit codes, no markdown parsing
- **Audit trail with provenance**: `ctx memory reject "$id" --json | jq '{when: .created_at_iso, who: .created_by, topic, decision: "rejected"}' >> review_audit.jsonl` — append-only audit log of rejected memories with their original creation provenance preserved
