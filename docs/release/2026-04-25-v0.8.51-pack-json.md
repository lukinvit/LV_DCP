# v0.8.51 ŌĆö `ctx pack --json` for scriptable structured pack payload (CLI/MCP parity)

**Date:** 2026-04-25
**Tag:** `v0.8.51`
**Theme:** CLI scriptability rollout ŌĆö primary retrieval surface

## Summary

Adds `--json` to `ctx pack`, exposing the full retrieval payload as a
single structured object that **mirrors the MCP `lvdcp_pack` response
schema 1:1**. The CLI and MCP surface now bind to one contract ŌĆö what
LLM agents see via `lvdcp_pack`, scripts and humans can now consume via
`ctx pack --json`.

This is the highest-leverage slice in the rolling `--json` rollout
(v0.8.42 ŌåÆ v0.8.50): `ctx pack` is the **primary retrieval interface**
of the system. Without `--json`, automation could see only the
human-readable assembled markdown ŌĆö no access to `coverage`,
`trace_id`, ranked file/symbol lists, or any of the structured signals
that drive lvdcp_explain, hash-comparison workflows, or downstream
re-ranking heuristics.

## Schema

`ctx pack --json` returns a single JSON object that exactly matches
`apps.mcp.tools.PackResult`:

```json
{
  "markdown": "# Context pack ŌĆö navigate\n\n**Project:** ...",
  "trace_id": "9ab55d4c-c2b6-4d02-8352-9abdddfbf911",
  "coverage": "medium",
  "retrieved_files": [
    "apps/cli/main.py",
    "libs/context_pack/builder.py",
    "..."
  ],
  "retrieved_symbols": [
    "apps.cli.commands.pack.pack",
    "libs.context_pack.builder.build_navigate_pack",
    "..."
  ]
}
```

### Why a single object (and why this exact shape)?

- **CLI/MCP parity is the design goal.** Two contracts for the same
  retrieval pipeline would invite subtle drift; one shared schema
  forces every shape change to be reviewed against both consumers.
- **Single object, not bare array.** `ctx pack` is a single-result
  retrieval call with a body (`markdown`) plus structured metadata
  (`trace_id`, `coverage`) plus two ranked sub-arrays. Same single-object
  discipline as v0.8.48 `wiki cross-project` (single-result generate)
  and v0.8.50 `obsidian status` (tri-state vault). Compare with the
  bare-array siblings (`wiki status`, `wiki lint`, `watch list`) which
  are pure per-row reads with no top-level metadata.
- **`markdown` field is included, not stripped.** A consumer that wants
  the rendered pack still gets it via `jq -r .markdown`; a consumer
  that wants only the structured signals can ignore the field. Stripping
  it would create a third contract divergent from MCP ŌĆö wrong tradeoff.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/context_pack/`,
  `libs/project_index/`, or any retrieval pipeline code.
- `_pack_to_json` lives next to `pack` in `apps/cli/commands/pack.py`,
  sister-helper to v0.8.46-v0.8.50 CLI-shell payload shapers.
- **Schema-locked tests.** `_PACK_JSON_KEYS` frozenset in
  `tests/integration/test_cli_pack.py` asserts the exact key set on
  every emission. `_VALID_COVERAGE_VALUES` frozenset locks the
  `coverage` enum domain. Adding a new field to either the MCP
  `PackResult` or the CLI helper without updating the other will fail
  the test instead of silently diverging.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  pure assembled markdown. Pre-existing pipes that consume `ctx pack`
  output as plain markdown see no diff.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.
- **Trace persistence unchanged.** `--json` mode still calls
  `idx.save_trace(trace_with_project)` so F1.B sparklines and
  `lvdcp_explain` see CLI-originated queries.

## Why this matters

Before this ship: every script, dashboard, or retrieval-quality
benchmark wanting structured pack data had to either (a) parse the
human-readable markdown back into structure (brittle, lossy ŌĆö `coverage`
and `trace_id` aren't even in the markdown body), or (b) duplicate the
retrieval call via the MCP server (operational overhead, requires the
server to be running). After this ship: `ctx pack ... --json` returns
the same dict the MCP tool returns. One contract, two transport modes.

This unblocks workflows that the rolling `--json` rollout's earlier
ticks couldn't reach:

- Retrieval-quality dashboards: `for q in queries.txt; do ctx pack . "$q" --json | jq -c '{q: $q, cov: .coverage, files: (.retrieved_files | length)}'; done > eval.ndjson`
- Hash-comparison drift detection: store `retrieved_files` per query, alert when set diff exceeds threshold
- Cross-project benchmark sweeps: same pack call across N projects, fold `coverage` distribution into hygiene reports
- LLM-on-LLM evaluation: feed the structured pack to a judge model without re-parsing markdown
