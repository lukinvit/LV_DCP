# v0.8.64 — `ctx mcp install --json` opens the MCP-binding surface

**Date:** 2026-04-25
**Tag:** `v0.8.64`
**Theme:** CLI scriptability rollout — MCP server registration

## Summary

Adds `--json` to `ctx mcp install`, the first scriptable surface for
the **MCP-binding** domain (Claude Code MCP server registration +
on-disk `CLAUDE.md` managed section + `~/.lvdcp/config.yaml`
bootstrap + hook scripts). This is the symmetric mirror of v0.8.62
`watch install-service --json` for the launchd-binding surface — both
are side-effecting installers wrapping a third-party tool's mutation
(`launchctl bootstrap` for the daemon, `claude mcp add` for the MCP
server).

A follow-up `ctx mcp uninstall --json` is queued separately and will
close the MCP-binding surface end-to-end (mirroring how v0.8.62
install + v0.8.63 uninstall closed the watch daemon-service surface).

## Schema

`ctx mcp install --json` returns a single JSON object with eight
top-level keys (locked by `_MCP_INSTALL_JSON_KEYS`):

```json
{
  "scope": "user",
  "entry_command": "/Users/<you>/.../python3.12",
  "entry_args": ["-m", "apps.mcp.server"],
  "claudemd_path": "/Users/<you>/.claude/CLAUDE.md",
  "config_path": "/Users/<you>/.lvdcp/config.yaml",
  "config_created": false,
  "version": "0.8.64",
  "hooks_installed": [
    "/Users/<you>/.claude/hooks/lvdcp-precheck.sh",
    "/Users/<you>/.claude/hooks/lvdcp-autoscan.sh"
  ]
}
```

- `scope` — the MCP scope the server was registered into (`user`,
  `project`, or `local`); round-trips so a follow-up uninstall can be
  scripted without re-deriving the flag
- `entry_command` — the Python interpreter `claude` will spawn for the
  MCP server (`sys.executable`); same post-`uv sync` interpreter-drift
  detection use case as v0.8.62 install-service's `program_arguments[0]`
- `entry_args` — JSON array `["-m", "apps.mcp.server"]`; stays a list
  so `jq -r '.entry_args[0]'` returns the module flag and
  `jq -r '.entry_args | join(" ")'` returns the rendered tail
- `claudemd_path` — absolute path of the `CLAUDE.md` file the managed
  section was written into (resolved via `--scope`); round-trips so a
  follow-up uninstall can target the same file without re-deriving
- `config_path` — absolute path of the lvdcp config file (always
  `~/.lvdcp/config.yaml`); round-trips for consistency with
  `claudemd_path`
- `config_created` — `true` if the install bootstrapped a new config,
  `false` if the file already existed (re-install branch); the explicit
  signal that distinguishes a fresh-install from a re-install
- `version` — the LV_DCP version recorded in the `CLAUDE.md` managed
  section's version marker (`<!-- lvdcp-managed-version: X.Y.Z -->`);
  lets a script verify the installed version matches what the user
  expected via `jq -e '.version == "0.8.64"'`
- `hooks_installed` — JSON array of absolute paths to the hook scripts
  copied into `~/.claude/hooks/` (PreToolUse `lvdcp-precheck.sh` +
  PostToolUse `lvdcp-autoscan.sh`); stays a list so individual hook
  paths are introspectable via `jq -r '.hooks_installed[]'`

## Dry-run interaction

`--json --dry-run` emits the planned-config snippet **alone** (no
`# Copy the following ...` comment header) so the output parses with
`jq` as-is:

```bash
$ ctx mcp install --json --dry-run | jq -r '.mcpServers.lvdcp.command'
/Users/.../python3.12
```

The dry-run path is a separate render — it does not perform the
install, does not bootstrap the config, does not write the managed
section, does not install hooks. The schema is the literal
Claude Desktop config snippet (already JSON-shaped via
`build_dry_run_snippet`); the only delta in JSON mode is suppressing
the comment header.

## Implementation discipline

- **CLI-only** change. `libs/mcp_ops/install.py` (`install_lvdcp`,
  `InstallResult`, `build_dry_run_snippet`) untouched.
- **Schema lock** via `_MCP_INSTALL_JSON_KEYS` (eight keys).
- **Error-vs-success boundary preserved.** `ClaudeCliError` →
  exit 1 with the error on stderr and **no** JSON payload on stdout —
  same v0.8.42-v0.8.63 discipline. A consumer chaining
  `--json | jq ...` hard-fails on a missing `claude` CLI rather than
  silently parsing prose.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  legacy chrome (`lvdcp MCP server registered ... / CLAUDE.md
  managed section: ... / config bootstrapped: ... / entry point: ... /
  hook installed: ... / hooks: PreToolUse ... / note: entry point
  contains ...`) bytewise. Pre-existing automation grepping these
  lines sees no diff. Locked by
  `test_mcp_install_text_output_unchanged_full_install`.

Tests stub `install_lvdcp` and `_install_hooks` so the real
`claude mcp add` is **never** invoked from unit tests, no real
`~/.claude/CLAUDE.md` is rewritten, no real `~/.lvdcp/config.yaml`
is bootstrapped, no real hooks land in `~/.claude/hooks`. Same
fixture-isolation discipline as v0.8.49 `_isolated_config` for the
watch registry and v0.8.62 `_isolated_launch_agent_paths` for the
launchd surface.

## Why this matters

Before this ship: every MCP-install automation had to grep seven
prose lines (`lvdcp MCP server registered (scope=...)`, `CLAUDE.md
managed section: <path>`, `config bootstrapped: <path>`,
`entry point: <interpreter> <args>`, `hook installed: <hook>` ×N,
`hooks: PreToolUse ...`, `note: entry point contains ...`) to figure
out what the install actually did. The recorded interpreter, the
resolved CLAUDE.md path, the per-hook script paths, and the
fresh-vs-reinstall signal were buried in unstructured stdout.

After this ship: `ctx mcp install --json` returns the eight-field
descriptor, so a provisioning script can reason about the install
state structurally. This unblocks workflows like:

- **Strict-version gate**: `ctx mcp install --json | jq -e '.version == "0.8.64"'` — exits 0 only when the recorded version matches the expected one (catches post-`pip install` version drift)
- **Stale-interpreter detection**: `[[ -x "$(ctx mcp install --json | jq -r .entry_command)" ]]` — fails if the recorded interpreter no longer exists (the post-`uv sync` Python-upgrade footgun)
- **Hook-installation audit**: `ctx mcp install --json | jq -r '.hooks_installed[]' | while read h; do test -x "$h" || echo "missing: $h"; done` — verifies every recorded hook actually landed and is executable
- **Fresh-vs-reinstall branching**: `if ctx mcp install --json | jq -e '.config_created'; then echo "first install"; else echo "re-install"; fi` — lets a CI step emit different telemetry on first install vs. update
- **Dry-run config synthesis**: `ctx mcp install --json --dry-run | jq '.mcpServers.lvdcp' > /tmp/lvdcp-mcp-snippet.json` — captures the snippet for manual paste into a host that doesn't have the `claude` CLI installed (exactly the use case the dry-run mode was designed for, now scriptable)

## Cross-surface schema discipline

`mcp install` is the third side-effecting installer to ship `--json`:

| Surface | Install | Uninstall | Inventory pattern |
|---|---|---|---|
| Watch daemon-service (launchd) | v0.8.62 | v0.8.63 | single-axis install + two-axis uninstall |
| Watch registry (lvdcp config) | v0.8.54 | v0.8.56 | single-result mutations |
| MCP-binding (claude mcp + CLAUDE.md + config + hooks) | v0.8.64 ← this ship | _queued_ | single-result side-effecting installer with stable descriptor |

This is the same closure pattern that v0.8.62 install + v0.8.63
uninstall applied to the watch daemon-service surface, that v0.8.59
enable + v0.8.60 disable applied to the timeline-flag surface, that
v0.8.57 accept + v0.8.58 reject applied to the memory-review
surface, and that v0.8.54 add + v0.8.56 remove applied to the
watch-registry surface — ship the opening side first, close with
the symmetric mirror in a follow-up.

## Pattern inventory after this ship

The seventh shape in the inventory now has its second occurrence:
**single-result side-effecting installer with stable descriptor**.
Both v0.8.62 (launchd) and v0.8.64 (MCP) share this shape — they
emit a flat descriptor on the success path and exit-non-zero-with-no-
payload on third-party-tool failure (`LaunchctlError` for launchd,
`ClaudeCliError` for `claude mcp add`).

Seven shapes:
- **Per-row reads** → bare arrays (v0.8.44, v0.8.49)
- **Single-result mutations w/ exception-on-absent** → bare entry
  (v0.8.54, v0.8.57, v0.8.58, v0.8.59, v0.8.60)
- **Single-result mutations w/ no-op-on-absent** → wrapper-with-null
  (v0.8.56)
- **Mutations w/ single count + invocation params** → flat dict
  (v0.8.38, v0.8.55)
- **Fleet-iteration with summary counters + per-entry array**
  (v0.8.61)
- **Single-result side-effecting installer with stable descriptor**
  (v0.8.62 launchd, v0.8.64 ← this ship MCP)
- **Single-result side-effecting uninstaller with two-axis success
  signal** (v0.8.63 launchd; MCP-side queued)
