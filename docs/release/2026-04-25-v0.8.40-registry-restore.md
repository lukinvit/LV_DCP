# v0.8.40 — `ctx registry restore` for prune undo

**Date:** 2026-04-25
**Previous:** [v0.8.39 — Dashboard surfaces tombstone counter](2026-04-25-v0.8.39-dashboard-tombstones.md)
**Type:** Feature / UX (registry hygiene — undo handle)
**Risk:** Low — new subcommand; default is dry-run; `*.bak` is left in place after apply (re-undo handle).

## Problem

`ctx registry prune --yes` has written a sibling `*.bak` snapshot before every mutation since v0.8.33. The recovery story has always been documented as a manual one-liner: `mv ~/.lvdcp/config.yaml{.bak,}`. That works, but:

- It's brittle — typo a brace and you've moved the wrong file or lost the live config entirely.
- It's invisible to scripts — there's no JSON contract for "the prune was rolled back."
- It's missing from the `ctx registry` subcommand surface — `ls`, `prune` exist; `restore` does not. Users discover the undo handle only by reading `prune --yes` output and noticing the `backup saved to ...` line.

The v0.8.38 release note explicitly listed `ctx registry restore` as a follow-up: *"useful but not blocking — the manual one-liner is documented in v0.8.33's release note."* Now blocking-enough — every other piece of the registry-hygiene quartet (write/read/script/UI) shipped in v0.8.36–v0.8.39, leaving restore as the last asymmetry.

## Solution

A new `ctx registry restore` subcommand that reads the `*.bak` and atomically writes its contents into the live config. Defaults to dry-run; `--yes` to apply; `--json` for scripts — same shape as the rest of the registry quartet.

```bash
# preview what restore would do
$ ctx registry restore
restore: would restore (dry-run) 5 entries from /Users/v/.lvdcp/config.yaml.bak
restore: live registry has 3 entries (delta: +2)

restore: dry-run — no changes written. Pass --yes to apply.

# apply
$ ctx registry restore --yes
restore: RESTORED 5 entries from /Users/v/.lvdcp/config.yaml.bak
restore: live registry has 3 entries (delta: +2)

# script consumption
$ ctx registry restore --json | jq '.delta'
2

# error path: no backup to restore from
$ ctx registry restore
restore: no backup found at /Users/v/.lvdcp/config.yaml.bak — run `ctx registry prune --yes` first to create one
$ echo $?
1
```

Library layer: `libs/status/registry_prune.py` gains a new `RestoreResult` dataclass and a `restore_from_backup(config_path, *, apply=False, backup_suffix=".bak")` function. The function reads both the backup and the live config to compute counts (`backup_entry_count`, `current_entry_count`, `delta = backup - current`), then on `apply=True` writes the backup contents through the same atomic `save_config` writer used everywhere else.

CLI layer: `apps/cli/commands/registry_cmd.py::restore_cmd` is a thin Typer shell over `restore_from_backup`. The `--json` path emits a 1:1 mirror of `RestoreResult` with `Path` fields stringified (same discipline as `_prune_result_to_json` in v0.8.38).

## Why the `*.bak` is left in place after restore

The simplest mental model: **at any moment, `*.bak` is the "last pre-mutation state."** After a restore, the live config now matches the backup byte-for-byte, so leaving `.bak` in place is harmless — same content lives in both files. This gives the user a re-undo handle: if they realize the restored state was wrong and they actually wanted what they had before the restore… well, the restored state and the backup are identical, so there's nothing further to undo. The `.bak` stays useful for the *next* prune cycle.

The alternative (deleting `.bak` after restore, like a true `mv`) was rejected because it asymmetrically destroys recovery information that costs nothing to keep. The next `prune --yes` overwrites `.bak` anyway with a fresh snapshot of the current state.

## The `delta` field

`delta = backup_entry_count - current_entry_count`. Always signed in the human-readable output (`+2`, `-1`, `0` → `"+0"`):

- **Positive delta** — restore *adds* entries (the typical "undo a prune" case).
- **Negative delta** — restore *removes* entries the user has registered since the backup was written. Surfaced in the dry-run preview so the user knows they're about to lose work — they can manually re-add any project after the restore.
- **Zero delta** — same number of entries on both sides, but the actual *contents* may differ (e.g., the same count via `prune` then `add`). Restore still proceeds but the user sees the count is unchanged.

A negative-delta restore is intentionally **not blocked** — the user explicitly invoked `restore --yes`, and the dry-run is right there to preview the loss. Adding a confirmation gate would make the JSON path harder to script and wouldn't actually prevent foot-shooting.

## New invariants

Five new tests in `tests/unit/status/test_registry_prune.py`:

- **`test_restore_dry_run_is_pure_read`** — `apply=False` reports counts without touching either file. Live config and backup byte-identical to before.
- **`test_restore_apply_replaces_config_with_backup`** — `apply=True` writes backup contents into live config, leaves `.bak` in place verbatim (re-undo handle).
- **`test_restore_no_backup_raises`** — `FileNotFoundError` with a "run `ctx registry prune --yes` first" hint when no `.bak` exists. Live config untouched.
- **`test_restore_handles_missing_live_config`** — if the user manually deleted the live config, `load_config` returns an empty registry → `current_count = 0` → restore re-creates the file from the backup. No crash.
- **`test_restore_negative_delta_when_user_added_after_prune`** — when the live config has more entries than the backup, `delta` is correctly negative. Surfaces "you'll lose 2 entries" in the preview.

Five new tests in `tests/unit/cli/test_registry_cmd.py`:

- **`test_registry_restore_dry_run_is_default`** — `ctx registry restore` (no `--yes`) prints `would restore (dry-run)`, the entry counts, and a signed delta. Both files untouched.
- **`test_registry_restore_yes_applies_and_keeps_backup`** — `--yes` writes the backup into the live config, `.bak` persists on disk.
- **`test_registry_restore_no_backup_exits_nonzero`** — exit code 1 when no `.bak` exists, with `prune --yes` hint in the error message.
- **`test_registry_restore_json_dry_run_shape`** — `--json` emits the exact key set `{backup_path, config_path, applied, backup_entry_count, current_entry_count, delta}` with `applied=false`, no hint text.
- **`test_registry_restore_json_yes_applies`** — `--yes --json` applies and reports `applied=true` with the same schema (invariant across dry-run / apply).

## Rejected alternatives

- **Delete the `*.bak` after restore (true `mv` semantics).** Asymmetrically destroys recovery information. The next `prune --yes` overwrites it anyway with a fresh snapshot — no value in eagerly removing it.
- **Block negative-delta restores behind a second confirmation flag.** The user explicitly invoked `restore --yes` and the dry-run preview already shows the loss. Extra gate would make the JSON path harder to script and wouldn't actually prevent foot-shooting.
- **Rename live config to `*.before-restore` before swapping in the backup.** Two backup files instead of one — confusing to reason about, and the user can always run `prune --yes` to capture the current state before restoring (which is exactly what produced the `.bak` in the first place).
- **Add `restore --to <path>` to restore to an arbitrary file.** Premature feature — the only documented backup location is the sibling `*.bak`. Add when a real use case appears.
- **Make `restore` an alias for `prune --restore`.** Conflates the two operations. They're inverses, deserve symmetric top-level subcommands. CLI surface symmetry beats flag-stuffing.
- **Schema-version the JSON output (`{"schema_version": 1, ...}`).** Same v0.8.38 argument — premature ceremony for a thin dataclass mirror that has no current versioning problem.
- **Auto-restore on `ctx registry prune` failure (e.g., if `save_config` raises).** The current code writes `.bak` *before* mutating, so a mid-write failure already leaves the user with both a `.bak` and (potentially) a partial live config. Adding implicit auto-restore would hide failure modes; explicit `restore` is the right primitive.

## Scope locks

- Library shape — only one new dataclass and one new function added to `registry_prune.py`. No removed fields, no renamed fields, no signature change on `prune_stale` / `plan_prune` / `PruneResult`.
- No registry schema change.
- No new on-disk artifacts beyond what the existing `*.bak` already provides.
- No public signature change on `audit_registry`, `is_stale`, `is_missing`, `list_projects`, `save_config`, `load_config`.
- Default behavior of `ctx registry prune` is unchanged — every existing test for prune still passes byte-identically.
- No MCP surface — destructive operations (and their inverses) stay CLI-only. Same lock as v0.8.33: `restore` mutates the live config, so it's CLI-only by the same argument.
- The `*.bak` is left in place after a successful restore (re-undo handle); the next `prune --yes` is the documented way to refresh it.

## Follow-ups explicitly NOT done here

- A `ctx registry diff` command that prints a side-by-side of live config vs `.bak` before restore. Useful for the negative-delta case where the user wants to see *which* entries they'd lose, not just how many. Plausible follow-up if the count-only preview turns out to be insufficient — but premature now (the JSON shape gives scripts everything they need to compute the diff via `comm` / `jq`).
- A `--keep-backup-as <suffix>` flag that renames the `.bak` to a timestamped sidecar before restore (e.g., `config.yaml.bak.2026-04-25T15:30:00Z`). Adds complexity for a problem that doesn't exist yet — the current "leave `.bak` in place" rule is simple and reversible.
- Schema versioning for the JSON output. Same deferral as v0.8.38 — premature ceremony.
- A dashboard widget surfacing "you have a `.bak` from N hours ago — restore?" Possibly useful for discoverability but adds UI surface for a recovery operation that should be deliberate, not nudged. The CLI is the right home.
