# v0.8.65 ‚Äî `CacheCorruptError` + daemon quarantine for corrupt SQLite caches

**Date:** 2026-04-25
**Tag:** `v0.8.65`
**Theme:** Reliability ‚Äî stop the SymbolType crash retry loop

## The bug

Field reports from a registered watch project (`X5_BM`) showed the
daemon log filling with cascading errors on every edit-block:

```
[error] processing failed for .../X5_BM: 147306 is not a valid SymbolType
[error] processing failed for .../X5_BM: 2 validation errors for File
  size_bytes: Input should be a valid integer, unable to parse string as an integer [input_value='class', input_type=str]
  role: Input should be a valid string [input_value=3279, input_type=int]
[error] processing failed for .../X5_BM: UNIQUE constraint failed: symbols.fq_name
[error] processing failed for .../X5_BM: database disk image is malformed
```

Three symptom classes, all rooted in the same cause: **physical
corruption of `.context/cache.db`** (B-tree page damage). `PRAGMA
integrity_check` on the offending DB returned dozens of `invalid page
number ...`, `Child page depth differs`, `Rowid ... out of order` ‚Äî
the file was past the point of in-place repair (32 MB cache.db with
extensive page corruption, schema version was current at v4).

The downstream symptoms made the root cause look like a Pydantic /
SymbolType bug, but the real failure was lower:

- `iter_symbols()` reads `row[2]` from a corrupt page ‚Üí SQLite returns
  an integer instead of the stored string ‚Üí `SymbolType(row[2])` raises
  `<int> is not a valid SymbolType`
- `get_file()` / `iter_files()` read column-misaligned values from a
  corrupt page ‚Üí `File(size_bytes='class', role=3279, ...)` fails
  Pydantic validation
- `replace_symbols` INSERT into a damaged index ‚Üí spurious `UNIQUE
  constraint failed` even on first write

The daemon caught these with a broad `except Exception` and continued
watching the project, so **every subsequent FS event re-triggered the
crash**. Log spam, wasted CPU, no user-visible recovery path.

## The fix

Three coordinated changes:

### 1. `libs/storage/sqlite_cache.py` ‚Äî typed corruption error

New `CacheCorruptError(RuntimeError)` raised by `SqliteCache.migrate()`
after a freshly-added `_check_integrity` step that runs `PRAGMA
quick_check` on the connection. `quick_check` is the cheap variant of
`integrity_check` (skips per-index validation) and catches the exact
failure mode that was producing garbage SELECT rows.

Two failure shapes are folded into one typed exception:

- **`sqlite3.DatabaseError` from the PRAGMA itself** ‚Äî e.g. `"file is
  not a database"` on a non-SQLite blob, or `"database disk image is
  malformed"` on severe header damage
- **Non-`"ok"` row from the PRAGMA** ‚Äî e.g. `"invalid page number
  8073"` for a damaged but readable B-tree

`_connect()` now suppresses `sqlite3.DatabaseError` from the
`PRAGMA journal_mode = WAL` setup write so corruption surfaces at the
typed `migrate()._check_integrity` boundary instead of as an untyped
connection-time crash.

The error message includes the recovery path:
> Delete `.context/` and re-run `ctx scan` to rebuild.

### 2. `apps/agent/daemon.py` ‚Äî in-process quarantine

Module-level `_QUARANTINED: dict[Path, str]` records projects that
raised `CacheCorruptError`. `process_pending_events` short-circuits
quarantined projects with a `[skip]` log line instead of re-attempting
the scan. The quarantine survives until daemon restart (the recovery
moment when the user has presumably deleted `.context/` and re-run
`ctx scan`).

`reset_quarantine()` test-only entry point clears the dict ‚Äî used by
the test fixture to keep tests isolated.

The catch is `except CacheCorruptError as exc: ...` placed **before**
the broad `except Exception` so corruption is funneled into quarantine
while other transient errors keep the previous "log + continue + retry
on next event" behavior.

### 3. Tests

**`tests/unit/storage/test_sqlite_cache.py`** ‚Äî 2 new tests:

| Test | Asserts |
|---|---|
| `test_migrate_raises_cache_corrupt_on_garbage_file` | non-SQLite bytes at `db_path` ‚Üí `CacheCorruptError` from quick_check, error message contains `db_path` and `ctx scan` recovery hint |
| `test_check_integrity_raises_on_non_ok_row` | a stub connection whose `quick_check` returns `("invalid page number 8073",)` ‚Üí `CacheCorruptError` with the page number echoed in the message |

**`tests/unit/agent/test_daemon_quarantine.py`** ‚Äî new file, 3 tests:

| Test | Asserts |
|---|---|
| `test_cache_corrupt_quarantines_project_and_skips_on_next_pass` | first edit-block raises ‚Üí `scan_project` called once, `[quarantine]` logged, second edit-block on same project does NOT call `scan_project` again, `[skip]` logged |
| `test_quarantine_is_per_project_not_global` | a corrupt project does not block scans of healthy projects in the same daemon process |
| `test_reset_quarantine_clears_state` | `reset_quarantine()` (modeling daemon restart) lets a previously quarantined project be re-attempted ‚Äî important because the user's recovery path is delete `.context/` + restart daemon |

Tests stub `daemon.scan_project` with a fake that raises
`CacheCorruptError` on demand so the real scanner is never invoked.
Tests use `monkeypatch.setattr(daemon_mod, "scan_project", ...)` for
clean isolation.

## Why this matters

Before this ship: a single corrupt `.context/cache.db` (from a half-
written commit during a daemon crash, a disk-level fault, or a
concurrent writer) would **silently wedge the daemon** ‚Äî every FS
event in that project would produce a fresh stack trace in the log
with no actionable signal. Other projects on the same daemon would
keep working, but the broken one would just keep crashing forever
with no clear recovery prompt.

After this ship:

- The first event that hits a corrupt cache produces **one** loud
  log line: `[quarantine] X5_BM: cache corrupt ‚Äî will skip until
  daemon restart. Recovery: delete .../X5_BM/.context/ and re-run
  \`ctx scan\`. (...)`
- Subsequent events on the same project produce a single `[skip]`
  line, not a stack trace
- The user has a documented, scriptable recovery path:
  ```bash
  rm -rf /path/to/X5_BM/.context/
  ctx scan /path/to/X5_BM
  launchctl kickstart -k gui/$(id -u)/tech.lvdcp.agent  # restart daemon
  ```
- Other projects on the same daemon keep scanning normally
- The error type is now part of the public surface ‚Äî `ctx scan`
  callers (CLI, future MCP tool) can catch `CacheCorruptError`
  specifically and surface the same recovery hint to humans

## Recovery path for users hitting the X5_BM symptom

If your daemon log shows any of:

- `... is not a valid SymbolType`
- `Input should be a valid integer, unable to parse string as an integer [input_value='...']`
- `UNIQUE constraint failed: symbols.fq_name`
- `database disk image is malformed`

‚Ä¶then the project's `.context/cache.db` is corrupt. Run:

```bash
rm -rf /path/to/your/project/.context/
ctx scan /path/to/your/project
```

Restart the daemon afterwards (`launchctl kickstart -k
gui/$(id -u)/tech.lvdcp.agent`) so it clears its in-process
quarantine and re-binds the (now-rebuilt) cache.

After v0.8.65 the daemon will **detect** the corruption on the next
event and quarantine the project with a clear recovery message,
instead of looping on the failure forever.
