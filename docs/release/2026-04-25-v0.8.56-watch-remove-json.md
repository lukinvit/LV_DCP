# v0.8.56 — `ctx watch remove --json` for scriptable daemon unregistration

**Date:** 2026-04-25
**Tag:** `v0.8.56`
**Theme:** CLI scriptability rollout — closes the daemon-registration operator surface

## Summary

Adds `--json` to `ctx watch remove`, completing the daemon-registration
operator surface. The wrapper-object payload `{"removed": <entry|null>}`
lets a script audit exactly which `ProjectEntry` was just deleted (or
gracefully no-op when the path was never registered) without grepping
the human `removed <path>` line.

This closes the daemon-registration write surface end-to-end:

- **Read side** (v0.8.49): `ctx watch list --json` — bare array of all
  registered projects
- **Add side** (v0.8.54): `ctx watch add --json` — single object,
  idempotent re-add returns the existing entry
- **Remove side** (this ship): `ctx watch remove --json` — wrapper
  object exposing the removed entry (or `null` for no-op)

## Schema

`ctx watch remove --json` returns a single JSON object with one
top-level key `removed` whose value is either a `ProjectEntry`-shape
dict (path was registered) or `null` (path was not registered):

```json
{
  "removed": {
    "root": "/abs/path/to/project",
    "registered_at_iso": "2026-04-25T10:14:00.123456+00:00",
    "last_scan_at_iso": null,
    "last_scan_status": "pending"
  }
}
```

For an unregistered path:

```json
{
  "removed": null
}
```

### Why the `{removed: <entry|null>}` wrapper?

- **`removed: null` for no-op success.** Removing an unregistered path
  is a successful no-op (same `remove_project` semantic as the text
  mode — silent filter-by-inequality on the list, save-back). The
  wrapper-with-null shape lets a script use `jq -e '.removed != null'`
  as the natural "did this remove do work" guard, parallel to v0.8.55
  prune's `jq -e '.deleted > 0'`. Same v0.8.45 / v0.8.49 / v0.8.55
  "no work done is still a successful run, surface the null/zero
  rather than erroring out" discipline.
- **Inner entry mirrors v0.8.49 `watch list --json` 1:1.** The
  `_project_to_json` helper introduced in v0.8.54 emits the same
  four-field schema (`root`, `registered_at_iso`, `last_scan_at_iso`,
  `last_scan_status`). Cross-surface schema parity: a future
  `ProjectEntry` field addition has **one** helper to bump (and the
  shared `_WATCH_LIST_JSON_KEYS` frozenset locks it across `list`,
  `add`, and now `remove`).
- **Wrapper, not bare entry.** A bare `ProjectEntry` would be
  ambiguous with `null` semantics — `null` and a missing-keys object
  would both signal "no entry", confusing consumers. The wrapper
  makes the no-op case unambiguous and lets future fields (e.g. a
  `reason` for richer audit) extend the wrapper without disturbing
  the inner entry shape.

## Capture-then-mutate ordering

The entry is read **before** `remove_project` mutates the config, so
the consumer sees what was just deleted. This is locked by
`test_watch_remove_json_emits_removed_entry`: a regression that
read-after-write would always emit `removed: null` and lose the audit
signal.

```python
if as_json:
    resolved = path.resolve()
    existing = next(
        (p for p in list_projects(DEFAULT_CONFIG_PATH) if p.root == resolved), None
    )
    remove_project(DEFAULT_CONFIG_PATH, path)  # mutation AFTER capture
    payload: dict[str, object] = {
        "removed": _project_to_json(existing) if existing is not None else None,
    }
    typer.echo(json.dumps(payload, indent=2))
    return
```

## Error-vs-success boundary

The `--json` flag is a **render switch**, not a semantic change:

- **Unregistered path** → exit 0 with `{"removed": null}` payload.
  Same no-op-as-success precedent.
- **Resolved-path resolution failure** (e.g. permission error on the
  parent directory) → propagated through Typer's `resolve_path=True`
  argument handling; non-zero exit, no JSON success-shape payload on
  stdout. The error-vs-success boundary stays at the exit-code gate.

## Implementation discipline

- **CLI-only** change. `remove_project` library signature unchanged.
- **Shared helper reused.** `_project_to_json` and `list_projects`
  imports were already in scope (used by `add` and `list`). No new
  imports, no new helpers.
- **Schema-locked test.** `_WATCH_REMOVE_JSON_KEYS = frozenset({"removed"})`
  in `tests/unit/cli/test_watch_cmd.py` asserts the exact
  one-top-level-key wrapper shape; the inner entry uses the existing
  `_WATCH_LIST_JSON_KEYS` frozenset (cross-surface lock now spans
  `list` / `add` / `remove`).
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  single-line `removed <path>` footer. Pre-existing pipes that
  consume `ctx watch remove` text output (or grep for "removed") see
  no diff. Locked by `test_watch_remove_text_output_unchanged`.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Why this matters

Before this ship: a script unregistering a project (a teardown hook
in CI when a worktree gets wiped, an operator's "decommission
project X" cron job, a fleet-rebalancing tool moving a project
between hosts) had to parse the `removed <path>` text line to
audit-log what was deleted, and had no machine-readable signal for
"the path was already absent" — `removed <path>` printed unchanged
whether the entry existed or not.

After this ship: `ctx watch remove --json` returns a structured
payload directly. The `removed` field is `null` when the path was
not registered (audit-log can skip the "what we lost" entry),
and a full `ProjectEntry` when there was an actual deletion. Same
hard-mutation semantic, same config writes, zero text parsing.

This unblocks workflows like:

- **CI teardown hook**: `ctx watch remove --project "$WORKTREE" --json | jq -e '.removed != null' && audit_log "decommissioned $WORKTREE at $(jq -r .removed.registered_at_iso)"` — emits an audit log entry only when an actual unregistration happened
- **Bulk decommission**: `for p in $(cat decommission_list.txt); do ctx watch remove "$p" --json; done | jq -s 'map(select(.removed != null)) | length'` — counts how many projects were actually present (vs. already-absent ghosts)
- **Closing the daemon-registration operator surface**: read via `watch list --json` (v0.8.49), add via `watch add --json` (v0.8.54), remove via `watch remove --json` (this ship) — together they let a script manage the daemon's project registry end-to-end with `jq` and exit codes, no markdown parsing
