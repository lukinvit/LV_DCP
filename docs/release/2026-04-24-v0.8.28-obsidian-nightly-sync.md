# v0.8.28 — Obsidian nightly sync (`ctx obsidian sync-all`)

**Date:** 2026-04-24
**Scope:** `apps/cli/commands/obsidian_cmd.py` adds a `sync-all` subcommand + tests in `tests/unit/cli/test_obsidian_sync_all.py`. No backend change, no daemon change, no TypeScript change.

## Why

The Obsidian publisher has been live since Phase 2, wired two ways:

1. **Manual:** `ctx obsidian sync <project> --vault <vault>` — explicit, one project at a time.
2. **After scan:** `apps.agent.obsidian_worker.run_obsidian_sync()` — the daemon fires it on every successful scan, debounced by `~/.lvdcp/config.yaml:obsidian.debounce_seconds` (default 1h) via `.context/obsidian_last_sync`.

What's missing is **nightly** — the case where a project was scanned yesterday, the debounce window elapsed, but no scan has fired since, so the vault is drifting. Real users (including the author) don't scan every project every day; without a scheduled sweep, Obsidian stops being a faithful mirror.

The v0.8.27 roadmap explicitly called out "Obsidian nightly sync" as the remaining non-parser Phase 10 item. This release closes it.

The design question is **who schedules**. Options:

- **In-process scheduler inside the daemon** — an extra moving part, an extra place for bugs, and it fires even when the daemon isn't running (laptop closed lid, user logged out of that account).
- **External cron / launchd** — user's own OS scheduler runs a CLI command. Dead-simple, already the macOS-native way, no new daemon code.

LV_DCP has always been "local-first, minimal moving parts" (constitution §I.3, `CLAUDE.md` Принцип 5: "Modular monolith, не микросервисы. Простота эксплуатации превыше модной архитектуры"). Shipping the CLI command and letting the user own the schedule is the smaller surface area — and it makes the nightly run testable from a shell without ever involving `launchd`.

## Design

### `ctx obsidian sync-all` — what it does

Walks every project in `~/.lvdcp/config.yaml:projects`, decides which ones need a sync, and shells out to `ctx obsidian sync <root> --vault <vault>` for each. Writes `.context/obsidian_last_sync` on success so the daemon's auto-sync debounce respects the nightly run.

```
ctx obsidian sync-all [--stale-hours N] [--dry-run] [--config PATH] [--timeout SECS]
```

| Flag | Default | Purpose |
| --- | --- | --- |
| `--stale-hours` | `24.0` | Sync only projects whose last Obsidian sync is older than N hours (or never synced). `0` = force-sync every registered project. |
| `--dry-run` | off | Print the plan (`sync` / `skip: reason` per project) and exit 0 without invoking the inner CLI. |
| `--config` | `~/.lvdcp/config.yaml` | Override for tests / non-standard setups. |
| `--timeout` | `300` | Per-project subprocess timeout in seconds. |

### Why these knobs

- **`--stale-hours` instead of `--force`.** A single numeric gate covers both "run me hourly, sync anything >1h old" and "force-sync everything right now" (`--stale-hours 0`) without a second flag. One less flag = one less interaction to test.
- **Missing marker = stale.** A project registered but never Obsidian-synced is the most common "please sync this" case. Treating a missing marker as stale means the first nightly run after project registration just works, no setup step required.
- **`--dry-run` is cheap but essential.** Scheduled tasks are the easiest thing in the world to get wrong silently. A user configuring launchd/cron should be able to see exactly which projects will sync before arming the schedule.
- **One bad project doesn't block the others.** `subprocess.run` per project, failures logged to stderr with the last line of the child's stderr. Exit code is `1` iff any project failed — the rest still ran.
- **Shells out to `ctx obsidian sync`, not in-process call.** Re-uses every tested pathway of the existing CLI command, gets per-project timeout isolation from `subprocess.run(timeout=...)`, and means a crash in one project's publisher can't poison the rest of the run. ~20 extra ms of Python startup per project is a non-issue for nightly scheduling.

### Marker semantics

`.context/obsidian_last_sync` already existed — written by `apps.agent.obsidian_worker.run_obsidian_sync` after every successful after-scan sync. `sync-all` reads the same file with the same float-epoch format and writes to it after every successful manual invocation. One marker, two writers, same debounce gate. The daemon won't re-fire an auto-sync inside its own debounce window after a nightly run completes — they cooperate rather than compete.

### Scheduling (out of repo)

The repo ships the command, not the schedule. Two canonical ways to arm it:

**launchd (macOS-native):** `~/Library/LaunchAgents/tech.lukinvit.lvdcp.obsidian.plist` with `StartCalendarInterval` at 03:00 local, invoking `/path/to/LV_DCP/.venv/bin/ctx obsidian sync-all`. `bootout`-safe — `launchctl unload` stops it cleanly.

**cron (cross-platform):** `0 3 * * * /path/to/LV_DCP/.venv/bin/ctx obsidian sync-all >> ~/.lvdcp/logs/obsidian-sync.log 2>&1`. Simplest possible setup.

Neither is installed by the LV_DCP installer — this is a deliberate choice. `ctx mcp install` touches `~/.claude/`, which is Claude's territory. A scheduled daemon touching `~/Library/LaunchAgents/` is a different trust level — user should own that decision.

## Manual verification path

```bash
# Sanity check without side effects.
ctx obsidian sync-all --dry-run

# Force every registered project to re-sync.
ctx obsidian sync-all --stale-hours 0

# Use a non-standard config.
ctx obsidian sync-all --config ~/tmp/test-config.yaml
```

Example `--dry-run` output:

```
Sync plan (dry-run, stale-hours=24.0):
  sync                           /Users/you/code/LV_DCP
  skip: fresh (2.1h old)         /Users/you/code/another-project
  skip: no cache.db              /Users/you/code/never-scanned-project
```

## Known limitations (explicit scope locks)

- **Sequential, not parallel.** One subprocess at a time. Even a 20-project nightly sweep runs in ≤20 × (publisher latency + 20 ms startup) — well under a minute on a warm SQLite cache. Parallelization would require a shared vault-lock strategy; not worth the complexity until we see >60 registered projects per user.
- **No per-project override.** The `--stale-hours` gate is global. A user who wants "small projects nightly, big project weekly" would need two launchd entries with different `--config` files or a wrapper script. Deemed low-priority.
- **No built-in scheduler.** On purpose — see "Scheduling (out of repo)" above. The day a Windows or Linux user hits this and the `launchd` path doesn't apply, a `ctx watch install-service`-style subcommand becomes the natural follow-up.
- **Exit-code granularity.** `0` or `1` — there's no "`2` means partial". Scripts that care about the failure count can parse the `"failed: N"` tail line.
- **Per-invocation subprocess overhead.** ~20 ms of Python startup per project. Irrelevant for nightly. A future increment could replace the `subprocess.run` loop with an in-process call to the same `ObsidianPublisher.sync_project` pathway, but that would lose the timeout-isolation + error-isolation properties we explicitly wanted.
