# v0.8.37 — `ctx registry ls --missing` for tombstone audit

**Date:** 2026-04-25
**Previous:** [v0.8.36 — `ctx registry prune --missing` for absent-root cleanup](2026-04-25-v0.8.36-prune-missing-roots.md)
**Type:** Feature / UX (registry hygiene — symmetric read surface)
**Risk:** Low — adds a new flag on a read-only command; additive JSON field on `ls` output; default behavior unchanged.

## Problem

v0.8.36 shipped `ctx registry prune --missing` so users can clean up tombstones (registry entries whose root directory no longer exists on disk). But there's no matching **read** surface — to audit which entries would be removed before running the destructive command, users have to shell out:

```bash
ctx registry ls --json | jq -r '.[] | .root' | while read r; do
  [ -d "$r" ] || echo "MISSING: $r"
done
```

The information is right there in the registry; the tool should surface it directly. And without a `MISS` marker in the default text output, a tombstone is indistinguishable from a merely-unscanned project — both show `SCAN=no`.

## Solution

A new `--missing` flag on `ctx registry ls` plus an additive `missing: bool` field on every `ProjectAudit` row:

- `ctx registry ls --missing` — filters to entries whose root is gone from disk.
- `ctx registry ls --missing --json` — machine-readable for `jq` pipelines.
- `ctx registry ls --missing --kind transient` — composable with the existing filters (AND).
- `ctx registry ls --missing --stale` — intersection: tombstones that *also* meet the 30d/zero-packs staleness gate.
- `ctx registry ls` (default) — now renders `MISS` in the `SCAN` column for tombstoned rows, so users spot them without needing the flag.
- Every `ls --json` row now carries `"missing": true|false` (additive, existing keys unchanged).

```bash
$ ctx registry ls --missing
NAME              KIND       SCAN    7D  TOTAL  LASTSCAN  PATH
-------------------------------------------------------------
deleted-today     transient  MISS     0      0   (never)  /tmp/scratch/worktrees/deleted-today
```

Library layer: `libs/status/registry_audit.py` — `ProjectAudit` gains `missing: bool = False`; `audit_registry` sets it via `not entry.root.exists()` and short-circuits `scanned`/`packs_*` so a tombstone row doesn't stat inside a ghost parent; new `is_missing(row) -> bool` helper mirrors the `is_stale(row)` shape.

CLI layer: `apps/cli/commands/registry_cmd.py::ls_cmd` adds `--missing`; `_apply_filters` gains a `missing: bool` parameter composed as AND with `kind` and `stale`; `_render_text` surfaces `MISS` in the `SCAN` column when `missing=True`.

## Why composable on `ls`, but independent on `prune`

`prune --missing` in v0.8.36 **ignores** `--kind` and `--older-than` on purpose — a mutation command with silently-compositional flags is a footgun (stray `--kind real` in shell history would silently leave transient tombstones behind).

`ls --missing` is **read-only** with no mutation risk, so AND-composition is strictly additive value: `ls --missing --kind transient` scopes an audit, `ls --missing --stale` spots entries that hit both predicates. Different semantics for different commands is justified by the asymmetric consequences.

## New invariants

Four new library-layer tests + five new CLI-layer tests lock the contract:

- **`test_audit_marks_absent_root_as_missing`** — `missing=True` for deleted roots, `False` otherwise.
- **`test_audit_missing_row_has_no_scan_signal`** — a tombstone row reports `scanned=False`, `packs_total=0`, `packs_7d=0` (cache-db probe short-circuits past a gone parent).
- **`test_is_missing_returns_the_flag`** — the helper returns the predicate boolean, doesn't re-probe the filesystem.
- **`test_missing_and_stale_are_independent_predicates`** — recent tombstone is missing-but-not-stale; 40-day-old tombstone is both.
- **`test_registry_ls_missing_filters_to_tombstones`** — `--missing --json` returns only deleted-root rows with `missing=True`.
- **`test_registry_ls_missing_text_shows_MISS_marker`** — the text table shows `MISS` for tombstones.
- **`test_registry_ls_missing_composes_with_kind`** — `--missing` + `--kind transient` returns the transient tombstone and no others.
- **`test_registry_ls_missing_empty_when_all_roots_exist`** — `--missing` returns `[]` when every root is alive.
- **`test_registry_ls_json_carries_missing_field`** — every `ls --json` row includes the new `missing` field (additive; old consumers still parse).

## Rejected alternatives

- **Add a dedicated `NAME  KIND  SCAN  MISS  7D  ...` column.** Extra width for a signal that matters only for ~1 % of rows. Overloading `SCAN=MISS` reads at a glance, keeps the table fitting in 120 cols, and the JSON output carries the boolean separately for scripts.
- **Expose `--missing` as `--kind missing`.** Same argument as v0.8.36: `--kind` is a classification axis (real vs transient vs all), not an eligibility axis. A tombstone is still *classified* as transient or real — the fact that it's gone is orthogonal.
- **Make `prune --missing` also compose with `--kind`.** Rejected in v0.8.36 for safety (footgun risk) and stays rejected here — the ls/prune asymmetry is the right trade.
- **Recompute `missing` lazily in `_apply_filters` from `Path.exists()` instead of storing on the dataclass.** Would make every CLI render stat the filesystem twice. Storing once during `audit_registry` keeps `is_missing` O(1) and gives JSON consumers the signal for free.
- **Emit a `missing_count` summary line.** Nice-to-have but adds a header line every invocation for rare information. The user can compute it trivially: `ctx registry ls --missing --json | jq length`.

## Scope locks

- No registry schema change — `~/.lvdcp/config.yaml` structure unchanged.
- `ProjectAudit` gains one optional field with a safe default (`False`), so pickled/cached instances from older code deserialize cleanly; existing JSON consumers see one additional key and nothing renamed.
- Default behavior of `ctx registry ls` (without `--missing`) is unchanged except for the `MISS` marker in the `SCAN` column for tombstones — which is pure additive signal, not a breaking layout change (column width stays the same).
- No mutation path touched — `prune` and `save_config` untouched.
- No MCP surface — `ls` remains CLI-only (same scope lock as v0.8.32).

## Follow-ups explicitly NOT done here

- A single `ctx registry audit` command that combines `ls --stale`, `ls --missing`, and a recommended remediation plan. Plausible but no driving signal yet — the flag composition already covers the audit workflows.
- Exposing `missing` in the `HealthCard` rendered on the dashboard. Low priority — the dashboard's "projects active" card already filters by `is_transient`; a third dimension risks crowding the card.
- Coloring `MISS` red in terminal output via `rich`. The current output is plain-text; colorization is a global policy decision (respect `NO_COLOR`, detect TTY, etc.) worth deferring until there's a broader ask.
