# v0.8.21 — Kotlin tree-sitter parser

**2026-04-24**

## Why

v0.8.20 opened Phase 10 with a Java parser and committed to Kotlin as
the next natural step — "shares the JVM story". This release follows
through: Kotlin projects (Android apps, Spring / Ktor backends, Gradle
build scripts, JVM libraries) now index to the same symbol graph and
import relations that Python / TypeScript / JavaScript / Go / Rust /
Java projects already enjoy.

Kotlin parser follows the exact `TreeSitterParser` hook-method shape
established by Java — same grammar binding, same symbol-type map
discipline, same role-detection strategy, same module-FQ derivation
that strips source roots anywhere in the path. The decision to do
Kotlin after Java proved right: ~150 lines of parser code, zero
infrastructure churn, extended the language allowlists in two
status files and was done.

## Design

`libs/parsers/kotlin.py` is a 116-line `TreeSitterParser` subclass.
All non-trivial Kotlin-specific decisions are hook overrides.

### Symbol type mapping

| Kotlin AST node | LV_DCP `SymbolType` |
|-----------------|---------------------|
| `class_declaration` | `CLASS` |
| `object_declaration` | `CLASS` |
| `function_declaration` | `METHOD` |
| `property_declaration` | `VARIABLE` |

`class_declaration` collapses regular / interface / enum / annotation
/ data / sealed classes onto `CLASS` — the Kotlin grammar uses one
AST node kind for all of these, distinguishing via modifier children
(`modifiers > class_modifier > sealed | enum | annotation | data`) or
a sibling keyword (`interface` vs `class`). For v1 we don't branch on
the modifier — a type declaration is a type declaration.

`object_declaration` covers named singleton objects (`object Foo`).
At runtime Kotlin compiles these to classes-with-one-instance, so
`CLASS` is the right mapping for graph purposes.

`companion_object` intentionally is **not** in the map. The grammar
gives companion objects no `name` field (they're implicitly called
`Companion`), so the base walker naturally skips them for symbol
emission but still recurses into their body. Methods and properties
defined inside a companion object therefore land with
`parent_fq_name = EnclosingClass`, which matches Kotlin semantics —
companion members are addressed as `EnclosingClass.member` without
an explicit `Companion` hop.

### Property extraction quirk

Kotlin's `property_declaration` doesn't carry a `name` field on the
node itself — the identifier lives inside a `variable_declaration`
child. The default base hook
(`node.child_by_field_name("name")`) returns `None` for properties.
`KotlinParser._extract_name()` overrides the property case to walk
children for a `variable_declaration`, then walk its children for
the `identifier`:

```python
if node.type == "property_declaration":
    for child in node.children:
        if child.type == "variable_declaration":
            for grand in child.children:
                if grand.type == "identifier" and grand.text:
                    return grand.text.decode("utf-8", errors="replace")
    return None
return super()._extract_name(node)
```

Parity with the Java parser's equivalent `field_declaration` /
`variable_declarator` dance — Java and Kotlin both bury the name
one level deeper than most tree-sitter grammars.

### Const val handling

Kotlin's `const val MAX_SIZE: Int = 1024` is the closest thing the
language has to Java's `static final`. v1 maps **all**
`property_declaration` nodes (including `const val`) to `VARIABLE`
uniformly — matches the Java v1 decision to map `static final` to
`VARIABLE`. Promoting `const val` to `CONSTANT` requires inspecting
the property's `modifiers` child at parse time; the test
`test_extracts_const_val_inside_companion` locks the current
behaviour so a future promotion is an explicit contract break.

### Import relations

`import` emits `RelationType.IMPORTS` edges with `dst_type="module"`:

| Kotlin import form | `dst_ref` |
|--------------------|-----------|
| `import java.util.List` | `java.util.List` |
| `import java.util.Map.*` (wildcard) | `java.util.Map` |
| `import com.example.Foo as FooAlias` (alias) | `com.example.Foo` |

Wildcard imports drop the `.*` suffix because the Kotlin grammar
places `.` and `*` as sibling tokens after the `qualified_identifier`
— reading the qualified_identifier's text naturally gives the
package path without the wildcard. Parity with Java.

Alias imports (`import ... as ...`) surface the original module
path. The graph tracks real dependencies, not local aliases — the
alias is a compile-time convenience that has no meaning outside the
file.

### Role detection

Kotlin projects span Gradle (Maven-like), Android Gradle (extra
`androidTest` source set), and mixed Java/Kotlin projects. Role
detection walks directory fragments first, then filename fallback:

**Directory fragments** that flip role to `test`:
- `src/test/kotlin/` (standard Gradle)
- `src/androidTest/kotlin/` (Android instrumented tests)
- `src/test/java/`, `src/androidTest/java/` (Kotlin files in
  mixed projects)
- `/tests/`, `/test/`

**Filename fallback** (strips `.kt` / `.kts` first so the extension
doesn't interfere):
- `*Test` / `*Tests` suffix
- `Test*` prefix

Everything else → `source`. The filename fallback catches
`app/src/ConfigTest.kt` even when it's outside a dedicated `test/`
directory.

Gradle Kotlin DSL scripts (`.kts`) are deliberately **not**
classified as tests — `build.gradle.kts` stays `source` so the
file is still indexed.

### Module FQ derivation

`_module_fq()` strips the first recognised source root it sees and
dots the remainder:

| Input path | `_module_fq` output |
|------------|---------------------|
| `src/main/kotlin/com/example/app/Config.kt` | `com.example.app.Config` |
| `src/test/kotlin/com/example/app/ConfigTest.kt` | `com.example.app.ConfigTest` |
| `src/androidTest/kotlin/com/example/Foo.kt` | `com.example.Foo` |
| `src/main/java/com/example/Bar.kt` | `com.example.Bar` |
| `modules/foo/src/main/kotlin/com/example/Bar.kt` | `com.example.Bar` |
| `com/example/Foo.kt` | `com.example.Foo` |
| `Foo.kt` | `Foo` |
| `build.gradle.kts` | `build.gradle` |

Nested Gradle multi-module projects work because `.find()` locates
the first occurrence of the source root anywhere in the path, not
just at the start — parity with Java's handling of
`modules/foo/src/main/java/…`.

Both `.kt` and `.kts` extensions are stripped before the source-
root logic runs.

## Known limitations / non-goals

1. **No `INHERITS` edges** — `class Loaded : State()`, `class Foo :
   Interface, OtherInterface` and delegation specifiers on object
   declarations (e.g. `object Loading : State()`) are visible in
   the AST but v1 doesn't emit graph edges for them. Java has the
   same v1 gap; both would benefit from a future inheritance
   extractor.
2. **No `SAME_FILE_CALLS` edges** — matches Go / Rust / Java
   shape; only the Python parser emits call-graph edges today.
3. **`const val` maps to `VARIABLE`, not `CONSTANT`** — documented
   above, locked by test, parity with Java `static final`.
4. **Primary constructor properties don't surface as separate
   symbols** — `class Config(val name: String)` extracts `Config`
   but not `name`. The `val name` is a `class_parameter` inside
   `primary_constructor > class_parameters`, not a
   `property_declaration`. Adding them requires a dedicated walker
   for constructor parameters; a natural future sub-increment.
5. **Data class components don't surface separately** — same
   reason as (4); `data class Point(val x: Int, val y: Int)`
   emits `Point` as `CLASS` but not `x` / `y`.
6. **Swift is the next language, not in scope for this release.**

## Roadmap positioning

Second shipped increment of Phase 10. With Java (v0.8.20) and
Kotlin (v0.8.21) done, the JVM pair is complete. Next language
target: **Swift**. After that, sub-increments to deepen language
support (inheritance edges for Java / Kotlin, data class / record
component extraction, call-graph edges) start to compete with the
VS Code marketplace / Obsidian nightly sync items in Phase 10 on
prioritisation — driven by concrete user signals rather than a
fixed order.
