# v0.8.26 — VS Code extension marketplace-readiness prep

**Date:** 2026-04-24
**Scope:** metadata + packaging only; no behaviour change on backend or extension runtime.

## Why

With the inheritance-edge axis closed across Java/Kotlin/Swift in v0.8.25, the next Phase 10 lane is **VS Code marketplace publication** (called out in the v0.8.22 Phase 10 close-out, the v0.8.24 roadmap line, and every release note since). The extension at `apps/vscode/` has been tracked in the repo since Phase 9 but has never been marketplace-ready: it was missing repository URL, keywords, categories, a listing README, a CHANGELOG, a LICENSE file, and a tightened `.vscodeignore` — all of which the VS Code Marketplace surfaces or requires on the extension's listing page.

This release does the **metadata and packaging prep** so that a future `vsce publish` is a one-command operation for the maintainer once the publisher account is registered and a Personal Access Token is provisioned. No backend code, no TypeScript code, no user-facing behaviour change.

### What this release deliberately does *not* do

- **Does not publish to the Marketplace.** `vsce publish` requires creating a Publisher on the Microsoft Marketplace (a user-level action outside the project) and an Azure DevOps Personal Access Token (a secret that must live on the maintainer's machine, not in CI). Both are explicitly out of scope for an autonomous-mode increment.
- **Does not add a marketplace icon asset.** A 128x128 PNG icon improves discoverability but requires a designer's call on branding. The current `activitybar` icon uses the built-in `$(symbol-structure)` codicon, which is sufficient for in-VS-Code UX. A PNG icon is tracked as a separate TODO.
- **Does not add extension settings (path config, backend URL).** Currently the extension shells out to `ctx` on `PATH`; that's a real gap (users without `uv run` shims have to wrapper it) but it's runtime behaviour, not packaging. Separate increment.

### `apps/vscode/README.md` — new file

The Marketplace renders the extension's `README.md` directly on the listing page as the primary marketing surface. The file covers:

- **What it does** — three-bullet summary (Context Pack, Impact Analysis, status bar)
- **Requirements** — Python 3.12, `uv`, backend stack, `ctx` on `PATH`, project indexed
- **Commands** — both registered commands with their palette titles
- **Design philosophy** — local-first, no network uploads
- **Known limitations** — `ctx` must be on `PATH`, `.vsix` install until marketplace-published, planned settings
- **Links** — docs, issues, discussions, LICENSE

Separate from the root `README.md` (which is 500+ lines of full-project context) — the extension README is ~60 lines, focused on someone deciding whether to install from the Marketplace.

### `apps/vscode/CHANGELOG.md` — new file

Marketplace also renders `CHANGELOG.md` on the listing's "Changelog" tab. Structure: version header, date, change bullets. Entries for v0.8.23–0.8.26; earlier versions point to the main repo release notes directory so the changelog doesn't duplicate 500 KB of history.

### `apps/vscode/LICENSE` — new file

Copy of the repo root `LICENSE` (Apache 2.0). The Marketplace's "License" link on the listing requires the file to be inside the packaged `.vsix` — linking to the repo LICENSE is not sufficient.

### `apps/vscode/.vscodeignore` — expanded

Before (4 lines): `.vscode/`, `src/`, `tsconfig.json`, `node_modules/`.

After (12 lines): added `.vscode-test/`, `package-lock.json`, `.claude/`, `.github/`, `**/.DS_Store`, `**/*.map`, `**/*.log`, `**/*.tsbuildinfo`, `*.vsix`. Prevents the built `.vsix` from bloating with source maps, lockfiles, the worktree `.claude/` dir, and the recursive self-reference to any previously packaged `.vsix`.

Verified with `npm run package`: the resulting artifact is **11.02 KB / 9 files** — `CHANGELOG.md`, `LICENSE.txt`, `README.md`, `package.json`, and the three compiled `.js` files. No sourcemaps, no `node_modules`, no `.ts` sources.

## Publishing runbook (for the maintainer)

Once ready to publish:

1. Register publisher `lv-dcp` on [marketplace.visualstudio.com/manage](https://marketplace.visualstudio.com/manage) (one-time, user-level)
2. Generate a Personal Access Token in Azure DevOps with `Marketplace: Manage` scope (one-time)
3. `export VSCE_PAT=<token>` (per-shell)
4. `cd apps/vscode && npm install && npm run publish` — runs `vscode:prepublish` (compile), then `vsce publish` with the token

If the publisher name `lv-dcp` is taken on the Marketplace, update `package.json`'s `publisher` field to an available name (e.g. `lukinvit`, `lv-dcp-oss`) before publishing. The rest of the manifest is publisher-agnostic.

## Known limitations (explicit scope locks)

- **No PNG icon.** Marketplace listing icon is missing; the activity-bar icon uses the built-in `$(symbol-structure)` codicon. Future increment: commission a 128x128 PNG and wire via `"icon": "media/icon.png"`.
- **No extension settings.** `ctx` must currently be on `PATH`; there's no `lvdcp.cliPath` configuration. Users running `ctx` via `uv run python -m apps.cli` need a wrapper shell script. Future increment will add `configuration` contribution points.
- **No marketplace screenshots.** The listing "Features" and "Overview" sections support inline images; none yet. Future increment when the UX is more photogenic.
- **Not yet published.** This release only makes the package marketplace-ready. Actual publish requires the user-level Publisher account + PAT described in the runbook above.
