# v0.8.34 — single atomic write path for `~/.lvdcp/config.yaml`

**Date:** 2026-04-24
**Previous:** [v0.8.33 — `ctx registry prune` with dry-run default](2026-04-24-v0.8.33-registry-prune.md)
**Type:** Bug fix / refactor (unify two divergent writers)
**Risk:** Low — the atomic version replaces a non-atomic one; every call site now strictly safer than before.

## Problem

v0.8.33 added an atomic `save_config()` to `libs/core/projects_config.py` for the new `ctx registry prune --yes` destructive path — tmp file → `fsync` → `rename`. But the repo also had a **second, older** `save_config()` in `apps/agent/config.py` — a simple `path.write_text()` with no fsync, no tmp, no atomicity — that was the writer used by every other code path:

- **Daemon scan completion** (`update_last_scan` via `apps.agent.config`) — every scan on every project.
- **Project add/remove** (`add_project`, `remove_project`) — every first-time scan, every unregister.
- **CLI setup wizard** (`apps/cli/commands/setup.py`).
- **UI settings form** (`apps/ui/routes/settings.py`).

So the user's `~/.lvdcp/config.yaml` was being written non-atomically dozens of times per day via the daemon and UI paths, while the prune path — used maybe once a month — was the only one with crash safety. A power loss or OOM kill mid-write could leave a truncated YAML, which `load_config()` would fail to parse on the next daemon boot, wiping out the entire project registry.

The divergence also meant two places to maintain the same logic, and the local `apps/agent` version shadowed the shared one in `__all__` — meaning anyone `from apps.agent.config import save_config` got the worse implementation.

## Solution

`apps/agent/config.py` now re-exports `save_config` from `libs.core.projects_config` instead of defining its own. Every writer goes through the atomic path. The one feature the old local version had — `chmod 0o600` — moves into the shared implementation, wrapped in `contextlib.suppress(OSError)` so filesystems without POSIX perms (FAT, some network mounts) don't crash the write.

The shared implementation also picks up `yaml.safe_dump(..., allow_unicode=True)` from the v0.8.33 path — which means Cyrillic / emoji project names (`/Users/me/Проект_X5`) now render as readable YAML instead of `\uXXXX` escape sequences.

## Before / after

**Before** (two implementations):
```python
# libs/core/projects_config.py  — atomic, v0.8.33
def save_config(path, config):  # tmp + fsync + rename
    ...

# apps/agent/config.py  — non-atomic, shadows the above in __all__
def save_config(path, cfg):
    path.write_text(yaml.safe_dump(cfg.model_dump(mode="json"), sort_keys=False))
    with contextlib.suppress(OSError):
        path.chmod(0o600)
```

**After** (one):
```python
# libs/core/projects_config.py  — atomic + chmod + allow_unicode
def save_config(path, config):
    # tmp + fsync + rename + chmod 0o600 (best-effort)

# apps/agent/config.py
from libs.core.projects_config import save_config  # re-export
```

## New invariants

Five new tests lock the contract:

1. **`save_config_applies_owner_only_perms`** — `path.stat().st_mode & 0o777 == 0o600` on POSIX (skipped on Windows).
2. **`save_config_preserves_original_on_rename_failure`** — simulate `Path.replace` raising `OSError("disk full")`, assert the pre-existing file bytes are byte-identical and the temp file is cleaned up. Locks the crash-atomicity property.
3. **`save_config_preserves_unicode_project_names`** — a project root named `проект-рф` round-trips through YAML without escape sequences. Locks the `allow_unicode=True` decision.
4. **`save_config_survives_chmod_failure`** — patch `Path.chmod` to raise, assert the save still succeeds. Locks the defense-in-depth semantics of the chmod call.
5. **`save_config_is_single_source_of_truth`** — `apps.agent.config.save_config is libs.core.projects_config.save_config`. This one-liner would catch any future regression that reintroduces a second writer.

Plus the three `save_config` tests from v0.8.33 (round-trip, no `*.tmp` leak, atomic overwrite) continue to pass.

## Rejected alternatives

- **Keep both implementations.** Divergence invites drift; the non-atomic path is strictly worse. No reason to preserve it.
- **Promote the non-atomic one.** Loses fsync + rename atomicity — the whole reason v0.8.33 needed the new path.
- **Use `fcntl.flock()` for exclusive access.** Overkill — the registry is single-writer (desktop agent). The atomic-rename pattern is enough. Flock would also not work across Docker volumes on some filesystems.
- **Write directly to `~/.lvdcp/config.yaml`.** Non-atomic by definition — a crash mid-write leaves corrupted YAML.
- **Move `add_project`/`remove_project`/`update_last_scan` down to `libs/core`.** Would pull the `datetime.now()` clock and the domain-level "register a project" semantics into the core layer, violating the minimum-deps rule (CLAUDE.md: "libs/core зависит только от stdlib + pydantic"). The current split — atomic primitive in `libs/core`, domain helpers in `apps/agent` — respects the layering.

## Scope locks

- No change to the registry file schema.
- No change to any public signature (`save_config(path, config)` unchanged, `add_project`/`remove_project`/`update_last_scan` unchanged).
- No new CLI surface — this is a bug fix, not a feature.
- No migration required — users don't need to do anything; the first write after upgrade is atomic.

## Follow-ups explicitly NOT done here

- `ctx registry clean` — on-disk cleanup of orphaned `.context/` directories for pruned roots.
- Prevent auto-registration of transient worktree paths in `add_project` itself — a structural fix that would prevent the noise that prompted v0.8.31–v0.8.33. Worth doing, but risks regressing intentional worktree scans (e.g. running `ctx scan` inside a worktree to test the changes before merge). Design needs more thought than this bug fix.
- Generalize the atomic-write helper to other files the daemon writes (`.context/cache.db` is SQLite so already atomic; `~/.lvdcp/claude_usage.db` same; `*.bak` write during prune is write-once so atomicity is moot).
