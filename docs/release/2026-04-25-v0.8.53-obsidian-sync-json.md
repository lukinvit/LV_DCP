# v0.8.53 — `ctx obsidian sync --json` for scriptable vault writes

**Date:** 2026-04-25
**Tag:** `v0.8.53`
**Theme:** CLI scriptability rollout — operator write surface (Obsidian)

## Summary

Adds `--json` to `ctx obsidian sync`, exposing the full `SyncReport`
payload as a single structured object so nightly cron jobs, deployment
hooks, and operational dashboards can consume per-project sync results
without parsing the human-readable footer.

This closes the Obsidian operator surface:

- **Read side** (v0.8.50): `ctx obsidian status --json` — "what's in
  the vault" tri-state contract
- **Write side** (this ship): `ctx obsidian sync --json` — "what just
  changed in the vault" per-sync payload

The pair lets a script `sync` a project, parse the `SyncReport` to see
how many pages moved, then `status` to verify the project subdir is
present — all with `jq` instead of regex-on-stdout.

## Schema

`ctx obsidian sync --json` returns a single JSON object that exactly
mirrors the `SyncReport` dataclass plus invocation-parameter
round-trip fields:

```json
{
  "vault": "/abs/path/to/vault",
  "project": "/abs/path/to/project",
  "project_name": "MyProject",
  "pages_written": 42,
  "pages_deleted": 0,
  "duration_seconds": 1.83,
  "errors": []
}
```

### Why this exact shape?

- **`vault` + `project` round-tripped.** Same v0.8.48 / v0.8.50 /
  v0.8.52 precedent for "did this run actually target the input I
  intended" — catches `--vault` typos and confirms the project root
  the publisher actually saw, without echoing arbitrary input flags
  back. A nightly script that fans out across N projects can confirm
  per-result which project produced each row.
- **`project_name` mirrored separately from `project`.** The text
  view's "Synced X to Y" message uses `project_path.name` as the
  vault subdirectory; mirroring it in JSON lets a consumer locate
  the written subdirectory under `vault/Projects/<project_name>/`
  without re-deriving the basename. Same string the publisher used
  internally.
- **`pages_written` is the canonical "did this sync do work"
  signal.** `jq -e '.pages_written > 0'` is the natural CI guard
  for "the publisher actually wrote pages, not a no-op". Locked
  by `test_obsidian_sync_json_clean_run_reports_pages_written`.
- **`pages_deleted` separated from `pages_written`.** Drift cleanup
  (source files no longer exist → vault page removed) is a
  semantically distinct event from "fresh write" load, so dashboards
  can split them. Will surface meaningfully once incremental drift
  detection lands; currently always `0` for full-resync semantics.
- **`duration_seconds` as float.** "Did this sync slow down" hygiene
  track via `jq '.duration_seconds'` over time. Float so 1.83s isn't
  silently truncated to 1s.
- **`errors` is always an array (never `null`).** Empty array for a
  clean sync — `jq -e '.errors == []'` works as the CI gate without
  a None-guard. `len(errors) > 0` does **not** flip the exit code:
  the publisher swallows per-page errors into the report rather than
  crashing, and the JSON contract preserves that semantic so
  consumers see the partial-success case as `exit 0` + non-empty
  errors array. Mirrors v0.8.50 `obsidian status` "missing Projects/
  is exit 0 + structured signal" discipline.

## Error-vs-success boundary

The `--json` flag is a **render switch**, not a semantic change:

- **`cache.db` missing** → exit 1, `error: no cache.db found at ...`
  on stderr, no JSON payload on stdout. The same exit-code gate the
  text mode uses. Locked by
  `test_obsidian_sync_json_missing_cache_exits_1_and_does_not_emit_json`.
- **Per-page sync errors** → exit 0, errors collected into the
  `errors` array. Partial success is the publisher's documented
  semantic; the JSON contract preserves it.
- **Full-publisher crash** (e.g. vault permission denied) →
  uncaught exception bubbles out of typer, exit 1. The `--json`
  flag does NOT swallow exceptions into a fake clean payload.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/obsidian/` —
  `SyncReport`, `ObsidianPublisher`, `sync_project` all unchanged.
  `_sync_report_to_json` lives next to the `sync` Typer command in
  `apps/cli/commands/obsidian_cmd.py` — sister-helper to the
  v0.8.46-v0.8.52 CLI-shell payload shapers.
- **Defensive `getattr` access.** The helper reads `SyncReport`
  fields via `getattr(report, "field", default)` rather than
  importing the dataclass and accessing attributes directly, so
  the helper's sole input contract is "object with the documented
  fields" — survives a future `SyncReport` move or rename without
  import-graph churn (same shape discipline as v0.8.45
  `ReconcileReport` shaping).
- **Schema-locked test.** `_SYNC_JSON_KEYS` frozenset in
  `tests/unit/cli/test_obsidian_sync.py` asserts the exact
  seven-key set on every emission. Adding a new field to
  `SyncReport` or the CLI helper without updating the frozenset
  will fail the test instead of silently diverging.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  three-line "Synced X to Y / Pages written: N / Duration: ..."
  footer. Pre-existing pipes that consume `ctx obsidian sync`
  output as plain text (or grep for "Synced") see no diff.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Why this matters

Before this ship: every Obsidian sync orchestration script (nightly
cron via `sync-all`, post-scan hooks, deployment automation) had to
either grep the text footer ("Synced X to Y" / "Pages written: N")
or rely on the exit code alone — losing the per-sync diagnostic
data (page counts, duration, accumulated errors) that the
`SyncReport` already carries internally.

After this ship: `ctx obsidian sync --vault V P --json` returns
the `SyncReport` dataclass shape directly. Same publisher, same
file output, zero text parsing.

This unblocks workflows that v0.8.50 (`obsidian status --json`)
couldn't reach on its own — `status` tells you which projects
exist in the vault, but not whether the last sync did meaningful
work. Now the operator surface is closed at both ends:

- **Operational dashboards**: per-project rolling
  `pages_written` / `duration_seconds` charts to spot vault drift
  or publisher slowdowns
- **Nightly cron with structured logging**:
  `for p in $(ls .); do ctx obsidian sync --vault $V "$p" --json; done | jq -s '.[] | select(.errors != [])'`
  — surface only projects whose sync had non-empty errors
- **Post-scan automation**: a CI hook that runs `ctx scan` then
  `ctx obsidian sync --json | jq -e '.pages_written > 0'` to gate
  the build on "scan results actually propagated to the vault"
- **Closing the operator surface**: read side via
  `obsidian status --json` (v0.8.50) + write side via
  `obsidian sync --json` (this ship) — together they let a script
  manage an Obsidian vault end-to-end with `jq` and exit codes,
  no markdown parsing
