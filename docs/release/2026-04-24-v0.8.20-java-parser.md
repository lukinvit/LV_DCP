# v0.8.20 — Java tree-sitter parser

**2026-04-24**

## Why

The v0.8.x arc closed the `wiki_refresh` story (visibility / interactivity /
telemetry / accessibility). Phase 10 opens on the language-coverage
axis — parser support for JVM / systems languages not yet indexed.
Java is the natural first step: a self-contained deliverable, the
tree-sitter grammar is stable and already a transitive LV_DCP dep,
and Java follows the same `TreeSitterParser` hook-method shape as
the Go and Rust parsers that shipped in Phase 6 — no new
infrastructure, just a new grammar + symbol-type map + role rules.

Projects mixing Python backends with Java integration layers
(Android, Spring-side adapters, legacy JVM glue) now get the same
symbol graph, import relations and coverage accounting that
Python / TypeScript / JavaScript / Go / Rust projects already enjoy.

## Design

`libs/parsers/java.py` is a `TreeSitterParser` subclass, ~150 lines
total. Every non-trivial decision is a hook-method override on the
base class — no new abstractions, no parser-shape drift.

### Symbol type mapping

| Java AST node type | LV_DCP `SymbolType` |
|--------------------|---------------------|
| `class_declaration` | `CLASS` |
| `interface_declaration` | `CLASS` |
| `enum_declaration` | `CLASS` |
| `annotation_type_declaration` | `CLASS` |
| `record_declaration` | `CLASS` |
| `method_declaration` | `METHOD` |
| `constructor_declaration` | `METHOD` |
| `field_declaration` | `VARIABLE` |

All five kinds of type declaration collapse onto `CLASS` — the
downstream graph treats them as top-level named scopes, which
matches how Rust maps `struct` / `enum` / `trait`. Constructors and
regular methods both surface as `METHOD`; the class prefix in
`fq_name` (`Config.Config` vs `Config.process`) is what tells them
apart.

### Field extraction quirk

Java's `field_declaration` AST node doesn't carry a direct `name`
field — the name lives inside a nested `variable_declarator` child.
The base class's default `_extract_name()` does
`node.child_by_field_name("name")` and returns `None` for fields.
`JavaParser._extract_name()` overrides the field case to walk
children until it finds a `variable_declarator`, then reads its
`name` field:

```python
if node.type == "field_declaration":
    for child in node.children:
        if child.type == "variable_declarator":
            name_node = child.child_by_field_name("name")
            if name_node is not None and name_node.text:
                return name_node.text.decode("utf-8", errors="replace")
    return None
return super()._extract_name(node)
```

Multi-declarator fields (`int a, b, c;`) surface only the first
declarator — documented non-goal, matches the base class's one-node
-> one-symbol contract.

### Static-final fields

`public static final int MAX_SIZE = 1024;` could plausibly promote
to `SymbolType.CONSTANT`, but that requires inspecting the field's
modifier children at parse time. v1 maps **all** `field_declaration`
nodes to `VARIABLE` uniformly — simpler, one less hook, and the
test `test_extracts_static_final_field` locks the behaviour in so a
future promotion is a conscious contract break rather than a silent
flip.

### Import relations

`import_declaration` emits `RelationType.IMPORTS` edges with
`dst_type="module"`:

| Java import form | `dst_ref` |
|------------------|-----------|
| `import java.util.List;` (scoped) | `java.util.List` |
| `import java.util.*;` (wildcard) | `java.util` |
| `import static java.util.Arrays.asList;` (static) | `java.util.Arrays.asList` |

Wildcard imports drop the `.*` suffix — graph consumers care about
the package, not whether every symbol is pulled. Static imports
surface the fully qualified symbol path; downstream callers can
split on `.` if they need to distinguish the class from its
imported member.

`_extract_import_ref()` walks the `import_declaration` children
looking for `scoped_identifier` (multi-segment) or plain
`identifier` (single-segment) and decodes whatever it finds. The
grammar places the `*` as a separate sibling token, so reading
`scoped_identifier.text` naturally yields the package path without
the wildcard suffix.

### Role detection

Java projects don't have a single canonical test layout. The
detector walks two signals:

1. **Directory fragments** — `/src/test/java/` (Maven / Gradle
   standard), `/tests/`, `/test/` → `role="test"`.
2. **Filename** (fallback when directory doesn't disambiguate) —
   `*Test.java` / `*Tests.java` (JUnit 4+ conventions), `Test*.java`
   (JUnit 3 convention) → `role="test"`.

Anything else → `role="source"`. The filename fallback ensures
`app/src/ConfigTest.java` classifies correctly even though it's
outside a `/test/` directory.

### Module FQ derivation

Maven / Gradle place sources under `src/main/java/` (or
`src/test/java/`). `_module_fq()` strips the first such prefix it
sees and converts the remainder to dotted form:

| Input path | `_module_fq` output |
|------------|---------------------|
| `src/main/java/com/example/app/Config.java` | `com.example.app.Config` |
| `src/test/java/com/example/app/ConfigTest.java` | `com.example.app.ConfigTest` |
| `modules/foo/src/main/java/com/example/Bar.java` | `com.example.Bar` |
| `com/example/Foo.java` | `com.example.Foo` |
| `Foo.java` | `Foo` |

Nested Gradle multi-module projects — `modules/foo/src/main/java/…`
— work because `.find()` locates the first occurrence of the source
root anywhere in the path, not just at the start.

## Known limitations / non-goals

1. **No `INHERITS` / `IMPLEMENTS` edges** — `class X extends Y` /
   `class X implements I, J` are visible in the AST but v1 doesn't
   emit graph edges for them. Adding an `_extract_inheritance()`
   hook is natural future work.
2. **No `SAME_FILE_CALLS` edges** — method-invocation detection
   would need a call-expression walker; the Python parser has one,
   Go / Rust don't, and Java follows the Go / Rust shape for v1.
3. **Static-final fields map to `VARIABLE`, not `CONSTANT`** —
   documented above, locked by test, deliberate v1 scope.
4. **Multi-declarator fields surface only the first name** — `int a,
   b` yields a single symbol for `a`. Matches the base class's
   one-node-one-symbol contract.
5. **Kotlin / Scala / Groovy are separate work items** — each needs
   its own grammar and hook subclass. v0.8.20 is Java-only.

## Roadmap positioning

First shipped increment of Phase 10 (Java / Kotlin / Swift parsers,
VS Code marketplace, Obsidian nightly sync). Java-only scope was
chosen deliberately: self-contained, grammar already a transitive
dep, and follows the Rust / Go parser shape exactly — lowest-risk
way to prove Phase 10 language expansion is tractable.

Next increments along the same axis: Kotlin (shares the JVM story),
then Swift. Inheritance / call-graph edges for Java are plausible
sub-increments if a driving use case arrives.
