# v0.8.35 — skip transient paths on auto-registration

**Date:** 2026-04-24
**Previous:** [v0.8.34 — single atomic write path for `~/.lvdcp/config.yaml`](2026-04-24-v0.8.34-atomic-config-write.md)
**Type:** Bug fix / design tightening
**Risk:** Low — changes implicit registration only; explicit `ctx watch add` preserves prior behavior via a new `allow_transient=True` flag.

## Problem

`ctx scan` auto-registers any path with a project marker (`.git`, `pyproject.toml`, etc.) into `~/.lvdcp/config.yaml` so the daemon knows to watch it. That heuristic was correct for first-time user projects but wrong for two well-known transient patterns:

1. **Ship-ceremony worktrees** — every release we cut a worktree under `.claude/worktrees/<branch>/`, run `ctx scan` there to test the changes, then delete the worktree. The registry accumulated dead entries for each release.
2. **`sample_repo` test fixture** — shared pytest fixture that lives under `tests/eval/fixtures/sample_repo`. Running the test suite with `ctx scan` in scope registered the fixture permanently.

`libs/status/value_metrics.py` already classified these as "transient" via a private `_is_transient()` helper — but only to separate them on the `/status` dashboard. The *write* path (`apps/agent/config.add_project`) had no such filter, so every scan inside a worktree kept silently growing the registry. v0.8.32 shipped `ctx registry ls` to make the pollution visible; v0.8.33 shipped `ctx registry prune --yes` to clean it up; this release is the structural fix that *prevents* the pollution from happening in the first place.

## Solution

Two small, symmetric changes:

1. **Promote `_is_transient` to public `is_transient` in `libs/core/projects_config`.** Pure pathlib, no new deps — still satisfies the core-layer contract (stdlib + pydantic only). `libs/status/value_metrics` and `libs/status/registry_audit` now import the canonical name. The old private symbol stays as a backward-compat alias inside `value_metrics` so any downstream importer keeps working.
2. **Add `allow_transient: bool = False` to `add_project`.** When the path looks transient and the caller didn't opt in, `add_project` returns silently. The `ctx watch add <path>` CLI — explicit user intent — passes `allow_transient=True`. The `ctx scan` auto-register path (`apps/cli/commands/scan._auto_register`) keeps the default and quietly skips worktree/fixture paths.

Both changes respect the `apps/* → libs/*` layering: the classifier is in `libs/core` (readable by both write and read sides), the domain helper remains in `apps/agent/config` where the single-writer rule lives.

## Why a flag, not a hard block

`ctx watch add ./.claude/worktrees/v0.9.0-something` is an unusual but legitimate workflow: a developer may deliberately want the daemon to index a worktree while debugging the ship-ceremony itself. Silently ignoring an explicit command would be a worse UX bug than the noise we're preventing. The flag makes the intent explicit at every call site.

## New invariants

Four new tests (plus one rewrite):

1. **`test_add_project_skips_worktree_paths_by_default`** — default-off filter drops `.claude/worktrees/*` paths.
2. **`test_add_project_skips_sample_repo_fixture`** — default-off filter drops `sample_repo` basename paths.
3. **`test_add_project_registers_transient_when_explicit`** — `allow_transient=True` overrides the filter.
4. **`test_add_project_registers_real_project_without_flag`** — regression guard that normal paths still register.
5. **`test_auto_register_skips_worktree_paths`** (new in `test_scan_auto_register.py`) — locks the CLI wiring: `ctx scan` must never grow the registry from inside a worktree.
6. **`test_watch_add_registers_worktree_path_explicitly`** (new `test_watch_cmd.py`) — locks the other direction: `ctx watch add` must still accept a worktree when the user asks for it.

Existing `value_metrics` classifier tests were renamed (`_is_transient` → `is_transient`) and moved to import from `libs.core.projects_config` to discourage reaching into the private-ish status module.

## Rejected alternatives

- **Silently register and mark the entry `kind=transient`.** Still pollutes the YAML and still requires `ctx registry prune` housekeeping. The point is to not write in the first place.
- **Filter only `.claude/worktrees/` and leave `sample_repo` alone.** Both patterns produce the same noise; gating one and not the other would just force us to prune the other next week.
- **Hard-block all transient paths with a stderr warning.** Breaks the `ctx watch add <worktree>` debugging workflow. The flag is cheaper.
- **Move the filter into `ctx scan._auto_register` only.** Would leave `apps/ui` or future callers re-implementing it. Putting the default at the `add_project` layer makes "don't register junk" a registry-wide invariant, not a one-caller heuristic.
- **Generalize to "only register paths with `.git`".** Too strict — LV_DCP intentionally supports non-Git projects (hg, plain directories with `pyproject.toml`). The transient classifier is narrow by design.

## Scope locks

- No registry schema change.
- No change to `ctx watch add`'s observable behavior on non-transient paths.
- No new CLI surface.
- No migration — users with existing transient entries can clean them with `ctx registry prune --yes`, which already shipped in v0.8.33.

## Follow-ups explicitly NOT done here

- Teach `ctx registry prune` to run opportunistically when transient entries are detected during `ctx status`. The dry-run-by-default contract from v0.8.33 is the right UX; automating deletion would violate it.
- Extend the transient classifier to Docker scratch mounts (`/tmp/...`, `/private/var/folders/...`). Those don't currently pollute the registry in practice; revisit if users report it.
- Surface a one-line hint when `add_project` silently skips a transient path during `ctx watch add`. Current behavior already refuses via the silent filter; a hint would require a logger in `libs/core`, which is stdlib-only. Worth re-evaluating when structlog moves into core.
