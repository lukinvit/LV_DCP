# v0.8.39 — Dashboard surfaces tombstone counter

**Date:** 2026-04-25
**Previous:** [v0.8.38 — `ctx registry prune --json` for scriptable cleanup](2026-04-25-v0.8.38-prune-json.md)
**Type:** Feature / UX (registry hygiene — discoverability via dashboard)
**Risk:** Low — additive default-zero field on `ValueMetrics`; pure read; UI is one inline conditional.

## Problem

v0.8.36 added `ctx registry prune --missing` and v0.8.37 added `ctx registry ls --missing`, but both live in the CLI. A user opening the dashboard at `localhost:8765` has no signal that their registry has accumulated tombstones — orphaned entries pointing at deleted worktrees or moved project folders. The hygiene loop is "remember to run a CLI command", which is exactly the kind of friction we already solved for the transient-vs-real classification in v0.8.31:

```
5/12  projects active
12 files indexed · +3 transient
```

That `+3 transient` subtext is what makes the CLI's `ls --kind transient` discoverable from the UI. We need the same passive nudge for tombstones.

## Solution

Add a fourth additive segment to the existing "projects active" card subtext:

```
5/12  projects active
12 files indexed · +3 transient · +2 tombstones
```

The `+2 tombstones` span carries a tooltip: *"Registered roots whose directory no longer exists on disk. Run `ctx registry prune --missing` to clean up."* — the discovery loop is closed without needing a separate UI element.

Library layer: `ValueMetrics` gets one new default-zero field, `projects_missing_count: int = 0`. The aggregator's existing `for entry in projects` loop adds one branch:

```python
if not entry.root.exists():
    missing_count += 1
```

That's the entire data-layer change. No new SQL, no new file walks — the existence check is a single `stat` per registered root, amortized inside a loop that already does per-entry work (`is_transient`, sqlite open, etc.).

UI layer: extend the existing transient-subtext conditional to chain a second optional segment. The render is a single line of Jinja with two independent `{% if %}` blocks — neither, one, the other, or both.

```jinja
<div style="font-size:11px;color:#999;">
    {{ metrics.total_files_in_projects }} files indexed{% if metrics.projects_transient_total > 0 %} &middot; +{{ metrics.projects_transient_total }} transient{% endif %}{% if metrics.projects_missing_count > 0 %} &middot; <span title="...">+{{ metrics.projects_missing_count }} tombstones</span>{% endif %}
</div>
```

## Schema

`ValueMetrics` gains exactly one field:

```python
projects_missing_count: int = 0
```

Default-zero, additive — every existing consumer of `ValueMetrics` (dashboard partial, integration tests, any future JSON export) continues to work without code change. The counter is intentionally *not* split by kind (real-tombstone vs transient-tombstone) — the prune action is the same, so the UX nudge is the same.

## Why a single counter, not two

Tombstone classification is independent of why-it-was-registered classification:

- A real user project moved to a new folder → tombstone, prune candidate.
- A worktree finished its lifecycle and got deleted → tombstone, prune candidate.

Both have the same recovery action: `ctx registry prune --missing`. Splitting the counter into `tombstones_real` / `tombstones_transient` would force the user to learn the distinction before acting on a fix that doesn't care about it. One counter, one tooltip, one CLI invocation — minimum cognitive surface.

## New invariants

Four new tests in `tests/unit/status/test_value_metrics.py`:

- **`test_collect_value_metrics_zero_missing_when_all_roots_present`** — when every registered root exists on disk, `projects_missing_count == 0`. Confirms the default-zero field stays at zero in the happy path.
- **`test_collect_value_metrics_counts_missing_root`** — registering a path that was never created on disk (or was deleted post-registration) increments the counter. Confirms the `not entry.root.exists()` predicate fires.
- **`test_collect_value_metrics_counts_missing_regardless_of_kind`** — a missing transient worktree and a missing real project both count toward `projects_missing_count`. Confirms the counter is independent of classification (single counter, not split).
- **`test_collect_value_metrics_empty_registry_has_zero_missing`** — empty registry → `projects_missing_count == 0`. Guards against divide-by-zero or sentinel surprises in the dashboard.

The 13 existing tests in `test_value_metrics.py` continue to pass unmodified — the added field is purely additive.

## Rejected alternatives

- **Split the counter into `tombstones_real` / `tombstones_transient`.** Forces the user to learn a distinction before acting on a fix that doesn't care about it. Same prune command works for both kinds.
- **Add a separate "Registry hygiene" card on the dashboard.** Real estate cost without a payoff — there's nothing else to put on it. The "projects active" card is already the natural home for a project-counter signal.
- **Show the tombstone count even when zero (e.g., `12 files indexed · +0 tombstones`).** Visual noise. The "passive nudge" pattern only fires when there's actually something to act on, mirroring the v0.8.31 transient subtext discipline.
- **Make the tombstone span clickable to trigger prune.** Tempting, but a destructive mutation behind a single click is exactly the pattern v0.8.33 explicitly locked out of the MCP/web surface. CLI-only for prune is the v0.8.33 invariant; the dashboard can nudge but cannot mutate.
- **Color the tombstone segment red.** Premature design — wait for user feedback on whether this counter is useful before tuning its visual weight. v0.8.39 ships the data plumbing and a neutral grey segment; visual emphasis can land in a follow-up if it earns its own release.

## Scope locks

- Library shape — only one default-zero field added to `ValueMetrics`. No removed fields, no renamed fields, no signature change on `collect_value_metrics`.
- No new SQL, no new file walks — the FS check rides inside the existing per-entry loop.
- CLI surface — unchanged. The dashboard surfaces the data; the CLI commands (`ls --missing`, `prune --missing`, `prune --missing --json`) already shipped in v0.8.36–v0.8.38.
- No MCP surface — the dashboard nudges, the CLI mutates. Same lock as v0.8.33.
- Visual layout — the new segment composes inline into the existing subtext line; no new card, no grid reflow, no width change.

## Follow-ups explicitly NOT done here

- A "1-click prune" button on the dashboard. Would violate the v0.8.33 lock that destructive operations are CLI-only. The tooltip already names the right command.
- An aggregator-level breakdown of which kinds of paths are tombstoned (e.g., "+2 tombstones (1 worktree, 1 user project)"). Useful diagnostic but not motivated by a current scripting need; the user can run `ctx registry ls --missing` to get the per-row detail.
- Coloring the tombstone segment differently from the transient segment. Same deferral as in v0.8.37 / v0.8.38 — global TTY/visual policy decision, not a one-flag drop-in. Wait for "is this segment actually useful" feedback first.
- A status badge on the topbar (e.g., red dot when `projects_missing_count > 0`). Potential follow-up but adds global UI surface; the inline subtext is the minimum-viable nudge that lives entirely inside the existing "projects active" card.
