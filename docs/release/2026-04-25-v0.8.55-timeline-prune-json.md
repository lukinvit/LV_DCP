# v0.8.55 — `ctx timeline prune --json` for scriptable timeline maintenance

**Date:** 2026-04-25
**Tag:** `v0.8.55`
**Theme:** CLI scriptability rollout — operator write surface (timeline maintenance)

## Summary

Adds `--json` to `ctx timeline prune`, exposing the prune result as a
single structured object so timeline-maintenance cron jobs, retention-
policy enforcement scripts, and operational dashboards can consume the
deletion count and round-trip the invocation parameters without parsing
the human `prune: deleted N events (...)` line.

This closes the timeline operator surface:

- **Read side**: `ctx timeline status --json` (pre-existing) — "what's
  in the timeline" tri-state contract
- **Reconciliation side** (v0.8.45): `ctx timeline reconcile --json` —
  "which events got newly flagged as orphaned"
- **Write side** (this ship): `ctx timeline prune --json` — "what
  just got hard-deleted from the timeline"

The triplet lets a script `reconcile` to flag stale events, then `prune
--json` to actually remove them, and finally `status --json` to verify
the resulting event-count baseline — all with `jq` and exit codes
instead of three different prose-grepping patterns.

## Schema

`ctx timeline prune --json` returns a single JSON object with five
fields — round-trips the invocation parameters plus the raw deletion
count:

```json
{
  "project_root": "/abs/path/to/project",
  "store_path": "/abs/path/to/timeline.db",
  "older_than_days": 90,
  "include_live": false,
  "deleted": 42
}
```

### Why this exact shape?

- **`project_root` round-tripped (resolved absolute path).** Same v0.8.48 /
  v0.8.50 / v0.8.52 / v0.8.53 / v0.8.54 precedent for "did this run
  actually target the input I intended". A nightly cron that fans out
  across N projects can confirm per-result which project produced
  each row.
- **`store_path` round-tripped.** The timeline DB lives at a platform
  cache path resolved at runtime (`LVDCP_TIMELINE_DB` env var,
  `--store` override, or platform default). Round-tripping it means
  a script can verify "this prune actually targeted the DB I think
  it did" without re-deriving the resolution rules.
- **`older_than_days` and `include_live` round-tripped.** Same
  precedent — confirms the cutoff and the destructive-vs-routine
  flag landed as intended. A future change to the default (`90`)
  would surface in the JSON output rather than silently changing
  cron behavior.
- **`deleted` is the canonical "did this prune do work" signal.**
  `jq -e '.deleted > 0'` is the natural CI guard for "the prune
  actually removed events". Locked by
  `test_prune_json_emits_well_formed_object` and
  `test_prune_json_include_live_round_trips_flag_and_count`.

## Error-vs-success boundary

The `--json` flag is a **render switch**, not a semantic change:

- **`--older-than 0` (or negative)** → exit 2 with `prune: --older-than
  must be positive` on stderr, no JSON payload on stdout. The same
  exit-code gate the text mode uses. Locked by
  `test_prune_json_validation_error_exits_two_no_payload`.
- **Successful prune (deleted == 0)** → exit 0 with the JSON payload
  showing `"deleted": 0`. Same v0.8.45 / v0.8.49 / v0.8.50 discipline
  of "no work done is still a successful run, surface the zero count
  not an error".

## Implementation discipline

- **CLI-only** change. No mutation to `libs/symbol_timeline/` —
  `prune_events` signature unchanged. The JSON dict is built inline
  in `apps/cli/commands/timeline_cmd.py::prune_cmd` next to the
  existing text-mode emission, sister to v0.8.45 `reconcile_cmd`'s
  `_reconcile_report_to_json` helper.
- **Inline dict, not a helper.** Unlike v0.8.45 reconcile (which has
  a `ReconcileReport` dataclass mirrored 1:1 by a helper), `prune`
  has no upstream dataclass — `prune_events` returns a bare `int`.
  Building the dict inline keeps the schema-as-source-of-truth at
  the command site, matching v0.8.38 `registry prune --json`'s
  inline-dict pattern.
- **Schema-locked test.** `_TIMELINE_PRUNE_JSON_KEYS` frozenset in
  `tests/unit/cli/test_timeline_cmd.py` asserts the exact five-key
  set on every emission. Adding a new field to the inline dict
  without updating the frozenset will fail the test instead of
  silently diverging.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  single-line `prune: deleted N events (orphaned only|orphaned + live,
  older than Nd)` footer. Pre-existing pipes that consume `ctx
  timeline prune` output as plain text (or grep for "deleted") see
  no diff. Locked by `test_prune_text_output_unchanged`.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.
- **Store-close discipline preserved.** `store.db_path` is captured
  inside the `try` block before `store.close()` in the `finally` —
  the read happens inside the with-resource discipline so a future
  refactor that invalidates the attribute on close doesn't silently
  break the JSON shape.

| Test | Mode | Asserts |
|---|---|---|
| `test_prune_text_output_unchanged` | text | `prune: deleted N events (orphaned only, ...)` line preserved bytewise; no JSON leak via `pytest.raises(json.JSONDecodeError)` |
| `test_prune_json_emits_well_formed_object` | json | `_TIMELINE_PRUNE_JSON_KEYS` schema lock; `project_root`/`store_path` round-trip; `older_than_days == 1`, `include_live is False`, `deleted == 1` (orphaned-only default mode) |
| `test_prune_json_include_live_round_trips_flag_and_count` | json | `--include-live` round-trips as `True` in the payload; full deletion count (4 events, not just 1 orphaned) |

## Why this matters

Before this ship: every timeline-maintenance script (nightly cron
running `ctx timeline prune --older-than 90` to bound DB growth,
fleet retention-policy enforcement folding per-project deletion
counts into a hygiene dashboard, post-reconcile cleanup hooks gating
on "did this prune actually remove the orphans we just flagged") had
to either grep the text footer (`deleted N events`) or rely on the
exit code alone — losing both the deletion count and the
round-tripped flags (`older_than_days`, `include_live`) that confirm
the prune was called as intended.

After this ship: `ctx timeline prune --json` returns a structured
payload directly. Same hard-delete semantic, same DB writes, zero
text parsing.

This unblocks workflows that v0.8.45 (`timeline reconcile --json`)
and the pre-existing `timeline status --json` couldn't reach on
their own — `reconcile` flags orphans, `status` shows the resulting
counts, but neither closes the loop on the actual destructive
operation that frees DB space. Now the timeline operator surface
is closed end-to-end:

- **Nightly retention cron**: `for proj in $(ctx watch list --json | jq -r '.[].root'); do ctx timeline prune --project "$proj" --older-than 90 --json; done | jq -s 'map({project: .project_root, freed: .deleted}) | sort_by(-.freed)'` — surfaces top-N projects by deletion volume
- **Post-reconcile cleanup hook**: `ctx timeline reconcile --project P --json | jq -e '.orphaned_newly_flagged > 0' && ctx timeline prune --project P --older-than 30 --json | jq -e '.deleted > 0'` — gate the prune on whether the reconcile actually flagged anything
- **Operational dashboards**: per-project rolling `deleted` charts to spot timeline-DB growth anomalies; aggregate `deleted` sums for fleet hygiene reporting
- **Closing the timeline operator surface**: read side via `timeline status --json`, reconciliation via `timeline reconcile --json` (v0.8.45), write side via `timeline prune --json` (this ship) — together they let a script manage timeline DBs end-to-end with `jq` and exit codes, no markdown parsing
