# v0.8.15 — toast-render telemetry on the wiki_refresh panel

**2026-04-24**

## Why

The v0.8.9 crash toast, the v0.8.11 recovery toast, and the v0.8.14 "Retry
now" escape hatch have been in production for five releases without a
single metric about whether they actually fire. From server logs we can
tell a refresh failed, and from a user's browser we can tell a toast
appeared — but we had no way to answer three operational questions:

1. Which projects flap their refresh binary often enough to surface
   crash toasts? (The red toast is supposed to be rare; a slug that
   ships one every few hours deserves a look at the refresh pipeline,
   not at the UI.)
2. On recoveries, is the 30 s auto-poll doing the work, or are users
   hitting the v0.8.14 Retry button? Near-zero manual share would
   suggest the button is invisible; 90 %+ manual share would suggest
   the poll cadence is too slow.
3. Does the crash-toast path even fire in practice, or are most
   crashes landing while the dashboard is closed (so the OOB flash is
   wasted DOM the user never sees)?

Without instrumentation we've been tuning the UX blind. This release
adds the minimum viable signal: one structured log event per render
of each toast, with enough fields to answer all three questions from
log aggregation alone.

## Shipped

**`apps/ui/routes/project.py` — two log events + a trigger classifier.**

- New constant `_RETRY_SOURCE_HEADER = "X-LV-DCP-Retry-Source"` alongside
  the existing `_WAS_DEGRADED_HEADER`.
- New helper `_classify_recovery_trigger(request)` returns `"manual"`
  when the request carries `X-LV-DCP-Retry-Source: manual`, `"poll"`
  otherwise. Conservative default: an unknown header value stays
  `"poll"`, so a garbled header can't inflate the manual-click count.
- The `wiki_refresh_fragment` success branch now computes
  `show_crash_toast` and `show_recovery_toast` into locals (they were
  inline kwargs before), and when either is True emits
  `log.info("ui.wiki_refresh.toast.rendered", extra={...})` with:
  - `event`: literal `"ui.wiki_refresh.toast.rendered"` (duplicated in
    `extra` so structured-log pipelines that key on `event` rather
    than `msg` can filter on it without a message parse).
  - `slug`: the project slug.
  - `kind`: `"crash"` or `"recovery"`.
  - `trigger` (recovery only): `"poll"` or `"manual"`.

**`apps/ui/templates/partials/wiki_refresh.html.j2` — Retry button
carries the new marker.**

The v0.8.14 Retry button's `hx-headers` gains a second key:

```
hx-headers='{"X-LV-DCP-Was-Degraded": "true", "X-LV-DCP-Retry-Source": "manual"}'
```

The auto-poll wrapper is deliberately unchanged — it still sends only
the degraded marker. The route treats that asymmetry as the "trigger=
poll" signal, so there's no need to wire a second attribute into every
poll site.

## Tests

Five new integration tests in `tests/integration/test_ui_wiki_refresh.py`
using pytest's built-in `caplog` fixture:

- `test_crash_toast_render_emits_telemetry_event` — seeds `exit_code=1`,
  fires an `HX-Request` poll, asserts exactly one `toast.rendered` log
  with `kind="crash"`, `slug=<seeded>`, and no `trigger` field.
- `test_recovery_toast_via_poll_emits_trigger_poll` — seeds a clean last
  run, sends only `X-LV-DCP-Was-Degraded: true`, asserts
  `kind="recovery"`, `trigger="poll"`.
- `test_recovery_toast_via_manual_retry_emits_trigger_manual` — same
  seed, adds `X-LV-DCP-Retry-Source: manual`, asserts
  `trigger="manual"`.
- `test_no_telemetry_emitted_on_idle_poll` — ordinary poll on a healthy
  panel emits **zero** `toast.rendered` records. Prevents future
  refactors from accidentally firing the event on every clean tick.
- `test_retry_button_carries_retry_source_header` — DOM-level contract:
  the Retry button's JSON has both keys, and only the button carries
  the `Retry-Source` marker (the polling wrapper must not, otherwise
  trigger classification breaks).

One existing test (`test_retry_now_button_carries_was_degraded_marker_
for_recovery_toast`, v0.8.14) was updated: it used to count occurrences
of the exact `hx-headers` attribute string, which now differs between
wrapper and button because the button's JSON gained a second key. The
assertion now counts the `"X-LV-DCP-Was-Degraded": "true"` key/value
substring, which still appears twice. The test's intent — marker
parity across auto-poll and manual-click paths — is unchanged.

Full suite: **1176 passed, 1 skipped, 1 deselected** (up from 1171).

## Design notes

**Why `log.info` with `extra=` and not structlog?** The route uses
stdlib `logging.getLogger(__name__)` and every other log call in this
module does the same. Switching one call to structlog would create an
inconsistent mix; switching the whole module would blow the scope of
a pure-observability release. The `extra={}` pattern writes keyword
fields into `LogRecord.__dict__`, which both structlog adapters and
the OpenTelemetry logging bridge pick up automatically — so when we
do migrate, these events move without a template change.

**Why two headers instead of one?** We could have folded trigger
classification into a single multi-value header (`X-LV-DCP-Refresh-
State: was-degraded,manual-retry`) but it would couple two orthogonal
concepts. `X-LV-DCP-Was-Degraded` means "this fetch came from a
degraded wrapper and expects a recovery toast on success"; that's a
*contract* between the partial and the route, and is the same for auto
and manual paths. `X-LV-DCP-Retry-Source` means "who triggered this
fetch" and is purely observability. Keeping them separate means the
recovery-toast contract is unchanged — v0.8.15 can be backed out by
deleting the trigger classifier and the two log calls, with the
template change surviving as a no-op marker.

**Why no dismiss-side telemetry?** The dismiss button is a pure
`onclick="this.parentElement.remove()"` DOM operation — there's no
server round-trip, so a server log would need a JS beacon (`navigator.
sendBeacon`) or an HTMX-wrapped dismiss endpoint to observe it.
Either would cost more than the signal is worth at current scale:
dismiss-rate only matters once we have a non-trivial impression count
to compare it against, and we don't yet. When the render-rate data
reveals enough impressions to warrant it, we can add a dismiss event
via the same `extra={}` pattern from a lightweight `POST /api/project/
<slug>/toast-dismiss` endpoint.

**Why `show_crash_toast` and `show_recovery_toast` locals?** Before
v0.8.15 the `TemplateResponse` computed both predicates inline in the
kwargs dict. To log whether a toast fired we have to know the result
*before* the template renders, so the predicates had to land in
locals. Side benefit: the route is now a touch easier to read — the
template-render call's context dict doesn't carry two function calls.

**Why does the `kind="crash"` record not carry a `trigger`?** Crash
toasts have no user-initiated path — they fire on the first poll tick
that sees a fresh `last_exit_code != 0`. There's no manual "test
crash" button and shouldn't be. A `trigger` field would have to be a
constant or absent; absent is cheaper and the log schema stays tight.

## Migration / compat

- No schema changes. No config changes.
- The new `X-LV-DCP-Retry-Source` header is client-authored; existing
  deployments of older templates keep working because the route's
  classifier defaults to `"poll"` when the header is missing.
- The old v0.8.14 test assertion counted a full `hx-headers`
  substring; if you've been pinning that string elsewhere (integration
  tests in forks, CI scripts, etc.) update it to count
  `"X-LV-DCP-Was-Degraded": "true"` instead. The contract — both
  wrapper and button carry the marker — is unchanged.
- Log aggregators should filter on `event == "ui.wiki_refresh.toast.
  rendered"` or parse records where `msg == "ui.wiki_refresh.toast.
  rendered"`. Either key works; they're redundant on purpose.

## Known gaps

- **No dismiss-event telemetry.** See Design notes. Deferred until the
  render-rate data justifies a beacon endpoint.
- **No outage-duration signal.** The recovery toast says "recovered"
  without saying "after X seconds of degraded state". A user who
  watched the panel turn yellow for 2 minutes vs. 20 seconds gets the
  same confirmation. Worth adding — likely a `degraded_at` timestamp
  on the polling wrapper (client-set on the first degraded render,
  server-echoed back on recovery) — but out of scope here.
- **`forced-colors` adaptation still pending.** v0.8.13 added
  `prefers-reduced-motion` but not `forced-colors: active`; the green
  and red toast backgrounds become system-`Canvas` in Windows
  high-contrast mode and the white foreground text stays white →
  invisible. Separate release.
- **Crash-toast auto-dismiss still not a thing.** v0.8.12 added it
  for the recovery toast only; the crash toast stays sticky. That's
  mostly correct (bad news deserves acknowledgement) but a user
  switching projects shouldn't have to clear ten stale red banners.
  Separate release.
- **Per-project dashboards still aggregate logs client-side.** Without
  a server-side log reader the telemetry added here only shows up in
  whatever log aggregator the operator wires to the UI worker. A
  future phase can surface this in the admin view directly.
