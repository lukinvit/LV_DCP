# v0.8.61 — `ctx obsidian sync-all --json` for nightly multi-project audit trails

**Date:** 2026-04-25
**Tag:** `v0.8.61`
**Theme:** CLI scriptability rollout — multi-project iteration surface

## Summary

Adds `--json` to `ctx obsidian sync-all`, the nightly multi-project
sync orchestrator. Single-object payload with a top-level summary plus
a per-project `results` array — `{vault, stale_hours, dry_run, synced,
skipped, failed, results}` where each result entry is
`{project_root, outcome, reason, error}`. The same payload shape covers
the success path, the dry-run plan path, and the no-op cases
(`obsidian.enabled=false`, no projects registered) — one schema, every
exit branch.

This is the first multi-project iteration surface to ship `--json`
(prior surfaces shipped per-row arrays for reads or single-object
mutations for writes; `sync-all` is per-project work over a fleet, so
it needs both summary counters AND per-entry results in one payload).

## Schema

`ctx obsidian sync-all --json` returns a single JSON object with seven
top-level keys (locked by `_SYNC_ALL_JSON_KEYS`):

```json
{
  "vault": "/abs/path/to/vault",
  "stale_hours": 24.0,
  "dry_run": false,
  "synced": 2,
  "skipped": 1,
  "failed": 1,
  "results": [
    {"project_root": "/abs/proj1", "outcome": "synced",  "reason": null,                "error": null},
    {"project_root": "/abs/proj2", "outcome": "skipped", "reason": "fresh (3.2h old)",  "error": null},
    {"project_root": "/abs/proj3", "outcome": "skipped", "reason": "no cache.db",       "error": null},
    {"project_root": "/abs/proj4", "outcome": "failed",  "reason": null,                "error": "timeout"}
  ]
}
```

Each `results` entry is a four-key object (locked by
`_SYNC_ALL_RESULT_KEYS`):

- `project_root` — absolute path of the project (round-trips so a
  script can correlate without re-deriving from registry order)
- `outcome` — exactly one of `"synced"` / `"skipped"` / `"failed"`;
  enum-shaped so a `jq -r '.results[] | select(.outcome == "failed")'`
  filter is unambiguous
- `reason` — populated only on `"skipped"` (mirrors the text view's
  `[skip] <root>: <reason>` line); `null` elsewhere
- `error` — populated only on `"failed"` (mirrors `[fail] <root>: <err>`);
  `null` elsewhere

Both `reason` and `error` are explicit `null` rather than missing keys
so consumers can `jq -r '.results[] | .error // empty'` without a
defined-key guard.

## Exit semantics preserved

- All projects synced cleanly → exit 0
- Any project failed → exit 1 (rest still run; one flaky project
  doesn't block the others)
- `obsidian.enabled=false` → exit 0 with empty `results: []` (no-work
  is a successful run; same v0.8.45/v0.8.49/v0.8.50 discipline)
- No projects registered → exit 0 with empty `results: []`
- Missing config / empty `vault_path` → exit 1 to stderr, **no** JSON
  payload on stdout (same v0.8.42-v0.8.60 error-vs-success boundary)

The `--json` flag is a render switch, not a behavior switch. Per-project
markers, subprocess timeouts, and the rest-of-fleet-still-runs
discipline all carry forward unchanged.

## Dry-run path

`--dry-run --json` emits the same schema with `dry_run: true` and every
`outcome` reflecting the *plan* (would-sync / would-skip), no
subprocess invocations, no on-disk markers written. Lets a script
distinguish "this is what would happen" from "this is what did happen"
in one filter:

```bash
ctx obsidian sync-all --dry-run --json | jq -r '.results[] | select(.outcome == "synced") | .project_root'
# → list of projects that would sync if we ran for real
```

Locked by `test_sync_all_json_dry_run_emits_plan_without_subprocess`
which asserts zero subprocess calls and no marker files written.

## Implementation discipline

- **CLI-only** change. Library code (`obsidian.publisher`,
  `_invoke_sync`) untouched.
- **`_run_plan` refactored** to return a fourth element — the
  per-project `results: list[dict[str, object]]` — alongside the
  existing `(synced, skipped, failed)` counters. The text path ignores
  the new return slot and behaves identically; the JSON path consumes
  it. Single source of truth, two render paths.
- **`quiet=as_json`** parameter on `_run_plan` suppresses the
  `[sync] / [skip] / [fail]` per-project prose so stdout stays pure
  data in JSON mode. Same stdout/stderr discipline as v0.8.42-v0.8.60.
- **Schema lock** via two frozensets:
  - `_SYNC_ALL_JSON_KEYS` for the seven top-level keys
  - `_SYNC_ALL_RESULT_KEYS` for the four per-entry keys
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  legacy `[sync] / Done. Synced: N, ...` lines. Pre-existing pipes
  see no diff. Locked by `test_sync_all_text_output_unchanged`.

## Why this matters

Before this ship: every nightly-cron / CI orchestrator wrapping
`ctx obsidian sync-all` had to grep the human chrome
(`[sync] / [skip] / [fail] / Done. Synced: N, ...`) to figure out
which projects synced, which were skipped, and which failed. The
text format is brittle (project paths containing the substring
`[fail]` or `Done.` would collide), and the per-project failure
*reason* was on stderr while the per-project success was on stdout —
no single place to assemble a structured audit trail.

After this ship: `ctx obsidian sync-all --json` returns a single
structured payload with the per-project outcome, reason, and error
populated explicitly. Same exit semantics, same side effects, same
subprocess fan-out, zero text parsing.

This unblocks workflows like:

- **Nightly audit append**: `ctx obsidian sync-all --json | jq -c '{when: now, synced, skipped, failed, results: [.results[] | {project: .project_root, outcome, reason, error}]}' >> nightly_sync_audit.jsonl` — append-only audit trail of every nightly-cron run with per-project provenance preserved
- **Failure-only paging**: `ctx obsidian sync-all --json | jq -e '.failed == 0' || ctx obsidian sync-all --json | jq -r '.results[] | select(.outcome == "failed") | "\(.project_root): \(.error)"' | mail -s "obsidian sync failures" ops@example.com` — page only on actual failures, with the per-project error included so operators can act without re-running
- **Pre-flight plan before bulk re-sync**: `ctx obsidian sync-all --dry-run --stale-hours 0 --json | jq -r '.results[] | select(.outcome == "synced") | .project_root'` — list every project that would sync under a force-re-sync, audit before committing to the fleet-wide run
- **Per-project dashboard tile**: `ctx obsidian sync-all --json | jq -r '.results[] | "\(.project_root)\t\(.outcome)\t\(.reason // .error // "-")"' > /var/www/sync_dashboard.tsv` — feed a per-project status dashboard from a single CLI invocation, no per-project follow-up calls
- **Skipped-rate hygiene**: `ctx obsidian sync-all --json | jq '.skipped / (.synced + .skipped + .failed)'` — track the "fresh skip" rate over time; if it climbs above ~0.95 the cron interval is too tight relative to `--stale-hours`

## Cross-surface schema discipline

`sync-all` is the first multi-project surface to ship `--json` —
prior surfaces split into:

- **Per-row reads**: bare arrays (v0.8.44 memory list, v0.8.49 watch
  list — the read-side row sets)
- **Single-result mutations with exception-on-absent**: bare entry
  (v0.8.54 watch add, v0.8.57 memory accept, v0.8.58 memory reject,
  v0.8.59 timeline enable, v0.8.60 timeline disable)
- **Single-result mutations with no-op-on-absent**: wrapper-with-null
  (v0.8.56 watch remove)
- **Mutations with single count + invocation params**: flat dict
  (v0.8.38 registry prune, v0.8.55 timeline prune)

`sync-all` introduces a fifth pattern — **fleet-iteration with
summary counters + per-entry array** — natural for any future surface
that fans out work across registered projects (e.g. a future
`ctx wiki update --all --json` or `ctx scan --all --json`). The
seven-key top-level + four-key per-entry shape established here is the
template.
