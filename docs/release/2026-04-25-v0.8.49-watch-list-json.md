# v0.8.49 — `ctx watch list --json` for scriptable registered-projects listing

**Date:** 2026-04-25
**Tag:** `v0.8.49`
**Theme:** CLI scriptability rollout — daemon-registry read side

## Summary

Adds `--json` to `ctx watch list`, so the auto-indexing daemon's registered
project set can be enumerated by automation (CI, dashboards, MCP tooling)
without parsing the human-friendly indented path list.

This is the next slice in the rolling `--json` rollout that started in
v0.8.42 (root-level structlog routing) and has now been threaded through
`memory list`, `timeline status`, `timeline reconcile`, `wiki status`,
`wiki lint`, and `wiki cross-project`. With this tick the **watch
subgroup gains its first scriptable read** — a natural pairing with the
`registry ls --json` audit surface that already existed.

## Schema

`ctx watch list --json` returns a **bare JSON array**, one object per
registered project, mirroring `ProjectEntry` 1:1:

```json
[
  {
    "root": "/Users/.../MyProject",
    "registered_at_iso": "2026-04-25T00:00:00Z",
    "last_scan_at_iso": null,
    "last_scan_status": "pending"
  }
]
```

An **empty registry** returns `[]` (not the text-mode `"no projects
registered"` sentinel). Consumers can rely on the array shape
unconditionally and need no special-casing for the empty path.

### Why a bare array, not an envelope?

Same reasoning as `wiki status --json` and `wiki lint --json`:
`ctx watch list` is a per-row read whose natural shape is "a sequence of
project rows". Wrapping in `{"projects": [...]}` would add zero
information and force consumers to dereference a single fixed key — the
classic anti-pattern called out in the v0.8.46 release note. Compare with
`ctx wiki cross-project --json` (v0.8.48), which intentionally emits a
single object because it is a *generate* command with one result.

## Implementation discipline

- **CLI-only** change. No mutation to `apps.agent.config.list_projects`
  or `libs.core.projects_config.ProjectEntry`. The helper
  `_project_to_json` lives in `apps/cli/commands/watch_cmd.py` next to
  `list_cmd` so future schema drift is reviewed against the dataclass at
  the same site.
- **Schema-locked tests.** A `_WATCH_LIST_JSON_KEYS` frozenset asserts
  the exact key set on every row — adding a new field to `ProjectEntry`
  forces an explicit, reviewed change to the JSON shape.
- **Text-mode UX preserved.** Both the empty-registry sentinel and the
  indented path list remain unchanged for human consumers.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Tests

Four new tests in `tests/unit/cli/test_watch_cmd.py`:

| Test | Mode | Asserts |
|---|---|---|
| `test_watch_list_text_output_unchanged_empty_registry` | text | `"no projects registered"` sentinel preserved |
| `test_watch_list_text_output_unchanged_with_projects` | text | indented path list preserved + no JSON leak (`pytest.raises(json.JSONDecodeError)`) |
| `test_watch_list_json_empty_registry_returns_empty_array` | json | `[]` (not the text sentinel) |
| `test_watch_list_json_emits_well_formed_array` | json | bare array, schema-locked keys, ISO timestamps preserved, defaults (`last_scan_status="pending"`, `last_scan_at_iso=None`) round-trip cleanly |

All 6 tests in the file pass (2 pre-existing add-flow + 4 new list-flow).
