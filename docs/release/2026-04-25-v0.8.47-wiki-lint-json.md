# v0.8.47 — `ctx wiki lint --json` for scriptable wiki hygiene gating

**Date:** 2026-04-25
**Previous:** [v0.8.46 — `ctx wiki status --json` for scriptable wiki state per module](2026-04-25-v0.8.46-wiki-status-json.md)
**Type:** UX (CLI scriptability — extends the v0.8.42-v0.8.46 `--json` rollout to the `ctx wiki lint` surface; the natural read-side companion to v0.8.46's `wiki status --json`)
**Risk:** Low — additive flag; existing text path byte-identical; only one new CLI flag; no library signature changes; no on-disk artifact changes; the exit-code gate is preserved unchanged across both modes (no semantic shift, only a render switch).

## Problem

`ctx wiki lint` has been the deterministic-pre-LLM hygiene check for wiki state since the wiki feature shipped — it surfaces five issue classes (orphaned articles, missing articles, stale articles, empty articles, INDEX mismatch) — but its CLI output stayed text-only:

```
  [WARN] pkg.alpha: Stale article: source changed since last generation.
  [WARN] pkg.beta: Missing article: module has source files but no wiki article.
  [ERROR] pkg.gamma: Empty article: file is less than 50 bytes.

1 error(s), 2 warning(s)
```

That's fine for humans, useless for automation that wants to act on the structured findings:

- A pre-commit hook that asserts "no `error`-severity wiki lint findings on this branch" needs `jq -e 'all(.[]; .severity != "error")'`, not a brittle `grep "ERROR"` over prose (which trips on a module path that happens to contain the substring).
- A multi-project hygiene dashboard — `find ~/dev -maxdepth 2 -name '.context' -exec ctx wiki lint {}/.. --json` — wants per-project per-severity counters folded into a single workspace-hygiene view.
- A nightly CI guard surfacing "stale wiki articles per module" needs `jq -r '.[] | select(.severity == "warning" and (.message | startswith("Stale"))) | .module_path'` — which requires the `severity` and `message` as raw enum-like strings, not formatted prefix tokens (`[WARN]`) the regex would have to strip.
- A `ctx wiki update --module` follow-up that targets only the modules with `Missing article` findings needs `jq -r '.[] | select(.message | startswith("Missing")) | .module_path' | xargs ctx wiki update --module` — a structured pipeline impossible with the text view.
- A workspace-wide "wiki hygiene summary" that buckets findings by severity and message-class needs `jq 'group_by(.severity) | map({severity: .[0].severity, count: length})'` — which requires the JSON contract.

`lint_wiki()` in `libs/wiki/lint.py` already returns a clean `list[LintIssue]` where `LintIssue` is a 3-field dataclass (`severity`, `module_path`, `message`) — we just weren't exposing it. Same gap that v0.8.38 closed for `prune`, v0.8.40 for `restore`, v0.8.42 for `scan`, v0.8.43 for `inspect`, v0.8.44 for `memory list`, v0.8.45 for `timeline reconcile`, v0.8.46 for `wiki status`. Time to close it for `wiki lint`.

This release was explicitly listed as a follow-up in the v0.8.46 release note, with the open question called out: *"deserves its own ship with careful thought on whether `--json` should also exit 1 on error rows or only emit the structured payload."* The answer this release commits to: **preserve the exit-code gate across both modes** (see "Exit-code semantics" below).

## Solution

A `--json` flag on `ctx wiki lint` that emits a bare JSON array — one entry per `LintIssue`:

```bash
$ ctx wiki lint /path --json
[
  {
    "severity": "warning",
    "module_path": "pkg.alpha",
    "message": "Stale article: source changed since last generation."
  },
  {
    "severity": "warning",
    "module_path": "pkg.beta",
    "message": "Missing article: module has source files but no wiki article."
  },
  {
    "severity": "error",
    "module_path": "pkg.gamma",
    "message": "Empty article: file is less than 50 bytes."
  }
]
$ echo $?
1
```

### Schema notes

- **Pass-through of the `LintIssue` dataclass.** Adding a field to `LintIssue` (and making `lint_wiki` populate it) is the single point that surfaces it in both text and JSON paths — `_issue_to_json(issue)` is a one-line dict-literal that walks the dataclass fields.
- **`severity` is the raw enum-like string** (`"warning"` / `"error"`) — pass-through from the `LintIssue.severity` field. Lets `jq 'map(select(.severity == "error"))'` work without nested object access. Different from the text view's `[WARN]` / `[ERROR]` formatted prefixes, which are render concerns.
- **`message` is the unformatted human string** — same string the text view shows, minus the `  [WARN] <module_path>: ` prefix. Consumers can parse the leading word (`Orphaned`, `Missing`, `Stale`, `Empty`, `INDEX`) for issue-class bucketing without needing a separate enum field — which would need a new dataclass field, a new contract, and a migration story for every new issue class. The string-prefix protocol is enough for current consumers; promote to a `class` field only when there's a real consumer that needs it.
- **Bare array, not `{"issues": [...], "summary": {error: N, warning: M}}`** — matches the v0.8.32 `ls --json`, v0.8.40 `restore --json`, v0.8.44 `memory list --json`, v0.8.46 `wiki status --json` precedent. Empty list (`[]`) is the contract for "no issues found" — never `null`, never the human-prose `No issues found.` marker. Consumers do `jq length` without a None-guard; summary counts are trivially recoverable via `jq 'group_by(.severity) | map({key: .[0].severity, value: length}) | from_entries'`.
- **No `module_path` namespacing or normalization** — the field carries whatever `lint_wiki` puts in it (which can be a `(project)` sentinel for project-level findings like the missing-`cache.db` error, or an article stem like `pkg.alpha` for per-article findings). Consumers that need the kind classification can `jq 'group_by(.module_path == "(project)")'`.

### Exit-code semantics — the deliberate choice

The v0.8.46 release note flagged this as the open question for the lint slice: *"whether `--json` should also exit 1 on error rows or only emit the structured payload."* This release **preserves the exit-code gate unchanged across both modes** — error rows exit 1 in both text and JSON; warnings-only and clean exits 0 in both.

**Why preserve the gate:**

1. **The exit code is the script's gating mechanism.** A CI guard like `ctx wiki lint . --json | jq ... && echo "ok"` works because `set -e` short-circuits on the lint exit. Stripping the gate from `--json` mode would force every consumer to additionally check the payload (`jq -e 'all(.[]; .severity != "error")'`) — duplicating logic the lint command already performs.
2. **Splitting the contract by render switch is a code-smell.** `--json` is a render concern (how the data is shaped on stdout); the exit-code gate is a semantic concern (does this command succeed or fail). Render switches must not change semantics — same v0.8.42-v0.8.46 discipline that keeps the missing-`cache.db` error path identical in both modes.
3. **The structured payload IS the error detail.** Stripping the gate would push consumers to the brittle pattern of "exit 0 + parse-then-check-keys." Keeping the gate gives them the clean pattern of "exit code is the gate, payload is the diagnostic" — same shape as `eslint`, `mypy`, `ruff`, and every other lint tool with a JSON output mode.

**The single subtle difference vs. the v0.8.42-v0.8.46 pattern:** in those releases the error path emitted a stderr message + exit 1 and stdout was empty (the missing-`cache.db` path). Here the error path emits a **valid JSON payload** on stdout (the structured findings) AND exits 1. That's not a contract split — it's a different error class. `cache.db` missing is an environment error (no data to render); `error`-severity lint findings are a data error (data exists, includes errors). The first deserves stderr + exit 1 + empty stdout; the second deserves payload + exit 1. Same exit-code gate, different stdout shape — which is the right Unix discipline.

### Error path

Missing `cache.db` (project not yet scanned) is handled by `lint_wiki()` itself (it appends a `LintIssue("error", "(project)", "No cache.db found. Run `ctx scan` first.")` and returns) — so in `--json` mode a missing-`cache.db` project surfaces as a single-row JSON array with `severity='error'`, exit 1. Consistent with the "error rows exit 1" contract above.

```bash
$ ctx wiki lint /not/scanned --json
[
  {
    "severity": "error",
    "module_path": "(project)",
    "message": "No cache.db found. Run `ctx scan` first."
  }
]
$ echo $?
1
```

This is intentionally different from the `wiki status --json` v0.8.46 contract where missing-`cache.db` emits stderr + exit 1 + empty stdout. The reason: `lint`'s job is to surface findings as data; "no cache.db" is itself a finding (the project is in a state lint can flag). `status`'s job is to render an existing state; "no cache.db" means there is no state to render — it's an environment error, not a finding. Two commands, two error models, both internally consistent.

## Library / CLI layering

CLI-only change. No new library helpers, no schema modifications to `libs/wiki/lint.py`. `lint_wiki()` already returned the right shape since the wiki feature shipped.

A small `_issue_to_json(issue: LintIssue) -> dict[str, object]` helper in `apps/cli/commands/wiki_cmd.py` builds the per-row payload. Lives next to `lint` (the only consumer). Sister-helper to `_module_to_json` (v0.8.46) — same pattern of per-row JSON shaping in the CLI shell.

## New invariants

Four new tests appended to `tests/unit/cli/test_wiki_cmd.py`:

- **`test_wiki_lint_text_output_unchanged`** — baseline: text path emits the legacy `  [WARN] <module>: <message>` bullet list byte-identically. Locks the v0.8.42-v0.8.46 "text path untouched" promise.
- **`test_wiki_lint_json_emits_well_formed_array_warnings_only`** — bare JSON array, exact key set per entry (`_WIKI_LINT_JSON_KEYS` frozenset = `{severity, module_path, message}`), warnings-only seeded data → exit 0 in both modes (locks the "warnings don't fire the gate" half of the exit-code contract). Also asserts no human-prose markers (`No issues found`, `warning(s)`) leak into the JSON output.
- **`test_wiki_lint_json_empty_returns_bare_list`** — clean project (no issues fire) returns `[]`, not `null`, not the `No issues found.` prose marker. Locks the empty-list contract so consumers can do `jq length` without a None-guard.
- **`test_wiki_lint_json_preserves_exit_code_gate_on_errors`** — error-severity finding (sub-50-byte article triggers the `Empty article` error rule) exits 1 in **both** text and JSON modes. Locks the deliberate exit-code preservation choice — a refactor that strips the `if errors: raise typer.Exit(code=1)` from the JSON path breaks this test instead of silently degrading every CI guard.

Existing 4 wiki tests + every other wiki test passes byte-identically — text path untouched.

## Rejected alternatives

- **Strip the exit-code gate in `--json` mode** ("only emit the structured payload"). Would force every consumer to duplicate the gate logic (`jq -e 'all(.[]; .severity != "error")'`) — and produce a contract split where `--json` and the text mode disagree on what "this command succeeded" means. The v0.8.42-v0.8.46 discipline of "render switch must not change semantics" applies here directly. See "Exit-code semantics" above for the full reasoning.
- **Wrap in `{"issues": [...], "summary": {error: N, warning: M, total: K}}`** — breaks the v0.8.32 / v0.8.40 / v0.8.44 / v0.8.46 bare-array precedent. Summary counts are trivially recoverable via `jq 'group_by(.severity) | map({key: .[0].severity, value: length}) | from_entries'` — no value in baking it into the payload shape.
- **Schema-version the JSON output (`"_schema_version": 1`).** Premature ceremony, same v0.8.38-v0.8.46 argument. Add when there's a real consumer that needs to gate on schema versions.
- **Add a structured `class` field (`"orphaned" | "missing" | "stale" | "empty" | "index_mismatch"`) per row.** The string-prefix protocol on `message` is enough for current consumers — `jq 'select(.message | startswith("Stale"))'` works today. Promoting to a `class` field would need a new dataclass field, a new test invariant per class, and a migration story for every new issue class added to `lint_wiki()`. Defer until a real consumer needs the explicit enum.
- **Add a `--severity error` / `--severity warning` filter flag.** Trivially recoverable via `jq 'map(select(.severity == "error"))'`; promote to a flag only if the pattern repeats across multiple consumers.
- **Add `--json` to `wiki update` / `wiki cross-project` in the same release.** Keep increments small. `lint` is the natural follow-up to v0.8.46's `wiki status` — both are read-side hygiene surfaces with closely related contracts; `update` has mutation semantics with progress streaming that benefits from a separate ship; `cross-project` is the next read-side slice but reads from a different layer (`libs/wiki/cross_project.py`) and deserves its own analysis.
- **Echo the `errors_count` / `warnings_count` totals in the JSON shape.** Inflates the payload with derived data; `jq length` and `jq 'group_by(.severity) | length'` recover both for free. Same v0.8.43 / v0.8.44 / v0.8.46 argument against baking summary stats into the row shape.
- **Make the JSON output suppress warnings unless `--include-warnings` is passed.** Breaks the symmetry with the text view (which surfaces both severities by default) and conflates the "what to render" axis with the "how to render" axis. Filter in `jq` if needed.

## Scope locks

- Only one new CLI flag (`--json`) on `wiki lint`.
- `wiki update`, `wiki status`, `wiki cross-project` subcommands **untouched**.
- `libs/wiki/lint.py::lint_wiki` and the `LintIssue` dataclass **unchanged**.
- No new on-disk artifacts.
- No MCP surface — wiki lint is a hygiene/observability surface for operators and CI, not retrieval context for LLMs.
- Text output **byte-identical** — the new `test_wiki_lint_text_output_unchanged` test locks this.
- Exit-code gate **preserved unchanged** — error rows → exit 1 in both modes; warnings/empty → exit 0 in both modes.
- structlog routing introduced in v0.8.42 (`apps/cli/main.py`) carries forward unchanged — JSON consumers benefit automatically.

## Follow-ups explicitly NOT done here

- `ctx wiki cross-project --json` — natural follow-up read-side slice; structured per-pattern payload would benefit cross-project hygiene dashboards.
- `ctx wiki update --json` — has mutation semantics with per-module progress streaming; deserves a separate ship to think through whether the JSON contract should be NDJSON (streaming progress events) or a single payload at the end (terminal summary). The streaming-vs-terminal question is the same one we'd hit for any long-running command's `--json` mode.
- Promote the `class` enum field on `LintIssue` (orphaned / missing / stale / empty / index_mismatch). String-prefix discipline is enough today; revisit when a consumer needs the explicit field.
- A README badge tracking the JSON-scriptable surface coverage (e.g. "JSON-scriptable: scan ✓, inspect ✓, prune ✓, restore ✓, ls ✓, memory ✓, reconcile ✓, wiki status ✓, wiki lint ✓"). Premature — release notes carry this discoverability and a badge would need maintenance every time another command grows `--json`.
