# v0.8.57 — `ctx memory accept --json` for scriptable memory-review mutations

**Date:** 2026-04-25
**Tag:** `v0.8.57`
**Theme:** CLI scriptability rollout — operator write surface (memory review)

## Summary

Adds `--json` to `ctx memory accept`, exposing the accepted `Memory`
entry as a single structured object so memory-review automation, batch
acceptance scripts, and IDE integrations can consume the post-mutation
state without parsing the human `accepted: <id>  <topic>` line.

This is the first half of closing the memory-review operator surface:

- **Read side** (v0.8.44): `ctx memory list --json` — bare array of all
  memory entries (composes with `--status` for filtering)
- **Accept side** (this ship): `ctx memory accept --json` — single
  object emitting the accepted entry with `status: "accepted"`
- **Reject side** (v0.8.58, next ship): `ctx memory reject --json` —
  symmetric mirror, will close the surface

## Schema

`ctx memory accept --json` returns a single JSON object — same per-row
schema as v0.8.44 `memory list --json` rows:

```json
{
  "id": "mem_abc123",
  "status": "accepted",
  "topic": "Auth flow",
  "tags": [],
  "created_at_iso": "2026-04-25T10:14:00.123456+00:00",
  "created_by": "claude-opus",
  "path": "/abs/path/to/.context/memory/mem_abc123.md"
}
```

### Why reuse the v0.8.44 list-row schema verbatim?

- **Cross-surface schema parity.** A future `Memory` field addition has
  **one** helper (`_memory_to_json`) to bump and **one** test frozenset
  (`_MEMORY_LIST_JSON_KEYS`) to update. Same pattern as v0.8.54 watch
  add reusing v0.8.49 watch list helper.
- **`status: "accepted"` confirms the mutation landed.** The post-
  mutation status round-trips so a script can confirm the state
  transition without a follow-up `list --json` call. Locked by
  `test_memory_accept_json_emits_well_formed_object`.
- **`body` intentionally omitted.** Same v0.8.44 precedent: arbitrary-
  size markdown is recoverable via `cat $(jq -r .path)`. Keeping the
  payload lean lets memory-review pipelines stay cheap.
- **`path` is the canonical recovery handle.** `cat $(jq -r .path)`
  reconstructs the full body when needed.

## Error-vs-success boundary

The `--json` flag is a **render switch**, not a semantic change:

- **Unknown memory id** → exit 2 with `error: <msg>` on stderr, no JSON
  payload on stdout. Same exit-code gate the text mode uses. Locked by
  `test_memory_accept_json_unknown_id_exits_nonzero_no_payload`.
- **Successful accept (idempotent re-accept)** → exit 0 with the JSON
  payload showing `status: "accepted"`. The library `accept_memory`
  has no state-machine guard — accepting an already-accepted memory
  is a no-op write that returns the same entry. Locked by
  `test_memory_accept_json_idempotent_re_accept_returns_accepted_entry`.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/memory/store.py` —
  `accept_memory` signature unchanged. The JSON object is emitted
  inline in `apps/cli/commands/memory_cmd.py::accept_cmd` next to the
  existing text-mode emission.
- **Helper reused, not duplicated.** `_memory_to_json` (introduced
  v0.8.44 for `memory list --json`) emits the single-object payload —
  cross-surface schema lock with no new helper.
- **Schema-locked test.** `_MEMORY_LIST_JSON_KEYS` frozenset in
  `tests/unit/cli/test_memory_cmd.py` (introduced v0.8.44) now locks
  **two** surfaces: list rows and the accept single-object emission.
  Adding a new field requires bumping ONE frozenset.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  single-line `accepted: <id>  <topic>` footer. Pre-existing pipes
  see no diff. Locked by `test_memory_accept_text_output_unchanged`.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Tests

Four new tests in `tests/unit/cli/test_memory_cmd.py`:

| Test | Mode | Asserts |
|---|---|---|
| `test_memory_accept_text_output_unchanged` | text | `accepted: <id>  <topic>` line preserved bytewise; no JSON leak via `pytest.raises(json.JSONDecodeError)` |
| `test_memory_accept_json_emits_well_formed_object` | json | `_MEMORY_LIST_JSON_KEYS` schema lock; `status: "accepted"` round-trip; cross-check via `list --json` confirms the on-disk mutation matches the emitted payload |
| `test_memory_accept_json_unknown_id_exits_nonzero_no_payload` | json + error | unknown id → exit 2, no JSON success-shape on stdout — error-vs-success boundary stays at the exit-code gate |
| `test_memory_accept_json_idempotent_re_accept_returns_accepted_entry` | json | re-accepting an already-accepted memory succeeds (exit 0), emits the same entry, `path` stable across the idempotent re-write |

All 12 tests in `test_memory_cmd.py` pass (8 pre-existing + 4 new).
All 153 tests in `tests/unit/cli/` pass (no regressions).

## Why this matters

Before this ship: every memory-review automation (a daily cron that
accepts proposed memories matching a pattern, an IDE integration that
flips a memory's status from a UI button click and reports back the
new state, a batch-acceptance script that walks a list of memory ids)
had to grep the text footer (`accepted: <id>`) or rely on the exit
code alone — losing the round-tripped `Memory` shape that confirms
which on-disk entry was just mutated and what its post-mutation
state is.

After this ship: `ctx memory accept --json` returns a structured
payload directly. Same mutation semantic, same on-disk write, zero
text parsing.

This unblocks workflows like:

- **Batch acceptance**: `for id in $(ctx memory list --status proposed --json | jq -r '.[] | select(.topic | test("auth")) | .id'); do ctx memory accept "$id" --json; done | jq -s 'map(.id)'` — accept all proposed auth-related memories and collect the resulting ids
- **IDE state confirmation**: a VS Code extension button that calls `ctx memory accept` can render the post-mutation status from the JSON payload directly, without a follow-up `list --json` round-trip
- **Audit log**: `ctx memory accept "$id" --json | jq '{when: .created_at_iso, who: .created_by, topic, path}' >> accept_audit.jsonl` — append-only audit trail of accepted memories with their creation provenance

This unblocks the same workflows v0.8.44 (`memory list --json`)
opened up on the read side — but for the destructive state-flip
operation. v0.8.58 will close the surface end-to-end with the
symmetric `memory reject --json`.
