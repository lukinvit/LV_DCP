# v0.8.41 — `ctx registry ls` surfaces backup-availability footer

**Date:** 2026-04-25
**Previous:** [v0.8.40 — `ctx registry restore` for prune undo](2026-04-25-v0.8.40-registry-restore.md)
**Type:** UX (registry hygiene — recovery handle discoverability)
**Risk:** Low — pure additive footer on text output; JSON unchanged; library adds one read-only function.

## Problem

v0.8.40 shipped `ctx registry restore` as the first-class inverse of `prune --yes`. The restore subcommand reads the sibling `*.bak` snapshot that `prune --yes` writes before every mutation and atomically replaces the live config. That closes the *write-side* asymmetry — but discoverability stayed implicit:

- A user who didn't read the `prune --yes` output (or who ran `prune` weeks ago and forgot) had no way to learn from the `ls` view that a `*.bak` recovery handle existed beside their live config.
- The CLI surface that users naturally reach for first ("show me what's in my registry") was silent about the recovery handle that protects them.
- The restore command was discoverable only via `ctx registry --help`, which a user has to know to run — circular.

The v0.8.40 release note had explicitly listed a dashboard widget for backup discoverability as a follow-up but rejected it ("adds UI surface for a recovery operation that should be deliberate, not nudged. The CLI is the right home"). That left the CLI-side discoverability gap unaddressed.

## Solution

A one-line footer on the `ctx registry ls` text output when (and only when) `~/.lvdcp/config.yaml.bak` exists:

```bash
$ ctx registry ls
NAME      KIND       SCAN  ...
X5_BM     real       yes   ...
LV_DCP    real       yes   ...
worktree  transient  no    ...

(backup: /Users/v/.lvdcp/config.yaml.bak — 3h old — `ctx registry restore` to undo last prune)
```

When no `*.bak` exists, the footer is absent — no clutter for fresh users who have never invoked `prune --yes`.

The footer fires regardless of which `--kind / --stale / --missing` filter was used. The backup is a property of the live config, not of any particular slice — and a user filtering by `--missing` finding zero rows still benefits from knowing recovery is available.

## Library layer

A single new function in `libs/status/registry_audit.py`:

```python
def backup_status(
    config_path: Path,
    *,
    backup_suffix: str = ".bak",
    now: float | None = None,
) -> tuple[Path | None, float | None]:
    """Return (backup_path, age_seconds) for the prune-undo sidecar."""
```

Pure read — never mutates either file. Returns `(None, None)` when no backup exists. `age_seconds` is `mtime`-delta clamped to non-negative for clock-skew safety (NFS, dual-boot, virtualization can transiently skew mtime ahead of `time.time()`). `now` kwarg is injectable for deterministic tests. `backup_suffix` kwarg matches the same default as `prune_stale` / `restore_from_backup` and keeps the door open for a future `--backup-suffix` flag without a library signature change.

## CLI layer

`apps/cli/commands/registry_cmd.py::ls_cmd` calls `backup_status` after `_render_text`, appending a one-line `(backup: ...)` footer when present. A new `_format_backup_age_seconds` helper uses the same compact `<1h` / `Nh` / `Nd` vocabulary as the table's `LASTSCAN` column — the two read consistently in one screen.

JSON output (`ls --json`) is **unaffected** — bare array of audit rows. Backup info is reachable via `restore --json` for scripts (which already returns `backup_path`).

## Why text-only, not JSON

Adding backup info to `ls --json` would force a wrapped `{"projects": [...], "backup": {...}}` schema and break every existing JSON consumer. Scripts that need backup info already have a clean surface in `restore --json`. The footer is a discoverability nudge for human readers — JSON consumers don't need a nudge, they bind to the API directly.

## New invariants

Four library tests in `tests/unit/status/test_registry_audit.py`:

- **`test_backup_status_returns_none_when_no_bak`** — no `*.bak` yields `(None, None)`, not an exception. The footer renderer needs a cheap "is there a backup?" check that doesn't raise.
- **`test_backup_status_returns_path_and_age_when_bak_exists`** — backup present returns absolute path + non-negative age; mtime-delta from injected `now`.
- **`test_backup_status_clamps_negative_age_to_zero`** — clock-skew safety: a future-dated backup mtime returns age 0, not negative.
- **`test_backup_status_respects_custom_suffix`** — `backup_suffix` kwarg honored; default `.bak` and custom `.saved` both work.

Four CLI tests in `tests/unit/cli/test_registry_cmd.py`:

- **`test_registry_ls_text_no_footer_when_no_bak`** — default state for fresh users: no backup line, no `ctx registry restore` mention. No clutter.
- **`test_registry_ls_text_shows_backup_footer_when_bak_exists`** — motivating case: footer surfaces path + `<1h` age tag + `ctx registry restore` command name.
- **`test_registry_ls_json_unaffected_by_backup_presence`** — `--json` stays a bare array even when `*.bak` exists; no backup keys leak into the row schema.
- **`test_registry_ls_footer_composes_with_filters`** — footer fires on `ls --kind real` (and by extension every other filter) when backup exists. Backup is a property of the live config, not the filter slice.

## Rejected alternatives

- **Emit backup info in `ls --json` too.** Would force a wrapped `{projects: [...], backup: {...}}` schema and break every existing JSON consumer. `restore --json` already provides this surface for scripts.
- **Surface backup status only when the `ls` table is empty.** Creates a "footer disappears when you have rows" UX confusion — the recovery handle is equally relevant whether you have 0 or 50 entries.
- **Color the footer differently (e.g., yellow).** Same TTY/`NO_COLOR` global decision deferred since v0.8.37 — wait for a broader colorization ask before introducing one-flag terminal styling.
- **Add a `--no-backup-footer` opt-out flag.** Premature — only suppress when there's a complaint. The footer is one line and only appears when there's something to say.
- **Expose backup info on `ctx status` instead of `ls`.** `status` is a cross-feature dashboard; users investigating registry state are already in `ls`. Footer lives where the user already is.
- **Add a separate `ctx registry status` overview command.** CLI surface bloat for one line of info that fits naturally on `ls`. If a real consolidated status surface is needed later, that's a separate design.
- **Auto-restore prompt on `ls` when backup exists ("Restore now? [y/N]").** Violates the v0.8.33 lock that destructive operations are explicit `--yes`-gated. Footer names the command; user invokes deliberately.

## Scope locks

- Library shape — only one new function added to `registry_audit.py`. No removed/renamed fields, no signature change on `audit_registry`, `is_stale`, `is_missing`, `iso_utc`, `ProjectAudit`.
- `ls --json` output schema **unchanged** — every existing JSON consumer passes byte-identically.
- Text output above the footer **unchanged** — `_render_text` and the table renderer are untouched.
- No registry schema change.
- No new on-disk artifacts.
- No MCP surface — discoverability is a CLI-only concern (same lock as v0.8.33: destructive operations and their inverses stay CLI-only).
- No new CLI flags.
- Dashboard UI untouched — CLI-side discoverability is the right surface for a CLI-only recovery operation, explicitly per v0.8.40's deferral of the dashboard backup widget.

## Follow-ups explicitly NOT done here

- A `--backup-suffix <suffix>` CLI flag that matches the library kwarg, useful for users who keep multiple manually-named backup snapshots. Trivial to add but motivated by zero current driver — the library kwarg is already there for a future ship.
- Showing the **count of entries** in the backup as part of the footer (e.g., `backup: 5 entries — 3h old`). Adds a `load_config` call on every `ls` — currently the footer is `O(1)` in registry size (just an `mtime` stat). Defer until "how many entries does my backup have?" is a real friction point; for now `restore --json | jq .backup_entry_count` answers it.
- A multi-backup history (`*.bak.1`, `*.bak.2`, `*.bak.3` rotated on each prune). Adds disk-space and complexity without a current driver — the current "one backup, overwritten on next prune" rule is simple and reversible.
- Surfacing the backup line on `ctx status`. The dashboard nudge was deferred in v0.8.40; surfacing on `ctx status` would be a partial reversal of that decision without a stronger driver.
- Auto-prompt restore on `ls` when the backup is older than N days (e.g., "your backup is 90 days old — refresh with `prune --yes`?"). Speculative UX without a real friction signal yet.
