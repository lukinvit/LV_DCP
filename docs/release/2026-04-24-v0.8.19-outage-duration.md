# v0.8.19 — outage-duration signal on the recovery toast

**2026-04-24**

## Why

Since v0.8.11 the `wiki_refresh` panel has fired a green recovery
toast on the degraded → normal transition: "Wiki refresh status
recovered — `<slug>` is reachable again. Resuming normal polling."
The copy told the user *that* recovery happened but never *how long*
they were looking at the yellow degraded card. v0.8.15 added
structured-log telemetry (`trigger=manual|poll`) for the same event
but left the user-facing duration story untouched — a documented
gap.

With the gap closed, a user returning to their dashboard after
lunch now sees "reachable again after 1h 14m" instead of a generic
"reachable again". For short blips the copy reads "reachable again
after 2m 17s" / "after 47s" / "after <1s". The distinction between a
30-second hiccup and an hour-long outage is exactly the kind of
signal the recovery toast was meant to carry — v0.8.19 makes it
visible.

## Design

The route has **no server-side state** (no session, no cookie, no
Redis entry) tracking outage starts — the dashboard is stateless
HTMX end-to-end. The outage-start timestamp instead **rides on the
browser** via an HTMX `hx-headers` round-trip, identical to the
v0.8.11 `X-LV-DCP-Was-Degraded` marker and the v0.8.15
`X-LV-DCP-Retry-Source` classifier:

1. **First degraded render** — route's `except` branch stamps
   `degraded_since = int(time.time())` into the template context.
   The degraded wrapper's `hx-headers` JSON includes
   `"X-LV-DCP-Degraded-Since": "<ts>"` alongside
   `"X-LV-DCP-Was-Degraded": "true"`.
2. **Every subsequent degraded tick** (while the outage continues)
   — HTMX echoes the header back on the next poll. The route calls
   `_parse_degraded_since(request)` which returns the original
   timestamp; the new degraded render re-stamps the **same** value
   so the original start time is preserved indefinitely across a
   long outage.
3. **Recovery tick** (degraded → normal, the first successful poll)
   — HTMX echoes the same header. The route parses it, computes
   `outage_seconds = now - since`, formats it via
   `_format_outage_duration(seconds)`, and the template splices the
   resulting label (`"2m 17s"`) into the toast copy:

   ```
   {% if outage_duration_label %}
     is reachable again after {{ outage_duration_label }}. Resuming normal polling.
   {% else %}
     is reachable again. Resuming normal polling.
   {% endif %}
   ```

4. **Manual retry** (the v0.8.14 "Retry now" button) — the button's
   `hx-headers` JSON now carries three keys:
   `X-LV-DCP-Was-Degraded`, `X-LV-DCP-Retry-Source`, and
   `X-LV-DCP-Degraded-Since`. The manual-retry recovery path
   surfaces the duration label identically to the poll-initiated
   recovery path — no split UX.

### Parser contract

`_parse_degraded_since(request) -> int | None` is defensive against
client-controlled garbage:

| Incoming header value | Parser result |
|-----------------------|---------------|
| absent / empty string | `None` |
| non-integer (`"abc"`, `"3.14"`) | `None` |
| zero or negative | `None` |
| future timestamp (> `int(time.time())`) | `None` |
| positive integer ≤ now | the value |

Every `None` result collapses the toast copy to the pre-v0.8.19
fallback — so a browser tab that opened on a pre-v0.8.19 build and
finally polls through a recovery still sees the toast (just without
the duration suffix). Rolling upgrades lose nothing.

### Formatter contract

`_format_outage_duration(seconds) -> str` returns:

| Range | Output | Example |
|-------|--------|---------|
| `< 1` | `"<1s"` | |
| `1-59` | `"Ns"` | `47s` |
| `60-3599`, `secs==0` | `"Xm"` | `3m` |
| `60-3599`, `secs>0` | `"Xm Ys"` | `2m 17s` |
| `>= 3600`, `mins==0` | `"Xh"` | `1h` |
| `>= 3600`, `mins>0` | `"Xh Ym"` | `1h 23m` |

Only the two largest non-zero units are surfaced — long outages
read as `1h 23m`, not `1h 23m 47s`, so the toast stays scannable.

### Telemetry

The existing `ui.wiki_refresh.toast.rendered` log event (kind
`recovery`) gains an `outage_seconds` field alongside the v0.8.15
`trigger` field. Set to the integer duration on a clean round-trip
and `None` when the header was absent / malformed — field shape
stays stable for downstream structured-log consumers.

## Shipped

**`apps/ui/routes/project.py`** — ~100 line net addition:

- New constant `_DEGRADED_SINCE_HEADER = "X-LV-DCP-Degraded-Since"`
  with docstring mirroring the v0.8.11 / v0.8.15 marker constants.
- New helper `_parse_degraded_since(request) -> int | None` with
  the defensive failure-mode table above.
- New helper `_format_outage_duration(seconds) -> str` with the
  range table above.
- `wiki_refresh_fragment`'s `except Exception` branch now stamps
  `degraded_since` into template context (preserving the incoming
  value when valid, falling back to `int(time.time())` otherwise).
- `wiki_refresh_fragment`'s success branch computes
  `outage_seconds` / `outage_duration_label` on the recovery path
  and extends the `ui.wiki_refresh.toast.rendered` log event with
  `outage_seconds`.

**`apps/ui/templates/partials/wiki_refresh.html.j2`** — three
string-level edits:

- Degraded wrapper's `hx-headers` JSON gains a third key:
  `"X-LV-DCP-Degraded-Since": "{{ degraded_since }}"`.
- Retry button's `hx-headers` JSON gains the same key — the button
  now round-trips all three markers.
- Recovery toast's second `<div>` wraps its copy in an
  `{% if outage_duration_label %}` / `{% else %}` split so the
  "reachable again" line gets the duration suffix when available,
  pre-v0.8.19 copy otherwise.

Version bumps on `pyproject.toml`, `apps/vscode/package.json`, and
`apps/ui/templates/base.html.j2` to **0.8.19**.

## Known limitations / non-goals

1. **No seconds-level precision beyond 59 s** — "reachable again
   after 1h 23m 47s" would be noise; only the two largest units
   ship. Design choice, not a bug.
2. **The timestamp is client-echoed** — a malicious tab can inflate
   its own outage label by sending an ancient `X-LV-DCP-Degraded-Since`
   value. Future-dated timestamps are rejected in the parser, but
   a very-old real timestamp is accepted. Outage length is a
   telemetry hint, not a security boundary, so the trade-off is
   explicit: a user who wants to lie to themselves can.
3. **No separate "long outage" escalation** — the copy reads the
   same regardless of duration. Surfacing an additional warning
   glyph / colour step after (say) 10 minutes is plausible future
   work; out of scope for this release.

## Roadmap positioning

Closes the "outage duration on recovery" item documented as
future work in the v0.8.15 telemetry release. The v0.8.x `wiki_refresh`
arc is now complete across four axes:

1. **Visibility** (v0.8.9 crash toast / v0.8.11 recovery toast).
2. **Interactivity** (v0.8.14 Retry button).
3. **Telemetry** (v0.8.15 trigger classification, v0.8.19
   outage_seconds).
4. **Accessibility** (v0.8.13 reduced-motion / v0.8.16 forced-colors
   on toasts / v0.8.17 pulse dot / v0.8.18 progress bar).

Phase 10 work — dashboard crash remediation with a retry-guarded
background re-parse — is the next incremental target.
