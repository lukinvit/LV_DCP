# v0.8.54 — `ctx watch add --json` for scriptable daemon registration

**Date:** 2026-04-25
**Tag:** `v0.8.54`
**Theme:** CLI scriptability rollout — operator write surface (watch daemon)

## Summary

Adds `--json` to `ctx watch add`, exposing the just-registered
`ProjectEntry` as a single structured object so provisioning scripts,
fleet registration tooling, and `ctx scan` follow-on automation can
consume the registration result without parsing the human `added X`
line.

This closes the watch operator surface:

- **Read side** (v0.8.49): `ctx watch list --json` — "what is the daemon
  watching" array contract
- **Write side** (this ship): `ctx watch add --json` — "what just got
  registered" per-call payload

The pair lets a script `add` a project, parse the returned
`registered_at_iso` to detect duplicate-add against wall-clock, then
`list` to verify the full set — all with `jq` instead of regex on
stdout.

## Schema

`ctx watch add --json` returns a single JSON object that exactly
mirrors the `ProjectEntry` dataclass — the **same** schema as one row
of `ctx watch list --json`:

```json
{
  "root": "/abs/path/to/project",
  "registered_at_iso": "2026-04-25T10:42:13.831947+00:00",
  "last_scan_at_iso": null,
  "last_scan_status": "pending"
}
```

### Why this exact shape?

- **1:1 `ProjectEntry` mirror.** Same v0.8.49 `_project_to_json`
  helper renders both surfaces — no shape drift between read and
  write paths is possible by construction.
- **Cross-surface schema lock.** Tests reuse the v0.8.49
  `_WATCH_LIST_JSON_KEYS` frozenset on **both** the `list`-row and
  the `add`-result emissions. A future `ProjectEntry` field addition
  requires bumping ONE frozenset; the test invariant catches divergence.
- **`root` round-tripped (resolved absolute path).** Same v0.8.48 /
  v0.8.50 / v0.8.52 / v0.8.53 precedent for "did this run actually
  target the input I intended". The Typer `resolve_path=True` gate
  on the argument means symlinks, `..`, and relative paths land
  normalized — the consumer sees what the daemon will actually watch,
  not what was typed.
- **`registered_at_iso` is the duplicate-add detector.** A consumer
  comparing the returned `registered_at_iso` against wall-clock can
  distinguish "first registration" from "already-watched, no-op"
  without an extra `list --json` call. Locked by
  `test_watch_add_json_idempotent_returns_existing_entry`.
- **`last_scan_at_iso: null` on fresh registration.** Always `null`
  until the daemon runs the first scan — explicit `null` (not absent
  key) so consumers `jq -e '.last_scan_at_iso == null'` works
  unconditionally.
- **`last_scan_status: "pending"` on fresh registration.** Default
  state for a never-scanned project; flips to `"ok"` / `"error"` once
  the daemon runs. Lets a script gate "is this project ready to
  query" without inspecting timestamps.

## Idempotent re-add semantic in JSON

`ctx watch add` is idempotent — re-adding an already-registered path
is a silent no-op in `add_project` (see `apps/agent/config.py`). The
JSON contract preserves this:

- **First add** → returns the freshly-written entry with the just-now
  `registered_at_iso`.
- **Second add of the same path** → returns the **same** entry, with
  the **original** `registered_at_iso` (not a fresh timestamp).

This is a stronger contract than the text mode (which emits an
indistinguishable `added X` line either way). Locked by
`test_watch_add_json_idempotent_returns_existing_entry`.

## Error-vs-success boundary

The `--json` flag is a **render switch**, not a semantic change:

- **Path does not exist** → exit 2 (Typer argument validation), no
  JSON payload on stdout. The Typer `exists=True` gate fires before
  the command body, so the JSON branch is never reached. Locked by
  `test_watch_add_json_error_path_exits_nonzero_no_payload`.
- **Path exists but read-back fails** (vanishingly rare —
  concurrent `remove_project` between write and read, or a corrupt
  config) → exit 1 with `error: registered ... but cannot read it
  back from ...` on stderr, no JSON payload. The defensive
  `next(...)` guard in `apps/cli/commands/watch_cmd.py::add` ensures
  the JSON contract never emits a malformed payload.
- **Idempotent re-add** → exit 0, returns the existing entry (see
  above).

Same v0.8.42–v0.8.53 discipline of "exit code is the gate, structured
payload is for the success path".

## `allow_transient=True` preserved

Per v0.8.35 (existing behavior, unchanged): `ctx watch add` is an
explicit user intent, so the CLI passes `allow_transient=True` to
`add_project` — the daemon will follow ship-ceremony worktrees and
the `sample_repo` fixture if the user explicitly asks. Only implicit
auto-registration (from `ctx scan` on an unregistered path) filters
transient roots; explicit invocation wins. The JSON mode preserves
this — no path filtering changes between text and JSON modes.

## Implementation discipline

- **CLI-only** change. No mutation to `apps/agent/config.py` —
  `ProjectEntry`, `add_project`, `list_projects` all unchanged.
  `as_json` parameter and the readback branch live next to the
  existing `add` Typer command in `apps/cli/commands/watch_cmd.py`,
  reusing the v0.8.49 `_project_to_json` helper.
- **Helper reuse, not duplication.** The v0.8.49 `_project_to_json`
  serializer is the single source of truth for the JSON shape on
  both surfaces. No copy-paste, no shape drift possible.
- **Schema-locked tests.** The reused `_WATCH_LIST_JSON_KEYS`
  frozenset in `tests/unit/cli/test_watch_cmd.py` asserts the
  same four-key set on both the v0.8.49 list-row emission and the
  v0.8.54 add-result emission. Adding a new field to `ProjectEntry`
  or the helper without updating the frozenset will fail BOTH test
  groups instead of silently diverging.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  single-line `added <path>` footer. Pre-existing pipes that consume
  `ctx watch add` output as plain text (or grep for `added`) see no
  diff. Locked by `test_watch_add_text_output_unchanged`.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.

## Why this matters

Before this ship: every `ctx watch add` orchestration script (fleet
provisioning, `ctx scan` follow-on auto-registration tooling, CI
hooks that register a project before invoking the daemon) had to
either grep the text footer (`added /abs/path`) or invoke a separate
`ctx watch list --json` afterwards to confirm the registration
landed. Either way, the `registered_at_iso` (the duplicate-add
detector) was not directly accessible from the `add` call.

After this ship: `ctx watch add /path --json` returns the
`ProjectEntry` shape directly. Same registration semantic, same
config write, zero text parsing.

This unblocks workflows that v0.8.49 (`watch list --json`) couldn't
reach on its own — `list` tells you what the daemon is currently
watching, but not whether the just-issued `add` was a fresh
registration or a no-op duplicate. Now the watch operator surface
is closed at both ends:

- **Fleet provisioning**: `for d in projects/*; do ctx watch add "$d" --json | jq -r '.root'; done`
  — capture exactly what the daemon will watch, normalized to absolute paths
- **Idempotent re-provisioning**: a script that re-runs `add --json` on every boot can detect "this was already registered" via `registered_at_iso` against `now - boot_time` without an extra `list` call
- **`ctx scan` follow-on**: a CI hook that runs `ctx scan` then `ctx watch add --json | jq -e '.last_scan_status == "pending"'` to confirm the daemon will pick up the next scheduled scan
- **Closing the operator surface**: read side via
  `watch list --json` (v0.8.49) + write side via
  `watch add --json` (this ship) — together they let a script
  manage daemon registrations end-to-end with `jq` and exit codes,
  no text parsing

## Deferred

- **`ctx watch remove --json`** — destructive command, different shape
  decision (return removed entry vs. `{root, removed: bool}`),
  small-increment discipline says ship one thing at a time. The
  read-side and additive-write-side surface is sufficient for the
  v0.8.x scriptability rollout; removal scripting can land
  separately when the shape question is settled.
