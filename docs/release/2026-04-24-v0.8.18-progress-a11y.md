# v0.8.18 — a11y parity for the `wiki_refresh` progress bar

**2026-04-24**

## Why

The Running card on the `wiki_refresh` panel renders a horizontal
module-progress bar whenever the worker reports a `modules_total` —
a 6 px-tall track (`<div>` with `background:#bbdefb`) containing a
fill (`<div>` with `background:#1976d2`, width computed from
`modules_done / modules_total`, with a `transition:width 0.3s` so
the bar animates the jump on each 2 s poll tick as the module count
advances).

Two a11y scenarios were broken:

1. **`forced-colors: active`** (Windows high-contrast mode and
   equivalent OS-level accessibility palettes). Both the `#bbdefb`
   track and the `#1976d2` fill get remapped by the UA to system
   `Canvas` — which equals the page background. The 6 px bar
   becomes an invisible sliver and the module-progress signal is
   lost. v0.8.17 closed this gap for the `lvdcp-pulse` running dot;
   the progress bar was left unadapted — documented as an explicit
   known gap in the v0.8.17 release notes.

2. **`prefers-reduced-motion: reduce`** (vestibular sensitivity,
   screen-reader users, anyone who's opted out of OS-level motion).
   The fill's `transition:width 0.3s` fires on every poll tick
   whenever `modules_done` advances — so a 100-module refresh shows
   ~100 short slide animations at 2 s intervals. That's exactly the
   kind of repeated motion that `prefers-reduced-motion` is meant
   to suppress. v0.8.13 / v0.8.17 honoured this preference for the
   recovery toast fade and the pulse dot; the bar transition was
   left running.

With v0.8.18 shipped the `wiki_refresh` subsystem is **a11y-complete
across all four animated / coloured interactive elements**: recovery
toast fade (v0.8.13 + v0.8.16), crash toast container (v0.8.16),
running-status pulse dot (v0.8.17), and the progress bar (this
release).

## Shipped

**`apps/ui/templates/partials/wiki_refresh.html.j2`** — two-class
template edit + three rule additions inside the existing
`<style>` block nested in `{% if wr.in_progress %}`:

- The progress-bar track `<div>` gains `class="lvdcp-progress-track"`
  alongside its existing inline `style`.
- The progress-bar fill `<div>` gains `class="lvdcp-progress-fill"`
  alongside its existing inline `style`.
- The `<style>` block that already carried the v0.8.17
  pulse-dot rules grows three sibling rules:
  ```css
  @media (prefers-reduced-motion: reduce) {
      .lvdcp-progress-fill { transition: none !important; }
  }
  @media (forced-colors: active) {
      .lvdcp-progress-track { border: 1px solid CanvasText !important; }
      .lvdcp-progress-fill { background: CanvasText !important; transition: none !important; }
  }
  ```

All rules stay scoped inside `{% if wr.in_progress %}` — idle /
clean / crashed / degraded responses ship zero progress-bar CSS.

## Tests

Seven new integration tests in `tests/integration/test_ui_wiki_refresh.py`:

- `test_progress_bar_carries_track_class_hook` — contract lock:
  track `<div>` renders with `class="lvdcp-progress-track"` on the
  Running card when `modules_total` is set.
- `test_progress_bar_carries_fill_class_hook` — contract lock: fill
  `<div>` renders with `class="lvdcp-progress-fill"`.
- `test_progress_bar_respects_forced_colors` — the
  `@media (forced-colors: active)` block ships with the exact
  track rule (`border: 1px solid CanvasText !important`) and the
  exact fill rule (`background: CanvasText !important; transition:
  none !important`).
- `test_progress_bar_respects_prefers_reduced_motion` — the
  `@media (prefers-reduced-motion: reduce)` block ships the fill's
  `transition: none !important` cancel rule.
- `test_no_progress_bar_dom_when_no_modules_total` — when the lock
  doesn't carry `modules_total` (e.g. early "starting" phase), the
  Running card still renders but the bar's class-hook `<div>`s are
  omitted per the `{% if wr.modules_total %}` guard. The a11y CSS
  block still ships (selectors are inert without their elements) —
  this test asserts the DOM contract, not the CSS payload.
- `test_no_progress_a11y_css_when_not_running` — clean / crashed /
  idle responses carry neither the class-hook DOM nor the
  progress-specific media-query rules (entire `<style>` block is
  scoped to the Running branch).
- `test_progress_bar_transition_still_present_for_running_card` —
  regression guard: the default `transition:width 0.3s` stays on
  the fill element; v0.8.18 adapts it, doesn't remove it.

Full suite: **1192 passed, 1 skipped, 1 deselected** (up from 1185).

## Design notes

**Why `background: CanvasText` on the fill and a border on the track?**
The track is a container (6 px tall, full panel width). In forced-
colors the inline `#bbdefb` gets remapped to `Canvas` — identical to
the page background. A 1 px `CanvasText` border (borders are exempt
from the UA color remap per spec) restores a visible container
frame without having to pick a specific fill colour. The inner fill
div is the *indicator*; it needs to be a solid dense shape against
the (now-Canvas-coloured) track, so `background: CanvasText` fills
it entirely with the system foreground token. Combined, the track
reads as a bordered rectangle and the fill reads as a solid bar
inside it — same mental model as the non-forced-colors render, just
using system tokens instead of blues.

**Why cancel the transition in forced-colors as well as in
prefers-reduced-motion?** Consistency with the v0.8.17 dot rule:
when in doubt, stay static in any a11y mode. A user on forced-
colors is almost certainly an a11y user, and the transition is
decorative — the width change alone conveys progress.

**Why `!important` on all rules?** Same reasoning as v0.8.13 /
v0.8.16 / v0.8.17: the inline `transition:width 0.3s` and
`background:#1976d2` / `background:#bbdefb` on the elements carry
specificity 1000 (inline style) which beats a plain class rule (10).
`!important` bumps the class rule above the inline style without
needing to break the inline shorthand into longhand and update
dozens of unrelated tests.

**Why one `<style>` block for pulse-dot AND progress-bar, not two?**
Both subsystems share the same outer guard (`{% if wr.in_progress
%}`) and ship together on every Running render. Merging them into a
single block saves one `<style>` open/close tag and one `@media`
duplication per media query. The cost: when `modules_total` is
absent, the progress-bar selectors are still present in the
stylesheet (but inert — they don't match any element). ~150 bytes
of dead-weight CSS in a specific edge-case; the payload-reduction
from the merged `@media` blocks more than pays for it on the common
case.

**Why *not* also adapt the track's `background:#bbdefb` explicitly
with a forced-colors override?** The UA remaps it to `Canvas`
automatically (the default remap for `background` author values),
which is exactly what we want — a track that matches the page
background, so the `CanvasText` fill reads as a solid shape against
it. Pinning `background: Canvas !important` would be redundant.

## Migration / compat

- Pure template change. No route change, no schema change, no
  behaviour change for users without a11y preferences.
- Adds two class attributes to two existing `<div>`s. The inline
  `style` attributes on both are unchanged — zero risk of breaking
  CSS pinning on the elements or on any external selector that
  might pin them by tag + parent structure.
- Existing tests that match `background:#1976d2` / `background:#bbdefb`
  / `transition:width 0.3s` keep passing — the new regression guard
  `test_progress_bar_transition_still_present_for_running_card`
  explicitly locks the inline transition string in place.

## Known gaps (carry forward)

- **`wiki_refresh` a11y is now complete.** All animated / coloured
  interactive elements on the panel honour both `prefers-reduced-
  motion` and `forced-colors` where relevant. No more a11y debt on
  this subsystem.
- **No outage-duration signal on recovery toast.** Still deferred
  from v0.8.15 / v0.8.16 — actual user-visible UX improvement
  candidate, likely v0.8.19.
- **No dismiss-event telemetry.** Blocked on render-data volume
  (v0.8.15 reasoning).
- **Crash toast still sticky, no auto-dismiss.** Project-switching
  users accumulate stale red banners — good v0.8.19 or v0.8.20
  candidate.
