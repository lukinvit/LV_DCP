# v0.8.36 — `ctx registry prune --missing` for absent-root cleanup

**Date:** 2026-04-25
**Previous:** [v0.8.35 — skip transient paths on auto-register](2026-04-24-v0.8.35-skip-transient-auto-register.md)
**Type:** Feature / UX (registry hygiene)
**Risk:** Low — adds a new flag; default behavior unchanged; inherits the dry-run + `*.bak` + `--yes` safety rails from v0.8.33.

## Problem

`ctx registry prune` (v0.8.33) gates pruning on two predicates:

1. **Staleness** — `packs_total=0 AND last_scan>30d` (never-scanned entries treated as stale by default).
2. **Kind** — `transient` (worktrees, fixtures), `real`, or `all`.

That works for entries whose *scanning* is dormant but whose *directory* still exists on disk. But there's a common failure mode v0.8.33 doesn't catch quickly: a worktree that got deleted the same day (`git worktree remove ship/v0.8.35-foo`, `rm -rf /tmp/scratch`, moved a project folder across a drive). The registry entry becomes a **tombstone** — the path is gone, the staleness gate doesn't fire for 30 days, and `ctx status` keeps pinging a dead filesystem root on every wake-up.

v0.8.35 prevents *new* transient entries from accruing, but existing tombstones (and any real-project paths a user moves / deletes) still need a catch-up command.

## Solution

A new `--missing` flag on `ctx registry prune` that treats **root directory absence** as an independent eligibility predicate:

- `Path(entry.root).exists() is False` → candidate.
- Ignores `--older-than` (a gone root is gone regardless of scan age).
- Ignores `--kind` (a gone transient worktree and a gone real project are equally dead).
- Preserves the existing safety contract: dry-run default, `--yes` gate, pre-mutation `*.bak` sidecar, atomic `save_config`.

```bash
# preview which registry entries reference deleted directories
$ ctx registry prune --missing

# actually clean them up
$ ctx registry prune --missing --yes
```

Library layer: `plan_prune` and `prune_stale` in `libs/status/registry_prune.py` gain a `missing_only: bool = False` kwarg. When `True`, the candidate filter checks `Path(row.root).exists()` and ignores `older_than_days` / `kind`.

CLI layer: `apps/cli/commands/registry_cmd.py::prune_cmd` adds `--missing` alongside the existing `--older-than` / `--kind` / `--yes` flags. Output copy clarifies `missing root` vs `>Nd` when reporting "nothing to do" or listing candidates.

## Why an independent predicate, not a new `--kind=missing`

`--kind` is a classification axis (real vs transient vs all), not an eligibility axis. A missing-root check cuts across classifications — a gone-from-disk path is pruneable regardless of whether it started as a worktree or a real project. Overloading `--kind` would force the user to pick one interpretation and lose the ability to combine "gone from disk" with "also transient" in a future compose-mode (not built yet, but easy from here).

## New invariants

Five new library-layer tests + three new CLI-layer tests lock the contract:

- **`test_plan_prune_missing_lists_absent_roots`** — the predicate catches deleted worktree paths while sparing live directories.
- **`test_prune_missing_ignores_older_than_gate`** — a worktree registered this morning, scanned at 9 a.m., deleted at noon: staleness-mode returns `[]`, missing-mode returns it.
- **`test_prune_missing_ignores_kind_filter`** — both a gone-real and a gone-transient entry are caught even with `kind="transient"`.
- **`test_prune_missing_applies_with_backup`** — `apply=True` mutates exactly the gone entries and writes `*.bak` with the pre-mutation bytes.
- **`test_prune_missing_no_candidates_is_graceful`** — when every root exists, missing-mode is a pure read.
- **`test_registry_prune_missing_dry_run_lists_deleted_root`** — CLI dry-run prints `missing root` + `dry-run` and leaves config untouched.
- **`test_registry_prune_missing_yes_removes_and_backs_up`** — CLI `--yes` applies + reports `REMOVED` + `backup saved`.
- **`test_registry_prune_missing_nothing_to_do`** — CLI no-op path prints the `missing root` variant of the nothing-to-do message.

## Rejected alternatives

- **Prune automatically on every `ctx status` / daemon wake.** Hidden destructive behavior. v0.8.33's dry-run-by-default contract is the right UX — the user chooses when to apply.
- **Check `.context/cache.db` existence instead of root directory.** A user who runs `rm -rf /path/to/proj/.context` but keeps the project would lose their registry entry too; not the intent. Root-directory absence is the correct signal.
- **Add a `--stale-or-missing` combined mode.** Overspecified — the two predicates ship as independent flags, and a future compose-mode can be added if a driving use case arrives.
- **Use `Path(root).is_dir()` instead of `.exists()`.** A regular-file tombstone (someone `touch`'d a file where a directory used to be) is still a dead registry entry worth cleaning; `.exists()` catches both.
- **Gate `--missing` behind `--kind all` so the user has to be explicit.** Adds friction for the common case (just want to clean up dead worktree entries) with no safety win — `--yes` is already the mutation gate.

## Scope locks

- No registry schema change.
- No public signature change on read-side helpers (`audit_registry`, `is_stale`, `list_projects`, `save_config`).
- Default behavior of `ctx registry prune` (without `--missing`) is unchanged.
- No new on-disk artifacts — only the `*.bak` sidecar from v0.8.33's contract.
- No `ctx registry ls --missing` yet (symmetric read-side surface is a plausible follow-up, but nobody's asked for it and `ls --json | jq` already handles the use case).

## Follow-ups explicitly NOT done here

- Symmetric `ctx registry ls --missing` read-side filter. Trivial to add but no driving signal yet.
- On-disk cleanup of orphaned `.context/cache.db` directories whose registry entry has been pruned (the "ctx registry clean" item from v0.8.34 follow-ups). Still needs a design call on what "orphan" means when the project root itself is gone vs merely unregistered.
- Extending `Path.exists()` to handle unreachable mount points (NFS, external drives) gracefully — currently a disconnected mount looks the same as a deleted directory. Low priority; dry-run default mitigates.
