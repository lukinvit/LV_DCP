# v0.8.31 — value metrics: separate transient projects from real ones

**Date:** 2026-04-24
**Previous:** [v0.8.30 — VS Code extension activation-event narrowing](2026-04-24-v0.8.30-vscode-activation.md)
**Type:** Read-side observability fix
**Risk:** Low (additive fields on `ValueMetrics`, single UI card rebind)

## Problem

The dashboard "projects active" card has been misleading for weeks. It reads straight off the registry — every `lvdcp_scan` adds an entry to `~/.lvdcp/config.yaml`, and that includes:

- **Ship-ceremony worktrees.** Every `v0.8.x-*` release spawns a temporary `.claude/worktrees/<name>/` clone, we scan it to dogfood the pipeline, and the entry sticks in the registry long after the worktree is gone. After 5+ ship rounds there were 50+ dead worktree rows.
- **Test fixture `sample_repo`.** Unit tests scan `tests/fixtures/sample_repo` on every worktree, so the same path shows up once per branch. Pack counts from test runs (~236 as of today) dominate any per-project ranking.

Result: the card read **18/79** ("18 active out of 79 registered"). The honest number — "real user projects that served a pack" — was closer to **5/8**. The noise made the card useless for answering "is LV_DCP actually being adopted?"

## Solution

Additive split. Existing `projects_total` / `projects_active` semantics unchanged (count everything, preserves any downstream consumers). Add four new fields:

```python
projects_real_active: int       # excludes worktrees & fixtures, served ≥1 pack
projects_real_total: int        # excludes worktrees & fixtures, registered
projects_transient_active: int  # worktree/fixture that served ≥1 pack
projects_transient_total: int   # worktree/fixture registered
```

Dashboard rebinds the "projects active" card to render `projects_real_active / projects_real_total` and, when `projects_transient_total > 0`, appends a `· +N transient` subtext so the noise is still visible but not leading.

## Classifier — `_is_transient(root: Path) -> bool`

Two rules, both narrow on purpose:

1. **Worktree marker** — any path containing consecutive segments `(".claude", "worktrees")`. We walk `root.parts` rather than checking `".claude/worktrees" in str(root)` because Pathlib normalises separators cross-platform and substring matches would false-positive on something like `.claude_worktrees_archive/...`.
2. **Fixture name** — `root.name in frozenset({"sample_repo"})`. A pure basename check. Any real user project that happens to be named `sample_repo` would be misclassified — the risk is small enough (zero evidence anyone ships a repo with that name) that we accept it. If this bites, we'll add a "real-project override" flag to the registry entry later.

Rejected classifiers:
- `".claude" in parts` alone — too broad, catches legitimate `.claude/` config directories inside real projects.
- Path-age heuristic (entry older than N days with 0 packs) — fires on dormant real projects too (`LV_Presentation` had 1 pack in 3 weeks), would incorrectly reclassify them as transient.
- Registry-flag at scan time — would require a migration and a hook in `lvdcp_scan`; we can do it later if the path-based classifier proves insufficient.

## Scope locks

- **No pruning.** We only _classify_, we do not delete. Auto-prune of dead worktrees is a separate, destructive operation and needs explicit user confirmation — flagged as a follow-up.
- **`projects_total` / `projects_active` stay backward compatible.** Any caller relying on the total count still gets it. Only the dashboard card changes its binding.
- **Template-only UI change.** No JS, no CSS, no new endpoints. One card on `apps/ui/templates/partials/value_metrics.html.j2`.
- **MCP `lvdcp_status` untouched.** It surfaces workspace-level counters, not adoption metrics, and the extra fields don't belong on its contract until we have a user story for them.

## Testing

New `tests/unit/status/test_value_metrics.py` — 6 cases:

1. `_is_transient` detects `.claude/worktrees/*` path segments
2. `_is_transient` detects `sample_repo` basename
3. `_is_transient` rejects a normal project path
4. `_is_transient` rejects `.claude/hooks/` (unrelated `.claude` subdirs)
5. `collect_value_metrics` with mixed real+transient roots produces correct four-counter split
6. `collect_value_metrics` with only real roots sets transient counters to 0

Each test seeds a minimal `.context/cache.db` with `files/symbols/relations/retrieval_traces` tables and a synthetic config — no network, no fixtures, fast (full suite <0.1s).

## Expected impact (measured from my own workspace)

Before: `18/79 active`, grossly diluted by 50+ dead worktrees and the `sample_repo` fixture.
After: card will read approximately `5/8 active · +61 transient` — the real-project ratio becomes visible, the transient count stays auditable for cleanup decisions.

## Follow-ups explicitly NOT done here

- Auto-prune of registry entries where `packs_served == 0` and `last_scan_at_iso > 30d` ago. Destructive, needs user ACK.
- Hiding transient rows in the per-project coverage table — left visible because debugging scans often needs them.
- Exposing the new fields on `lvdcp_status` MCP tool — no user story yet.
