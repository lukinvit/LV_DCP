# v0.8.17 — a11y parity for the `lvdcp-pulse` running indicator

**2026-04-24**

## Why

The blue "Running" card on the `wiki_refresh` panel (v0.8.7) has had
a pulsing status dot since day one — a 10×10 round `<span>` with an
infinite `@keyframes lvdcp-pulse 1.5s infinite ease-in-out` animation
cycling opacity between 1 and 0.4. Every browser renders it fine in
standard themes; two accessibility scenarios were broken:

1. **`prefers-reduced-motion: reduce`** — users who've opted out of
   OS-level motion (vestibular sensitivity, screen-reader users,
   anyone who just doesn't want a perpetual flicker in the corner
   of their eye) were still subjected to the infinite animation.
   v0.8.13 closed this gap for the recovery toast fade; the pulse
   dot was left animating.
2. **`forced-colors: active`** — Windows high-contrast mode remaps
   the dot's inline `background:#1976d2` (solid blue) to system
   `Canvas`, which *is* the page background. The dot becomes
   invisible. v0.8.16 closed this gap for the crash + recovery
   toasts; the pulse dot was left un-adapted.

This release closes both gaps. After v0.8.17 the wiki_refresh
subsystem is a11y-complete: all three animated elements (recovery
toast fade, crash toast container, running-status dot) honour
`prefers-reduced-motion` and `forced-colors: active`.

## Shipped

**`apps/ui/templates/partials/wiki_refresh.html.j2`** — one-line
template change + extended `<style>` block inside the existing
`{% if wr.in_progress %}` branch:

- The pulsing `<span>` gains `class="lvdcp-pulse-dot"` alongside
  its existing inline `style`.
- The `<style>` block that carries the `@keyframes lvdcp-pulse`
  definition grows two sibling rules:
  ```css
  @media (prefers-reduced-motion: reduce) {
      .lvdcp-pulse-dot { animation: none !important; opacity: 1 !important; }
  }
  @media (forced-colors: active) {
      .lvdcp-pulse-dot { animation: none !important; opacity: 1 !important; background: CanvasText !important; }
  }
  ```

Both rules stay scoped inside `{% if wr.in_progress %}` — the
keyframes never ship on idle/clean/crashed responses, and the
media queries don't either. No style bleed, no dead CSS.

## Tests

Five new integration tests in `tests/integration/test_ui_wiki_refresh.py`:

- `test_running_dot_carries_pulse_class_hook` — contract lock:
  the span renders with `class="lvdcp-pulse-dot"` on the Running
  card.
- `test_running_dot_respects_prefers_reduced_motion` — the
  `@media (prefers-reduced-motion: reduce)` rule is shipped
  with the exact body that cancels `animation` and pins
  `opacity: 1`.
- `test_running_dot_respects_forced_colors` — the
  `@media (forced-colors: active)` rule is shipped with the
  full three-declaration body (cancel animation, pin opacity,
  `background: CanvasText !important`).
- `test_no_pulse_a11y_css_when_not_running` — a clean last-run
  response must not carry the pulse-dot class or its keyframes.
- `test_pulse_keyframes_still_present_for_running_card` — the
  default animation is preserved for non-a11y users; v0.8.17
  adapts it, doesn't remove it.

Full suite: **1185 passed, 1 skipped, 1 deselected** (up from 1180).

## Design notes

**Why `background: CanvasText` and not a border?** The pulse dot
is a 10×10 round element. A `border: 1px solid CanvasText` on it
would be ~40 % of the visual mass, and the inner area would
still be `Canvas` (page background) — a hollow ring that's
harder to read as a status indicator. A solid fill in
`CanvasText` gives a clear, dense shape against the page. The
`CanvasText` system-color token survives the forced-colors
remap (author RGB values are overridden, system tokens aren't),
so this is the reliable fix.

**Why cancel the animation in forced-colors?** The animation
only mutates `opacity`, which is preserved under forced-colors
(unlike `background` which is remapped). So technically the
dot would still pulse. But: a user on forced-colors is almost
certainly an a11y user, and the animation is decorative — the
word "Running · phase: X" next to the dot conveys the state
regardless. Consistent with the v0.8.13 / v0.8.16 reasoning:
when in doubt, stay static.

**Why `!important` on all three rules?** The inline `style`
attribute on the span carries `animation:lvdcp-pulse 1.5s
infinite ease-in-out`, `background:#1976d2`, and implicit
opacity 1. Inline style specificity is 1000; a plain class rule
is 10. Without `!important` the class rules lose every time.
Same trade-off as v0.8.13 and v0.8.16 — three tokens beat
breaking the inline shorthand into longhand and updating a
dozen unrelated tests.

**Why not also adapt the progress bar?** The running card
renders a horizontal progress bar (`background:#bbdefb` track +
`background:#1976d2` fill) when `wr.modules_total` is set. In
forced-colors mode both become `Canvas` → invisible bar. Worth
fixing, but a different class of element (nested divs, width
computed from Jinja expression), so I'd rather land it in
v0.8.18 with its own tests than widen this release. Added to
known gaps below.

## Migration / compat

- Pure template change. No route change, no schema change, no
  behaviour change for users without a11y preferences.
- Adds one class attribute to one span. Zero risk of breaking
  CSS pinning on the element (the inline `style` is unchanged).
- Existing tests that match `animation:lvdcp-pulse 1.5s infinite
  ease-in-out` keep passing — the new test
  `test_pulse_keyframes_still_present_for_running_card`
  explicitly guards that the inline animation string isn't
  accidentally removed.

## Known gaps (carry forward)

- **Progress bar forced-colors adaptation.** The track + fill
  both remap to `Canvas` in high-contrast; users lose the
  module-progress signal. Likely v0.8.18 or v0.8.19 target.
- **No outage-duration signal on recovery toast.** Still
  deferred from v0.8.15 / v0.8.16 — actual user-visible UX
  improvement candidate.
- **No dismiss-event telemetry.** Blocked on render-data volume
  (v0.8.15 reasoning).
- **Crash toast still sticky, no auto-dismiss.** Project-
  switching users accumulate stale red banners.
- **With v0.8.17 shipped, all *toast* a11y gaps from v0.8.16
  are closed.** The remaining a11y debt is on the progress bar
  only.
