# v0.8.29 — Obsidian sync observability on the project card

**Date:** 2026-04-24
**Scope:** `libs/status/models.py` + `libs/status/health.py` + `apps/ui/templates/partials/health_card.html.j2` + `tests/unit/status/test_health.py`. No CLI change, no daemon change, no TypeScript change, no new migration.

## Why

v0.8.28 shipped `ctx obsidian sync-all` — the nightly-sweep command, scheduled by the user via `launchd`/`cron`. The release-note roadmap explicitly listed:

> **Nightly-sync observability in the dashboard.** The existing `obsidian_last_sync` marker is already on disk; surfacing "last Obsidian sync: 3h ago" next to "last scan: 12min ago" on the project card is a ~10-line template change.

That's this release. It is the smallest, highest-leverage follow-through on v0.8.28: the data is already on disk, we just weren't reading it. Without it, a user who armed `sync-all` in `launchd` has no way to confirm from the UI that last night's run actually fired.

The "last scan" timestamp on the project card has existed since Phase 3. Putting "last Obsidian sync" next to it is the same information architecture — two cheap heartbeat signals the user reads at a glance to decide whether the background machinery is alive.

## Design

### Two new nullable fields on `HealthCard`

```python
obsidian_last_sync_at_iso: str | None = None   # ISO 8601 UTC, ends with "Z"
obsidian_sync_age_hours: float | None = None   # precomputed: now - ts, in hours
```

Both default to `None` so:
- Projects that have never Obsidian-synced (most projects today) get the fields serialized as `null` and the template hides the row entirely.
- Every existing caller — dashboard, `lvdcp_status` MCP tool, `ProjectStatus.card`, `WorkspaceStatus.projects` — works unchanged. No code had to be taught the new fields exist; they just come along for the ride.

Why **two** fields instead of one:

- The ISO timestamp goes into the `<time datetime="...">` attribute so browsers/readers can localize it and so we don't lose precision.
- The precomputed `age_hours` keeps Jinja template-side code free of datetime math. `build_health_card` already has a `now` reference — computing `(now - ts) / 3600` once in Python is cleaner than importing a `now` filter into the template layer.

### `_read_obsidian_sync(root, *, now=None)` helper

Lives in `libs/status/health.py` next to the existing `_parse_iso`:

```python
def _read_obsidian_sync(
    root: Path, *, now: float | None = None
) -> tuple[str | None, float | None]:
    marker = root / ".context" / _OBSIDIAN_MARKER
    try:
        raw = marker.read_text(encoding="utf-8").strip()
    except (FileNotFoundError, OSError):
        return None, None
    try:
        ts = float(raw)
    except ValueError:
        return None, None
    current = now if now is not None else time.time()
    age_hours = max(0.0, (current - ts) / 3600.0)
    iso = datetime.fromtimestamp(ts, tz=UTC).isoformat().replace("+00:00", "Z")
    return iso, age_hours
```

Design points:
- **Returns `(None, None)` on missing / unreadable / non-float marker.** The dashboard must degrade gracefully; a corrupted marker can't crash the status aggregation that renders every project card.
- **Accepts an injected `now`** so the helper is deterministic in tests — no `freezegun` dependency needed.
- **`max(0.0, …)` clamps future timestamps to zero.** Clock skew, VM time jumps, user manually editing the marker — any of these can produce a marker dated in the future. Showing "-0.3h ago" is worse than showing "<1h ago".

### Marker filename is canonical, duplicated in constants

Three files now name the marker:
- `apps/agent/obsidian_worker._OBSIDIAN_MARKER` — daemon after-scan writer.
- `apps/cli/commands/obsidian_cmd._OBSIDIAN_MARKER` — `sync-all` writer.
- `libs/status/health._OBSIDIAN_MARKER` — this release's reader.

A constant isn't exported to a shared module because the two writers existed before the reader and didn't need coordination. A third touch-point still doesn't justify a shared module — but the docstring on the new constant explicitly says "rename all three", so a future rename doesn't silently break only the dashboard.

### Template rendering

The partial renders three tiers based on age:

- `< 1h` → `<1h ago`
- `1h–48h` → `Nh ago` (rounded to whole hours)
- `≥ 48h` → `Nd ago` (rounded to whole days)

Tier breakpoints chosen to match the user's mental model: for Obsidian sync, hour-precision matters for "did last night run", day-precision matters for "is this stale". Minute-precision is noise — the sync is debounced at 1h anyway.

The row is wrapped in `{% if card.obsidian_last_sync_at_iso %}` — projects with no marker render exactly like before. No CSS class change, no layout change, one row appears under the existing "last scan" line when (and only when) there's data to show.

## Known limitations (explicit scope locks)

- **Relative age is computed server-side at render time.** If a dashboard HTML page is left open in a tab for an hour, the rendered "2h ago" text stays stale until the user refreshes. The existing refresh button and 30s auto-reload already cover this; a live timer would be over-engineering for a heartbeat signal.
- **No colour coding for staleness.** Unlike `last scan` which has a `stale` banner for >24h, Obsidian sync doesn't get one. Different projects can reasonably have different sync cadences (a weekly project vs a daily one), so a global 24h threshold would generate noisy banners on perfectly healthy projects.
- **`lvdcp_status` MCP tool gets the new fields free.** Because `HealthCard` is used by both the dashboard and the MCP tool, agent consumers automatically see `obsidian_last_sync_at_iso` / `obsidian_sync_age_hours` on every card. No MCP-schema migration needed. No MCP docs updated because the field names are self-descriptive.
- **Workspace-level aggregate not shown.** We don't surface "N of M projects out-of-sync" on the overview header. One line per card is enough; the overview counter would duplicate information the user already scans card-by-card.
