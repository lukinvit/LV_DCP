# v0.8.62 — `ctx watch install-service --json` for scriptable launchd registration

**Date:** 2026-04-25
**Tag:** `v0.8.62`
**Theme:** CLI scriptability rollout — daemon-service registration surface

## Summary

Adds `--json` to `ctx watch install-service`, the launchd LaunchAgent
installer for the LV_DCP background daemon. Single-object payload
describing the just-installed service:
`{label, plist_path, uid, program_arguments, log_dir, bootstrapped}`.

`bootstrapped: true` is the explicit success signal — the only path
that emits a JSON payload at all is the success path
(`launchctl bootstrap` returned 0). On `LaunchctlError` the existing
exit-3-to-stderr contract is preserved with **no** JSON payload on
stdout, so a script piping `--json | jq ...` hard-fails on `launchctl`
errors rather than silently parsing prose.

This is the seventh `--json` ship in the watch/agent/registry domain
and the first to wrap a launchd-side mutation. Prior watch surfaces
covered the registry side (`add`, `remove`, `list`); `install-service`
covers the daemon-binding side.

## Schema

`ctx watch install-service --json` returns a single JSON object with
six top-level keys (locked by `_WATCH_INSTALL_SERVICE_JSON_KEYS`):

```json
{
  "label": "tech.lvdcp.agent",
  "plist_path": "/Users/<you>/Library/LaunchAgents/tech.lvdcp.agent.plist",
  "uid": 502,
  "program_arguments": [
    "/Users/<you>/.../python3.12",
    "-m",
    "apps.agent.daemon"
  ],
  "log_dir": "/Users/<you>/Library/Logs/lvdcp-agent",
  "bootstrapped": true
}
```

- `label` — the launchd label (`tech.lvdcp.agent`); matches what
  `launchctl print gui/<uid>/<label>` would resolve
- `plist_path` — absolute path of the plist file actually written
  (round-trips so a follow-up `ctx watch uninstall-service` can be
  scripted from the same payload without re-deriving the location)
- `uid` — the gui/<uid> launchd domain the agent was bootstrapped into
- `program_arguments` — the argv launchd will invoke; `[0]` is the
  Python interpreter (lets ops verify the recorded interpreter still
  exists — catches the `uv sync`-after-Python-upgrade footgun where
  the old interpreter path is now stale)
- `log_dir` — `~/Library/Logs/lvdcp-agent`; round-trips so a tailing
  script doesn't have to hard-code it
- `bootstrapped` — always `true` on the JSON path (the error path
  exits 3 before any payload). Explicit positive signal:
  `jq -e '.bootstrapped'` is the natural "did launchctl really
  happen" gate

## Exit semantics preserved

- `launchctl bootstrap` returned 0 → exit 0 + JSON payload on stdout
  (or text chrome on stdout in the no-`--json` path)
- `LaunchctlError` (launchctl non-zero) → exit 3 + error message on
  stderr, **no** payload on stdout (same v0.8.42-v0.8.61 boundary)

The `--json` flag is a render switch, not a behavior switch. Plist is
still written before `launchctl bootstrap` runs; failure semantics,
the GUI-session-required note, and exit codes all carry forward
unchanged.

## Implementation discipline

- **CLI-only** change. `libs/mcp_ops/launchd.py` (the `write_plist`,
  `bootstrap_agent`, `bootout_agent` helpers) untouched — JSON is a
  pure render concern.
- **Schema lock** via `_WATCH_INSTALL_SERVICE_JSON_KEYS` (six keys).
  Any future field (e.g. a `keep_alive` toggle, a `bootstrapped_at_iso`
  timestamp) becomes a reviewed schema change rather than a silent
  payload drift.
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  legacy `plist written: ... / launchctl bootstrap gui/<uid>
  succeeded` lines bytewise. Any pre-existing automation greppi​ng
  these lines sees no diff. Locked by
  `test_install_service_text_output_unchanged`.

All 18 tests in `test_watch_cmd.py` pass (14 pre-existing + 4 new).
All 174 tests in `tests/unit/cli/` pass (no regressions).

Tests use a `_isolated_launch_agent_paths` fixture that monkey-patches
`LAUNCH_AGENT_DIR` and `AGENT_LOG_DIR` to `tmp_path` subdirectories,

## Why this matters

Before this ship: every install-service automation (CI image
provisioning, dev-machine bootstrap scripts, developer onboarding
runbooks) had to grep the human chrome (`plist written: <path>` and
`launchctl bootstrap gui/<uid> succeeded`) to confirm the install
landed. The text format is brittle (paths containing spaces or special
characters could collide with the parser), and the actual Python
interpreter that launchd recorded was nowhere in the output — leaving
the post-`uv sync` Python-upgrade footgun (the one where `which
python` updates but the launchd-recorded interpreter doesn't) silent
until the daemon next failed to start.

After this ship: `ctx watch install-service --json` returns a
structured payload that scripts can verify without text parsing. The
`program_arguments[0]` field exposes the recorded interpreter, so a
post-install check can compare it against `which python3.12` and
re-install if they diverge. The `plist_path` round-trips so a
companion `uninstall-service` script can target the exact same file.

This unblocks workflows like:

- **Post-install verification**: `ctx watch install-service --json | jq -e '.bootstrapped and (.program_arguments[0] | test("3\\.12"))'` — one-line gate for "service is installed AND the recorded interpreter is the expected major.minor"
- **Idempotent CI bootstrap**: `ctx watch install-service --json > /tmp/svc.json && jq -r '.plist_path' /tmp/svc.json` — capture the install descriptor for later scripting (uninstall on rollback, log-tail on demand)
- **Stale-interpreter detection**: `ctx watch install-service --json | jq -r '.program_arguments[0]'` and compare against the current `python3.12 -c 'import sys; print(sys.executable)'` to catch interpreter drift after a `uv sync`
- **Log-dir round-trip for tailing**: `tail -F "$(ctx watch install-service --json | jq -r '.log_dir')/lvdcp-agent.out.log"` — start the service and follow its logs from a single composed pipeline

## Cross-surface schema discipline

`install-service` is a single-result success-path mutation that, on
the success path, always returns the same shape (no
`removed`-style wrapper because there is no "no-op" semantics —
re-installing always rewrites the plist and re-bootstraps). Closest
prior pattern: v0.8.59 `timeline enable --json` and v0.8.60
`timeline disable --json` — single-result mutations with a stable
descriptor on success and exit-non-zero-with-no-payload on error.

Pattern inventory after this ship:
- **Per-row reads**: bare arrays (v0.8.44, v0.8.49)
- **Single-result mutations w/ exception-on-absent**: bare entry
  (v0.8.54, v0.8.57, v0.8.58, v0.8.59, v0.8.60)
- **Single-result mutations w/ no-op-on-absent**: wrapper-with-null
  (v0.8.56)
- **Mutations w/ single count + invocation params**: flat dict
  (v0.8.38, v0.8.55)
- **Fleet-iteration with summary counters + per-entry array**:
  (v0.8.61)
- **Single-result side-effecting installer w/ stable descriptor**:
  (v0.8.62 ← this ship — first daemon-binding `--json` surface)
