# v0.8.25 — Swift `INHERITS` edges (single-colon heritage)

**Date:** 2026-04-24
**Scope:** single-language deepening; no new dependencies; no new route; no config changes.

## Why

v0.8.23 closed the inheritance-edge gap for Java; v0.8.24 did the same for Kotlin. Swift was the last of the Phase 10 JVM/Apple trio still carrying the explicit v1 scope-lock — "no inheritance edges yet" — from v0.8.22. This release delivers it.

With this ship, every tree-sitter parser on the platform that targets a language with a class hierarchy (Java, Kotlin, Swift) emits `INHERITS` edges. Go / Rust have structural conformance rather than nominal inheritance; the Python parser treats call-graph as its primary edge type and inheritance as a separate future pass. So this release closes the **inheritance-edge axis across language coverage**, not a single-parser gap.

## Design

`SwiftParser` now overrides `parse()` to call the base `TreeSitterParser.parse()` first, then walks the CST a second time with a scope-aware stack to emit `Relation(relation_type=INHERITS)` edges for every heritage clause it finds. Structural shape: identical to the Java (v0.8.23) and Kotlin (v0.8.24) ports — `_TYPE_DECLARATION_NODES` frozenset, `parse()` override, `_extract_inherits()` scope walker, `_collect_heritage()`, `_user_type_name()`.

### Heritage-clause shape (direct sibling, no wrapper node)

This is the one structural difference from the JVM pair. Java wraps heritage entries in `super_interfaces`, Kotlin in `delegation_specifiers`; Swift has **no wrapper**. Every `inheritance_specifier` is a direct sibling of the type name inside `class_declaration` / `protocol_declaration`, separated by `,` tokens:

```
class_declaration
├── class / struct / actor / enum / extension    (declaration keyword)
├── type_identifier  (or user_type for extensions)    (source type name)
├── :
├── inheritance_specifier
│   └── user_type
├── ,
├── inheritance_specifier
│   └── user_type
└── class_body
```

`_collect_heritage` therefore iterates `type_node.children` directly and picks out every `inheritance_specifier`. Non-heritage siblings (`type_parameters` on the source type, `type_constraints` for a `where` clause, `class_body` / `enum_class_body` / `protocol_body`, `:` / `,` tokens) are all filtered by the node-type check.

### One syntax covers five declaration kinds

Swift overloads the `:` heritage syntax across an unusually broad set of declarations — `class`, `struct`, `enum`, `actor`, `protocol`, and `extension`. All land on `class_declaration` (one AST node kind for the first five) or `protocol_declaration`, and every one can carry heritage:

| Declaration | Example | Edge |
| --- | --- | --- |
| Class inheritance | `class Dog: Animal` | `Dog INHERITS Animal` |
| Struct conformance | `struct User: CustomStringConvertible` | `User INHERITS CustomStringConvertible` |
| Enum raw-value | `enum Color: Int, Serializable` | `Color INHERITS Int` + `Color INHERITS Serializable` |
| Actor conformance | `actor Counter: Sendable` | `Counter INHERITS Sendable` |
| Protocol composition | `protocol P: Q, R` | `P INHERITS Q` + `P INHERITS R` |
| Extension conformance | `extension User: Equatable, Hashable` | `User INHERITS Equatable` + `User INHERITS Hashable` |

### Extensions: edge source is the extended type

Swift extensions land on `class_declaration` in the grammar, same as the primary declaration. The node's `name` field is set to the **extended** type — for `extension User: P`, `child_by_field_name("name").text` is `b"User"`, not `b"UserExtension"`. The base walker (from v0.8.22) already emits a duplicate CLASS symbol named after the extended type for every extension; this release's heritage walker therefore produces `<module>.User INHERITS P` for the conformance, which correctly matches the duplicate symbol's `fq_name`.

The UX consequence: a graph query for "does this codebase conform `User` to `Equatable`?" now answers yes whether the conformance is declared in `User`'s primary definition or in a retroactive `extension User: Equatable {}` — regardless of which file the extension lives in. That is the shape Swift users actually reason about.

### Type-name extraction from `user_type`

Swift's `user_type` grammar uses `type_identifier` rather than Kotlin's `identifier` for each dotted-path segment. Generics appear as a trailing `type_arguments` child that must be ignored:

| Node layout | Example | `dst_ref` |
| --- | --- | --- |
| `user_type` with one `type_identifier` | `Animal` | `"Animal"` |
| `user_type` with `type_identifier . type_identifier . type_identifier` | `pkg.lib.Deep` | `"pkg.lib.Deep"` |
| `user_type` with `type_identifier` + `type_arguments` | `Container<String>` | `"Container"` (args stripped) |
| Nested type_arg → deeper `user_type` | `Comparable<Box<T>>` | `"Comparable"` (base only) |

Implementation: `_user_type_name` walks only direct `type_identifier` children and joins them with `.`. `type_arguments` is never iterated into, so generics are stripped without special-casing. `test_generic_type_arguments_stripped` asserts `String` and type-parameter substitutions never surface as bases — same safety shape the Java and Kotlin releases shipped.

### `where` clauses must not leak

`class Constrained<T>: Base where T: Hashable {}` puts the `where` clause in a `type_constraints` sibling after the last `inheritance_specifier`. Inside, `inheritance_constraint` contains a `user_type` (`Hashable`) that is syntactically identical to a heritage entry — but it's a generic-parameter constraint, not an inheritance edge. The node-type filter in `_collect_heritage` (`child.type != "inheritance_specifier"`) excludes `type_constraints` naturally, so no `Constrained INHERITS Hashable` edge leaks. `test_where_clause_does_not_leak` locks this.

### Scope-aware source_ref derivation (identical to Java/Kotlin)

The walker tracks an `enclosing_fq` alongside each CST node on its work stack. When entering a `class_declaration` or `protocol_declaration`:

1. Computes `type_fq = f"{enclosing_fq}.{name}"` — matching the base walker's `parent_fq.name` convention
2. Emits every heritage edge with `src_ref=type_fq`
3. Propagates `type_fq` as the enclosing scope for the node's children

`class Outer { class Inner: pkg.lib.Deep {} }` therefore produces `<module>.Outer.Inner INHERITS pkg.lib.Deep`. The locked invariant `test_inherits_src_ref_matches_symbol_fq` asserts every `INHERITS.src_ref` resolves to an actual Symbol in the same file — any divergence between the symbol walker's scope logic and the heritage walker's scope logic is caught by that test.

### INHERITS vs. CONFORMS_TO: still one edge type

Same decision as v0.8.23 Java and v0.8.24 Kotlin: class inheritance, protocol conformance, enum raw-value bases, actor conformance, and extension conformance all collapse onto `RelationType.INHERITS`. A future `CONFORMS_TO` sub-type would require a `RelationType` enum addition and a coordinated downstream audit — no concrete consumer driving it yet.

## Known limitations (explicit scope locks)

- `INHERITS` collapses class inheritance + struct/enum/actor protocol conformance + protocol composition + extension conformance. A future `CONFORMS_TO` sub-type would require a `RelationType` enum addition.
- `dst_ref` is a bare type name (`Equatable`, `pkg.lib.Deep`), not a fully-qualified symbol. Swift lets you refer to `Data` after `import Foundation`, so `: Data` emits `dst_ref="Data"`, not `"Foundation.Data"`. Import-based resolution is a separate graph-enrichment pass (same gap as Java / Kotlin).
- `typealias` and `associatedtype` declarations are not in `_symbol_type_map` and therefore cannot be INHERITS sources. Future sub-increment.
- `deinit_declaration` has no name field and is effectively an anonymous destructor; excluded from symbol emission and therefore from edges.
- No `SAME_FILE_CALLS` edges — same gap as Java / Kotlin / Go / Rust; the Python parser's call-graph remains the only one generalised to method invocations.
- Where-clause conformances (`class C<T>: Base where T: Hashable`) do **not** emit `C INHERITS Hashable` — the `where` clause describes a constraint on `T`, not an inheritance of `C`. This is deliberate (locked by `test_where_clause_does_not_leak`).
