# v0.8.46 — `ctx wiki status --json` for scriptable wiki state per module

**Date:** 2026-04-25
**Previous:** [v0.8.45 — `ctx timeline reconcile --json` for scriptable orphan-flagging](2026-04-25-v0.8.45-reconcile-json.md)
**Type:** UX (CLI scriptability — extends the v0.8.42-v0.8.45 `--json` rollout to the `ctx wiki` group)
**Risk:** Low — additive flag; existing text path byte-identical; only one new CLI flag; no library signature changes; no on-disk artifact changes.

## Problem

`ctx wiki status` has been the human-readable per-module wiki-state report since the wiki feature shipped, but its CLI output stayed text-only:

```
Module                                   Status     Last Generated
---------------------------------------------------------------------------
pkg.alpha                                current    2026-04-20 09:12:44
pkg.beta                                 dirty      never
pkg.gamma                                current    2026-04-22 15:33:01
```

That's fine for humans, useless for automation that wants to act on the structured wiki state:

- A pre-commit hook that asserts "no dirty wiki modules accumulated since the last `ctx wiki update`" needs `jq -e 'all(.[]; .status == "current")'`, not a brittle `grep "dirty"` over prose.
- A workspace-sweep dashboard — `find ~/dev -maxdepth 2 -name '.context' -exec ctx wiki status {}/.. --json` — wants per-project wiki-staleness counters folded into a single hygiene view.
- A nightly CI job that surfaces "wiki articles older than N days" needs `jq -r '.[] | select(.last_generated_ts < (now - 7*86400)) | .module_path'` — which requires the timestamp as a raw POSIX float, not a formatted human string the regex would have to parse back to epoch.
- A `ctx wiki update` follow-up that targets only specific modules needs `jq -r '.[] | select(.status == "dirty") | .module_path' | xargs ctx wiki update --module` — a structured pipeline impossible with the text view.

`get_all_modules()` in `libs/wiki/state.py` already returns a structured per-row payload (`module_path`, `wiki_file`, `last_generated_ts`, `source_hash`, `status`) — we just weren't exposing it. Same gap that v0.8.38 closed for `prune`, v0.8.40 for `restore`, v0.8.42 for `scan`, v0.8.43 for `inspect`, v0.8.44 for `memory list`, v0.8.45 for `timeline reconcile`. Time to close it for `wiki status`.

## Solution

A `--json` flag on `ctx wiki status` that emits a bare JSON array — one entry per tracked module:

```bash
$ ctx wiki status /path --json
[
  {
    "module_path": "pkg.alpha",
    "wiki_file": "modules/pkg.alpha.md",
    "status": "current",
    "last_generated_ts": 1745149964.0,
    "source_hash": "abc123def456..."
  },
  {
    "module_path": "pkg.beta",
    "wiki_file": "modules/pkg.beta.md",
    "status": "dirty",
    "last_generated_ts": 0,
    "source_hash": "789xyz..."
  }
]
```

### Schema notes

- **Pass-through of the `wiki_state` row schema.** Adding a column to `wiki_state` (and surfacing it via `get_all_modules`) is the single point that surfaces it in both text and JSON paths — `_module_to_json(mod)` is a one-line dict-literal that walks the dict keys returned by `get_all_modules`.
- **`last_generated_ts` is the raw POSIX float** — not the formatted `YYYY-MM-DD HH:MM:SS` string the text view shows. Dashboards and scripts can do age math (`now - last_generated_ts`) without re-parsing a formatted string back to epoch seconds. The formatted view is trivially recoverable via `jq -r '.[].last_generated_ts | strftime(...)'` or shell `date -r $(jq -r ...)`. **`0`** = never generated (sentinel matching the text view's `never` rendering).
- **`status` is the raw enum-like string** (`"current"` / `"dirty"`) — pass-through from the SQL `status` column. Lets `jq 'map(select(.status == "dirty"))'` work without nested object access.
- **Bare array, not `{"modules": [...]}`** — matches the v0.8.32 `ls --json`, v0.8.40 `restore --json`, v0.8.44 `memory list --json` precedent. Empty list (`[]`) is the contract for "no modules tracked" — never `null`, never the human-prose `No modules tracked.` marker. Consumers do `jq length` without a None-guard.
- **`source_hash` is the full hex digest** — not truncated. The text view doesn't show it; consumers that want a short prefix can `jq -r '.[].source_hash | .[0:8]'`. Keeping the full hex enables hash-comparison workflows (e.g. "did the wiki source change between PRs?").

### Error path

Missing `cache.db` (project not yet scanned) surfaces error on stderr + exit 1 in **both** text and JSON modes:

```bash
$ ctx wiki status /not/a/scanned/project --json
$ echo $?
1
$ # stderr: "error: no cache.db found at /not/a/scanned/project/.context/cache.db. Run `ctx scan` first."
$ # stdout: empty
```

Discipline shared with v0.8.42-v0.8.45: `--json` does **not** swallow the error into stdout JSON (`{"error": "..."}`). Scripts gate on exit code (`set -e`) and read the human message from stderr.

## Library / CLI layering

CLI-only change. No new library helpers, no schema modifications to `libs/wiki/state.py`. `get_all_modules()` already returned the right shape since the wiki feature shipped.

A small `_module_to_json(mod: dict[str, Any]) -> dict[str, object]` helper in `apps/cli/commands/wiki_cmd.py` builds the per-row payload. Lives next to `status` (the only consumer). Sister-helper to `_memory_to_json` (v0.8.44) — same pattern of per-row JSON shaping in the CLI shell.

## New invariants

A new test module `tests/unit/cli/test_wiki_cmd.py` (4 tests, focused on the new `--json` slice — older surfaces `update` / `lint` / `cross-project` already have integration coverage elsewhere):

- **`test_wiki_status_text_output_unchanged`** — baseline: text path emits the legacy table format byte-identically. Locks the v0.8.42-v0.8.45 "text path untouched" promise.
- **`test_wiki_status_json_emits_well_formed_array`** — bare JSON array, exact key set per entry (`_WIKI_STATUS_JSON_KEYS` frozenset = `{module_path, wiki_file, status, last_generated_ts, source_hash}`), `status='current'` rows have non-zero `last_generated_ts` (raw POSIX float), `status='dirty'` rows have `last_generated_ts=0` (sentinel for "never generated"). Schema-locked surface — adding a key requires bumping the frozenset + the helper at the same time.
- **`test_wiki_status_json_empty_returns_bare_list`** — empty `wiki_state` table returns `[]`, not `null`, not the `No modules tracked.` prose marker. Locks the empty-list contract so consumers can do `jq length` without a None-guard.
- **`test_wiki_status_no_cache_db_fails_in_both_modes`** — missing `cache.db` exits 1 in both text and JSON modes, surfaces `no cache.db` on the merged stderr/stdout output, **never** emits a `{"error": "..."}` payload (no `"error"` key, `json.loads(output)` raises). Locks the discipline shared with v0.8.42-v0.8.45.

## Rejected alternatives

- **Format `last_generated_ts` as the same `YYYY-MM-DD HH:MM:SS` string the text view shows.** Loses the script-side ergonomic of doing age math directly (`now - ts`); forces consumers to re-parse a formatted string back to epoch seconds. The text view's formatting is a render concern; the JSON contract is data.
- **Truncate `source_hash` to 8 chars (matching common wiki UI patterns).** Loses the full hex digest that enables hash-comparison workflows; truncation is trivially recoverable via `jq -r '.[].source_hash | .[0:8]'` for display.
- **Wrap in `{"modules": [...], "summary": {dirty: N, current: M}}`** — breaks the v0.8.32 / v0.8.40 / v0.8.44 bare-array precedent. Summary counts are trivially recoverable via `jq 'group_by(.status) | map({key: .[0].status, value: length}) | from_entries'` — no value in baking it into the payload shape.
- **Schema-version the JSON output (`"_schema_version": 1`).** Premature ceremony, same v0.8.38-v0.8.45 argument. Add when there's a real consumer that needs to gate on schema versions.
- **Add `--status dirty` / `--status current` filter flag.** Trivially recoverable via `jq 'map(select(.status == "dirty"))'`; promote to a flag only if the pattern repeats across multiple consumers.
- **Add `--json` to `wiki update` / `wiki lint` / `wiki cross-project` in the same release.** Keep increments small. `status` is the natural read-side companion (mirrors the v0.8.45 read-side-first pattern); `update` has mutation semantics with progress streaming that benefits from a separate ship; `lint` has its own exit-code gating that needs careful thought; `cross-project` is the natural next read-side slice.
- **Make the error path emit `{"error": "...", "exit_code": 1}` JSON in `--json` mode.** Would split the contract — exit code is the script gate, stderr carries the human message, stdout is exclusively the success payload. Same discipline as v0.8.42-v0.8.45.

## Scope locks

- Only one new CLI flag (`--json`) on `wiki status`.
- `wiki update`, `wiki lint`, `wiki cross-project` subcommands **untouched**.
- `libs/wiki/state.py::get_all_modules` and the `wiki_state` SQL schema **unchanged**.
- No new on-disk artifacts.
- No MCP surface — wiki state is a hygiene/observability surface for operators, not retrieval context for LLMs (the wiki content itself is already accessible via the file system).
- Text output **byte-identical** — the new `test_wiki_status_text_output_unchanged` test locks this.
- structlog routing introduced in v0.8.42 (`apps/cli/main.py`) carries forward unchanged — JSON consumers benefit automatically.

## Follow-ups explicitly NOT done here

- `ctx wiki lint --json` — has its own exit-code gating (`exit 1` when errors exist, `exit 0` with warnings) and a richer per-row schema (severity, message, module_path); deserves its own ship with careful thought on whether `--json` should also exit 1 on error rows or only emit the structured payload.
- `ctx wiki cross-project --json` — natural follow-up read-side slice; structured per-pattern payload would benefit dashboards.
- A summary header `{generated_at, total, dirty_count, current_count}` wrapping the array. `jq` slicing handles this without baking it into the payload.
- A README badge tracking the JSON-scriptable surface coverage (e.g. "JSON-scriptable: scan ✓, inspect ✓, prune ✓, restore ✓, ls ✓, memory ✓, reconcile ✓, wiki status ✓"). Premature — release notes carry this discoverability and a badge would need maintenance every time another command grows `--json`.
