# v0.8.23 — Java `INHERITS` edges (`extends` / `implements`)

**Date:** 2026-04-24
**Scope:** single-language deepening; no new dependencies; no new route; no config changes.

## Why

Phase 10 closed the language-coverage axis in v0.8.22 (Java / Kotlin / Swift parsers all live). The three parsers all shipped with the same explicit v1 scope-lock: **no inheritance edges**. That was the right call at the time — it kept each release reviewable and gave the graph a new language in a single commit — but it leaves a concrete capability gap. Without `extends` / `implements` edges, the relation graph can tell you "file A imports file B" but not "class Child specialises class Parent", so impact analysis can't answer "what concrete subclasses of `Serializable` exist in this repo?" or "if I change `AbstractHandler.process()`, what subclasses override it?" — the two questions where an inheritance edge is the only available signal.

Java is the natural first language to close the gap: oldest of the three parsers (shipped first in v0.8.20), the most common JVM target in the wild, the clearest tree-sitter heritage-clause structure of the three, and — most importantly — the TypeScript parser already emits `INHERITS` edges from `extends_clause` / `implements_clause` nodes, so the pattern is proven in-codebase, not borrowed from a spec. This release ports that pattern to Java. Kotlin and Swift get the same treatment as separate follow-up increments once Java's behaviour is real-world-tested.

## Design

`JavaParser` now overrides `parse()` to call the base `TreeSitterParser.parse()` first, then walks the CST a second time with a scope-aware stack to emit `Relation(relation_type=INHERITS)` edges for every `extends` / `implements` clause it finds.

### Heritage clause shapes (from the tree-sitter-java grammar)

- **`class_declaration`** carries `superclass` (single, `extends`) and/or `super_interfaces` (multiple, `implements`)
- **`interface_declaration`** carries `extends_interfaces` (multiple, `extends`)
- **`enum_declaration`** carries `super_interfaces` (multiple, `implements`)
- **`record_declaration`** carries `super_interfaces` (multiple, `implements`)
- **`annotation_type_declaration`** (`@interface`) cannot inherit — never appears as a source

`_collect_heritage(type_node)` consumes these three field-named children (`superclass` / `super_interfaces` / `extends_interfaces`), visits every non-keyword child of the clause, and calls `_heritage_type_name` on it. Single-base `extends` short-circuits after the first successful extraction (`class X extends Y extends Z` is not legal Java, but guarding against it anyway keeps the walker defensive).

### Type-name extraction

`_heritage_type_name(node)` handles the three type shapes the grammar produces inside a heritage clause:

| Node kind | Example | Emitted `dst_ref` |
| --- | --- | --- |
| `type_identifier` | `extends Parent` | `"Parent"` |
| `scoped_type_identifier` | `extends pkg.Outer.Inner` | `"pkg.Outer.Inner"` (full dotted path) |
| `generic_type` | `extends Container<String>` | `"Container"` (type arguments stripped) |

Generic stripping is handled by finding the first `type_identifier` or `scoped_type_identifier` child of the `generic_type` and recursing on it — so `pkg.Box<Integer>` emits `pkg.Box` (the scoped base), not `pkg` or `Box` or any of the arguments. The key test `test_generic_type_arguments_stripped` asserts `String` and `Integer` never surface as bases; this is the invariant that guards against the naive "flatten all type_identifiers" mistake.

### Scope-aware source_ref derivation

The walker tracks an `enclosing_fq` alongside each CST node on its work stack. When it enters a type declaration, it:

1. Computes `type_fq = f"{enclosing_fq}.{name}"` — matching the base walker's `parent_fq.name` convention
2. Emits every heritage edge with `src_ref=type_fq`
3. Propagates `type_fq` as the enclosing scope for the node's children

This means a nested class inherits correctly: for `class Outer { class Inner extends Base {} }`, the edge is `<module>.Outer.Inner INHERITS Base` — and crucially, `<module>.Outer.Inner` is exactly what the base symbol walker stores in the `Symbol.fq_name` field for `Inner`. The `test_inherits_src_ref_matches_symbol_fq` invariant locks this: every INHERITS `src_ref` must resolve to an actual symbol in the same file's parse result. This is the test that catches any divergence between the symbol walker's scope logic and the heritage walker's scope logic.

This is an improvement over the TypeScript parser's current behaviour, which uses `module_fq.name` uniformly — fine for flat TS files but incorrect for nested classes. Java has a much stronger cultural habit of static nested classes (e.g. Builder patterns, sealed hierarchies), so the extra scope-tracking overhead is worth the correctness.

### INHERITS vs. IMPLEMENTS: one edge type, not two

Java distinguishes `extends` (class inheritance) from `implements` (interface conformance) at the syntax level, and so does Kotlin (both under one keyword) and Swift (both under `:`). In the `libs.core.entities.RelationType` enum, these all collapse onto a single `INHERITS` variant — the graph consumer asks "is-a" questions and treats interface conformance as a special case of inheritance for impact-analysis purposes. Adding an `IMPLEMENTS` variant is a plausible future refinement if a consumer asks "strictly-subclass" vs. "strictly-implements" separately, but doing so now would force the downstream consumers (Obsidian wiki sync, context pack enrichment, dashboard graph rendering) to decide the same split, with no concrete driving use case. Staying at one edge type is the minimum-commitment choice.

## Known limitations (explicit scope locks)

- Only Java is ported in this release. Kotlin (`class Foo : Base()`, `object X : Interface`) and Swift (`class Foo: Base, P`, `struct S: P, Q`) use different heritage syntax — same concept, different AST — and are separate sub-increments. Both parsers already document the gap in their v0.8.21 / v0.8.22 release notes.
- `INHERITS` collapses `extends` + `implements`. A future `IMPLEMENTS` sub-type would require a `RelationType` enum addition and a downstream-consumer audit.
- `dst_ref` is a bare type name (`Parent`, `pkg.Outer.Inner`, `Container`), not a fully-qualified symbol. Java allows referring to `List` after `import java.util.List`, so `extends List<String>` emits `dst_ref="List"`, not `"java.util.List"`. Resolving bare names against the file's import list is a separate graph-enrichment pass.
- No `SAME_FILE_CALLS` edges yet — same gap as Kotlin / Swift / Go / Rust parsers; the Python parser's call-graph remains the only one generalised to method invocations.
- Record components (`record Point(int x, int y)`) still don't surface as individual `VARIABLE` symbols — they live in `formal_parameters` under `record_declaration`, not in a `field_declaration`, so the base walker's single-node-one-symbol assumption doesn't reach them. Adding them is orthogonal to inheritance edges.
