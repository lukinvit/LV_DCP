# v0.8.45 — `ctx timeline reconcile --json` for scriptable orphan-flagging

**Date:** 2026-04-25
**Previous:** [v0.8.44 — `ctx memory list --json` for scriptable memory review queue](2026-04-25-v0.8.44-memory-list-json.md)
**Type:** UX (CLI scriptability — completes the read-side `--json` pair on the `ctx timeline` group: `status` since spec-010, now `reconcile`)
**Risk:** Low — additive flag; existing text path byte-identical; only one new CLI flag; no library signature changes; no on-disk artifact changes.

## Problem

`ctx timeline status` has had `--json` since the timeline subgroup shipped (spec-010 T034) — but its read-side companion `ctx timeline reconcile` (which walks `git rev-list --all` + `git reflog` and flags symbol-timeline events whose `commit_sha` is no longer reachable as orphaned) stayed text-only:

```
reconcile: 7 newly flagged orphaned
  reachable commits: 142
  orphaned added    : 3
  orphaned removed  : 4
```

That's fine for humans, useless for any cron-driven hygiene pipeline that wants to act on the counters:

- A nightly job that runs `ctx timeline reconcile` after a `git gc --prune=now` and posts a Slack alert when `orphaned_newly_flagged > 50` (signal of a large rebase / force-push) needs `jq '.orphaned_newly_flagged'`, not a brittle `grep "newly flagged" | awk '{print $2}'`.
- A multi-project sweep — `find ~/dev -maxdepth 2 -name '.context' -exec ctx timeline reconcile --project {}/.. --json` — wants per-project payloads it can fold into a single workspace-hygiene dashboard.
- A `ctx timeline prune` follow-up that bases its `--older-than` on the actual reachable-commit window needs the prior reconcile's `reachable_commit_count` as a structured int.
- A CI guard that asserts "reconcile didn't accidentally orphan more than N events on this branch" needs `jq -e '.orphaned_newly_flagged < 10'`, not a parsed table.

`ReconcileReport` already carried every field a script needs (`project_root`, `git_available`, `reachable_commit_count`, `orphaned_newly_flagged`, `orphaned_by_event_type`) — we just weren't exposing it. Same gap that v0.8.38 closed for `prune`, v0.8.40 for `restore`, v0.8.42 for `scan`, v0.8.43 for `inspect`, v0.8.44 for `memory list`. Time to close it for `timeline reconcile`.

## Solution

A `--json` flag on `ctx timeline reconcile` that emits a 1:1 mirror of the `ReconcileReport` dataclass:

```bash
$ ctx timeline reconcile /path --json
{
  "project_root": "/path",
  "git_available": true,
  "reachable_commit_count": 142,
  "orphaned_newly_flagged": 7,
  "orphaned_by_event_type": {
    "added": 3,
    "removed": 4
  }
}
```

### Schema notes

- **1:1 mirror of `ReconcileReport`.** Adding a field to the dataclass is the single point that surfaces it in both text and JSON outputs by design — `_reconcile_report_to_json(report)` is a one-line dict-comprehension-equivalent that walks the dataclass fields. No drift between the two views.
- **`orphaned_by_event_type` is alphabetically sorted.** The dataclass `dict[str, int]` carries whatever insertion order the SQL `GROUP BY` emits (implementation-dependent across SQLite versions and table statistics). The CLI helper explicitly sorts so consumers can `jq -r '.orphaned_by_event_type | keys[]'` without an explicit `sort` filter — the script-side ergonomic that matches v0.8.43's `most_common()` ordering for inspect counts and v0.8.42's deterministic field ordering for scan results.
- **`git_available` is always `true` in the success payload.** When git is unavailable the command exits 1 before the JSON path runs — keeping `git_available` in the schema documents the field for consumers reading the dataclass and lets a script defensively assert intent (`jq -e '.git_available == true'`) without having to also branch on exit code in interactive contexts.

### Error path

`git unavailable` (no `git` on `PATH`, or rev-list fails) surfaces error on stderr + exit 1 in **both** text and JSON modes:

```bash
$ ctx timeline reconcile /not/a/repo --json
$ echo $?
1
$ # stderr: "reconcile: git unavailable — no events flagged"
$ # stdout: empty
```

Discipline: `--json` does **not** swallow the error into stdout JSON (`{"error": "..."}`). Scripts gate on exit code (`set -e`) and read the human message from stderr. Splitting that contract — sometimes JSON, sometimes prose, both on stdout — would force every consumer to parse-then-check-keys instead of just relying on the gate. Same scriptability discipline as v0.8.42 / v0.8.43 / v0.8.44.

## Library / CLI layering

CLI-only change. No new library helpers, no schema modifications to `libs/symbol_timeline/reconcile.py` — `ReconcileReport` already had the right shape since spec-010 T033.

A small `_reconcile_report_to_json(report: ReconcileReport) -> dict[str, object]` helper in `apps/cli/commands/timeline_cmd.py` builds the per-call payload. Lives next to `reconcile_cmd` (the only consumer). Sister-helper to the `_event_counts` and `_orphaned_count` private helpers already in the same module — keeps render-shape concerns in the CLI shell layer where they belong.

## New invariants

Three new tests in `tests/unit/cli/test_timeline_cmd.py` (9 existing + 3 new):

- **`test_reconcile_json_emits_report_payload`** — exact key set on the JSON payload (`_RECONCILE_JSON_KEYS` frozenset = `{project_root, git_available, reachable_commit_count, orphaned_newly_flagged, orphaned_by_event_type}`), values match the `ReconcileReport` injected via monkeypatch (avoiding the cost of spinning up a real git repo for a pure render-layer test). Schema-locked surface — adding a key requires bumping the frozenset + the helper at the same time.
- **`test_reconcile_json_git_unavailable_does_not_emit_error_json`** — `--json` mode with no git on `PATH` exits 1, surfaces `git unavailable` on the merged stderr/stdout output, **never** emits a `{"error": "..."}` payload (no leading `{`, no `"error"` key, `json.loads(output)` raises). Locks the discipline shared with v0.8.42 / v0.8.43 / v0.8.44.
- **`test_reconcile_json_orphaned_by_event_type_is_alphabetically_sorted`** — inject a `ReconcileReport` whose `orphaned_by_event_type` insertion order is intentionally non-alphabetical (`{"removed": 1, "added": 2, "modified": 1}`); assert the JSON output emits `["added", "modified", "removed"]`. Locks the explicit sort in `_reconcile_report_to_json` so a refactor that drops it breaks this test instead of silently reshuffling consumer dashboards.

## Rejected alternatives

- **Schema-version the JSON output (`"_schema_version": 1`).** Premature ceremony, same v0.8.38 / v0.8.40 / v0.8.42 / v0.8.43 / v0.8.44 argument. Add when there's a real consumer that needs to gate on schema versions.
- **Make the error path emit `{"error": "...", "exit_code": 1}` JSON in `--json` mode.** Would split the contract — exit code is the script gate, stderr carries the human message, stdout is exclusively the success payload. Forcing JSON consumers to also branch on `if .error` defeats the gate. Same discipline as v0.8.42 / v0.8.43 / v0.8.44.
- **Sort `orphaned_by_event_type` by descending count instead of alphabetically.** Different semantic from v0.8.43's `most_common()` for inspect counts: there, the dimension is *cardinality* (which language has the most files); here, it's *typology* (added/modified/removed/renamed/moved are a closed enum-like set, alphabetic ordering reads more predictably). A consumer that wants frequency ordering can `jq '.orphaned_by_event_type | to_entries | sort_by(-.value) | from_entries'`.
- **Add `--json` to `reconcile` and `prune` and `enable` and `disable` in the same release.** Keep increments small. `reconcile` is the natural read-side companion to the already-`--json`-enabled `status`; `prune` is destructive and deserves its own ship with the discipline already proven on `registry prune --json` (v0.8.38). `enable` / `disable` are pure side-effect side-effects with no payload to emit (success means the flag file landed; `cat $project/.context/timeline.enabled` recovers the state).
- **Include `elapsed_seconds` in the JSON payload.** Reconcile timing is observable via shell (`time ctx timeline reconcile`); adding it to the payload adds noise without serving a script-side use case (no current consumer pattern wants to alert on reconcile latency).
- **Echo invocation parameters (`--project`, `--store`) in the JSON.** Brittle — schema would change with every new flag. `project_root` already round-trips the resolved scan target.

## Scope locks

- Only one new CLI flag (`--json`) on `reconcile_cmd`.
- `enable`, `disable`, `status`, `prune`, `backfill` subcommands **untouched**.
- `libs/symbol_timeline/reconcile.py::reconcile` and `ReconcileReport` library signatures **unchanged**.
- No new on-disk artifacts.
- No MCP surface — reconcile is a hygiene operation triggered by operators / cron, not by an LLM session asking for context.
- Text output **byte-identical** — every existing timeline test passes.
- structlog routing introduced in v0.8.42 (`apps/cli/main.py`) carries forward unchanged — JSON consumers benefit automatically.

## Follow-ups explicitly NOT done here

- `ctx timeline prune --json` — destructive operation with the same dry-run-vs-apply axis as `registry prune`; deserves its own ship with the v0.8.38 discipline (preview JSON when no `--include-live`, applied JSON with deletion counts on `--include-live`). Bundling it here would conflate a render-only change (this PR) with a mutation-shape change.
- A `--quiet` flag that suppresses the "reconcile: N newly flagged" header text. `--json` is already that gate (no human chrome on stdout); a separate `--quiet` would be redundant.
- Time-series metadata in the payload (`reconcile_started_at`, `reconcile_completed_at`). Useful for dashboard "last reconcile" widgets but better surfaced via `ctx timeline status --json` on the actual scan-state record rather than recomputed at reconcile time.
- A README badge tracking the JSON-scriptable surface coverage (e.g. "JSON-scriptable: scan ✓, inspect ✓, prune ✓, restore ✓, ls ✓, memory ✓, reconcile ✓"). Premature — release notes carry this discoverability and a badge would need maintenance every time another command grows `--json`.
