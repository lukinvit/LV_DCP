# v0.8.50 — `ctx obsidian status --json` for scriptable vault project enumeration

**Date:** 2026-04-25
**Tag:** `v0.8.50`
**Theme:** CLI scriptability rollout — Obsidian vault read side

## Summary

Adds `--json` to `ctx obsidian status`, exposing the vault's `Projects/`
directory state as a single structured object with a tri-state contract:

```json
{
  "vault": "/Users/.../vault",
  "projects_dir": "/Users/.../vault/Projects",
  "projects_dir_exists": true,
  "projects": ["AlphaProject", "BetaProject"]
}
```

This is the next slice in the rolling `--json` rollout (v0.8.42 →
v0.8.49) and the **first scriptable surface in the obsidian subgroup** —
analogous to `wiki status` and `watch list` but for the Obsidian sync
target.

## Why a single object (not a bare array)?

`obsidian status` has a **tri-state contract** that a bare array of
project names cannot encode without a sentinel value:

| State | Text mode | JSON mode |
|---|---|---|
| `Projects/` missing (vault never set up) | `"No Projects/ directory found in vault."` | `projects_dir_exists: false`, `projects: []` |
| `Projects/` exists but empty (configured, no syncs) | `"No project directories found."` | `projects_dir_exists: true`, `projects: []` |
| `Projects/` populated | indented bullet list with count | `projects_dir_exists: true`, `projects: [sorted names]` |

Without the `projects_dir_exists` flag, a script could not distinguish
"vault was never set up" from "vault is configured but no projects have
been synced yet" — both would collapse to `[]`. That's a real ergonomic
loss for onboarding scripts that need to differentiate "create the
Projects/ directory" from "run the first sync".

Compare with the bare-array siblings (`wiki status`, `watch list`):
those are pure per-row reads with no meta-state to encode. Same
discipline as v0.8.48 `wiki cross-project` which also uses a single
object because it carries run-level summary fields.

## Implementation discipline

- **CLI-only** change. No mutation to `libs/obsidian/`.
- `_vault_status_to_json` lives next to `status` in
  `apps/cli/commands/obsidian_cmd.py`, sister-helper to v0.8.46-v0.8.49
  CLI-shell payload shapers.
- **Schema-locked tests.** `_OBSIDIAN_STATUS_JSON_KEYS` frozenset asserts
  the exact key set on every emission — adding a new field to the vault
  state contract forces an explicit, reviewed change.
- **Text-mode UX preserved.** Both sentinel strings and the populated
  bullet list remain unchanged. Pre-existing consumers see no diff.
- **Stdout/stderr split.** Pure JSON on stdout (no logger leaks);
  `structlog` is already routed to `sys.stderr` since v0.8.42.
- **Exit code unchanged.** `Projects/` missing still exits 0 — a missing
  `Projects/` is a valid vault configuration, not an error condition.

## Tests

Six new tests in `tests/unit/cli/test_obsidian_status.py` (new file):

| Test | Mode | Asserts |
|---|---|---|
| `test_obsidian_status_text_unchanged_no_projects_dir` | text | legacy "No Projects/ directory found" sentinel preserved + no JSON leak |
| `test_obsidian_status_text_unchanged_empty_projects_dir` | text | legacy "No project directories found." sentinel preserved |
| `test_obsidian_status_text_unchanged_populated` | text | indented bullet list with count header preserved + no JSON leak |
| `test_obsidian_status_json_no_projects_dir` | json | `projects_dir_exists=false`, `projects=[]`, schema-locked keys |
| `test_obsidian_status_json_empty_projects_dir` | json | `projects_dir_exists=true`, `projects=[]` (distinguishable from missing case) |
| `test_obsidian_status_json_emits_well_formed_object` | json | sorted project list, loose files at projects root excluded, schema lock |

All 6 tests pass.
