# v0.8.63 — `ctx watch uninstall-service --json` closes the daemon-service surface

**Date:** 2026-04-25
**Tag:** `v0.8.63`
**Theme:** CLI scriptability rollout — daemon-service teardown surface

## Summary

Adds `--json` to `ctx watch uninstall-service`, the symmetric mirror of
v0.8.62's `install-service --json`. Closes the watch daemon-service
surface end-to-end (alongside the pre-existing `add` v0.8.54, `remove`
v0.8.56, `list` v0.8.49 registry triplet that closed the watch
registry surface).

The payload introduces a **two-axis success signal** —
`booted_out` (launchd-side) AND `plist_removed` (on-disk-side) are
independent booleans, so the JSON contract captures the genuinely
partial states (bootout failed but plist removed, or vice versa) that
the existing text mode collapses into prose footers.

## Schema

`ctx watch uninstall-service --json` returns a single JSON object with
seven top-level keys (locked by `_WATCH_UNINSTALL_SERVICE_JSON_KEYS`):

```json
{
  "label": "tech.lvdcp.agent",
  "plist_path": "/Users/<you>/Library/LaunchAgents/tech.lvdcp.agent.plist",
  "uid": 502,
  "plist_existed": true,
  "plist_removed": true,
  "booted_out": true,
  "bootout_error": null
}
```

- `label` / `plist_path` / `uid` — same shape as v0.8.62 install
  descriptor; lets a script verify the uninstall targeted the same
  artifact the install wrote
- `plist_existed` — was a plist file present at uninstall time? `false`
  on a no-op uninstall (never installed)
- `plist_removed` — was the plist actually unlinked? `true` only when
  `plist_existed` AND the unlink succeeded
- `booted_out` — did `launchctl bootout` succeed? `false` on
  `LaunchctlError` (no such service, transient launchd error, etc.)
- `bootout_error` — `null` on success, populated with the launchctl
  stderr on failure (mirrors v0.8.61 `results[].error` discipline so
  `jq -r '.bootout_error // empty'` works without a defined-key guard)

## Two-axis success signal

The four real states the schema can express:

| State | `booted_out` | `plist_removed` | Meaning |
|---|---|---|---|
| Full uninstall | `true` | `true` | Daemon stopped + disk clean |
| No-op | `false` | `false` | Nothing was installed (typical: never-installed system) |
| Partial (disk-only) | `false` | `true` | Bootout failed but disk cleaned — daemon may run until next reboot |
| Partial (daemon-only) | `true` | `false` | Hypothetical: bootout succeeded but plist already gone — schema covers it |

No prior watch surface needed two independent success axes (registry
mutations either succeed atomically or raise). Uninstall is the first
to inherit launchd's "domain cleanup vs file cleanup are separate
operations" semantic and surface it structurally.

## Exit semantics preserved

The existing **bootout-failure-is-non-fatal** behavior is preserved
unchanged in both modes — uninstall always exits 0 if it makes any
forward progress (or completes the no-op). A consumer that wants to
gate on full success uses `jq -e '.booted_out and .plist_removed'`
explicitly. Mirrors the v0.8.42-v0.8.62 render-switch-not-behavior-
change discipline.

## Implementation discipline

- **CLI-only** change. `libs/mcp_ops/launchd.py` (`bootout_agent`,
  `LaunchctlError`) untouched.
- **Schema lock** via `_WATCH_UNINSTALL_SERVICE_JSON_KEYS` (seven keys).
- **Cross-surface install↔uninstall round-trip** locked by
  `test_uninstall_service_json_round_trips_with_install_descriptor` —
  verifies the v0.8.62 install descriptor's `plist_path` / `label` /
  `uid` match the v0.8.63 uninstall descriptor's, proving the
  daemon-service surface is closed bidirectionally through the same
  on-disk artifact and launchd label (no path drift between the two
  surfaces).
- **Text-mode UX preserved.** Default (no `--json`) still emits the
  legacy `note: <exc> / removed plist: <path> / no plist at <path>`
  chrome bytewise. Pre-existing automation greppi​ng these lines sees
  no diff. Locked by
  `test_uninstall_service_text_output_unchanged_full_uninstall`.

## Why this matters

Before this ship: every uninstall automation had to grep three
mutually-exclusive prose footers (`removed plist: <path>`, `no plist
at <path>`, `note: launchctl bootout failed (exit N): <stderr>`) to
figure out what actually happened. The two-axis success state
(daemon-cleanup vs disk-cleanup independently succeeding) was
invisible — the text mode collapsed `partial` into the same
`removed plist:` line as `full uninstall`.

After this ship: `ctx watch uninstall-service --json` returns the
two-axis state explicitly, so a teardown script can reason about
"did the daemon really stop" separately from "is the plist gone".

This unblocks workflows like:

- **Strict-success gate**: `ctx watch uninstall-service --json | jq -e '.booted_out and .plist_removed'` — exits 0 only on a true full uninstall (lets a CI rollback step page on a partial)
- **Idempotent teardown**: `ctx watch uninstall-service --json | jq -e '.plist_existed == false' || ctx watch uninstall-service --json` — the first pass cleans up, the second pass confirms no-op (both shapes parse with the same `jq` filter; no need to switch on text vs JSON)
- **Diagnostic on partial**: `ctx watch uninstall-service --json | jq -r 'select(.booted_out == false) | .bootout_error'` — surfaces the exact launchctl stderr only on the partial state (no need to stitch stderr + stdout from prose mode)
- **Cross-surface verification**: install + uninstall JSON descriptors share `plist_path`, so `[install_payload.plist_path, uninstall_payload.plist_path] | unique | length` should equal 1 — locks against path drift between the two surfaces in operational scripts

## Cross-surface schema discipline

`uninstall-service` is the symmetric mirror of v0.8.62 install — both
ship `--json` over the same launchd LaunchAgent label and on-disk
plist artifact, with the descriptors sharing `label` / `plist_path` /
`uid` keys (round-trip-verified by
`test_uninstall_service_json_round_trips_with_install_descriptor`).

This is the same closure pattern that v0.8.59 enable + v0.8.60 disable
applied to the timeline-flag surface, that v0.8.57 accept + v0.8.58
reject applied to the memory-review surface, and that v0.8.54 add +
v0.8.56 remove applied to the watch-registry surface — ship the
opening side first, close with the symmetric mirror.

After this ship the **watch daemon-service surface is fully closed**:
- `install-service --json` (v0.8.62) — opening
- `uninstall-service --json` (v0.8.63 ← this ship) — closing

The watch-registry surface (`add` v0.8.54 / `remove` v0.8.56 / `list`
v0.8.49) and the watch daemon-service surface
(`install-service` v0.8.62 / `uninstall-service` v0.8.63) together
mean **every** `ctx watch` subcommand has `--json` (modulo `start`,
which is a long-running foreground process that has no JSON output
shape).

## Pattern inventory after this ship

The seventh shape pattern: **single-result side-effecting installer
with two-axis success signal**. The v0.8.62 install descriptor was
single-axis (`bootstrapped: true` only, `LaunchctlError → exit 3 no
payload`); v0.8.63 uninstall introduces the two-axis variant for the
case where launchd-side and disk-side failures are independent and
both states are recoverable.

Six prior patterns:
- **Per-row reads** → bare arrays (v0.8.44, v0.8.49)
- **Single-result mutations w/ exception-on-absent** → bare entry
  (v0.8.54, v0.8.57, v0.8.58, v0.8.59, v0.8.60)
- **Single-result mutations w/ no-op-on-absent** → wrapper-with-null
  (v0.8.56)
- **Mutations w/ single count + invocation params** → flat dict
  (v0.8.38, v0.8.55)
- **Fleet-iteration with summary counters + per-entry array**
  (v0.8.61)
- **Single-result side-effecting installer with stable descriptor**
  (v0.8.62)
- **Single-result side-effecting uninstaller with two-axis success
  signal** (v0.8.63 ← this ship)
