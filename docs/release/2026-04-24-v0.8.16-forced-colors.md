# v0.8.16 — `forced-colors: active` adaptation on wiki_refresh toasts

**2026-04-24**

## Why

The crash toast (v0.8.9) and recovery toast (v0.8.11) have been
shipping with solid-color backgrounds (`#d32f2f` red, `#2e7d32` green)
and drop shadows for five releases. Works fine in every standard
browser theme — but in Windows high-contrast mode (and equivalent
OS-level accessibility palettes on macOS and Linux) the UA:

1. Remaps both `background` and `color` to system colors (`Canvas`
   and `CanvasText`), so the banner text/background combo is
   guaranteed to contrast against each other — but both lose their
   original hues, so the banner blends visually into the page.
2. Forcibly drops `box-shadow` (per the forced-colors-mode spec), so
   even the subtle "floating card" edge that made the banner feel
   like a discrete container disappears.

Net effect: a high-contrast user sees the banner text, but the banner
itself looks like free-floating text pasted onto the dashboard — no
visual container, no signal that this is an alert. The red/green
semantic is gone too, since both are now `Canvas`.

We closed the companion gap (`prefers-reduced-motion`) in v0.8.13.
This release closes the last carry-forward a11y gap on the toast
subsystem.

## Shipped

**`apps/ui/templates/partials/wiki_refresh.html.j2` — two scoped
`<style>` blocks.**

- Crash toast gains a `class="lvdcp-crash-toast"` hook (previously
  had only an ID) and a fresh `<style>` block inside the
  `{% if show_crash_toast %}` guard:
  ```css
  @media (forced-colors: active) {
      .lvdcp-crash-toast { border: 1px solid CanvasText !important; box-shadow: none !important; }
      .lvdcp-crash-toast code { border: 1px solid CanvasText !important; }
      .lvdcp-crash-toast button { opacity: 1 !important; }
  }
  ```
- Recovery toast's existing `<style>` block (which already carried
  the v0.8.12 fadeout keyframes + v0.8.13 hover + reduced-motion
  rules) gains a third `@media` query:
  ```css
  @media (forced-colors: active) {
      .lvdcp-recovery-toast { animation: none !important; opacity: 1 !important; pointer-events: auto !important; border: 1px solid CanvasText !important; box-shadow: none !important; }
      .lvdcp-recovery-toast code { border: 1px solid CanvasText !important; }
      .lvdcp-recovery-toast button { opacity: 1 !important; }
  }
  ```

Both rules are scoped inside their respective `{% if %}` guards, so
healthy idle polls still ship zero a11y CSS.

## Tests

Four new integration tests in `tests/integration/test_ui_wiki_refresh.py`:

- `test_crash_toast_carries_forced_colors_rule` — seeds a crash,
  asserts the class hook, the media query, the border/shadow rule
  on the banner itself, the border rule on `<code>`, and the
  opacity pin on the dismiss button.
- `test_recovery_toast_carries_forced_colors_rule` — seeds a
  recovery, asserts the same checks as above plus the
  animation-cancel + opacity-1 + pointer-events-auto triad that
  makes the toast sticky in forced-colors mode (matching the
  reduced-motion treatment).
- `test_no_forced_colors_rule_when_no_toast_rendered` — idle clean
  poll must ship zero forced-colors CSS, no class hooks, no
  `CanvasText` tokens. Pattern-match to
  `test_no_a11y_css_when_no_recovery_toast_rendered` from v0.8.13.
- `test_forced_colors_rule_scoped_to_each_toast_class` — crash-only
  render must not carry `.lvdcp-recovery-toast` and vice versa.
  Prevents a future refactor from accidentally promoting either
  rule to a shared global selector.

Full suite: **1180 passed, 1 skipped, 1 deselected** (up from 1176).

## Design notes

**Why a border instead of changing the system colors?** In forced-
colors mode the author can't control which system colors are used
for `background` and `color` — the UA decides, and any author-set
RGB is overridden. What the author *can* reliably control is the
`border`, which is preserved as written. So the fix is not to pick
a different color palette but to add a visual frame that survives
the remap.

**Why cancel the fade on the recovery toast but not add any timer
cancel on the crash toast?** The crash toast has no timer — it's
sticky by design (bad news needs acknowledgement). Only the
recovery toast has the v0.8.12 fadeout animation, and that's the
one we cancel. The reasoning mirrors the v0.8.13
`prefers-reduced-motion` fallback: a user who turned on high-
contrast is almost certainly an a11y user; banners vanishing on a
timer are hostile to screen-reader and low-vision workflows. Stay
sticky; let the user dismiss manually.

**Why `!important`?** Same reason as v0.8.13: the toast elements
carry inline `style` attributes with `animation`, `box-shadow`,
and `opacity` shorthand properties. Inline style has specificity
1000, class rules have 10. Without `!important`, the class rules
lose. Breaking the inline shorthand into longhand would avoid
`!important` but blow the existing template format and require
updating a dozen unrelated tests. The three tokens are cheaper.

**Why pin button `opacity: 1`?** The crash toast's dismiss button
has `opacity: 0.8` inline for subtle de-emphasis. In high-contrast
that reduced opacity often reads as a "disabled" visual state to
assistive tech heuristics, and it reduces legibility. Pinning it
to 1 doesn't hurt the normal theme (the button is still small and
subtle by virtue of its size and position) and fixes the a11y
signal.

**Why don't we add a `@media (forced-colors: active)` to the
degraded card too?** The degraded card uses `background:#fff8e1`
and `border-left:4px solid #f9a825`. In forced-colors mode:
- `background` is remapped to `Canvas` (fine).
- `border-left` is preserved as `CanvasText`-ish (fine — borders
  are exempt from the color remap by spec).
- The text uses `color:#795548` which becomes `CanvasText` (fine).

So the degraded card is already legible in high-contrast: a clear
banner-like block with a visible left-border accent. No fix needed,
no test added.

**Why not extract the two `<style>` blocks into a shared partial?**
Tempting, but the rules reference class names that are specific to
each toast (`.lvdcp-crash-toast` vs `.lvdcp-recovery-toast`), and
the recovery toast's forced-colors rule includes animation-cancel
logic that's meaningless for the crash toast. A shared partial
would grow an `{% if kind == "crash" %}` split inside CSS, which is
worse than two parallel inline blocks.

## Migration / compat

- Pure CSS addition. No route change, no JSON schema change, no
  behaviour change in standard themes.
- The new `class="lvdcp-crash-toast"` attribute is additive; the
  existing ID selector `#crash-toast-<slug>-<timestamp>` still
  works. Existing tests that match on the ID or the red background
  keep passing.
- Forks that pin on the exact inline `<style>` blocks (e.g.
  snapshot tests) may need to regenerate — the crash-toast block
  is new, and the recovery-toast block grew a third media query.

## Known gaps (carry forward)

- **No dismiss-event telemetry.** Still deferred — v0.8.15 added
  render telemetry, dismiss telemetry requires a JS beacon or a
  dedicated POST endpoint that's only worth it once render data
  justifies it.
- **No outage-duration signal on the recovery toast.** "Recovered
  after 12s of degraded state" vs "recovered after 4 minutes" is
  a useful user signal that's still missing. Likely a
  client-set timestamp on the degraded wrapper + server echo on
  recovery.
- **Crash toast still sticky, no auto-dismiss.** Mostly correct
  (bad news demands acknowledgement) but project-switching users
  shouldn't accumulate ten stale red banners. Separate release.
- **`lvdcp-pulse` running indicator still animates unconditionally
  in forced-colors + reduced-motion.** Used by the blue "Running"
  card's status dot — not part of the toast subsystem, so out of
  scope here but worth a dedicated pass.
